# [버그] ConfirmMissedPaymentScheduleJobService가 환불 불가능한 오래된 주문을 반복 취소 시도 (좀비 호출)

| **항목** | **내용** |
| --- | --- |
| **배경** | `ConfirmMissedPaymentScheduleJobService`는 부트페이 confirm 미호출 건을 10분마다 자동 보정하는 스케줄러입니다. 그러나 현재 이 스케줄러가 환불 불가능한 **2021년 이전의 매우 오래된 주문**에 대해서도 반복적으로 `getReceipt` → `refund` 호출을 시도하고 있습니다. |
| **현재 문제점** | • 10분마다 약 10건의 무의미한 PG API 호출이 발생 (확인된 주문: orderId 6028, 5697 등 2021년 생성 건)<br>• 부트페이 응답은 매번 `RC_CANCEL_SERVER_ERROR` 또는 `RC_NOT_FOUND`로 실패<br>• PG사 Rate Limit(`30req/s`) 예산을 낭비하여, 동시간대 실행되는 다른 스케줄러(소셜링 폐강 환불 등)의 API 호출에 영향을 줄 가능성이 있음<br>• CloudWatch 로그에서 `ConfirmMissedPaymentScheduleJobService` 에러가 10분 간격으로 반복 확인됨 |
| **개선 방향** | 1. 쿼리 조건에 `updatedAt` 또는 `createdAt` 상한선 추가 (예: 최근 24시간 이내 주문만 대상)<br>2. 일정 횟수 이상 환불 실패한 주문은 별도 상태(`CANCEL_FAILED` 등)로 전환하여 반복 시도 대상에서 제외<br>3. 환불 실패 시 재시도 카운터를 두어 임계치 초과 시 알림 후 스킵 |
| **고려요소** | • WEBB-1272 (PG API Rate Limit 관리 체계 구축)와 연관된 이슈<br>• 좀비 호출이 Rate Limit 위반의 직접적 원인은 아니지만, 한정된 API 예산을 소모하여 간접적으로 다른 작업의 실패를 유발할 수 있음<br>• 기존 `WAITING_PAYMENT` 상태의 정합성을 해치지 않도록 상태 전환 설계 필요 |
| **기대효과** | • 10분마다 발생하는 불필요한 PG API 호출 제거 (일 약 1,440건 절감)<br>• PG Rate Limit 여유분 확보로 동시간대 실행되는 다른 스케줄러 안정성 향상<br>• 불필요한 에러 로그 제거로 모니터링 신호 대 잡음 비율 개선 |
