# 호스트(HOST/OWNER) 정원 포함 정책을 모든 도메인 코드에 일관되게 적용한다

| 항목 | 내용 |
| --- | --- |
| 배경 | 노션 정책 문서(문토 모임 참여/취소 정책문서, §2-3, §6-4)는 **"호스트(OWNER)는 모든 모임에서 정원에 1슬롯 포함된다"**고 명시한다. 그러나 실제 코드는 클럽·챌린지·소셜링 도메인이 호스트 표현·정원 차감·정산 캡 공식에서 비대칭으로 동작한다. 도메인별 비대칭은 호스트 정산금 과지급, 챌린지 유저/어드민 생성 경로의 availCount 시작값 불일치 등 운영 영향을 만들고 있다. |
| 현재 문제점 | 도메인별 현황은 다음과 같다.<br>1) 클럽: ClubMember에 grade=OWNER row가 존재하고 availCount = max - 1로 초기화된다. 정원 정합은 OK이나 정산 쿼리에 정원 상한 자체가 미적용되어 정원 초과분이 모두 정산된다 (settlement.schedule-job.service.ts:185-190).<br>2) 챌린지: ChallengeMember에 grade 필드가 없고 OWNER row도 없다. 어드민 생성은 availCount = max - 1, 유저 생성은 availCount = max로 두 경로가 1슬롯 어긋난다 (challenge.service.ts:252 vs challenge.admin.service.ts:308). 정산 캡은 settlementMaxCount = maximumMembers로 OWNER 1슬롯이 차감되지 않는다 (settlement.schedule-job.service.ts:100).<br>3) 소셜링: SocialingMemberGrade enum이 STAFF·MEMBER 2종뿐이라 HOST 값 자체가 없다. HOST는 SocialingMember 테이블에 row 없이 Socialing.userId로만 식별된다. availCount = maximumPerson으로 HOST 슬롯이 차감되지 않는다 (socialing.entity.ts:182, socialing.transformer.ts:58, socialing.v4.service.ts:1972). 정산 캡은 maximumPerson - staffCount로 HOST 1슬롯이 차감되지 않는다 (socialing-settlement.service.ts:65 외 2곳).<br>결과적으로 모임당 약 1슬롯치 호스트 정산금이 과지급되고 있으며, 클럽은 정원 초과분도 정산되어 추가 과지급이 발생한다. |
| 개선 방향 | 본 이슈는 결정 항목 3개를 정한 뒤 Phase별로 진행한다.<br>**결정 1 — 호스트 네이밍 통일(HOST/OWNER):** ClubMemberGrade.OWNER를 HOST로 rename한다<br>**결정 2 — 호스트를 정원에 포함하는지에 대한 통일 단위:** <br>(a) 모든 도메인 Member 테이블에 HOST row 백필<br>(b) 개념·정합식만 통일하고 데이터 표현은 도메인별 자유<br>(c) 클럽 OWNER row를 제거해 모든 도메인이 호스트를 상위 모델로만 식별.<br>**결정 3 — ChallengeMember.grade 필드 추가 여부** |
| 고려요소 | 본 작업은 호스트 정산금에 직접 영향을 주는 변경이므로 배포 전 검증이 필수다. 스테이징에서 직전 1주(소셜링·챌린지) 및 직전 1개월(클럽) 정산을 본 PR 적용 전/후 양쪽 공식으로 재계산해 before/after diff를 확보하고, 영향 호스트·금액 규모를 이해관계자에게 사전 공유한 뒤 배포 의사결정을 한다.<br>API 응답 호환성도 점검 대상이다. currentMembers, availCount를 노출하는 API의 클라이언트 호환성을 확인하고, +1 보정 코드를 제거한 뒤 모바일·웹 양쪽에서 회귀 테스트를 진행한다.<br>Sendbird 채널 가입/탈퇴 루프와 멤버 조회 API는 본 이슈가 (b)안을 채택해 HOST row 백필을 하지 않으므로 추가 영향이 없다. 만약 (a)안을 채택한다면 SocialingMember.orderId NOT NULL @unique 제약 해소, 멤버 조회 결과에 HOST 포함 여부, Sendbird 루프 중복 처리 방지 등을 별도 점검해야 한다. |
| 기대효과 | 정책 문서(노션)와 실 코드의 정합성이 회복되어 호스트 정산금 과지급이 해소된다. 모임당 1슬롯치, 클럽의 경우 정원 초과분만큼 추가로 정산금이 정상화된다.<br>도메인 간 호스트 모델·정원 식·정산 캡 공식이 통일되어 신규 기능 개발 시 도메인별 분기 비용이 감소한다. 향후 (a)안 데이터 표현 통일이 필요해질 경우에도 invariant 식 변경 없이 백필만으로 진행 가능한 구조가 마련된다 |


## comment by 김범진

도메인별 정원 로직이 차이가 발생한 이유는 다음과 같이 추측해볼 수 있습니다.

1. 처음 소셜링이 만들어질 때 "호스트는 운영자라서 별도 모델, 일반 멤버만 `Member` 테이블에 둔다"는 단순한 설계로 출발 → `availCount = maximumPerson`이 자연스러웠음
2. 이후 정원 표시 요구사항이 들어오자 "호스트도 같이 보여야 한다" → 매퍼에서 `+1` 임시 보정
3. 클럽이 나중에 추가되면서 "권한 관리 편의상 `ClubMember` row 만들자" → 클럽만 다른 표현
4. 챌린지는 백오피스 생성 경로가 먼저, 유저 경로가 나중에 추가되며 한쪽만 `1` 보정
5. 정산 로직은 나중에 추가되면서 raw `maximumPerson`을 그대로 캡으로 사용 → 호스트 미차감
6. 누구도 "이거 다 합쳐 보면 어디서 어디까지 호스트가 카운트되는 건지" 점검 안 함

다만 사유 표현이 "Prisma가 partial unique index를 지원하지 않음"으로 적혀 있는데 정확히는 "Prisma schema 표현 불가(`extendedIndexes` preview 미활성), raw SQL로는 가능"입니다. "schema 표현 불가 → drift 위험 → nullable unique로 단순화"로 다듬어 주세요.
