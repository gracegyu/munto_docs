# ElastiCache RI를 cache.m6g.large 대신 cache.r6g.large로 착오 구매한 건을 환불 요청하고 적정 노드로 재구매한다

| 항목 | 내용 |
| --- | --- |
| 배경 | munto-prod ElastiCache·RDS의 기존 1년 RI가 2025-05-27에 만료되어 전 노드가 On-Demand로 청구되고 있었고, 2026-06-10에 Option A(1년 / All Upfront) 기준으로 RI 재약정을 진행했습니다. 관련 검토 보고서는 AWSI-83 트랙의 ElastiCache·RDS RI 비용 최적화 문서입니다. |
| 현재 문제점 | redis-2(`prod-munto-redis-2`)에 적용할 `cache.m6g.large` ×2($1,013/노드)를 구매하려 했으나, 콘솔에서 `cache.r6g.large` ×2($1,371/노드)로 노드 타입을 착오 입력해 구매했습니다.<BR>ElastiCache RI는 노드 타입이 정확히 일치해야 적용되므로, 구매한 r6g.large RI가 m6g.large 노드인 redis-2에 적용되지 않고 미사용 상태입니다. 과지불액은 2개 기준 약 **$716**이며, 해당 RI 2건은 이미 active로 전환되어 선납 $2,742가 확정 청구되었습니다. |
| 조치 현황 | AWS Support에 구매 당일·미사용 사유로 환불(one-time exception) 케이스를 접수했습니다. ElastiCache RI는 정책상 환불 불가 항목이라 성사가 보장되지 않으며, 결정까지 통상 1~3주가 소요됩니다.<BR>대상 RI ARN: `arn:aws:elasticache:ap-northeast-2:215467263292:reserved-instance:ri-2026-06-10-01-20-32-344` |
| 후속 작업 | **환불 성사 시:** 원안으로 lounge redis 사이즈업을 검토하고 노드를 적정 사이즈로 운영합니다.<BR>**환불 거부 시:** 메모리 사용률 88%(잔여 약 0.4GiB)로 사이즈업이 필요한 lounge(prod-munto-lounge-redis, cache.t4g.medium ×2)를 cache.r6g.large로 올려 미사용 RI를 흡수하고 OOM 리스크를 동시에 해소합니다. <BR>• **현재 lounge 사용 메모리**: `cache.t4g.medium` ×2 (2 vCPU / 3.09 GiB) — 노드당 약 **2.05 GiB 사용**(사용률 약 88.6%, 잔여 약 0.4 GiB)<BR>• **대조 단가** (Redis·서울·1yr All Upfront RI / OD):<BR>    ◦ `cache.m6g.large` — **2 vCPU / 6.38 GiB** · RI $1,013 · OD $0.181/hr(연 $1,585)<BR>    ◦ `cache.r6g.large` — **2 vCPU / 13.07 GiB** · RI $1,371 · OD $0.246/hr(연 $2,155) |
| 고려요소 | ElastiCache RI는 size/family flexibility를 지원하지 않아 r6g↔m6g 간 전환·부분 적용이 불가능합니다.<BR>lounge 사이즈업은 Multi-AZ·auto-failover 구성이라 수초 내 failover 1회를 동반하므로 트래픽이 적은 시간대에 수행합니다. |
| 기대효과 | 환불이 성사되면 약 $716을 절약하고 redis-2를 적정 사이즈(m6g.large)로 운영합니다. 환불이 거부되더라도 미사용 RI($2,742)를 lounge에 활용해 폐기 손실을 막고 lounge의 메모리 한계 문제를 해결합니다. |

## comment by 전규현

@홍진영 두 갈래 대응 방향에는 **동의**합니다. 아래 세 가지만 보완 부탁드립니다.

- **redis-2용 m6g.large RI는 어떻게 처리하실 계획인가요?** 반려 시 r6g는 lounge로 가고 redis-2엔 r6g가 안 맞아, m6g는 환불 결과와 무관하게 어느 경우든 필요해 보입니다. 환불 결과(1~3주)를 기다리는 동안 redis-2는 계속 On-Demand로 청구되니, 환불 건과 분리해서 m6g를 먼저 약정하는 게 낫지 않을까요? 두 안에 redis-2 처리 시점이 빠져 있어 확인차 물어봅니다.
- **반려 시 손해가 1년에 그치도록 "내년 RI 만료 후 다운사이즈" 계획을 후속작업에 넣어주세요.** 본문에 과지불 $716·선납 $2,742가 정리돼 있는데, lounge 실사용이 노드당 약 2GiB라 r6g(13GiB)는 과대 사이즈입니다. 이 손해가 1회로 끝나려면 내년 약정 만료 때 lounge를 적정 타입(m6g.large 등)으로 내리고 재약정해야 합니다. 지금 대응안에는 이 계획이 없는데, 다운사이즈 없이 r6g를 그대로 갱신하면 과대 사이즈와 손해가 **매년 반복**됩니다. 1년짜리 임시 흡수임을 분명히 하고, 만료 시점 정상화를 함께 적어주세요.
- **lounge 사이즈업(t4g.medium → r6g.large) 이관 계획도 구체적으로 작성해주세요.** 이건 무중단 작업이 아닙니다. lounge는 Multi-AZ·auto-failover 구성이라 변경 시 **수초간 failover가 1회** 발생하고, t4g→r6g는 패밀리 변경이라 데이터 이관을 동반합니다. 게다가 현재 메모리 88%(OOM 임박) 상태라 작업 자체에 리스크가 있습니다. 후속작업에 ① **사전 스냅샷(백업)**, ② **트래픽 최저 시간대 점검 창**, ③ 롤백 기준을 포함해 주세요.
    - **DNS/엔드포인트 전환**: failover가 나도 primary 엔드포인트(주소)는 유지되고 내부적으로 새 노드로 DNS만 갱신됩니다. 다만 연결 주체가 **DNS를 길게 캐싱**하면(예: JVM 기본 DNS 캐시) 옛 노드로 붙으려다 재연결이 지연·실패할 수 있어, 연결 주체의 DNS TTL·재연결 동작 점검이 필요합니다.
    - **lounge redis에 실제 연결하는 주체가 무엇인가요?** 최종 사용자 클라이언트가 아니라 **BE 애플리케이션**이 붙는 구조로 보이는데, 맞다면 failover·DNS 전환의 영향 범위는 해당 **BE 서비스의 재연결·재시도 로직**입니다. 어떤 서비스가 lounge redis에 연결하는지와 그 재연결 처리 여부를 함께 확인 부탁드립니다.
