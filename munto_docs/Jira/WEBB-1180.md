# [개선] WAITING_PAYMENT 이탈 건 자동 정리 스케줄러 추가

| 항목 | 내용 |
| --- | --- |
| 배경 | Order v3(서버사이드 결제) 배포 이후, 주문 생성 후 결제창 진입 전 이탈한 `WAITING_PAYMENT` 건이 DB에 지속 누적되고 있음. 3월 9일 배포 이후 현재까지 3,226건 이상이 정리되지 않은 채 적체 중. |
| 현재 문제점 | 기존 `PaymentUpdateJobService` 스케줄러는 `bootpayReceiptId IS NOT NULL` 조건으로만 WAITING_PAYMENT 건을 정리함. 결제창 진입 전 이탈한 건은 `bootpayReceiptId = null`, `confirmRetryCount = 0` 상태로, 현재 정리 대상에 포함되지 않아 DB에 영구 누적됨. 해당 건들의 총 주문 금액 합산이 약 1억원 이상으로, 거래액 집계 등에 혼선을 줄 수 있음. |
| 개선 방향 | `PaymentUpdateJobService`에 `bootpayReceiptId = null`인 WAITING_PAYMENT 건을 대상으로 하는 정리 로직 추가. Redis 세션 TTL(15분) 만료 기준으로, `updatedAt`이 15분 이상 경과한 건을 `COMPLETED_CANCEL_PAYMENT`로 상태 전환. |
| 고려요소 | • 기존 `bootpayReceiptId IS NOT NULL` 정리 로직과 분리하여 독립적으로 처리할 것 (영향 범위 최소화)<br>• 배치 처리 시 대량 건 존재 가능성 있으므로 페이지네이션 적용 필요<br>• 정리 기준 시간: Redis TTL(15분)과 동일하게 `updatedAt <= now - 15분` 조건 적용<br>• 처음 실행 시 기존 누적된 3,226건+ 일괄 처리 필요 |
| 기대효과 | • DB WAITING_PAYMENT 건 미정리 누적 해소<br>• 거래액 집계 및 모니터링 지표의 정확도 향상<br>• Order v3 플로우 이탈 케이스에 대한 명시적 상태 관리 확립 |




## Comment by 김범진

### **다음 두 항목에 대해 검토 요청드립니다.**

1. 기존 누적 3,226건 처리를 `COMPLETED_CANCEL_PAYMENT` 상태 전환 vs **삭제** 중 어느 방향으로 진행할지 결정이 필요합니다.
2. 상시 스케줄러(신규 발생 건)도 동일 방식으로 처리할지 확인이 필요합니다.

### **이슈 상황**

WEBB-1180 이슈에서는 기존 누적된 3,226건을 `COMPLETED_CANCEL_PAYMENT`로 상태 전환하는 방식을 제안하고 있습니다. 그런데 이렇게 처리하면 **어드민 거래액 집계에서 취소 건으로 잡혀 왜곡이 발생**할 수 있습니다. (이슈 목적 자체가 "거래액 집계 혼선 해소"인데, 오히려 취소 거래액이 갑자기 수억 원 증가하는 문제)

### **대안: 삭제 처리**

`WAITING_PAYMENT` + `bootpayReceiptId = null` + `confirmRetryCount = 0` 건은 **결제창 진입 전 이탈 건**으로, 실제 결제 데이터가 전혀 없습니다.

- `Payment` 레코드 없음 (`paymentId = null`)
- `OrderHistory`는 생성 이력 1건만 존재
- 재무/정산 영향 없음

프로덕션 DB에서 `UserCoupon` 연결 여부도 확인했으며 **0건**으로 확인됐습니다.

```
SELECT COUNT(*) FROM "UserCoupon" uc
JOIN "Order" o ON uc."orderId" = o.id
WHERE o."orderStatus" = 'WAITING_PAYMENT'
  AND o."bootpayReceiptId" IS NULL
  AND o."confirmRetryCount" = 0
  AND o."updatedAt" <= NOW() - INTERVAL '15 minutes';
-- 결과: 0건
```

따라서 상태 전환 대신 **OrderHistory → Order 순서로 삭제**하는 방식이 더 안전합니다.

- 거래액 집계에 영향 없음
- 쿠폰 유실 위험 없음 (확인 완료)
