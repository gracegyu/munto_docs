3. **pushEventMulticast FCM 호출 비동기화**

# comment by 전규현

@홍진영 분석 보고서 잘 확인했습니다. 브로커 정상 / 컨슈머 병목이라는 원인 진단과 제안된 3가지 개선 방향에 대해 대체로 동의합니다. 추가로 아래 사항을 검토 부탁드립니다.

**1. CloudWatch 알람 설정 즉시 적용 필요**

보고서에서 "현재 알람이 없어 지연 발생을 인지하지 못하고 있었다"고 지적되었는데, 제안 3가지 개선안에 알람 설정이 빠져 있습니다. 개선 작업과 별개로 EstimatedMaxTimeLag 기반 알람(pushEvent/smsEvent > 300s)을 우선 설정해 주세요. 지연 재발 시 즉시 인지할 수 있어야 합니다.

**2. Auto Scaling 최대 인스턴스 수 재검토**

제안이 "최대 9 → 3"인데, 현재 파티션 수(3개) 기준으로는 맞지만, 향후 트래픽 증가에 따른 파티션 추가 확장 가능성을 고려해서 최대값에 여유를 두는 것이 좋을지 검토 부탁드립니다. 예를 들어 최대 6 정도로 설정하고, 파티션 확장 시 같이 조정하는 방식도 고려해 주세요.

**3. 사용자 영향 범위 확인**

pushEvent 최대 9.8시간, smsEvent 최대 6.1시간 지연이 실제 사용자에게 어떤 영향을 끼쳤는지 확인이 필요합니다. 해당 시점에 CS 인입이 있었는지, 알림 미수신으로 인한 사용자 이탈이나 전환율 저하가 있었는지 확인할 수 있을까요?

**4. 파티션 확장 시 주의사항**

Kafka 특성상 파티션 수는 증가만 가능하고 축소가 불가능합니다. smsEvent 1 → 3 확장은 합리적이라 동의하지만, 적용 전에 컨슈머 그룹 리밸런싱 영향과 메시지 순서 보장 여부(SMS 발송 순서가 중요한 경우)를 확인해 주세요.

**5. 단계적 적용 및 효과 검증**

3가지 조치를 동시에 적용하면 문제 발생 시 원인 특정이 어렵습니다. 아래 순서로 단계별 적용을 제안합니다.

- 1단계: CloudWatch 알람 설정 (즉시)
- 2단계: smsEvent 파티션 확장 1 → 3 + ECS 인스턴스 2 → 3 (파티션 변경과 인스턴스를 함께 반영)
- 3단계: pushEventMulticast FCM 비동기화 (코드 변경이므로 QA 후 배포)

각 단계 적용 후 1~2일간 모니터링하여 지연 개선 효과를 확인한 뒤 다음 단계로 진행해 주세요.

**6. 현재 제안은 임시 미봉책임을 인지해야 합니다**

위 1~5 조치에 대해서는 동의하며 즉시 진행해 주세요. 다만, 이 조치들은 현재 아키텍처(MSK + ECS) 내에서의 튜닝이며 **근본적인 구조 한계를 해결하지 못합니다.**

현재 구조의 한계:
- 파티션 수 = 동시 소비 인스턴스 상한 → 파티션을 3으로 늘려도 최대 3개 인스턴스까지만 유효
- Kafka 파티션은 증가만 가능하고 축소 불가 → 파티션을 계속 늘리는 것은 지속 가능한 전략이 아님
- ECS 기반 Consumer는 await 블로킹 구조이므로 외부 API(FCM/SMS) 지연이 곧 전체 처리량 저하로 직결
- 150만 유저 동시 알림 발송 같은 피크 시나리오에서는 파티션/인스턴스를 아무리 늘려도 지연이 재발할 수밖에 없음

**7. 근본 해결: 알림 이벤트의 Serverless 전환**

근본적으로는 pushEvent, smsEvent, pushEventMulticast를 **Serverless 아키텍처(SQS + Lambda 또는 EventBridge + Lambda)로 전환**해야 합니다.

- Lambda는 파티션 수에 종속되지 않고 메시지당 개별 실행 → 사실상 무한 수평 확장
- 피크 시 수천 개 동시 실행으로 지연 없이 처리 가능
- ECS처럼 인스턴스 기동 대기 없이 즉시 스케일
- 외부 API 호출(FCM/SMS)이 개별 Lambda에서 격리되어 하나의 지연이 전체 처리량에 영향을 주지 않음

이미 Dating 추천/발견 작업에서 동일한 MSK 지연 우려로 인해 MSK → SQS + Lambda로 전환한 선례가 있습니다. ([MSK(Kafka)에서 SQS로의 메시징 아키텍처 마이그레이션 OnePager](https://www.notion.so/MSK-Kafka-SQS-OnePager-2d9e2bc7639d802a83dae27783f8da91?source=copy_link)) 해당 전환은 이미 프로덕션에 적용되어 정상 운영 중이므로, **같은 방식으로 알림 이벤트(pushEvent/smsEvent/pushEventMulticast)도 Serverless로 전환하면 이번 지연 문제를 근본적으로 해결**할 수 있습니다.

정리하면:
- **단기(즉시)**: 위 1~3단계 조치를 단계적으로 적용하여 당장의 지연 완화
- **중기(조속히 기획 착수)**: 알림 이벤트 Serverless 전환 기획 및 실행 → 기존 Dating 전환과 동일한 패턴 적용으로 구조적 문제 근본 해결
