# 프로덕션에서 POST /api/v2/auth/refresh 호출이 지난 12시간 동안 로그 기준 2xx 성공 없이 전부 실패(4xx 77건·5xx 2건, 합계 79건)로 집계된다

| 항목 | 내용 |
| --- | --- |
| 배경 | 프로덕션 ECS 서비스 munto-be-api-prod-cdk가 제공하는 POST /api/v2/auth/refresh 엔드포인트는 클라이언트가 액세스 토큰을 갱신할 때 호출한다.<br>운영자가 CloudWatch Logs Insights에서 API_REQUEST 로그만 필터링하여 해당 URL과 statusCode를 집계하였다. |
| 현재 문제점 | 지난 12시간 구간에서 동일 엔드포인트에 대한 요청이 로그 기준 79건 집계되었으며, statusCode가 200~299인 성공 응답은 0건이었다.<br>400번대는 77건, 500번대는 2건이었고 success_pct는 0%, fail_pct는 100%로 계산되었다. 따라서 관측 구간에서는 토큰 갱신이 정상 완료된 사례가 없었다.<br>참고: 사용된 Logs Insights 쿼리는 level이 API_REQUEST이고 url에 /api/v2/auth/refresh가 포함된 이벤트만 포함하였다. |
| 개선 방향 | 원인을 특정하기 위해 동일 시각대의 ERROR/API_ERROR 로그, 스택 트레이스, 그리고 refresh 토큰 검증·캐시(Auth · Redis 등) 의존 구간을 조사한다.<br>결과에 따라 애플리케이션 코드 또는 설정을 수정하고, 수정 후 동일 쿼리와 스테이징·프로덕션 검증으로 성공률을 확인한다. |
| 고려요소 | 갱신 실패는 사용자에게 재로그인으로 이어질 수 있어 수정 일정과 영향 범위를 클라이언트·기획과 공유하는 것이 좋다.<br>로그 집계와 ALB 타깃 HTTP 코드 메트릭을 교차 확인하여 표본 편향이 없는지 검토한다. |
| 기대효과 | 토큰 갱신 요청이 정상적으로 2xx로 완료되어 불필요한 세션 단절과 사용자 불편이 줄어든다. |


Logs Insights 쿼리 요약: fields @message | filter "API_REQUEST" 및 경로 필터 | parse statusCode | stats total/ok_2xx/fail_4xx/fail_5xx 및 비율

### **3) 코드 분석**

### **백엔드**

- `apps/api/src/auth/v2/auth-user.v2.controller.ts:79` `@Post('refresh')`
- `apps/api/src/auth/payload/v2/user.refresh-token.payload.ts`: `@IsString() refreshToken!: string` — DTO 검증 정상 작동 중
- 따라서 검증이 의도대로 통과/실패하고 있을 뿐, 백엔드 자체 결함 없음

### **웹 프론트엔드 (munto_web, `lib/api/api.ts:114-136`)**

```
const updateToken=async():Promise<string|null>=>{if(isWebView()){sendToApp({type:'REFRESH_TOKEN'});returnnull;}const originRefreshToken=getCookie('refreshToken');// 쿠키 없으면 undefinedconst response=await api.post('/api/v2/auth/refresh',{    refreshToken: originRefreshToken,// undefined 그대로 전송});...
```

문제 메커니즘:

1. `getCookie('refreshToken')`이 쿠키 없을 때 `undefined` 반환
2. **가드 없이** `refreshToken: undefined` 그대로 axios POST
3. JSON serialize 시 키 누락 → 백엔드 DTO 검증 실패 → 400 (`refreshToken must be a string`)

발생 시나리오:

- 비로그인/만료 사용자가 인증 API 호출 → 401 → 응답 인터셉터가 `updateToken()` 호출 → 쿠키 없음 → 잘못된 페이로드로 백엔드 호출 → 400
- 봇/스캐너의 빈 페이로드 호출도 동일 경로

### **4) 결론**

- 백엔드 측 추가 조치 없음 (DTO 검증은 의도대로 동작 중)
- 근본 원인은 munto_web의 401 인터셉터에서 refreshToken 누락 가드 부재
- WebView 22% 비중은 `isWebView()` 판정 실패 케이스가 있을 가능성 — 추가 조사 필요

후속 작업은 별도 WEBF 이슈로 분리 예정.
