### 시나리오 A — CPU 2배 업그레이드만

**장점:** 배포만으로 즉시 적용

**단점:**

- 파티션 3개 한계 미해소 (동시 소비자 여전히 3개)
- 비용 2배 (월 +$74)
- 평균 3.45% CPU에 2 vCPU는 심각한 낭비

**결론:** 비효율

### 시나리오 B — CPU 다운 + 인스턴스 6개 (권장)

**장점:**

- 동시 소비자 3 → 6 (2배 향상)
- 파티션 6개로 피크 부하 분산
- 비용 현재 대비 50% 절감 (Min 기준)
- 인스턴스당 부하 1/3로 감소

**단점:**

- 0.5 vCPU 인스턴스당 catch-up 속도 절반 (단, 6개 인스턴스 병렬로 총 처리량 3배)

**결론:** 최적 (비용 절감 + 성능 향상)

### 시나리오 C — CPU 유지 + 인스턴스 증가

**장점:**

- 동시 소비자 6개
- 인스턴스당 catch-up 속도 유지

**단점:**

- 비용 +50%
- 평균 사용률이 워낙 낮아 비효율

**결론:** 예산 여유 시 고려

### 시나리오 D — CPU 소폭 다운 + 인스턴스 3개

**장점:**

- 비용 절감 (-25%)
- 동시 소비자 6개

**단점:**

- B보다 최대 동시 소비자는 동일하나 Min 인스턴스가 적어 cold start 영향

**결론:** B와 거의 동일, 차별점 미미

파티션 증가와 인스턴스 확장은 **임시방편**입니다.

`pushEvent`가 계속 성장하면 파티션을 또 늘려야 하고, Kafka 파티션은 **축소 불가**이므로 지속 가능한 전략이 아닙니다.

**근본 해결: SQS + Lambda 전환**

```
현재:  API Server → Kafka (pushEvent) → ECS Consumer → Firebase
개선:  API Server → SQS → Lambda (메시지당 개별 실행) → Firebase
```

Lambda: 파티션 개념 없음, 메시지당 독립 실행, 사실상 무한 확장

## comment by 전규현

@홍진영 smsEvent/pushEventMulticast 안정화 결과 잘 확인했습니다. pushEvent 지연 추가 대응 방향(시나리오 B)에 대해서도 대체로 동의합니다. 다만 적용 전에 아래 사항을 검토해 주세요.

**1. 메모리 1GB 축소 전 OOM 리스크 확인 필수**

현재 pushEvent 컨슈머가 MongoDB insertMany + Redis 조회 + FCM 호출을 수행하고 있습니다. 메모리를 2GB → 1GB로 절반 줄이면 배치 처리 중 OOM Kill이 발생할 수 있습니다. **적용 전에 반드시 현재 ECS Task의 메모리 사용률 메트릭(7일간 평균/최대)을 확인**하고, 피크 시 메모리 사용량이 1GB 이내인지 검증해 주세요.

**2. MinCapacity 3 + 0.5 vCPU = 총 1.5 vCPU → 피크 초기 대응력 감소 주의**

현재 Min 2 x 1 vCPU = 총 2 vCPU인데, 변경 후 Min 3 x 0.5 vCPU = 총 1.5 vCPU가 됩니다. Scale-out Cooldown을 120s로 줄였지만 Fargate 인스턴스 기동 시간(~30초~1분)까지 감안하면, 09:00/12:00/20:00 급격한 스파이크 초반에 Min 3개가 버텨야 하는 시간이 있습니다. **Scale-out이 완료되기 전 구간에서는 현재보다 오히려 대응력이 25% 감소**하는 점을 인지하고, 적용 후 해당 시간대의 lag 추이를 집중 모니터링해 주세요.

**3. WEBB-1285 좀비 호출 병행 해결 요청**

WEBB-1285에서 `ConfirmMissedPaymentScheduleJobService`가 10분마다 2021년 오래된 주문에 PG API 좀비 호출을 반복하고 있어 Rate Limit(30req/s) 예산을 불필요하게 소모 중입니다. pushEvent 처리량이 2배로 늘어나면 알림 → 사용자 행동 → 결제/환불 트래픽도 간접적으로 증가할 수 있으므로, PG Rate Limit 여유 확보를 위해 WEBB-1285를 이번 작업과 병행 해결해 주세요.

**4. 파티션 증설은 새벽 시간에 적용**

파티션 3→6 변경 시 Consumer Group 리밸런싱이 발생하고, 리밸런싱 중에는 일시적으로 모든 파티션의 소비가 중단됩니다. **트래픽이 가장 낮은 새벽 시간대**에 적용해 주세요.

**5. 적용 후 모니터링**

WEBB-1212 때와 마찬가지로, 적용 후 1~2일간 아래 지표를 집중 모니터링한 뒤 결과를 공유해 주세요.
- 피크 시간대(09:00, 12:00, 20:00) pushEvent EstimatedMaxTimeLag
- 0.5 vCPU 환경에서의 CPU/메모리 사용률
- Scale-out 트리거 시점과 실제 인스턴스 기동까지 소요 시간

**6. 시나리오 B는 임시 조치 — SQS+Lambda 전환 기획 병행 착수 요청**

위 조치에 대해서는 동의하며 진행해 주세요. 다만 WEBB-1212에서 이미 논의한 대로, 파티션 증설은 임시방편입니다. Kafka 파티션은 증가만 가능하고 축소가 불가능하므로, 6개로도 부족해지면 또 늘려야 하는 악순환이 반복됩니다. await 블로킹 구조와 외부 API 의존성이 남아 있는 한 근본적인 해결이 아닙니다.

이미 Dating 추천/발견에서 MSK → SQS+Lambda 전환을 성공 적용한 선례가 있으므로, **알림 이벤트(pushEvent/smsEvent/pushEventMulticast)의 Serverless 전환 기획도 이번 안정화 작업과 병행하여 착수**해 주세요.

정리:
- **즉시**: 시나리오 B 적용 (메모리 사용률 확인 → 새벽 시간 파티션 증설 → 1~2일 모니터링)
- **병행**: WEBB-1285 좀비 호출 해결
- **조속히 기획 착수**: 알림 이벤트 SQS+Lambda 전환
