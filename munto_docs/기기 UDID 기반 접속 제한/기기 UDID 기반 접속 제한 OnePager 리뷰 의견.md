## 2. 정책 대비 스펙 공백 (보강 필요)

### 2-1. `## 2-3. 카운트 기준`이 기술 설계에 없음

정책은 다음을 **명시**한다.

- **기기 카운트:** 최근 **90일** 이내 로그인 이력이 있는 기기만 포함(그 외 기기는 집계에서 제외).
- **계정 카운트(동일 기기):** **기간 제한 없이** 해당 기기에서 로그인한 계정 이력 전체를 기준.

OnePager는 `DeviceUserActivation` 집계를 **`WHERE deletedAt IS NULL`** 수준으로만 서술한다.

- 동일 기기·다계정: **「계정 이력 전체」** 를 무엇으로 세는지(행 정의, 탈퇴 소프트 삭제와의 관계)가 없음.
- 동일 계정·다기기: **90일 활동 기기만 슬롯에 넣는지** 여부가 `## 2-3`과 연결되어 있지 않음.
- **`UserDeviceLog` vs `DeviceUserActivation`** 중 어느 것이 **판정의 진실(SOT)** 인지, 백필·정합성 요구가 없음.

→ 구현자가 90일·무기한 규칙을 **임의 해석**할 수 있는 상태다.

### 2-2. `📌 계정 / 디바이스 정책 정리` 미반영

정책 말미는 신규 `## 3` 절만으로는 덮기 어려운 **운영 규칙**이다.

- **`📌 2`:** 동일 디바이스에서 A·B **동시 이용 중**이면 추가 로그인·신규 회원가입 제한.
- **`📌 3`:** A·B가 동일 디바이스에 연결된 상태에서 **A가 90일 미사용**이면, C 추가 및 B 외 계정 확장 불가(미사용 계정도 **슬롯 점유**).

OnePager에는 **「2계정 찼을 때 3번째 차단」** 만 있고, 위 **미사용 90일·동시 이용**에 대한 **판정 조건·데이터 필드·예외**가 없다.  
QA 표의「90일」이 **어느 규칙**을 검증하는지 Technical Description과 연결되어 있지 않다.

### 2-3. `## 5. 앱 내 다이얼로그`와 OnePager 불일치·구식 표현

- 정책 `## 5-1`·`## 5-2`에 **제목·본문·버튼**이 이미 있다.
- OnePager는 여전히 **「Notion 5장, 수치 PM 확정」** 등으로 남아 있고, Swagger 예시 표는 **에러 코드 칸에 본문 문장**이 들어가 **코드 vs 사용자 문구**가 섞여 있다.
- 정책 `## 3-1-2` 처리 방식의 앱 문구 한 줄은 **오타/미완** 가능성이 있어, **단일 소스는 `## 5`로 통일**한다는 문장이 OnePager에 있으면 API·앱·정책 정합에 유리하다.

### 2-4. 참고 문서·우선순위

- OnePager는 주로 **Notion 링크**만 있다. 로컬 **`기기 UDID 기반 접속 제한 정책.md`** (버전·수정일)을 **준수 사양 후보**로 명시하고, Notion과 불일치 시 **어느 쪽이 우선인지**를 박아 두는 것이 좋다.

### 2-5. 정책의 90일 — 적용 축 정리 (다계정·다기기·📌)

「90이 다계정·다기기 판정에 동일하게 쓰인다」고 한 문장으로 묶기 어렵다. **`## 2-3. 카운트 기준`**을 기준으로 보면 다음이 맞다.

| 구분 | 정책 요지 |
|------|-----------|
| **기기 카운트** | 최근 **90일** 이내 로그인 이력이 있는 **기기만** 집계에 포함. 그보다 오래 활동이 없는 기기는 제외. |
| **계정 카운트(동일 기기)** | **기간 제한 없이**, 해당 기기에서 로그인한 **계정 이력 전체**를 기준. |

따라서:

- **다계정(한 기기, `## 3-1`):** “몇 계정까지 허용인가”는 **`## 2-3`상 무기한 이력**이 기준에 가깝다. 90일은 여기서 **‘기기’ 자체의 생존**과는 다른 축이다.
- **다기기(한 계정, `## 3-2`):** 본문에 90일 문장이 **다시 없어** 애매하다. 다만 **`## 2-3`의「기기 카운트 = 90일」**을 전역 규칙으로 읽으면, “계정에 묶인 기기 수”를 셀 때 **「90일 내 활동한 기기만 친다」는 해석**이 자연스럽다. **구현·감사를 위해 정책 `## 3-2` 처리 방식에 90일을 한 줄 명시**하는 것을 권장한다.
- **`📌 3`:** “A가 **90일** 미사용” 시 C 추가 불가 등 — 여기서 90은 **미사용 판정**이며, 위 **집계 윈도**와 **상수는 같을 수 있으나 역할이 다르다.**

### 2-6. 90일 구현 방식·데이터 보존 — OnePager 공백

OnePager에서 **「90일」**은 사실상 **QA 한 줄**에만 등장한다. 아래는 **문서에 결정되어 있지 않다.**

- 90일을 **어느 판정**에 적용하는지(계정당 기기만 / `📌 3`만 / 전부 등).
- **`lastUsedAt`(또는 동등 필드) 갱신 시점:** 로그인·가입 시에만인지, 인증된 API 요청마다인지 — 갱신 규칙에 따라 90일 의미가 바뀐다.
- **데이터 보존 전략:**
  - **A안 — 쿼리 시 필터:** `lastUsedAt >= now() - interval '90 days'` 등으로 집계만 제한하고 **행은 유지**.
  - **B안 — TTL/배치 삭제:** 90일 지난 바인딩·로그를 주기적으로 **물리 삭제**해 테이블을 슬림하게 유지.
  - **혼합**(판정은 필터, 아카이브는 배치 등).

문서에 나온 **탈퇴 후 30일 하드 삭제**는 **정책 90일과 별개**이며, **90일 초과 데이터를 지울지 말지**에 대한 서술이 없다.

**OnePager에 넣을 최소 예시(형태만):**

- 동일 계정·다기기(90일을 기기 슬롯에 적용한다고 가정할 때):  
  `활성 기기 수 = COUNT(DISTINCT deviceId) WHERE userId = ? AND deletedAt IS NULL AND lastUsedAt >= :cutoff`
- 동일 기기·다계정(계정 이력 무기한):  
  `연결 계정 수 = COUNT(DISTINCT userId) WHERE deviceId = ? AND deletedAt IS NULL`  
  (탈퇴 소프트 삭제 시 해당 행을 집계에서 빼는지까지 한 줄로 고정.)

### 2-7. 인덱스 — 현재 OnePager안으로는 「90일 쿼리」까지 완비라고 보기 어렵다

`DeviceUserActivation` 제안에는 대략 `@@unique([userId, deviceId])`, `@@index([deviceId])`, `@@index([userId])`, `@@index([deletedAt])` 만 있다.

| 이슈 | 설명 |
|------|------|
| **`lastUsedAt` 비인덱스** | 90일 조건이 `lastUsedAt >= ?` 이면, 실행 계획은 `(userId)` 또는 `(deviceId)`로 범위를 좁힌 뒤 필터에 의존할 수 있다. **읽기 패턴이 확정되면** `(userId, deletedAt, lastUsedAt)` / `(deviceId, deletedAt, lastUsedAt)` 등 **복합 인덱스 후보**를 스펙에 써야 한다. |
| **쿼리 패턴 부재** | “완벽한 인덱스”는 **실제 SELECT/COUNT**가 정해져야 논의 가능. 지금은 **집계 쿼리 자체가 스펙에 없음**. |
| **`UserDeviceLog`** | 정책 판정을 로그 테이블로만 풀 경우 `(userId, loggedAt)`, `(deviceId, loggedAt)` 등이 중요하나, OnePager는 Activation을 SOT라고만 하고 **90일과의 관계**가 없다. |

→ **인덱스는 “마이그레이션 확정 전에 EXPLAIN으로 검증”**이 전제이며, 그 전에 **§2-6의 집계 정의**가 먼저 문서화되어야 한다.

## 3. 스펙 품질(API·DB·용어)

- **API:** 컨트롤러·`POST .../login` **「등」** 수준 → 정책을 끝까지 지우려면 **적용 엔드포인트 전부 + 403 응답 JSON 스키마(필드명·에러 코드 enum)** 확정이 필요하다.
- **DB:** `DeviceAccessDenyLog`는 정책의「차단 이력 (UDID, 시도 일시 등)」과 **대체로 호환**되나, **「필드명은 팀 컨벤션에 맞게 조정 가능」** 문구는 확정 SRS와 상충한다 → 별도 **DB 확정 문서**를 두거나 본문에서 조건을 좁힐 것.
- **용어:** Risk 표의 **`revokedAt`** 과 스키마의 **`deletedAt`** 이 혼재 → 하나로 통일할 것.
- **링크:** `libs/prisma/schema.prisma` 등이 **Notion URL**로 되어 있으면 레포 경로로 바로잡을 것.

## 4. 종합 판단

- **정책의「무엇을 막는지(2/2, 웹 제외, 소급 분리, 식별자)」** 는 OnePager가 **상당 부분 반영**했다.
- **정책이 구현을 결정하는 지점(`## 2-3`, `📌 2·3`, `## 5`)** 은 OnePager에 **부족하거나 구식·모호**하여, **「현재 정책 본문을 충분히 지원하는 스펙」이라고 보기 어렵다.**
- **90일**은 정책 `## 2-3`·`📌 3`에 분명히 등장하지만, OnePager에는 **적용 범위·TTL vs 조회 필터·`lastUsedAt` 정의·인덱스**가 없어, **「90일 기준 제한을 어떻게 할지」가 기술 스펙으로는 제시되어 있지 않다.**

## 5. 수정 우선순위 (체크리스트)

문서 수정 시 아래 순서를 권장한다.

| 순위 | 조치 |
|------|------|
| 1 | Technical Description에 **`## 2-3` 반영:** 기기 90일·계정 무제한을 **집계 정의**(테이블, `WHERE`, 탈퇴/미사용 시 행 상태)로 적을 것. |
| 2 | **`📌 2`·`📌 3`** 을 판정 규칙으로 추가: 특히 **90일 미사용이 슬롯을 유지하는지**, `lastUsedAt` 갱신 범위(로그인만 vs 인증된 요청 전체). |
| 3 | **90일 구현 결정:** TTL(배치 삭제) vs **쿼리 시에만** `lastUsedAt`/기준 시각 필터할지 OnePager에 **한 가지로 확정**하고, 무기한 계정 이력과 **충돌하지 않는지** 검토할 것. |
| 4 | **인덱스:** 실제 판정용 COUNT/SELECT 확정 후 **`lastUsedAt` 포함 복합 인덱스** 후보와 EXPLAIN 검증 계획을 스펙에 남길 것. |
| 5 | **정책 보강(기획):** `## 3-2`에 **90일 적용 여부**를 `## 2-3`과 맞게 한 줄 명시해 해석 분쟁을 줄일 것. |
| 6 | **`## 5`와 API/클라이언트 매핑 표:** 에러 코드 ↔ 제목/본문/버튼;「PM 확정」문구는 정책 `## 5`와 맞출 것. |
| 7 | **참고 문서:** 로컬 정책 `.md` 버전·날짜를 단일 소스 후보로 명시하고 Notion과의 우선순위를 정할 것. |
| 8 | (선행) 정책 파일에서 **수정 내역 1차안과 본문**의 숫자·문구 충돌을 정리해 **단일 진실**을 만든 뒤 OnePager를 그에 맞출 것. |
