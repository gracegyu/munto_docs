## 1. 핵심 개념

### 1-1. 관련 테이블

| 테이블 | 역할 |
| --- | --- |
| `Order` | 주문. 도메인별 참여권의 결제 기록 |
| `Payment` | PG 결제 레코드. CARD 결제 시 생성 |
| `OrderHistory` | Order 상태 변경 이력. 모든 전이 시 반드시 기록 |
| `PaymentRequest` | Bootpay 웹훅 수신 임시 저장. 처리 후 삭제 |

### 1-2. 결제 방식

```
단건 결제 (소셜링, 챌린지)
  - OrderKind.CARD: Bootpay confirmReceipt → Payment 생성
  - OrderKind.FREE: PG 호출 없음 (price = 0)

빌링키 방식 (클럽)
  - 빌링키 등록 후 정기/즉시 청구
  - 세부 규칙 → order-club.md 참조
```

### 1-3. OrderStatus 정의

```
WAITING_PAYMENT          결제 대기 (주문 생성 직후)
COMPLETED_PAYMENT        결제 완료 (참여 유효)
FAILED_PAYMENT           Bootpay 결제 승인 실패 (카드 거절, 한도 초과 등 PG 거절)
EXPIRED                  결제창 진입 없이 이탈
COMPLETED_CANCEL_PAYMENT 취소 및 환불 완료
FAILED_CANCEL_PAYMENT    취소 완료, 그러나 PG 환불 실패 (수동 처리 필요)
PARTIAL_REFUNDED         부분 환불 완료 (얼리버드 등)
PARTIAL_REFUND_FAILED    부분 환불 실패 (수동 처리 필요)

-- 제거 예정 (레거시) --
NONE                     무료 주문 완료를 나타내던 구 값. v3 전환 후 COMPLETED_PAYMENT로 통일됨.
                         프로덕션 잔존 약 150건(유저 2명) → COMPLETED_PAYMENT로 마이그레이션 후 제거
REQUEST_CANCEL_PAYMENT   사용된 적 없는 값. 프로덕션 0건 확인 → 제거
COMPLETE_PAYMENT         COMPLETED_PAYMENT와 혼동되는 구 값. 프로덕션 0건 확인 → 제거
```

## 2. OrderStatus 전이 규칙

```
WAITING_PAYMENT
  ├→ COMPLETED_PAYMENT  (결제 성공 + 도메인 후처리 완료)
  ├→ FAILED_PAYMENT     (결제 확정 실패 — 카드 거절 / confirm 미완료로 스케줄러 정리)
  └→ EXPIRED            (스케줄러: 결제창 미진입 이탈)

COMPLETED_PAYMENT
  ├→ COMPLETED_CANCEL_PAYMENT (취소 + PG 환불 성공)
  ├→ FAILED_CANCEL_PAYMENT    (취소 완료, PG 환불 실패)
  └→ PARTIAL_REFUNDED         (부분 환불 성공 — 얼리버드 등)

PARTIAL_REFUNDED
  ├→ COMPLETED_CANCEL_PAYMENT (잔여분 전액 환불 성공)
  └→ PARTIAL_REFUND_FAILED    (잔여분 환불 실패)

FAILED_CANCEL_PAYMENT
  └→ COMPLETED_CANCEL_PAYMENT (재환불 성공 — 스케줄러 재시도 또는 수동)

PARTIAL_REFUND_FAILED
  └→ PARTIAL_REFUNDED         (재환불 성공 — 스케줄러 재시도 또는 수동)
```

역방향 전이는 불가하다.

모든 전이 시 `OrderHistory` 레코드를 반드시 생성한다.

## 3. 환불 처리 원칙

```
[CARD 결제인 경우]
  PG 환불 실행 (Bootpay cancelReceipt)

  성공 → Order.orderStatus = COMPLETED_CANCEL_PAYMENT
         OrderHistory 기록

  실패 → Order.orderStatus = FAILED_CANCEL_PAYMENT
         OrderHistory 기록 (실패 사유 포함)
         Slack 알림 발송 (수동 처리 채널)

[FREE 결제인 경우 (price = 0 또는 OrderKind.FREE)]
  PG 호출 없이 Order.orderStatus = COMPLETED_CANCEL_PAYMENT
  OrderHistory 기록

공통 원칙:
  - 환불 결과를 삼키지(silently swallow) 않는다
  - 성공/실패 모두 OrderHistory에 기록한다
  - PG와 DB 사이는 원자적이지 않다
    → PG 성공 + DB 실패 시 Slack 알림 후 수동 조치
    → PG 실패는 DB에 FAILED_CANCEL_PAYMENT로 기록하고 재시도 대기
```

## 4. 주문 상태별 처리 규칙

> 현재 코드의 문제점과 개선 방향 → `improvements.md` 섹션 1, 2 참조
> 

### 4-1. WAITING_PAYMENT

```
[웹훅 수신 시 선행 처리]
  payload.order_id로 Order 조회
  → WAITING_PAYMENT 상태인 경우: Order.bootpayReceiptId = payload.receipt_id 세팅
  → 그 외 (이미 처리됨): 무시

[WAITING_PAYMENT 통합 스케줄러]

  CASE A: 결제 미발생 순수 이탈
    조건: bootpayReceiptId IS NULL AND updatedAt <= NOW() - 15분
    처리: PG 호출 없음 → EXPIRED, OrderHistory 기록

  CASE B: 결제 완료, confirm 미완료
    조건: bootpayReceiptId IS NOT NULL AND updatedAt <= NOW() - 10분
    처리: Bootpay.getReceipt(bootpayReceiptId)
          → 결제완료(1) 또는 승인대기(2): PG 전액 환불
          → 그 외 (이미 취소됨): 환불 스킵
          Order → FAILED_PAYMENT, PaymentRequest 삭제, OrderHistory 기록

[고아 PaymentRequest 정리] (보조 안전망)
  조건: PaymentRequest.createdAt <= NOW() - 30분 AND Payment(receiptId) 없음
  처리: PG 전액 환불 → PaymentRequest 삭제

공통 원칙:
  confirm을 재호출하지 않는다
  도메인 후처리(멤버 생성, 정원 차감)는 스케줄러에서 재현 불가 → 유저가 재결제
  모든 전이 시 OrderHistory 기록 필수
```

---

### 4-2. FAILED_CANCEL_PAYMENT

```
조건: orderStatus = FAILED_CANCEL_PAYMENT AND updatedAt <= NOW() - 10분

이 상태는 도메인 취소는 완료됐으나 PG 환불만 실패한 것이다.
멤버 상태는 이미 CANCEL/KICK/REJECT → 도메인 후처리 불필요.

처리:
  PG 환불 재시도
  → 성공: Order → COMPLETED_CANCEL_PAYMENT, OrderHistory 기록
  → 실패: OrderHistory에 재시도 실패 기록
          최대 재시도 횟수 초과 시 Slack 알림 + 수동 처리
```

---

### 4-3. PARTIAL_REFUND_FAILED

```
조건: orderStatus = PARTIAL_REFUND_FAILED AND updatedAt <= NOW() - 10분

처리:
  PG 부분 환불 재시도
  → 성공: Order → PARTIAL_REFUNDED, OrderHistory 기록
  → 실패: OrderHistory에 재시도 실패 기록
          최대 재시도 횟수 초과 시 Slack 알림 + 수동 처리
```

## 5. 유령 결제 정리 규칙

```
조건:
  PaymentRequest 레코드가 존재
  AND createdAt <= NOW() - 30분
  AND 동일 receiptId를 가진 Payment 레코드가 없음

이 케이스는 Bootpay 웹훅은 수신됐으나 Payment로 전환되지 않은 고아 건이다.

처리:
  PG 전액 환불 (실패해도 계속)
  PaymentRequest 레코드 삭제
```

## 6. 동시성 보호 원칙

```
결제 확정 시 도메인 후처리(정원 차감, 멤버 생성)는 반드시 DB 트랜잭션 내에서 처리한다.

PG 결제 승인(외부 API)과 DB 트랜잭션은 원자적으로 묶을 수 없으므로:
  - PG 승인 직전: 가용 여부 재검증 (availCount > 0, status = RECRUITING 등)
  - PG 승인 후: DB 트랜잭션 내에서 원자적으로 처리
  - PG 승인 성공 + DB 트랜잭션 실패: Slack 알림, 수동 환불 필요
```
