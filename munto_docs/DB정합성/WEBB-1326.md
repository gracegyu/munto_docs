# WAITING_PAYMENT 처리 스케줄러 3개를 통합하고 환불 실패 자동 재시도 스케줄러를 신규 구현한다

| 항목 | 내용 |
| --- | --- |
| 배경 | WAITING_PAYMENT 상태의 주문을 처리하는 스케줄러가 3개(ExpireAbandonedOrder, ConfirmMissedPayment, PaymentUpdateJob)로 분산되어 있어, 각 스케줄러가 동일 주문을 서로 다른 상태로 처리하는 불일치가 발생하고 있다. 또한 FAILED_CANCEL_PAYMENT, PARTIAL_REFUND_FAILED 상태의 주문을 자동으로 재시도하는 스케줄러가 없어, Slack 알림 수신 후 수동으로 환불을 처리하고 있다 (기존 조사된 미환불 건 3건 확인). |
| 현재 문제점 | 1. ExpireAbandonedOrder가 bootpayReceiptId 유무를 확인하지 않고 WAITING_PAYMENT 주문을 일괄 EXPIRED 처리한다. PG 웹훅을 수신하여 bootpayReceiptId가 존재하는(= 실결제된) 주문도 EXPIRED가 되어, 이후 PaymentUpdateJob이 환불하면 Order는 EXPIRED이지만 실제로는 환불 완료인 불일치 상태가 된다. 2. ConfirmMissedPayment가 bootpayReceiptId가 있는 주문에 대해 confirm을 재시도하고, 실패 시 환불 후 COMPLETED_CANCEL_PAYMENT로 기록한다. 그러나 상태 전이가 WAITING_PAYMENT에서 FAILED_PAYMENT를 거치지 않고 곧바로 COMPLETED_CANCEL_PAYMENT로 직통하여, 중간 실패 이력이 누락된다. 3. PaymentUpdateJob.refundExpiredPaymentRequests가 EXPIRED 상태 주문의 PG 환불을 실행하지만, 환불 후 Order.orderStatus를 갱신하지 않아 EXPIRED 상태로 방치된다. 또한 EXPIRED 처리(15분)와 환불(30분)의 실행 주기 차이로 15분간 '실결제 + EXPIRED' 상태가 노출된다. 4. FAILED_CANCEL_PAYMENT 재시도 스케줄러가 없어 환불 실패 건이 수동 처리에 의존하고 있다. |
| 개선 방향 | 1. 3개 스케줄러를 WAITING_PAYMENT 통합 스케줄러 1개로 합친다. bootpayReceiptId 유무로 'PG 웹훅 수신 여부'를 판별하고, (A) bootpayReceiptId 존재 시 confirm 재시도, (B) 미존재 시 만료 처리를 분기한다. 이를 위해 PG 웹훅 수신 시점에 Order.bootpayReceiptId를 즉시 세팅하도록 변경한다 (현재는 confirm 성공 후 세팅). 2. FAILED_CANCEL_PAYMENT 재시도 스케줄러를 신규 구현한다. PG 환불 API를 지수 백오프로 재시도하고, 최대 재시도 횟수(3회) 초과 시 Slack 알림을 발송한다. 3. PARTIAL_REFUND_FAILED 재시도 스케줄러도 동일한 패턴으로 구현한다. 4. 고아 Order sweeper 스케줄러를 신규 구현한다. 일괄 취소(폐강/CLOSE_BY_SYSTEM) TX 커밋 후 환불 루프 크래시로 발생한 고아 Order(멤버 종료 상태 + Order COMPLETED_PAYMENT)를 탐지하여 FAILED_CANCEL_PAYMENT로 전환하고, 2번 재시도 스케줄러에 편입시킨다. |
| 고려요소 | 스케줄러 통합 시 기존 3개 스케줄러의 실행 주기와 타이밍을 분석하여 데이터 유실 없이 전환해야 한다. PG 환불 재시도 시 멱등성을 보장하여 이중 환불을 방지해야 한다. 고아 Order sweeper의 탐지 조건(멤버 종료 상태 + Order COMPLETED_PAYMENT + updatedAt 임계값)이 정상 흐름의 일시적 상태와 겹치지 않도록 충분한 대기 시간(10분)을 설정해야 한다. |
| 기대효과 | WAITING_PAYMENT 처리가 단일 스케줄러로 통합되어 상태 불일치가 구조적으로 방지된다. 환불 실패 건이 자동 재시도되어 수동 처리 업무가 제거된다. 일괄 취소 중 크래시로 방치되는 고아 Order가 자동 복구된다. |

## 대안 1(Sweeper) 방향에 동의합니다

크래시로 인한 고아 Order는 드문 예외 케이스이므로, 정상 흐름에 새 상태를 끼워넣는 것(대안 3)이나 status와 flag를 이원화하는 것(대안 2)은 과도합니다. Sweeper가 예외만 탐지해서 기존 재시도 파이프라인에 편입시키는 구조가 가장 깔끔합니다.

다만 구현 시 아래 4가지를 보완해주시면 좋겠습니다.

### 1. 무료 주문 분기 처리

소셜링 결제/환불 규칙 명세서 섹션 3-1에서 무료 취소는 PG 호출 없이 바로 `COMPLETED_CANCEL_PAYMENT`로 전환됩니다. Sweeper가 무료 주문까지 `FAILED_CANCEL_PAYMENT`로 전환하면, 재시도 스케줄러가 PG 환불을 시도하다가 에러가 발생합니다.

- **무료 주문(Order.orderPrice = 0)**: Sweeper에서 바로 `COMPLETED_CANCEL_PAYMENT`로 전환
- **유료 주문**: `FAILED_CANCEL_PAYMENT`로 전환 후 재시도 스케줄러에 편입

이 분기를 Sweeper 수도코드에 명시해주세요.

### 2. 탐지 조건의 false positive 허용 선언

10분 임계값은 정상 PG 환불(수 초)보다 충분히 크지만, PG 장애 시에는 정상 흐름의 "환불 진행 중" 건도 탐지 대상이 됩니다. 이 경우 `FAILED_CANCEL_PAYMENT`로 전환되어 재시도 스케줄러가 다시 환불을 시도하게 되는데, PG 환불 API가 멱등성을 보장한다면 실질적 문제는 없습니다.

이것이 **의도된 동작**이라면 그 점을 명세에 한 줄 선언해두면 좋겠습니다. 예: "PG 장애로 인한 false positive도 재시도 스케줄러가 처리하므로 허용한다."

### 3. 도메인 범위 확인

소셜링(6-2 폐강, 8-2 CLOSE_BY_SYSTEM)뿐만 아니라, 클럽(CLOSE_BY_HOST)과 챌린지(CLOSE_BY_SYSTEM)의 일괄 취소에서도 같은 고아 패턴이 발생할 수 있습니다. Sweeper가 `SocialingMember`, `ClubMember`, `ChallengeMember` 전부를 커버할 건지, 도메인별로 별도 sweeper를 둘 건지 범위를 정해주세요.

### 4. updatedAt 기준 필드 명확화

탐지 조건의 `updatedAt`이 `member.updatedAt`인지 `Order.updatedAt`인지에 따라 결과가 달라집니다. 멤버 TX에서 `member.updatedAt`은 갱신되지만 `Order.updatedAt`은 갱신되지 않으므로, `member.updatedAt` 기준이 적절합니다. 수도코드에 명시해주세요.

### 5. 스케줄러 실행 주기 정의

현재 이슈에 신규 스케줄러가 3종 추가되는데, 각각의 실행 주기가 명시되어 있지 않습니다. 아래 기준으로 정의를 제안합니다.

| 스케줄러 | 권장 주기 | 근거 |
| --- | --- | --- |
| WAITING_PAYMENT 통합 | 5분 | 기존 ExpireAbandonedOrder(15분) + ConfirmMissedPayment(10분)를 하나로 합치므로, 두 기존 주기의 최소 공약 수준인 5분이면 기존보다 빠르게 정리하면서도 DB 부하는 기존 3개 스케줄러 합산보다 낮다. 탐지 조건의 updatedAt 임계값(CASE A: 15분, CASE B: 10분)이 주기보다 크므로 동일 건을 중복 처리할 위험 없다. |
| FAILED_CANCEL_PAYMENT 재시도 | 10분 | order-common 섹션 4-2에서 `updatedAt <= NOW() - 10분` 조건이므로 10분 주기가 자연스럽다. 지수 백오프는 스케줄러 주기가 아니라 개별 Order의 재시도 간격(OrderHistory 기반 카운팅)으로 구현하면 된다. |
| PARTIAL_REFUND_FAILED 재시도 | 10분 | FAILED_CANCEL_PAYMENT와 동일한 패턴 |
| 고아 Order sweeper | 30분 | 크래시로 인한 고아는 극히 드문 케이스이므로 자주 돌 필요 없다. 탐지 임계값(10분)보다 주기가 길어도 최대 복구 지연이 40분(10분 임계 + 30분 주기) 수준이므로, 고아 상태 허용 시간으로 충분하다. |

특히 지수 백오프와 스케줄러 주기를 혼동하지 않도록 주의가 필요합니다. 스케줄러 주기는 일정하게 유지하고, 개별 Order 단위로 OrderHistory의 재시도 횟수를 카운팅해서 "이번 주기에 이 Order를 재시도할지"를 판단하는 구조가 적절합니다.

```
예시: FAILED_CANCEL_PAYMENT 재시도 (10분 주기 스케줄러)
  1회차: 실패 후 10분 경과 → 재시도
  2회차: 1회차 실패 후 20분 경과 → 재시도 (지수 백오프)
  3회차: 2회차 실패 후 40분 경과 → 재시도 (지수 백오프)
  3회 초과: Slack 알림 + 수동 처리 전환
```

### 6. 스케줄러 호출 방식 권장

현재 결제 스케줄러는 `apps/scheduler/` 내 NestJS Cron 기반으로 ECS에서 실행되고 있습니다. 신규 스케줄러의 호출 방식을 어떻게 할지 결정이 필요합니다.

**결론부터: 이번 이슈는 기존 ECS Cron 방식으로 구현하되, 향후 Serverless 전환 시 함께 마이그레이션하는 것을 권장합니다.**

근거는 아래 비교를 참고해주세요.

| 항목 | 기존 방식 (ECS Cron) | EventBridge + SQS + Lambda |
| --- | --- | --- |
| 구현 비용 | 낮음. 기존 `apps/scheduler/` 패턴 그대로 추가 | 높음. Lambda 함수 + CDK 인프라 + SQS DLQ 구성 필요 |
| DB 접근 | 기존 Prisma 클라이언트 그대로 사용 | Lambda에서 VPC 접근 + DB 커넥션 풀 별도 관리 필요 |
| 실패 처리 | 다음 주기에 자연스럽게 재시도 (updatedAt 조건으로 이미 멱등) | Lambda 재시도 + DLQ로 더 정교하지만 구성 복잡 |
| 확장성 | 단일 인스턴스 의존 | Lambda 수평 확장 가능 |
| 운영 일관성 | 기존 스케줄러와 동일한 배포/모니터링 파이프라인 | 별도 인프라 스택 관리 필요 |

**ECS Cron을 권장하는 이유:**

1. 이번 이슈의 핵심은 **스케줄러 통합과 재시도 로직 구현**이지, 인프라 전환이 아닙니다. 인프라까지 동시에 바꾸면 변경 범위가 커져서 문제 발생 시 원인 격리가 어렵습니다.
2. Sweeper와 재시도 스케줄러 모두 도메인별 멤버 테이블 조인 + Order 상태 갱신이 필요한데, 기존 `apps/scheduler/`의 Prisma 클라이언트와 서비스 레이어를 그대로 재사용하는 것이 구현이 빠릅니다.
3. WEBB-1212에서 이미 알림 이벤트(pushEvent)의 SQS+Lambda 전환이 기획 중이므로, 해당 전환이 안정화된 후 결제 스케줄러 전체를 함께 마이그레이션하는 것이 리스크를 분산하는 방법입니다.

**다만 Lambda 전환이 적합한 시점:**

- WEBB-1212의 SQS+Lambda 전환이 프로덕션 안정화된 이후
- 결제 스케줄러 전체(통합 스케줄러 + 재시도 + Sweeper)를 한 번에 전환
- 이때 CDK 기반으로 EventBridge Rule(주기 트리거) + Lambda + SQS DLQ 구성
