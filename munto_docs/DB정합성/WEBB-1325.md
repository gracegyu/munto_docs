# 소셜링·챌린지의 취소/환불 로직을 공통 서비스로 재설계하여 상태 변경 순서 버그와 환불 정책 오적용을 구조적으로 방지한다

| 항목 | 내용 |
| --- | --- |
| 배경 | 소셜링·클럽·챌린지의 취소/환불 로직이 도메인마다 제각각으로 구현되어 있어, 상태 변경 순서 버그·환불 정책 오적용·이중 호출 등 정합성 문제가 반복적으로 발생하고 있다. |
| 현재 문제점 | 1. 소셜링 V4에서는 멤버 상태를 먼저 변경한 뒤 cancelOrder를 호출하여, DB에서 변경된 상태로 조회되어 환불 정책이 잘못 적용된다 (REQUEST 멤버가 APPROVED_NON_REFUNDABLE로 기록된 건이 prod에 450건 존재).<br>2. 챌린지에서는 afterDeleteChallenge가 멤버 상태를 DELETE로 먼저 변경한 뒤 주문 취소를 시도하여, REQUEST/APPROVE 조회 결과가 0건이 되어 주문 취소가 실행되지 않았다.<br>3. 클럽 cancelClubMembership은 SUBSCRIBED 상태만 처리하고 GRACE_PERIOD는 order.update를 건너뛴다.<br>4. 취소/환불 로직이 V2(소셜링·챌린지·클럽)에만 있고 V3에는 미구현 상태이다. |
| 개선 방향 | 1. 도메인 공통 취소/환불 서비스(OrderCancelService)를 설계하여 상태 변경 → 환불 계산 → PG 환불 → Order 업데이트 순서를 단일 트랜잭션으로 보장한다.<br>2. cancelStatus 결정 로직을 멤버 상태 변경 전에 원본 상태를 기준으로 계산하도록 통일한다.<br>3. V3 Order 서비스에 취소/환불 로직을 포함하여 재작성한다. |
| 고려요소 | 운영 중인 결제/환불 흐름을 변경하므로, V2 → V3 전환 시 기존 주문과의 하위 호환성을 보장해야 한다.<br>PG사(Bootpay) 환불 API의 멱등성과 재시도 정책을 함께 고려해야 한다. |
| 기대효과 | 취소/환불 로직이 단일 서비스로 통합되어, 상태 변경 순서 버그나 환불 정책 오적용이 구조적으로 방지된다.<br>도메인별 중복 코드가 제거되어 유지보수성이 향상된다. |
