## 총평

992줄에 달하는 매우 상세하고 체계적인 OnePager입니다. 이전 Jira comment에서 제안한 6개 항목(무료 주문 분기, false positive 선언, 도메인 범위, updatedAt 기준 필드, 실행 주기, 호출 방식)이 모두 본문에 잘 반영되어 있습니다. 특히 다음 부분이 인상적입니다:

- **"1 Case = 1 Job" 원칙(§2.1)**: 분기 내 분기를 금지하고, Case 단위로 Feature Flag·장애 격리·람다 이관을 설계한 것이 전체 구조의 명확성을 담보합니다.
- **전체 케이스 매트릭스(§3.1)**: 그룹 A~E로 예외 성격을 분류하고, 행렬·다이어그램·대상/PG/도메인 요약 표를 다각도로 제시하여 11개 Case의 관계를 한눈에 파악할 수 있습니다.
- **원자적 check+update 원칙(§2.6)**: Order 2938107 실측 race condition(0.55초 간극)을 근거로 blind UPDATE를 금지하고 WHERE 가드를 의무화한 것은 실 사례 기반의 설득력 있는 설계입니다.
- **암묵적 백오프 설계(§2.6, Case 4·5)**: 이전 제안한 지수 백오프 대신 "스케줄러 주기 = 암묵적 백오프 + 좀비 수용 + BACK-202 운영자 수동 판정"을 채택했는데, WEBB-1325 멱등 계약에 의존하는 근거가 충분히 설명되어 있어 합리적 판단으로 봅니다.
- **BACK-202 연계(§7.1)**: 자동 재시도 인터페이스와 백오피스 수동 판정 책임의 경계를 명확히 정의한 것이 좋습니다.
- **이전 comment 반영**: 무료 주문 분기(Case 6·7), false positive 허용 선언(Case 6), 도메인 분리(Case 6·7·12), grace 기준 필드 명확화(Case 6·7), 실행 주기 정의(§3.1), ECS Cron 유지 판단(§2.5 + 각 Case "람다 이관 시") 모두 잘 반영되었습니다.

수정이 필요한 항목은 2건(MEDIUM)이며, 나머지는 보충 사항입니다.

## 잘 된 점

- **"1 Case = 1 Job" + Feature Flag 개별화(§2.1·§2.6)**: 장애 격리·단계 롤아웃·람다 이관 단위가 일치하는 깔끔한 설계입니다.
- **전체 케이스 매트릭스(§3.1)**: 그룹별 요약(A~E) → Order 상태 전이 행렬 → 대상·PG·도메인 요약 → 예외 성격 다이어그램의 4단 구성이 11개 Case의 관계와 경계를 명확히 합니다.
- **원자적 check+update(§2.6)**: Order 2938107 실측 사례를 기반으로 blind UPDATE를 금지하고, PostgreSQL의 UPDATE WHERE 재평가 동작을 활용한 race 차단 설계가 강력합니다.
- **SOT=PG 원칙(§2.6)**: Case 2가 `getReceipt`로 PG에 직접 상태를 물어보는 것은 ASIS의 웹훅 간접 추론 방식 대비 근본적 개선입니다.
- **암묵적 백오프 + 좀비 수용(§2.6, Case 4·5)**: 명시적 retryCount를 두지 않는 대신 WEBB-1325 멱등 계약에 의존하는 설계가 시스템 복잡도를 낮추면서도 안전을 유지합니다. BACK-202 운영 연계로 영구 실패 건을 관리하는 출구도 마련되어 있습니다.
- **Case 6·7 무료 주문 분기**: 이전 comment에서 제안한 무료 주문 처리가 명확하게 반영되었습니다.
- **레거시 데이터 cutoff 원칙(§2.2)**: 과거 EXPIRED 오분류 건을 본 이슈에서 정정하지 않고 WEBB-1346으로 이관하는 판단과 그 근거(PG 금전 정합 확인, 스키마 전환 1회 집중)가 합리적입니다.
- **WEBB-1325 계약 사용 매핑(§5)**: Case별로 어떤 API와 멱등성 키를 사용하는지 표로 정리한 것이 두 이슈 간 경계를 명확히 합니다.
