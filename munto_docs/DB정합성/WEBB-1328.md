# 클럽·코스 정산 스케줄러가 orderStatus를 검증하지 않아 취소된 주문이나 이중결제 건이 호스트에게 정산될 수 있다

| 항목 | 내용 |
| --- | --- |
| 배경 | 정산 스케줄러가 도메인별로 Order 필터링 기준이 다르며, 클럽·코스는 orderStatus를 검증하지 않고 remainPrice > 0만 체크하여 취소된 주문이나 이중결제 건이 호스트에게 정산될 수 있다.<br>정산 규칙 분석 문서: munto-dev-assistant/document/munto/order/settlement.md |
| 현재 문제점 | 1. 챌린지만 orderStatus = COMPLETED_PAYMENT를 명시적으로 필터링하며, 클럽·코스는 orderStatus를 전혀 체크하지 않는다. 소셜링은 PARTIAL_REFUNDED 등 특수 상태만 분기하고 나머지는 remainPrice > 0으로만 판단한다.<br>2. 이중결제로 Order가 2건 생성되면 둘 다 remainPrice > 0이므로 클럽·코스 정산에서 둘 다 호스트에게 지급된다.<br>3. FAILED_CANCEL_PAYMENT 상태(PG 환불 실패)인 주문도 remainPrice가 그대로 남아있어 정산에 포함될 수 있다. 이후 수동 환불 시 정산 차감 프로세스가 별도로 없다.<br>4. 소셜링에서 STAFF 멤버를 정산 인원에서 차감하지만, STAFF가 유료 결제한 경우에는 결제 금액이 정산에서 누락될 수 있다. |
| 개선 방향 | 1. 모든 도메인의 정산 대상 Order 필터에 orderStatus 조건을 추가한다. COMPLETED_PAYMENT와 PARTIAL_REFUNDED만 정산 대상으로 포함하도록 통일한다.<br>2. 환불 처리 시 Order.remainPrice를 반드시 갱신하여, 전액 환불 시 remainPrice = 0이 되도록 보장한다.<br>3. 정산 후 환불이 발생한 경우 정산 차감(Settlement 수정 또는 다음 정산에서 차감) 프로세스를 설계한다. |
| 고려요소 | 기존 정산 데이터 중 잘못 포함된 건이 있는지 확인하고, 이미 지급된 건에 대한 차액 정산 방안을 마련해야 한다.<br>orderStatus 필터 추가 시 기존에 remainPrice > 0으로만 정산되던 정상 건이 누락되지 않는지 영향 분석이 필요하다. |
| 기대효과 | 취소·환불된 주문이 정산에 포함되지 않아 호스트에게 초과 지급되는 문제가 방지된다.<br>도메인별 정산 필터링 기준이 통일되어 유지보수성이 향상된다. |
