# WEBB-1325 소셜링·챌린지 취소/환불 로직 V3 공통화 OnePager — 리뷰

- **대상 문서**: 소셜링·챌린지 취소/환불 로직 V3 공통화 OnePager (v1.0.0, 26.04.22, 김범진)
- **리뷰어**: 전규현
- **리뷰 일자**: 2026-04-27

---

## 총평

925줄에 달하는 매우 상세하고 완성도 높은 OnePager입니다. WEBB-1321 규칙 문서와의 정렬이 철저하며, 특히 다음 부분이 인상적입니다:

- **용어 정의(§1)**: "취소"와 "환불"의 의미 경계를 명확히 구분하고, 클래스·enum 이름별로 어떤 의미인지 일일이 매핑한 것이 이 문서 전체의 가독성을 높입니다.
- **C안 인터페이스 분리 + 두 조립자**: `OrderCancelV3Service`(멤버 상태 전이 있음)와 `OrderRefundV3Service`(전이 없음) 분리로 "스태프를 실수로 CANCEL 처리"하는 경로를 컴파일 레벨에서 차단한 설계가 적절합니다.
- **사가 원자성·멱등성 계약(§5.9)**: 크래시 지점별 DB/PG 상태와 복구 경로를 행렬로 정리한 것이 구현자·리뷰어 모두에게 큰 도움이 됩니다.
- **WEBB-1326과의 계약(§7)**: Case별 소비 매핑 표가 두 이슈 간 경계를 명확히 합니다.
- **V2→V3 이관 맵(§6.7)**: 이관/신규/WEBB-1326 영역을 체계적으로 구분하여 구현 시 빠짐없이 추적할 수 있습니다.
- **WEBB-1321 리뷰 미해소 CH-M2 해소**: CHOICE→COMMON 전환 시 환불 경로 누락 문제를 §6.3·§6.7에서 명시적으로 흡수한 것이 좋습니다.

수정이 필요한 항목은 3건(HIGH 1, MEDIUM 2)이며, 나머지는 보충·확인 사항입니다.

## 잘 된 점

- **용어 정의(§1)의 철저함**: "취소"(멤버 상태 전이) vs "환불"(멤버 상태 유지) vs "PG 환불"(하위 단계)의 3단 구분이 925줄 문서 전체의 일관성을 담보합니다. 클래스·enum별 의미 매핑(§1.5)까지 있어 혼동의 여지가 거의 없습니다.
- **C안 채택 근거(§5.1)**: 3가지 대안을 장단점으로 비교한 뒤 인터페이스 분리를 선택한 과정이 투명합니다.
- **두 조립자 분리(§5.2)**: `OrderRefundV3Service`의 의존성에 멤버 상태 변경 메서드가 물리적으로 없는 것은 "스태프를 CANCEL 처리하는" 유형의 실수를 컴파일 레벨에서 차단하는 강력한 설계입니다.
- **사가 크래시 복구 행렬(§5.9.1·§5.9.2)**: 크래시 지점 × DB 상태 × PG 상태 × 복구 경로의 조합을 빠짐없이 나열한 것이 이 OnePager의 핵심 가치입니다. WEBB-1326과의 복구 경로 대응(Case 6/7, Case 10/11)도 명확합니다.
- **멱등성 계약(§5.9.3)**: 각 Stage의 멱등 규약이 재진입 시 안전을 보장하는 방식으로 설계되었습니다. 특히 `cancelStatus`/`refundAmount` 결정 시점을 Stage 1(원본 상태)으로 고정한 것이 소셜링 V4 450건 버그의 근본 원인을 직접 해소합니다.
- **WEBB-1326 계약 표(§7.2)**: Case별 소비 매핑이 두 이슈 간 경계와 책임을 명확히 하여 병렬 개발이 가능합니다.
- **이관 맵(§6.7)**: "이관", "신규", "WEBB-1326 영역"으로 체계적 분류하여 구현 시 누락 추적이 용이합니다.
- **CHOICE→COMMON 흡수(§6.3·§6.7)**: WEBB-1321 리뷰 미해소 항목 CH-M2(CHOICE→COMMON 환불 경로 미명시)를 본 이슈에서 실질적으로 해소합니다.
- **`getReceipt` 재조회(§5.3 불변식 5)**: PG 응답 불명 시 SOT=PG 원칙을 재조회로 구현한 것은 V2에 없던 중요한 개선입니다.
