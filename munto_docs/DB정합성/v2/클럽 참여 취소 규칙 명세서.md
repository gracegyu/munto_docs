## 1. 핵심 개념

### 1-1. 관련 테이블

| 테이블 | 역할 |
| --- | --- |
| `Club` | 클럽 |
| `ClubMember` | 클럽 참여 멤버 |
| `ClubMembership` | 유료 멤버십 (결제 정보, 빌링키) |
| `Order` | 멤버십 결제 주문. `ClubMembership`을 통해 연결 |

### 1-2. ClubMember 상태

```
REQUEST   가입 신청 (승인제 클럽)
APPROVE   승인된 활성 멤버
REJECT    호스트에 의한 거절/강퇴
CANCEL    본인 취소
DELETE    소프트 삭제 (CANCEL/REJECT 이후 정리)
```

### 1-3. ClubMember 등급

```
OWNER   클럽장. 1명. 클럽 생성자. availCount에 포함되지 않음
STAFF   운영진. 최대 10명. OWNER가 MEMBER → STAFF 승격
        일반 MEMBER 승인/강퇴 가능. STAFF 간 권한 행사 불가 (STAFF → STAFF 승인/강퇴 불가)
MEMBER  일반 멤버
```

> OWNER는 클럽 생성 시 자동으로 APPROVE 상태가 되며 availCount를 차지하지 않는다.
STAFF는 일반 멤버와 동일하게 결제하고 가입한다 (소셜링 STAFF와 달리 완료 시 환불 없음).
*이 비대칭은 **현행 정책이며 고도화 단계에서 재검토 대상**이다 (PM 확인 완료, 2026-04-21).*
OWNER는 본인 취소로 클럽에서 빠질 수 없다. 탈퇴하려면 클럽 삭제(§5-5)가 유일한 경로이다.
등급 변경(§5-4)도 MEMBER ↔ STAFF만 대상이며, OWNER는 변경 대상이 될 수 없다.
OWNER 양도는 어드민API(`changeClubOwner`)로만 가능하며 유료 클럽은 양도 불가하다.
*(현행 정책, 고도화 단계 재검토 대상 — PM 확인 완료, 2026-04-21).*
> 

### 1-4. ClubMembership 상태

```
PAID          일회성 결제 완료 (ONETIME 타입)
SUBSCRIBED    구독 활성 (SUBSCRIPTION 타입)
EXPIRED       구독 만료
GRACE_PERIOD  결제 실패, 유예 기간 중
```

### 1-5. ClubMembership 타입

```
ONETIME       단건 결제. 기간 무제한 (expireAt = 2099-12-31)
SUBSCRIPTION  월 정기 결제. expireAt = 가입일 + 1개월 단위 갱신
```

### 1-6. 모집 방식

```
COMMON (선착순)  가입 즉시 APPROVE. availCount 즉시 차감
CHOICE (승인제)  가입 시 REQUEST. 호스트 승인 후 APPROVE. availCount는 승인 시 차감
                 recruitQuestion 필수 설정
```

## 2. 핵심 불변식

#### Invariant 1 (정원 정합성)

```
  availCount = maximumMembers - COUNT(ClubMember WHERE grade != OWNER AND status = APPROVE)
  → 항상 0 ≤ availCount ≤ maximumMembers
  → OWNER는 정원에 포함되지 않음
```

#### Invariant 2 (멤버-멤버십 정합성)

```markdown
  유료 클럽 APPROVE 멤버는 반드시 PAID 또는 SUBSCRIBED 상태의 ClubMembership을 가진다
  무료 클럽 APPROVE 멤버는 ClubMembership이 없다
```

#### Invariant 3 (멤버십-주문 정합성)

```markdown
  ClubMembership이 존재하면 연결된 Order가 반드시 존재한다
  Order.orderStatus = COMPLETED_PAYMENT
```

#### Invariant 4 (구독 상태 정합성)

```markdown
  GRACE_PERIOD 멤버는 ClubMember.status = APPROVE를 유지한다
  (유예 기간 중에는 강퇴하지 않고 재결제 기회 부여)
```

## 3. 상태 전이 규칙

```
ClubMember

REQUEST
  ├→ APPROVE  (CHOICE: 호스트 승인)
  ├→ REJECT   (CHOICE: 호스트 거절)
  └→ CANCEL   (본인 취소, 24시간 미승인 스케줄러)

APPROVE
  ├→ REJECT   (호스트 강퇴)
  └→ CANCEL   (본인 취소, 클럽 삭제)

REJECT / CANCEL
  └→ DELETE   (정리)

재신청: CANCEL/REJECT/DELETE → REQUEST (CHOICE) 또는 → APPROVE (COMMON)

ClubMembership

SUBSCRIBED
  ├→ GRACE_PERIOD  (갱신 실패)
  └→ EXPIRED       (OWNER 또는 수동 처리)

GRACE_PERIOD
  ├→ SUBSCRIBED    (재결제 성공)
  └→ EXPIRED       (수동 처리)
```

## 4. 가입 플로우

### 4-1. 가입 가능 여부 검증

```
조건 (모두 충족):
  Club.deletedAt IS NULL (삭제되지 않음)
  Club.isShow = true
  availCount > 0
  신청 유저가 이미 APPROVE 또는 REQUEST 상태가 아님
  성별 정책(genderPolicy) 충족 (MALE/FEMALE 클럽인 경우)
```

### 4-2. COMMON (선착순) 가입

```
[단계 A] 가입 가능 여부 검증

[단계 B] 결제 처리 (order-club.md 참조)

[단계 C] DB 트랜잭션
  [정원 차감 — 원자적 처리]
  updated = tx.Club.updateMany(
    WHERE id = clubId AND availCount > 0,
    DATA { availCount: decrement(1) }
  )
  IF updated.count ≠ 1 → TX ROLLBACK  ← 정원 초과 방어

  ClubMember 생성 (status = APPROVE, grade = MEMBER)
  ClubMembership 생성 (유료인 경우)
  Order.orderStatus = COMPLETED_PAYMENT
  OrderHistory 생성

[단계 D] 후처리 (트랜잭션 외부)
  Sendbird 채널 참여
  Push 발송
```

### 4-3. CHOICE (승인제) 가입

```
[단계 A] 가입 가능 여부 검증

[단계 B] 결제 처리 (order-club.md 참조)
  → REQUEST 상태이므로 availCount 차감 없음

[단계 C] DB 트랜잭션
  ClubMember 생성 (status = REQUEST, grade = MEMBER)
  ClubMembership 생성 (유료인 경우, 멤버십은 승인 전에도 생성)
  Order.orderStatus = COMPLETED_PAYMENT
  OrderHistory 생성

[단계 D] 후처리
  호스트에게 신청 알림 Push 발송
※ 이 경로의 availCount 차감은 호스트 승인 단계(§5-1)에서 발생한다.
  원자적 차감 패턴은 §5-1 참조.
```

## 5. 호스트 작업

### 5-1. 멤버 승인 (REQUEST → APPROVE)

```
FUNCTION approve(clubId, memberId):

  권한: OWNER 또는 STAFF (단, STAFF는 STAFF를 승인 불가)

  [선행 검증]
  - member.status = APPROVE → 이미 승인됨, 멱등 처리 후 종료
  - member.status ≠ REQUEST → 승인 불가

  [트랜잭션]
  BEGIN:
    [정원 차감 — 원자적 처리]
    updated = tx.Club.updateMany(
      WHERE id = clubId AND availCount > 0,
      DATA { availCount: decrement(1) }
    )
    IF updated.count ≠ 1 → TX ROLLBACK  ← 정원 없음

    ClubMember.status = REQUEST → APPROVE, approvedAt = NOW()
  COMMIT

  [후처리 — 트랜잭션 외부]
  예약 대기 중이던 소셜링 참여 처리 (deleteClubReservation)
  Sendbird 채널 참여
  Push 발송 (승인 알림)
```

### 5-2. 멤버 강퇴 (APPROVE → REJECT)

```
권한: OWNER 또는 STAFF (단, STAFF는 STAFF를 강퇴 불가)

처리:
  availCount 복구
  ClubMember.status = REJECT
  ClubMembership → EXPIRED, 결제 취소/환불 처리 (order-club.md 참조)
  유저의 클럽 피드 삭제
  유저가 모집 중인 소셜링 편집/분리
  Sendbird 채널 퇴장 (reason = KICK)
  Push 발송 (강퇴 알림)
```

### 5-3. 멤버 거절 (REQUEST → REJECT)

```
권한: OWNER 또는 STAFF (단, STAFF는 STAFF를 거절 불가)

처리:
  availCount 변동 없음 (REQUEST는 차감 안 됐으므로)
  ClubMember.status = REJECT
  결제 취소/환불 처리 (order-club.md 참조)
  Push 발송 (거절 알림)
```

### 5-4. 등급 변경 (MEMBER ↔ STAFF)

```
권한: OWNER만 가능

제약:
  STAFF 총 인원 최대 10명
  STAFF → MEMBER 강등도 가능

처리:
  ClubMember.grade 업데이트
  결제/환불 변동 없음
  (클럽 STAFF는 소셜링 STAFF와 달리 완료 시 환불 없음)
```

### 5-5. 클럽 삭제

```
FUNCTION deleteClub(clubId, ownerUserId):

  [선행 검증]
  - 호출자 = Club.ownerUserId → 아니면 ForbiddenException
  - Club.deletedAt ≠ NULL → 이미 삭제됨, 멱등 처리 후 종료

  [단일 트랜잭션 — 상태 전이 원자화]
  BEGIN TX:
    Club.deletedAt = NOW()

    FOR EACH member WHERE status ∈ {REQUEST, APPROVE}:
      wasApprove = (member.status = APPROVE)
      ClubMember.status → CANCEL, canceledAt = NOW()

      IF wasApprove AND member.grade != OWNER:
        Club.availCount += 1
        ※ OWNER는 availCount에 포함되지 않으므로 복구 대상 아님 (Invariant 1)

      IF member의 ClubMembership 존재:
        ClubMembership.status → EXPIRED
  COMMIT

  Club 상태 전이와 모든 멤버·멤버십 상태 전이를 하나의 TX에 묶어
  부분 실패로 인한 상태 불일치를 방지한다.

  [취소 처리 위임 → 주문 도메인] (TX 외부)
  FOR EACH 유료 멤버:
    PG 환불 실행 (order-club.md §3 참조)
    → 성공: Order.orderStatus → COMPLETED_CANCEL_PAYMENT
    → 실패: Order.orderStatus → FAILED_CANCEL_PAYMENT
            재시도 스케줄러가 복구 (주문/결제 공통 규칙 명세서 5-2)

  [예외 복구 — Sweeper 스케줄러]
  TX 커밋 후 환불 루프 크래시 등으로 "멤버 종료 상태 + Order=COMPLETED_PAYMENT"
  상태 불일치 Order가 발생한 경우, Sweeper 스케줄러가 이를 탐지하여
  FAILED_CANCEL_PAYMENT로 전환 후 기존 재시도 루프에 편입한다 (WEBB-1326).

	[후처리 — 트랜잭션 외부]
  유저별 클럽 피드 삭제
  유저가 모집 중인 소셜링 편집/분리
  예약 삭제
  Sendbird 채널 퇴장 처리 (모든 멤버)
  Push 발송 (전 멤버 대상)
```

## 6. 본인 취소

```
[선행 검증]
- member.grade = OWNER → 취소 불가
  (클럽 삭제 §5-5를 통해서만 OWNER 탈퇴 가능)
- member.status ∉ {REQUEST, APPROVE} → 취소 불가
- member.status = CANCEL → 이미 취소됨, 멱등 처리 후 종료

취소 가능 상태: REQUEST 또는 APPROVE

APPROVE 추가 제약:
  진행 중인 클럽 소셜링(모집 중 또는 참여 중)이 있으면 취소 불가

처리:
  ClubMember.status = CANCEL
  결제 취소/환불 처리 (order-club.md 참조)
  유저의 클럽 피드 삭제
  유저가 모집 중인 소셜링 편집/분리
  Sendbird 채널 퇴장 (reason = LEAVE)
  Push 발송 (취소 확인)
```

## 7. 정원 변경 규칙

정원 변경 시 `availCount`는 반드시 `newMax - currentApprovedCount`로 **재계산**한다.
`oldMax → newMax` delta 방식은 실제 인원을 반영하지 못하므로 사용 금지.
**OWNER는 정원에 포함되지 않으므로 카운트에서 제외**한다(Invariant 1).

정원 변경과 4-2/5-1의 정원 차감은 같은 `Club` 행을 수정하므로, 모든 조회/업데이트는
**TX 내부**에서 실행하고 **`SELECT ... FOR UPDATE`로 Club 행을 잠가 직렬화**한다.
TX 내부 조회만으로는 PostgreSQL READ COMMITTED 격리 수준에서 COUNT 후 UPDATE 사이에
다른 트랜잭션의 APPROVE 커밋이 끼어들어 availCount가 실제보다 크게 기록될 수 있다.

### 7-1. 정원 변경

```
FUNCTION updateMaximumMembers(clubId, newMax):

  BEGIN TX:
    SELECT * FROM Club WHERE id = clubId FOR UPDATE   ← 행 잠금

    currentApprovedCount = COUNT(ClubMember
                                 WHERE clubId = id
                                   AND grade != OWNER
                                   AND status = APPROVE)

    IF newMax < currentApprovedCount → BadRequestException (ROLLBACK)

    Club.maximumMembers = newMax
    Club.availCount = newMax - currentApprovedCount
  COMMIT
```

> • 정원 축소는 현재 OWNER 제외 APPROVE 인원 이상으로만 허용된다.
• 이미 APPROVE된 인원을 강제로 내보낼 기준(누구를 내보낼지)이 없기 때문이다.

## 8. 스케줄러 처리 규칙

### 8-1. REQUEST 24시간 미승인 자동 취소

```
[스케줄러: 10분마다]

대상:
  ClubMember.status = REQUEST
  AND updatedAt < NOW() - 24시간

처리:
  ClubMember.status = CANCEL
  결제 취소/환불 처리 (order-club.md 참조)
  Push 발송 (C07: 미승인 자동 취소 안내)
```

### 8-2. 구독 자동 갱신

```
[스케줄러: 매일 오전 5시]

대상: ClubMembership.status = SUBSCRIBED AND expireAt < NOW() + 1일
     (※ GRACE_PERIOD는 대상에서 제외 — 아래 GRACE_PERIOD 처리 참조)

처리:
  빌링키로 Bootpay 자동 결제 시도
  → 성공: ClubMembership.expireAt += 1개월
          새 Order 생성 (orderStatus = COMPLETED_PAYMENT)
          OrderHistory 생성
          Push 발송 (갱신 성공)
  → 실패: ClubMembership.status = GRACE_PERIOD
          ClubMember.status = APPROVE 유지 (강퇴 안 함)
          SMS_066 발송 (유예기간 진입 안내)
```

#### GRACE_PERIOD 상태 처리 (현재)

```markdown
GRACE_PERIOD 진입 후:
  - 자동 재결제 재시도 없음 (갱신 스케줄러 대상 아님)
  - 자동 만료(EXPIRED) 전환 없음 — 유예 기간 일수 기반 자동 전환 스케줄러 미구현
  - EXPIRED로 전환되는 유일한 경로는 멤버의 수동 취소(cancelClubMembership API)
  - SUBSCRIBED로 복귀하는 유일한 경로는 유저가 수동 재결제 API를 호출하는 것

※ PM 확인 완료 (2026-04-21): 재결제 유예 기간은 운영상 필요하다고 확정.
  현재 상태(유저 액션 없을 시 GRACE_PERIOD 영구 체류)는 미완성이며,
  자동 재결제 재시도 + 일수 기반 자동 EXPIRED 전환을 **고도화 단계에서 구현 예정**.
  세부 설계는 개선 목록 §11 참조.
```

## 9. availCount 요약

| 이벤트 | availCount 변화 |
| --- | --- |
| COMMON 가입 | -1 |
| CHOICE 가입 (REQUEST) | 변동 없음 |
| REQUEST → APPROVE | -1 |
| APPROVE → CANCEL | +1 |
| APPROVE → REJECT (강퇴) | +1 |
| REQUEST → CANCEL/REJECT | 변동 없음 |
| maximumMembers 변경 | `availCount = newMax - COUNT(APPROVE)` |
| MEMBER → STAFF 또는 STAFF → MEMBER | 변동 없음 |
