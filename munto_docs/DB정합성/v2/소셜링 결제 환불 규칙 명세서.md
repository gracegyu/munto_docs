## 1. 소셜링 결제 구조

```
SocialingMember (1:1) Order (1:1) Payment
```

- 소셜링 참여권은 Order로 표현된다
- `SocialingMember.orderId`는 `NOT NULL @unique`
- 유료·무료 구분 없이 Order는 반드시 존재한다

### 결제 방식 분기

```
Socialing.price > 0  →  단건 결제 (Bootpay confirmReceipt)
Socialing.price = 0  →  무료 (PG 호출 없음, Order만 생성)
```

## 2. 핵심 불변식

`주문/결제 공통 규칙 명세서 §2`의 불변식이 모두 적용된다. 소셜링 특화 구체화는 아래와 같다.

### Invariant 1 — 소셜링 멤버-Order 엄격 1:1

```
  SocialingMember.orderId NOT NULL @unique
  재신청 시 새 Order가 생성되고 SocialingMember.orderId는 새 orderId로 교체된다.
  (socialing-pseudocode.md §4-5 참조)
```

### Invariant 2 — 얼리버드 부분환불 금액 보존

```
  부분환불 완료 후:
    remainPrice = orderPrice - 할인액(환불된 금액)
    refundPrice = 할인액
  order-common.md Invariant 6 (금액 정합성)을 유지한다.
  부분환불 전이(COMPLETED_PAYMENT → PARTIAL_REFUNDED)도 order-common.md
  Invariant 4(OrderHistory 완전성)에 따라 OrderHistory를 반드시 기록한다.
```

### Invariant 3 — 얼리버드 부분환불 후 멤버 상태

```
  얼리버드 부분환불은 "취소(이탈)"가 아니라 "혜택 지급" 성격이다.
  환불 후에도 SocialingMember.status는 변하지 않으며 APPROVE가 유지된다.
  멤버 종료 상태가 아니므로 order-common.md §5-4 Sweeper 대상이 아니다.
```

### Invariant 4 — 스태프 환불 시점

```
  grade = STAFF AND status = APPROVE 멤버는 소셜링 COMPLETE 전이 시점에
  일괄 전액 환불된다. 중간 시점(grade 변경 등) 환불 없음.
  환불 후에도 SocialingMember.status = APPROVE는 유지된다
  (멤버 종료 상태가 아니므로 §5-4 Sweeper 대상이 아님).
```

## 3. 결제 확정 플로우

소셜링 도메인의 confirm 흐름에서 호출되는 결제 처리 순서.

```
[단계 A] 소셜링 가용 여부 재검증 (소셜링 도메인)
  - Socialing.status = RECRUITING
  - Socialing.availCount > 0

[단계 B] 결제 처리
  유료 (price > 0):
    Bootpay.confirmReceipt(receiptId)
    → 성공: receipt 데이터 확보
    → 실패: 전체 플로우 중단, 유저에게 에러 반환

  무료 (price = 0):
    이 단계 건너뜀

[단계 C] DB 트랜잭션
  Order.orderStatus = WAITING_PAYMENT → COMPLETED_PAYMENT
  Order.orderPrice = 결제된 금액

  [금액 산출]
  얼리버드 할인 적용 시:
    Order.refundPrice = 얼리버드 할인액
                        (취소 시 부분 환불될 예정 금액, §4-2 참조)
    Order.remainPrice = orderPrice - refundPrice
  얼리버드 미적용:
    Order.refundPrice = 0
    Order.remainPrice = orderPrice
  (Invariant S2 및 order-common.md Invariant 6 유지)

  OrderHistory 생성

  유료인 경우 추가:
    Payment 생성 또는 기존 Payment 재사용 (receiptId 기준)

  소셜링 도메인 후처리 (같은 트랜잭션):
    SocialingMember 생성/UPSERT
    Socialing.availCount 차감 (APPROVE인 경우)
```

## 4. 취소/환불 플로우

소셜링 멤버 취소(본인·호스트 킥·폐강)는 소셜링 도메인에서 트리거된다.

Order 상태 변경은 아래 규칙을 따른다.

### 4-1. 일반 취소 (본인 취소, 킥, 거절, 폐강)

환불 금액은 취소 주체와 시점에 따라 결정된다.

### 본인 취소 (USER) — 시간 컷오프 적용

```
IF 무료 (price = 0)                              → 환불 없음
IF 모임 진행 당일 (startDate와 같은 날)           → 환불 없음 (최우선)
IF 결제 후 30분 이내                              → 전액 환불
IF 원본 member.status = REQUEST (승인 대기)       → 전액 환불
IF 원본 member.status = APPROVE:
   IF 진행일까지 4일 이상 남음                    → 전액 환불
   그 외 (3일 이하)                              → 환불 없음
```

"당일" 규칙이 "30분 이내"보다 우선한다. 당일에 결제하고 즉시 취소해도 환불되지 않는다.

### 호스트 주도 취소 (HOST) — 항상 전액 환불

호스트 거절(REQUEST→REJECT), 킥(APPROVE→KICK), 호스트 폐강(CLOSE)은 사용자 귀책이 아니므로 유료였다면 항상 전액 환불.

### 스케줄러 주도 취소 (SYSTEM) — 항상 전액 환불

CONFIRM_PLAYING 전이 시 REQUEST 자동 취소, CLOSE_BY_SYSTEM 일괄 취소, 24h 미승인 자동 취소는 시스템 귀책이므로 유료였다면 항상 전액 환불.

### 환불 실행

공통 환불 원칙은 주문/결제 공통 규칙 명세서 §4 참조. Order 상태 전이와 실패 복구(재시도 스케줄러, Sweeper)도 공통 규칙을 따른다.

```
유료이고 환불 금액 > 0인 경우:
  주문/결제 공통 규칙 명세서 §4 에 따라 PG 환불 실행
  성공 → COMPLETED_CANCEL_PAYMENT
  실패 → FAILED_CANCEL_PAYMENT → §5-2 재시도 스케줄러

무료이거나 환불 금액 = 0인 경우:
  PG 호출 없이 Order.orderStatus → COMPLETED_CANCEL_PAYMENT
  OrderHistory 기록

일괄 취소(폐강/CLOSE_BY_SYSTEM) 시 TX 커밋 후 환불 루프 크래시로 발생한
상태 불일치 Order는 주문/결제 공통 규칙 명세서 §5-4 Sweeper 스케줄러가 복구 (WEBB-1326)
```

### 4-2. 스태프 환불 (COMPLETE 전이 시)

```
스태프(grade = STAFF)는 결제 후 참여하지만 소셜링이 완료되면 전액 환불된다.
grade 변경(MEMBER → STAFF) 시점에 즉각 환불하지 않고, COMPLETE 전이 시 일괄 처리한다.
(Invariant S3)

소셜링이 COMPLETE 전이될 때 (socialing-pseudocode.md §8-3):

  대상: SocialingMember WHERE grade = STAFF AND status = APPROVE

  [단일 트랜잭션 — 상태 전이]
  BEGIN TX:
    Socialing.status = CONFIRM_PLAYING → COMPLETE
  COMMIT

  [환불 처리 — 트랜잭션 외부]
  FOR EACH 대상 스태프:
    유료인 경우:
      PG 전액 환불 실행 (cancelReason = SYSTEM_REFUND_BY_STAFF)
      → 성공: Order.orderStatus → COMPLETED_CANCEL_PAYMENT
      → 실패: Order.orderStatus → FAILED_CANCEL_PAYMENT
              주문/결제 공통 규칙 명세서 §5-2 재시도 스케줄러가 복구
      OrderHistory 기록

    무료인 경우:
      PG 호출 없이 Order.orderStatus → COMPLETED_CANCEL_PAYMENT
      OrderHistory 기록

  ※ 주의: 스태프는 환불 후에도 SocialingMember.status = APPROVE 유지.
    따라서 §5-4 Sweeper(멤버 종료 상태 조건)로는 복구되지 않는다.
    COMPLETE 전이와 환불 루프 사이 크래시 복구가 필요하면 별도 스케줄러 설계 필요
```

## 5. 얼리버드 부분 환불

얼리버드 할인 대상 멤버에게 **할인액만큼 부분 환불**하여 혜택을 지급한다.
일반 취소(§4-1)와 달리 **"참여 유지 + 혜택 지급"** 성격이며, 환불 후에도 멤버는 APPROVE를 유지한다 (Invariant S3).

```
[대상]
  SocialingMember.status = APPROVE (얼리버드 혜택 보유)
  AND Order.orderKind = CARD
  AND Order.refundPrice > 0  (결제 시점에 얼리버드 할인액이 세팅된 Order — §3 단계 C 참조)

[금액 산출]
  refundPrice = orderPrice − 할인 적용 후 가격 = 얼리버드 할인액
  remainPrice = orderPrice − refundPrice        = 할인 적용 후 실제 결제 유지 금액
  (Invariant S2 및 order-common.md Invariant 6 유지)

  예: orderPrice = 10,000, 할인 적용 후 가격 = 8,000
      → refundPrice = 2,000, remainPrice = 8,000

[처리]
  PG 부분 환불 실행 (Bootpay cancelReceipt with refundPrice)

  → 성공: Order.orderStatus → PARTIAL_REFUNDED
          Order.refundPrice, cancelledPrice, remainPrice 갱신
          OrderHistory 기록 (Invariant 4)
  → 실패: Order.orderStatus → PARTIAL_REFUND_FAILED
          OrderHistory 기록 (실패 사유 포함, Invariant 4)
          order-common.md §5-3 재시도 스케줄러가 복구

※ 환불 후에도 SocialingMember.status = APPROVE 유지 (Invariant S3).
  Sweeper(order-common.md §5-4) 복구 대상이 아니다.

※ refundPrice, remainPrice는 정산(settlement) 로직에서 직접 참조되므로
  부분 환불 후 값이 Invariant 6을 위반하지 않도록 동일 TX에서 원자적으로 갱신한다.
```

## 6. 미처리 주문 스케줄러 (소셜링 한정 고려사항)

공통 스케줄러 규칙(주문/결제 공통 규칙 명세서 5)에 추가되는 소셜링 특화 주의사항.

### WAITING_PAYMENT → confirm 미호출 처리 시

```
confirm 미호출 건을 스케줄러가 환불 처리할 때:
  - SocialingMember 생성이 완료되지 않은 상태이므로 도메인 후처리 불필요
  - Order만 FAILED_PAYMENT로 전환하면 됨
  - (SocialingMember가 없으므로 availCount 복구 불필요)
```

### FAILED_CANCEL_PAYMENT 재처리 시

```
환불 실패 재처리 스케줄러가 실행될 때:
  - SocialingMember는 이미 CANCEL/KICK/REJECT 상태 (또는 스태프 환불 후 APPROVE 유지)
  - Socialing.availCount는 이미 복구됨 (취소 트랜잭션 시점에 처리)
  - Order 상태만 COMPLETED_CANCEL_PAYMENT로 전환
  - 도메인 후처리 없음
```

## 7. 주문 상태와 멤버 상태 정합성

소셜링 도메인 Invariant 3(참여권 정합성) 및 주문/결제 공통 규칙 명세서 Invariant 1(Order-Member 정합성)과 연계되는 구체 매핑.

| SocialingMember.status | Socialing.status | Order.orderStatus | 비고 |
| --- | --- | --- | --- |
| APPROVE (grade ≠ STAFF) | * | COMPLETED_PAYMENT | 정상 |
| REQUEST | RECRUITING | COMPLETED_PAYMENT | 승인 대기 |
| APPROVE (grade = STAFF) | COMPLETE | COMPLETED_CANCEL_PAYMENT 또는 FAILED_CANCEL_PAYMENT | 스태프 환불 |
| CANCEL (유료) | * | COMPLETED_CANCEL_PAYMENT 또는 FAILED_CANCEL_PAYMENT |  |
| CANCEL (무료) | * | COMPLETED_CANCEL_PAYMENT |  |
| KICK (유료) | * | COMPLETED_CANCEL_PAYMENT 또는 FAILED_CANCEL_PAYMENT |  |
| REJECT (유료) | * | COMPLETED_CANCEL_PAYMENT 또는 FAILED_CANCEL_PAYMENT |  |
| CANCEL (얼리버드 부분환불) | * | PARTIAL_REFUNDED 또는 PARTIAL_REFUND_FAILED |  |

`FAILED_CANCEL_PAYMENT`, `PARTIAL_REFUND_FAILED` 상태는 주문/결제 공통 규칙 명세서 §5-2, §5-3 재시도 스케줄러가 처리한다.
크래시로 인한 상태 불일치 Order는 `주문/결제 공통 규칙 명세서 §5-4` Sweeper가 복구 (WEBB-1326).
