## 1. 챌린지 결제 구조

```
ChallengeMember (1:1) Order (1:1) Payment
```

- `ChallengeMember.orderId IS NOT NULL @unique`
- 유료·무료 구분 없이 Order는 반드시 존재한다

### 결제 방식 분기

```
Challenge.price > 0  →  단건 결제 (Bootpay confirmReceipt)
Challenge.price = 0  →  무료 (PG 호출 없음, Order만 생성)
```

## 2. 핵심 불변식

`주문/결제 공통 규칙 명세서 §2`의 불변식이 모두 적용된다. 챌린지 특화 구체화는 아래와 같다.

### Invariant 1 — 챌린지 멤버-Order 엄격 1:1

```
  ChallengeMember.orderId NOT NULL @unique
  재신청 시 새 Order가 생성되고 ChallengeMember.orderId는 새 orderId로 교체된다.
```

### Invariant 2 — 환불 규칙 — 도메인 간 일관

```
  규칙은 소셜링과 동일하다 (order-socialing.md §4-1 참조):
    - 본인 취소 → 시간 컷오프(당일/30분/4일)
    - 호스트 주도 취소 → 항상 전액 환불 (구간 무관)
    - 시스템 주도 취소 → 항상 전액 환불 (구간 무관)
  challenge-pseudocode.md Invariant 7(본인 취소 가능 구간)에 의해
  본인 취소는 RECRUITING 구간에서만 가능하다.
```

### Invariant 3 — 부분환불 미사용

```
  챌린지는 PARTIAL_REFUNDED / PARTIAL_REFUND_FAILED 상태로 전이되지 않는다.
  환불은 전액 환불 또는 환불 없음 둘 중 하나다.
```

## 3. 결제 확정 플로우

```
[단계 A] 챌린지 가용 여부 재검증 (챌린지 도메인)
  - Challenge.status = RECRUITING
  - availCount > 0
  - startDate > NOW() + 1시간

[단계 B] 결제 처리
  유료 (price > 0):
    Bootpay.confirmReceipt(receiptId)
    → 성공: receipt 데이터 확보
    → 실패: 전체 플로우 중단, 유저에게 에러 반환

  무료 (price = 0):
    이 단계 건너뜀

[단계 C] DB 트랜잭션
  Order.orderStatus = WAITING_PAYMENT → COMPLETED_PAYMENT
  Order.orderPrice = 결제된 금액

  [금액 필드 세팅]
  Order.refundPrice = 0
  Order.remainPrice = orderPrice
  (주문/결제 공통 규칙 명세서 Invariant 6 유지)
  
  OrderHistory 생성
  유료: Payment 생성

  챌린지 도메인 후처리 (같은 트랜잭션):
    ChallengeMember 생성
      - COMMON: status = APPROVE → ChallengeRank 생성, availCount 차감
      - CHOICE: status = REQUEST
```

## 4. 취소/환불 플로우

챌린지 도메인(challenge-pseudocode.md)에서 취소를 트리거하면 결제 도메인에 위임된다.

### 4-1. 환불 금액 계산

환불 금액은 취소 주체와 시점에 따라 결정된다. **규칙은 소셜링과 동일하다** (`order-socialing.md` 4-1 참조).

### 본인 취소 (USER) — 시간 컷오프 적용

```
IF 무료 (price = 0)                              → 환불 없음
IF Challenge.status ≠ RECRUITING                 → 환불 없음 (본인 취소 불가 구간)
IF 모임 진행 당일 (startDate와 같은 날)           → 환불 없음 (최우선)
IF 결제 후 30분 이내                              → 전액 환불
IF 원본 member.status = REQUEST                   → 전액 환불
IF 원본 member.status = APPROVE:
   IF 진행일까지 4일 이상 남음                    → 전액 환불
   그 외 (3일 이하)                              → 환불 없음
```

챌린지는 CONFIRM_PLAYING 이후 본인 취소가 불가하므로 실제로는 RECRUITING 구간에서만 위 공식이 평가된다.

### 호스트 주도 취소 (HOST) — 항상 전액 환불

호스트 거절(REQUEST→REJECT), 강퇴(APPROVE→REJECT), 호스트 폐강(CLOSE)은 유료였다면 항상 전액 환불.

### 스케줄러 주도 취소 (SYSTEM) — 항상 전액 환불

CONFIRM_PLAYING 전이 시 REQUEST 자동 취소, CLOSE_BY_SYSTEM, 24h 미승인 자동 취소는 유료였다면 항상 전액 환불.

### 4-2. 환불 실행

공통 환불 원칙은 주문/결제 공통 규칙 명세서 §4 참조.

```
유료:
  주문/결제 공통 규칙 명세서 §4 에 따라 PG 환불 실행
  성공 → COMPLETED_CANCEL_PAYMENT + OrderHistory (Invariant 4)
  실패 → FAILED_CANCEL_PAYMENT + OrderHistory → §5-2 재시도 스케줄러

무료:
  PG 호출 없이 Order.orderStatus → COMPLETED_CANCEL_PAYMENT
  OrderHistory 기록

일괄 취소(호스트 폐강/CLOSE_BY_SYSTEM) 시 TX 커밋 후 환불 루프 크래시로 발생한
상태 불일치 Order는 주문/결제 공통 규칙 명세서 §5-4 Sweeper 스케줄러가 복구 (WEBB-1326).
```

### 4-3. CLOSE_BY_SYSTEM (인원 미달 자동 폐강)

```
챌린지 모집 마감 스케줄러(금요일 23시)가 CLOSE_BY_SYSTEM으로 전환 시:
  전체 APPROVE/REQUEST 멤버 → CANCEL (단일 TX, Invariant Ch1 유지)
  → 전액 환불 (Challenge 귀책)
  OrderHistory 기록
  SMS_062 발송 (인원 미달 안내)
```

### 4-4. 호스트 폐강 (CLOSE)

```
호스트가 수동 폐강 시:
  전체 REQUEST/APPROVE 멤버 → CANCEL (단일 TX)
  → 환불 없음 (OrderClaim = COMPLETED_CANCEL_HOST)
  OrderHistory 기록
```

## 5. 미처리 주문 스케줄러 (챌린지 한정 고려사항)

공통 스케줄러 규칙(주문/결제 공통 규칙 명세서 섹션 4)에 추가되는 챌린지 특화 주의사항.

### WAITING_PAYMENT → confirm 미완료 처리 시

```
confirm 미완료 건을 스케줄러가 환불 처리할 때:
  - ChallengeMember가 생성되지 않은 상태이므로 도메인 후처리 불필요
  - Order만 FAILED_PAYMENT로 전환
  - ChallengeRank 없음, availCount 복구 불필요
```

### FAILED_CANCEL_PAYMENT 재처리 시

```
환불 실패 재처리 스케줄러가 실행될 때:
  - ChallengeMember는 이미 CANCEL/REJECT 상태(또는 얼리버드 환불 후 APPROVE 유지)
  - availCount는 이미 복구됨 (취소 트랜잭션 시점에 처리)
  - Order 상태만 COMPLETED_CANCEL_PAYMENT로 전환
  - 도메인 후처리 없음
```

## 6. 주문 상태와 멤버 상태 정합성

챌린지 도메인 Invariant 2(참여권 정합성) 및 주문/결제 공통 규칙 명세서 Invariant 1(Order-Member 정합성)과 연계되는 구체 매핑.

| ChallengeMember.status | Challenge.status | Order.orderStatus |
| --- | --- | --- |
| APPROVE | RECRUITING / CONFIRM_PLAYING / COMPLETE | COMPLETED_PAYMENT |
| REQUEST | RECRUITING | COMPLETED_PAYMENT |
| CANCEL (환불 성공) | - | COMPLETED_CANCEL_PAYMENT |
| CANCEL (환불 실패) | - | FAILED_CANCEL_PAYMENT |
| CANCEL (환불 없음) | - | COMPLETED_CANCEL_PAYMENT |
| APPROVE (얼리버드 환불 성공) | CONFIRM_PLAYING 이후 | PARTIAL_REFUNDED |
| APPROVE (얼리버드 환불 실패) | CONFIRM_PLAYING 이후 | PARTIAL_REFUND_FAILED |
|  |  |  |

`FAILED_CANCEL_PAYMENT` 상태는 주문/결제 공통 규칙 명세서 §5-2 재시도 스케줄러가 처리한다.
스케줄러 재시도 후에도 해소되지 않으면 수동 환불 후 `COMPLETED_CANCEL_PAYMENT`로 전환한다.
크래시로 인한 상태 불일치 Order는 `order-common.md §5-4` Sweeper가 복구 (WEBB-1326).

챌린지는 부분환불 기능이 없으므로 `PARTIAL_REFUNDED` / `PARTIAL_REFUND_FAILED` 상태는 발생하지 않는다 (Invariant 3).
