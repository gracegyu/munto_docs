## 1. 결제 스케줄러 통합 (order-common)

### 문제

WAITING_PAYMENT 처리가 3개 스케줄러로 분산돼 있고 상태값 불일치가 있다.
미환불은 발생하지 않으나 Order 상태가 실제 의미와 다르게 기록되는 케이스가 있다.

```
[ExpireAbandonedOrderScheduleJobService]
  조건: bootpayReceiptId IS NULL AND paymentId IS NULL AND updatedAt <= NOW() - 15분
  처리: PG 호출 없음 → EXPIRED

  문제: 웹훅 수신됐으나 confirm 미호출인 경우(카드 승인됨)도 이 조건에 걸림
       → EXPIRED 처리 후 PaymentRequest 스케줄러가 뒤늦게 환불
       → Order는 EXPIRED지만 실제로는 환불된 불일치 상태가 됨

[ConfirmMissedPaymentScheduleJobService]
  조건: bootpayReceiptId IS NULL AND paymentId IS NOT NULL AND updatedAt <= NOW() - 30분
  처리: Bootpay 영수증 조회 → 환불 성공 시 COMPLETED_CANCEL_PAYMENT

  문제: 환불 성공 시 COMPLETED_CANCEL_PAYMENT로 기록 → FAILED_PAYMENT가 맞음
       (confirm이 완료된 적 없으므로 주문 확정 실패)

[PaymentUpdateJobService.refundExpiredPaymentRequests]
  조건: PaymentRequest.createdAt <= NOW() - 30분 AND Payment(receiptId) 없음
  처리: PG 전액 환불 → PaymentRequest 삭제

  문제: Order 상태 미업데이트
       EXPIRED 처리(15분)와 환불(30분) 타이밍 불일치
```

### 해결 방향

웹훅 수신 시점에 `Order.bootpayReceiptId`를 즉시 세팅하여 케이스를 단순화한다.

```
[웹훅 수신 시 선행 처리]
  payload.order_id로 Order 조회
  → WAITING_PAYMENT 상태인 경우: Order.bootpayReceiptId = payload.receipt_id 세팅
  → 그 외 (이미 처리됨): 무시

[WAITING_PAYMENT 통합 스케줄러]

  CASE A: 결제 미발생 순수 이탈
    조건: bootpayReceiptId IS NULL AND updatedAt <= NOW() - 15분
    처리: PG 호출 없음 → EXPIRED, OrderHistory 기록

  CASE B: 결제 완료, confirm 미완료
    조건: bootpayReceiptId IS NOT NULL AND updatedAt <= NOW() - 10분
    처리: Bootpay.getReceipt(bootpayReceiptId)
          → 결제완료(1) 또는 승인대기(2): PG 전액 환불
          → 그 외: 환불 스킵
          Order → FAILED_PAYMENT, PaymentRequest 삭제, OrderHistory 기록

[고아 PaymentRequest 정리] (보조 안전망)
  조건: PaymentRequest.createdAt <= NOW() - 30분 AND Payment(receiptId) 없음
  처리: PG 전액 환불 → PaymentRequest 삭제
```

## 2. 환불 실패 재시도 스케줄러 구현 (order-common)

### 문제

`FAILED_CANCEL_PAYMENT`, `PARTIAL_REFUND_FAILED` 상태 재처리 스케줄러가 미구현.
현재 Slack 알림 수신 후 수동 환불 처리에 의존하고 있다.

### 해결 방향

```
[FAILED_CANCEL_PAYMENT 재시도 스케줄러]
  조건: orderStatus = FAILED_CANCEL_PAYMENT AND updatedAt <= NOW() - 10분
  처리:
    PG 환불 재시도
    → 성공: Order → COMPLETED_CANCEL_PAYMENT, OrderHistory 기록
    → 실패: OrderHistory에 재시도 실패 기록
            최대 재시도 횟수 초과 시 Slack 알림 + 수동 처리

[PARTIAL_REFUND_FAILED 재시도 스케줄러]
  조건: orderStatus = PARTIAL_REFUND_FAILED AND updatedAt <= NOW() - 10분
  처리:
    PG 부분 환불 재시도
    → 성공: Order → PARTIAL_REFUNDED, OrderHistory 기록
    → 실패: OrderHistory에 재시도 실패 기록
            최대 재시도 횟수 초과 시 Slack 알림 + 수동 처리
```

## 4. 클럽 테이블 구조 단순화 (club)

### 문제

```
Club → ClubMember → ClubMembership → Order[]
                                   → billingKey (멤버십당 1개)

- ClubMember와 Order가 직접 연결되지 않음 (ClubMembership 경유)
  소셜링/챌린지: SocialingMember.orderId, ChallengeMember.orderId (직접 연결)
  클럽: ClubMember → ClubMembership → Orders[] (3단계 간접 참조)
- 클럽 가입 시마다 Bootpay 빌링키를 새로 등록해야 함
- 빌링키가 ClubMembership에 종속돼 다른 도메인에서 재사용 불가
- 카드 변경 시 구독 중인 모든 ClubMembership의 billingKey를 개별 교체해야 함
```

### 해결 방향

```
Club → ClubMember → Order  (소셜링/챌린지와 동일한 패턴)
       ClubMember → ClubMembership[]  (갱신 이력 로그로 역할 변경)
User  → UserBillingKey  (유저 레벨 빌링키로 격상)
```

**ClubMember** — 멤버 상태의 단일 소스

```
신규 필드:
  orderId          Int? @unique        최신 활성 Order FK
  membershipType   ONETIME | SUBSCRIPTION
  membershipStatus PAID | SUBSCRIBED | EXPIRED | GRACE_PERIOD
  expireAt         DateTime?
  startAt          DateTime?
```

**ClubMembership** — 갱신 이력 로그 (역할 변경)

```
clubMemberId  FK → ClubMember
orderId       FK → Order
amount        Int
periodStart   DateTime
periodEnd     DateTime
createdAt
```

**UserBillingKey** — 유저 레벨 빌링키 (v3에서 부분 도입, 격상)

```
- 클럽 가입 시 UserBillingKey가 이미 있으면 재사용 (PG 재등록 불필요)
- 카드 변경 시 isDefault 교체 한 번으로 모든 구독에 자동 반영
- 빌링키 파기 시점: 유저의 모든 활성 구독이 종료됐을 때
```

**구독 갱신 스케줄러 변화**

```
ASIS: ClubMembership.billingKey로 청구 → ClubMembership.expireAt 갱신
TOBE: UserBillingKey(isDefault=true)로 청구
      → ClubMember.expireAt += 1개월
      → ClubMembership 이력 한 줄 추가
      → ClubMember.orderId = 새 Order.id
```

## 5. ONETIME/GRACE_PERIOD 취소 시 Order 상태 미변경 (order-club)

```

→ Order.orderStatus가 COMPLETED_PAYMENT로 남음
```

```
  OrderHistory 기록

GRACE_PERIOD 취소:
  환불 없음은 동일하나 Order.orderStatus → COMPLETED_CANCEL_PAYMENT로 변경
  OrderHistory 기록
```

## 6. availCount 초기화 불일치 (challenge)

### 문제

```
유저 생성 챌린지 (challenge.service.ts):
  availCount: payload.maximumMembers   ← OWNER 슬롯 포함

어드민 생성 챌린지 (challenge.admin.service.ts):
  availCount: payload.maximumMembers - 1   ← OWNER 슬롯 제외

동일한 maximumMembers 값으로 챌린지를 생성해도 availCount 시작값이 다름.
getMyChallenges에서 currentMembers = maximumMembers + 1 - availCount로 계산하는데
어드민 챌린지는 이 계산이 맞지 않음.
```

### 해결 방향

```
OWNER가 availCount에 포함되는지 여부를 클럽과 동일하게 명확히 정의:
  - 클럽: OWNER는 availCount에 포함되지 않음 (Invariant 1 기준)
  - 챌린지도 동일하게 통일

유저/어드민 생성 모두:
  availCount = maximumMembers (OWNER 미포함 기준)
  currentMembers 계산: maximumMembers - availCount + COUNT(OWNER)
```

## 9. 챌린지 일괄 취소 TX 원자화 (challenge)

### 문제

호스트 폐강(`closeChallengeByHost`) 및 모집 마감 전환(`CLOSE_BY_SYSTEM`) 시
현재 코드는 Challenge 상태 전이와 멤버 상태 전이를 **각각 별개 TX**으로 처리한다.
부분 실패 시 `Challenge.status = CLOSE`인데 일부 멤버가 `APPROVE` 잔존 →
Invariant 3(상태별 멤버 정합성) 위반 가능.

### 해결 방향

```

재시도 스케줄러(WEBB-1326)가 처리.

상태 불일치 Order가 발생할 수 있는데, WEBB-1326 스코프의 Sweeper 스케줄러가
감지 → FAILED_CANCEL_PAYMENT로 전환 → 재시도 루프 편입.

세부 스펙은 challenge-pseudocode.md §5-4, §7-2 참조.
```

## 11. GRACE_PERIOD 자동 만료/재시도 스케줄러 미구현 (club)

### 문제

```
ClubMembership 구독 갱신 실패 시 GRACE_PERIOD로 전이되지만,
이후 자동 만료/재시도 스케줄러가 구현되어 있지 않다.

club-membership.schedule-job.ts:
  - expireSubscription() 메서드는 존재하나 "TEMP: expired는 현재 미사용" 주석
  - 갱신 스케줄러는 status = SUBSCRIBED만 조회 → GRACE_PERIOD 제외

결과:
  - GRACE_PERIOD 진입 후 자동 재결제 재시도 없음
  - 유예 기간 일수 기반 자동 EXPIRED 전환 없음
  - 유저가 수동 취소(cancelClubMembership) 또는 수동 재결제 API를 호출하지 않으면
    GRACE_PERIOD에 영구 체류
  - "유예 기간"이라는 개념 자체가 운영상 기능하지 않음
```

### 해결 방향 (PM/기획 확인 후 결정)

```
옵션 A: 유예 기간 일수 기반 자동 EXPIRED 전환 스케줄러 구현
  - 조건: status = GRACE_PERIOD AND gracePeriodEnteredAt < NOW() - N일
  - 처리: ClubMembership.status → EXPIRED, ClubMember 강퇴 또는 유지(정책 결정 필요)
  - N값은 운영 정책으로 결정 (일반적으로 3~7일)

옵션 B: GRACE_PERIOD 중 재결제 재시도 스케줄러 추가
  - 일정 주기(예: 매일)로 재결제 시도
  - 최대 재시도 횟수 초과 시 EXPIRED 전환
  - UserBillingKey 격상(improvements.md §4)과 함께 설계 검토

옵션 C: 현재 동작(수동 처리만)을 의도로 확정하고 문서만 명시

먼저 "유예 기간"의 운영상 의도 확인 필요.
```

## 12. 정산 orderStatus 필터링 통일 (settlement)

### 문제

```
정산 로직(settlement.md §4)에서 소셜링/클럽/코스가 orderStatus를 명시적으로
필터링하지 않아, FAILED_CANCEL_PAYMENT(환불 실패) 상태의 Order도 정산 대상에 포함된다.

영향:
  - 환불 실패 건이 정산에 포함되어 호스트에게 선지급
  - 이후 수동 환불이 완료되면 정산과 실제 지급 사이에 불일치
  - 이중결제로 Order가 2건 생기면 둘 다 정산될 수 있음

챌린지(§4-2)만 orderStatus = COMPLETED_PAYMENT로 필터링하고 있어 도메인 간 정합성 없음.
```

### 해결 방향

```
모든 도메인(소셜링 §4-1, 클럽 §4-3, 코스 §4-4)에서
  orderStatus IN (COMPLETED_PAYMENT, PARTIAL_REFUNDED)
필터를 명시적으로 적용한다.

FAILED_CANCEL_PAYMENT / PARTIAL_REFUND_FAILED / COMPLETED_CANCEL_PAYMENT 제외.
환불 실패 건은 order-common.md §5-2/§5-3 재시도 스케줄러로 해소 후
다음 정산 주기에 자연스럽게 대상 상태로 복귀한다.

settlement.md §4는 이미 TOBE 기준으로 이 필터를 반영했다.
실 코드 마이그레이션이 이 항목의 추적 대상이다.

추가 확인:
  챌린지(§4-2)의 현행 필터 `orderStatus = COMPLETED_PAYMENT`는 PARTIAL_REFUNDED를 제외하는데
  얼리버드 부분환불된 Order가 정산에서 누락될 가능성 있음.
  `orderStatus IN (COMPLETED_PAYMENT, PARTIAL_REFUNDED)`로 확장 검토 필요.
  단, 챌린지 계산식은 SUM(orderPrice) 기반이라 PARTIAL_REFUNDED 포함 시
  할인액도 호스트에게 지급되는 문제가 생기므로 소셜링 방식(remainPrice + discountAmount 분리)으로
  계산 로직 변경이 동반되어야 한다.
```
