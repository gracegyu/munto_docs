# [작업] DB 정합성 개선을 위한 도메인별 비즈니스 로직 명세 정리

| 항목 | 내용 |
| --- | --- |
| 배경 | DB 정합성 종합 점검(WEBB-1307) 결과 소셜링/챌린지/클럽/결제 도메인에서 총 70개 취약점 발견. 취약점 수정 전 각 도메인의 올바른 동작을 먼저 명세화하는 것이 필요. |
| 현재 문제점 | • 상태 전이 플로우, availCount 변경 경로, 트랜잭션 범위 등 핵심 비즈니스 로직이 문서화되어 있지 않음.<br>• 기준 명세 없이 버그를 수정하면 의도한 동작인지 검증할 수 없고, 동일 패턴의 취약점이 반복 유입될 수 있음. |
| 개선 방향 | 소셜링 / 챌린지 / 클럽 / 결제 4개 도메인에 대해 아래 산출물 작성:<br>1. 상태 전이 플로우 수도코드 (API / 스케줄러 / 어드민 경로 포함)<br>2. availCount 변경 경로 전체 매핑 및 경로별 올바른 동작 정의<br>3. TDD 기반 버그 수정에 활용할 테스트 케이스 정의 |
| 고려요소 | • v3 / v4 / v5 버전별 로직 차이를 모두 반영해야 함.<br>• 어드민 / 스케줄러 / 사용자 API 경로를 모두 포함해야 함. |
| 기대효과 | • 이후 버그 수정(Phase 1~3) 시 명확한 명세 기준 제공.<br>• 테스트 케이스 기반 TDD로 수정 후 회귀 방지.<br>• 코드 리뷰 시 명세 기반 검토 가능. |
