# [개선] 데이터베이스 정합성 종합 점검 및 개선


| 항목 | 내용 |
| --- | --- |
| 배경 | 2026년 1월 CX팀에서 "승인제 소셜링에서 호스트 승인 없이 정산된 케이스" 발견(WEBB-967) 후, 이중결제·결제 취소 실패·부분 환불 실패에 대한 전수 조사를 진행했다.<br>총 235건 / 유저 미환불 6,225,690원 + 호스트 과다 정산 5,479,448원 = 회사 총 손실 11,705,138원이 확인되었다.<br>환불 누락 보고서 작성 과정에서 문제의 근본 원인이 단순 결제 로직 버그가 아니라 시스템 전반의 DB 정합성 구조적 취약점에 있다는 것이 드러나, 결제/소셜링/챌린지/클럽 4개 핵심 도메인에 대해 정합성 종합 분석을 실시했다. |
| 현재 문제점 | 4개 도메인에서 총 70건의 정합성 취약점 발견 (CRITICAL 13건, HIGH 24건, MEDIUM 23건, LOW 10건).<br>주요 공통 패턴:<br>1. `availCount` 음수 방지 조건 누락 — v3 결제 확정 시 `> 0` 검증 없어 정원 초과 승인 가능<br>2. PG 환불 Fire-and-Forget — 환불 실패 시 에러만 로깅하고 DB는 취소 처리되어 금전↔상태 불일치<br>3. delta 기반 정원 조정 — 어드민/API에서 정원 변경 시 실제 인원 무시한 산술 조정<br>4. 일괄 취소 시 `availCount` 역보정 누락 — 스케줄러/어드민 폐강 경로<br>5. 일괄 처리 비원자성 — 멤버별 순차 await 루프에 전체 TX 없음, 부분 실패 시 3중 불일치<br>6. 도메인 간 enum 혼용 — 클럽 스케줄러에서 `ChallengeMemberStatus` 사용 |
| 개선 방향 | Phase 1(긴급): PG 환불 fire-and-forget 제거, enum 혼선 수정, where 조건 버그 수정, 일괄 취소 `availCount` 역보정, 환불 실패 처리 보강.<br>Phase 2(우선): `availCount` > 0 조건 추가, 재계산 함수 공통화, TX 내부 이동, Order 상태 전이 검증, 스케줄러 분산 락, 정산 멱등성.<br>Phase 3(중기): 환불 공통 메서드 추출, Summary 재집계 스케줄러, 레거시 정리, 모니터링 배치 |
| 고려요소 | • 결제 CRITICAL은 WEBB-1272(PG Rate Limit 관리)로 일부 해소 예정(2026-04-16).<br>• 코스 도메인은 서비스 종료 예정으로 제외. 정산은 결제 보고서에서 커버.<br>• 각 도메인별 상세 분석은 하위 이슈(WEBB-1295, WEBB-1296, WEBB-1297, WEBB-1298) 참조.<br>• 종합 보고서: [Notion](https://www.notion.so/342e2bc7639d80acba14da065497cac1?pvs=21) |
| 기대효과 | 1. 정원 초과 승인·음수 `availCount` 근절<br>2. PG↔DB 금전 정합성 보장 — 환불 누락으로 인한 금전 손실 재발 방지<br>3. 스케줄러/어드민 경로 원자성 확보<br>4. 운영 안정성 향상 — 모니터링·알림 체계로 이상 징후 조기 감지 |
