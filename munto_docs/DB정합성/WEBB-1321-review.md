| CHO-C1 | HIGH | 섹션 3-4 vs 3-3 | 섹션 3-3 CLOSE_BY_SYSTEM에서 "전액 환불(Challenge 귀책)"이라고 되어 있으나, 섹션 3-1 환불 금액 계산에서 `Challenge.status ≠ RECRUITING → 환불 없음`이라고 되어 있습니다. 챌린지 참여/취소 문서 섹션 7-2의 처리 순서가 `Challenge.status = CLOSE_BY_SYSTEM` 변경 → 멤버 취소이므로, 멤버 취소 시점에는 이미 `CLOSE_BY_SYSTEM`이어서 3-1 규칙에 의해 환불 0원이 됩니다. 다음 중 하나의 방향으로 해소해 주시면 좋겠습니다: (A) 섹션 3-1의 조건을 `Challenge.status ∉ {RECRUITING, CLOSE_BY_SYSTEM} → 환불 없음`으로 예외 추가, (B) 참여/취소 문서 섹션 7-2에서 멤버 환불 처리 후 상태 변경하도록 순서 변경, (C) 섹션 3-3이 3-1과 별개의 분기임을 명시 (예: "CLOSE_BY_SYSTEM 취소는 섹션 3-1 규칙을 적용하지 않고 무조건 전액 환불"). |

## 8. 정산 비즈니스 규칙 명세서

## 도메인 간 교차 검증

### Cross-1. availCount 원자적 처리 패턴 불일치

소셜링 문서에서는 `updateMany(WHERE availCount > 0, DECREMENT 1)` + `updated.count ≠ 1 → ROLLBACK` 패턴을 명확히 기술하고 있습니다. 클럽과 챌린지에서는 "availCount 차감"이라고만 되어 있습니다. WEBB-1307에서 지적된 "availCount 음수 방지 조건 누락"이 클럽/챌린지 명세에서도 반영되어야 합니다. 소셜링 문서 섹션 4-4의 패턴을 참고하여 통일해 주시면 좋겠습니다.

- 해당: C-M1, CH-M1

### Cross-2. 호스트 폐강 시 환불 정책 도메인 간 차이

| 도메인 | 호스트 폐강 환불 |
| --- | --- |
| 소셜링 | 환불 있음 (유료 참여 시 환불 처리 요청) |
| 챌린지 | 환불 없음 (OrderClaim = COMPLETED_CANCEL_HOST) |
| 클럽 | 환불 있음 (CLOSE_BY_HOST → 전액 환불) |

챌린지만 호스트 귀책 폐강 시 환불이 없습니다. 의도된 비즈니스 규칙이라면 그 근거를 명세에 기술해 주시고, 아니라면 소셜링/클럽과 통일이 필요합니다.

- 해당: CH-C1

### Cross-3. 챌린지 CLOSE_BY_SYSTEM 환불과 상태 전이 순서

챌린지 참여/취소 문서 섹션 7-2에서 `Challenge.status = CLOSE_BY_SYSTEM` 변경 후 멤버 취소를 진행합니다. 챌린지 결제/환불 문서 섹션 3-1에서 `Challenge.status ≠ RECRUITING → 환불 없음`이라고 정의되어 있습니다. 따라서 상태 변경이 먼저 이루어지면 환불 금액 계산 시 이미 `CLOSE_BY_SYSTEM`이므로 환불이 0원이 됩니다. 이는 섹션 3-3의 "전액 환불"과 충돌합니다. CHO-C1에 기술한 (A)/(B)/(C) 중 하나의 방향으로 해소해 주시면 좋겠습니다.

- 해당: CHO-C1

### Cross-4. 클럽 ONETIME 취소 시 Order 상태 미변경

클럽 결제/환불 문서에서 ONETIME 취소 시 `Order.orderStatus`가 `COMPLETED_PAYMENT`로 남습니다. 정산 문서에서 클럽 정산은 `remainPrice > 0`만 체크하므로, ONETIME 취소 후에도 `remainPrice`가 남아있으면 정산에 포함됩니다. ONETIME 취소 시 `remainPrice = 0`으로 업데이트하는지 여부를 클럽 결제/환불 문서 섹션 3-1에 명시해 주시면 좋겠습니다.

- 해당: CO-C1

### Cross-5. 소셜링 STAFF 환불과 정산 타이밍

소셜링 결제/환불 문서 섹션 3-3에서 STAFF 환불은 COMPLETE 전이 시 처리됩니다. 정산 문서에서 소셜링 정산 대상은 `status = COMPLETE`이고 `startDate`가 직전 1주인 소셜링입니다. COMPLETE 전이(스케줄러)와 정산 생성(화요일 04:00) 사이에 STAFF 환불이 실패(`FAILED_CANCEL_PAYMENT`)하면 해당 Order의 `remainPrice`가 그대로 남아 정산에 포함될 수 있습니다. ST-C1의 해소 방향(orderStatus 필터링 통일 또는 의도 선언)이 적용되면 함께 해소될 것으로 보입니다.

## 총평

8개 문서 전반적으로 도메인 구조, 상태 전이, 불변식 정의가 체계적으로 작성되어 있습니다. 이전 리뷰(5개 문서)에서 지적했던 결제/환불 상세 규칙 누락이 3개 결제/환불 문서 추가로 대부분 해소되었습니다.

주요 개선 필요 사항 (우선순위순):

1. **[HIGH] 챌린지 CLOSE_BY_SYSTEM 환불 순서 모순** (CHO-C1, Cross-3): 상태 변경과 환불 처리의 순서가 모순되어 있어, 실제 코드에서 전액 환불이 의도대로 이루어지는지 검증이 필요합니다.

2. **[HIGH] availCount 원자적 처리 통일** (C-M1, CH-M1, Cross-1): 클럽/챌린지에도 소셜링 수준의 `WHERE availCount > 0` + ROLLBACK 패턴 명시가 필요합니다.

3. **[HIGH] 일괄 처리 부분 실패 대응** (S-C1): 폐강/시스템 취소 시 멤버별 개별 TX의 부분 실패가 Invariant 위반을 유발할 수 있습니다. 전체 TX 또는 재시도 전략 명시가 필요합니다.

4. **[MEDIUM] 정산 orderStatus 필터링 불일치** (ST-C1, ST-M2, Cross-4): 도메인별로 정산 대상 필터 기준이 다르며, 의도인지 버그인지 선언이 필요합니다.

5. **[MEDIUM] 클럽 ONETIME Order 상태 미변경** (CO-C1, Cross-4): 주문/결제 공통 문서에 "환불 없는 취소" 경로 추가가 필요합니다.

6. **[LOW] 문서 형식 통일**: 소셜링 섹션 번호 중복(S-M3), 특정 클럽 하드코딩 배경 누락(CO-M1) 등입니다.
