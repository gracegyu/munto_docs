## Executive Summary

**4개 핵심 도메인(결제, 소셜링, 챌린지, 클럽)에서 총 70건의 DB 정합성 취약점이 발견되었으며, 이 중 13건은 금전 손실에 직결되는 CRITICAL 등급입니다.**

2026년 4월 소셜링 이중결제 CS(WEBB-967)를 기점으로 전수 조사를 실시한 결과, PG 환불 실패가 조용히 무시되는 구조, 정원(availCount) 음수 허용, 상태-금액 불일치 등 **실제 매출과 고객 신뢰에 영향을 미치는 구조적 결함**이 확인되었습니다.

현재까지 CS 인입이 발생한 건(WEBB-1292)이 있으며, 동시성 트래픽 증가 시 피해 규모가 확대될 수 있습니다.

## 2. 가장 시급한 리스크 TOP 3

| # | 리스크 | 영향 | 도메인 |
| --- | --- | --- | --- |
| 1 | **PG 환불 실패가 조용히 무시됨** (fire-and-forget) | 환불 처리했다고 기록되지만 실제 고객에게 돈이 안 돌아감 | 결제 |
| 2 | **정원(availCount) 음수 허용** | 정원 초과 승인 → 초과 결제 발생 가능 | 소셜링/챌린지/클럽 |
| 3 | **일괄 취소 시 정원 미복구** | 폐강 후 정원이 영구 소진 상태로 남음 | 챌린지/소셜링 |

---

## 3. 공통 구조적 문제 (4개 도메인 반복 패턴)

1. **PG 환불 ↔ DB 상태 변경이 분리되어 있음** → 돈과 상태가 불일치할 수 있음
2. **일괄 처리에 트랜잭션 없음** → 중간 실패 시 일부만 처리되어 3중 불일치
3. **정원 관리가 delta 방식** → 실제 인원과 무관하게 숫자가 왜곡됨
4. **Summary 카운트 비동기 누적** → 장애 시 집계 데이터 영구 왜곡

---

## 4. 개선 로드맵

### Phase 1: 긴급 (CRITICAL) — 금전/데이터 손실 직결

- PG 환불 fire-and-forget 패턴 제거
- where 조건 버그 및 enum 혼용 수정
- 일괄 취소 시 availCount 역보정 추가
- 환불 실패 처리 보강 및 레코드 보존
- 폐강 → 환불 순서 개선

### Phase 2: 우선 (HIGH) — 동시성/정합성 리스크

- availCount > 0 조건 추가 (전 도메인)
- availCount 재계산 함수 공통화
- Order 상태 전이 검증, 스케줄러 분산 락, 정산 멱등성

### Phase 3: 중기 (MEDIUM/LOW) — 운영 안정화

- 환불 공통 메서드 추출, Summary 재집계 스케줄러
- 레거시 코드 정리, 모니터링 배치 + 슬랙 알림
