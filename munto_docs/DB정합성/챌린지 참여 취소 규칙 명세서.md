## 1. 핵심 개념

### 1-1. 관련 테이블

| 테이블 | 역할 |
| --- | --- |
| `Challenge` | 챌린지 |
| `ChallengeMember` | 참여 멤버. `orderId NOT NULL @unique` |
| `ChallengeRank` | 멤버별 인증 횟수 리더보드 |
| `ChallengeHistory` | 개별 인증 기록 (피드 연결) |
| `Order` | 참여권의 존재 증명. 무료·유료 구분 없이 반드시 존재 |

### 1-2. Challenge 상태

```
PLANNING          관리자 생성 초기 상태
RECRUITING        모집 중 (참여 신청 가능)
CONFIRM_PLAYING   모집 마감, 최소 인원(2명) 충족 → 진행 확정
PLAYING           진행 중 (현재 미사용, 확장 예정)
COMPLETE          종료 완료
CLOSE             호스트 수동 폐강
CLOSE_BY_SYSTEM   최소 인원 미달로 자동 폐강
DELETE            소프트 삭제
EVENT             특수 이벤트 챌린지
```

### 1-3. ChallengeMember 상태

```
REQUEST  가입 신청 (승인제 챌린지)
APPROVE  승인된 활성 참여자
REJECT   호스트에 의한 거절/강퇴
CANCEL   본인 취소 또는 시스템 취소
DELETE   소프트 삭제
```

### 1-4. 모집 방식

```
COMMON (선착순)  가입 즉시 APPROVE. availCount 즉시 차감
CHOICE (인터뷰)  가입 시 REQUEST. 호스트 승인 후 APPROVE. availCount는 승인 시 차감
                 모집 방식을 CHOICE → COMMON으로 변경 시 기존 REQUEST 멤버 전원 자동 취소
```

## 2. 핵심 불변식

```
Invariant 1 (정원 정합성)
  availCount = maximumMembers - COUNT(ChallengeMember WHERE status = APPROVE)
  → 항상 0 ≤ availCount ≤ maximumMembers

Invariant 2 (최소 인원)
  CONFIRM_PLAYING 전이 조건: APPROVE 멤버 수 ≥ 2
  미달 시 CLOSE_BY_SYSTEM으로 전이

Invariant 3 (참여권 정합성)
  ChallengeMember.orderId IS NOT NULL @unique
  모든 멤버(무료·유료)는 반드시 하나의 Order를 가진다

Invariant 4 (랭킹-멤버 정합성)
  ChallengeRank는 APPROVE 상태 멤버만 존재한다
  APPROVE → REJECT/CANCEL/DELETE 시 ChallengeRank 삭제

Invariant 5 (환불 가능 구간)
  환불은 Challenge.status = RECRUITING 구간에서만 발생한다
  CONFIRM_PLAYING 이후 취소는 환불 없음
```

## 3. 상태 전이 규칙

```
Challenge

PLANNING → RECRUITING (관리자 전환)
RECRUITING
  ├→ CONFIRM_PLAYING  (매주 금요일 23시 스케줄러: APPROVE ≥ 2)
  ├→ CLOSE_BY_SYSTEM  (매주 금요일 23시 스케줄러: APPROVE < 2)
  └→ CLOSE            (호스트 수동 폐강)
CONFIRM_PLAYING
  ├→ COMPLETE  (매일 자정 스케줄러: endDate 경과)
  └→ CLOSE     (호스트 수동 폐강)

ChallengeMember

REQUEST
  ├→ APPROVE  (CHOICE: 호스트 승인)
  ├→ REJECT   (CHOICE: 호스트 거절)
  └→ CANCEL   (본인 취소, 24시간 미승인 스케줄러, 챌린지 폐강)

APPROVE
  ├→ REJECT   (호스트 강퇴)
  └→ CANCEL   (본인 취소, 챌린지 폐강/CLOSE_BY_SYSTEM)
```

## 4. 참여 신청 플로우

### 4-1. 가입 가능 여부 검증

```
조건 (모두 충족):
  Challenge.status = RECRUITING
  availCount > 0
  startDate > NOW() + 1시간  (시작 1시간 전부터 신청 불가)
  신청 유저가 이미 APPROVE 또는 REQUEST 상태가 아님
```

### 4-2. COMMON (선착순) 가입

```
[단계 A] 가입 가능 여부 검증

[단계 B] 결제 처리 (order-challenge.md 참조)

[단계 C] DB 트랜잭션
  ChallengeMember 생성 (status = APPROVE)
  ChallengeRank 생성
  availCount 차감
  Order.orderStatus = COMPLETED_PAYMENT
  OrderHistory 생성

[단계 D] 후처리 (트랜잭션 외부)
  Sendbird 채널 참여
  Push 발송
```

### 4-3. CHOICE (인터뷰) 가입

```
[단계 A] 가입 가능 여부 검증

[단계 B] 결제 처리 (order-challenge.md 참조)

[단계 C] DB 트랜잭션
  ChallengeMember 생성 (status = REQUEST)
  availCount 변동 없음
  Order.orderStatus = COMPLETED_PAYMENT
  OrderHistory 생성

[단계 D] 후처리
  호스트에게 신청 알림 Push 발송
```

## 5. 호스트 작업

### 5-1. 멤버 승인 (REQUEST → APPROVE)

```
권한: OWNER만 가능

처리:
  availCount 차감
  ChallengeMember.status = APPROVE, approvedAt = now()
  ChallengeRank 생성
  Sendbird 채널 참여
  Push 발송 (H12: 승인 알림)
```

### 5-2. 멤버 강퇴/거절

```
권한: OWNER만 가능

APPROVE → REJECT (강퇴):
  availCount 복구
  ChallengeMember.status = REJECT
  ChallengeRank 삭제
  ChallengeHistory 삭제
  결제 취소/환불 처리 (order-challenge.md 참조)
  Sendbird 채널 퇴장 (reason = KICK)
  Push 발송

REQUEST → REJECT (거절):
  availCount 변동 없음
  ChallengeMember.status = REJECT
  결제 취소/환불 처리 (order-challenge.md 참조)
  Push 발송
```

### 5-3. 챌린지 폐강 (호스트)

```
권한: OWNER만 가능

처리:
  Challenge.status = CLOSE, deletedAt = now()
  모든 REQUEST/APPROVE 멤버 → CANCEL
    → 환불 없음 (호스트 귀책: OrderClaim = COMPLETED_CANCEL_HOST)
    → ChallengeHistory 삭제, ChallengeRank 삭제
  이메일/Push 발송 (전 멤버 대상)
```

## 6. 본인 취소

```
취소 가능 상태: REQUEST 또는 APPROVE
취소 가능 챌린지 상태: RECRUITING (CONFIRM_PLAYING 이후 본인 취소 불가)

APPROVE 취소 추가 처리:
  ChallengeRank 삭제
  ChallengeHistory 삭제
  availCount 복구
  Sendbird 채널 퇴장

환불 규칙 → order-challenge.md 참조
```

## 7. 스케줄러 처리 규칙

### 7-1. REQUEST 24시간 미승인 자동 취소

```
[스케줄러: 10분마다]

대상:
  ChallengeMember.status = REQUEST
  AND updatedAt < NOW() - 24시간
  AND Challenge.status = RECRUITING  (CONFIRM_PLAYING 이후는 처리 대상 아님)

처리:
  ChallengeMember.status = CANCEL
  결제 취소/환불 처리 (order-challenge.md 참조)
  Push 발송 (H13: 미승인 자동 취소 안내)
```

### 7-2. 모집 마감 전환 (매주 금요일 23시)

```
대상:
  Challenge.status = RECRUITING
  AND startDate ≤ NOW() + 2시간

처리:
  APPROVE 멤버 수 집계

  ≥ 2명: Challenge.status = CONFIRM_PLAYING
         REQUEST 멤버 전원 자동 취소 (메모: "호스트미승인 - 자동취소")
         → 환불 처리 (order-challenge.md 참조)

  < 2명: Challenge.status = CLOSE_BY_SYSTEM
         전체 APPROVE/REQUEST 멤버 취소
         → 전액 환불 (order-challenge.md 참조)
         SMS_062 발송 (인원 미달 안내)
```

### 7-3. 챌린지 완료 전환 (매일 자정)

```
대상:
  Challenge.status = CONFIRM_PLAYING
  AND endDate < NOW()

처리:
  Challenge.status = COMPLETE
  호스트의 userSummary.challengeCount + 1
  다음 주 일요일: Sendbird 봇 메시지 (챌린지 후기 링크)
```

## 8. ChallengeRank 및 인증 규칙

```
ChallengeRank
  생성: REQUEST → APPROVE 시점
  삭제: APPROVE → REJECT/CANCEL/DELETE 시점

  count 증가: ChallengeHistory 생성 시 (피드 인증)
  랭킹: count DESC 기준, 동점 시 동일 순위 (1, 1, 3, ...)

ChallengeHistory
  생성: 멤버가 챌린지 피드 태그로 인증 제출 시
  삭제: 멤버가 강퇴/취소될 때 (APPROVE → REJECT/CANCEL)
  제약: (challengeId, userId, confirmedAt) UNIQUE
```

## 9. availCount 요약

| 이벤트 | availCount 변화 |
| --- | --- |
| COMMON 가입 | -1 |
| CHOICE 가입 (REQUEST) | 변동 없음 |
| REQUEST → APPROVE | -1 |
| APPROVE → CANCEL | +1 |
| APPROVE → REJECT (강퇴) | +1 |
| REQUEST → CANCEL/REJECT | 변동 없음 |
| maximumMembers 변경 | `availCount = newMax - COUNT(APPROVE)` |
