# WEBB-1328 정산 대상 Order 필터 통일 OnePager — 리뷰

- **대상 문서**: 정산 대상 Order 필터 통일 OnePager (v1.0.0, 26.04.22, 김범진)
- **리뷰어**: 전규현
- **리뷰 일자**: 2026-04-17

---

## 총평

전체적으로 매우 잘 작성된 OnePager입니다. WEBB-1321 규칙 문서화(Invariant SE6)와의 연결이 명확하고, 도메인별 갭 분석·테스트 매트릭스·배포 전 검증 프로세스가 체계적으로 설계되어 있습니다. 서비스 분리를 "순수 이동"으로 먼저 수행하고 동작 변경을 분리한 점, Rule of Three 기준으로 공통 인터페이스 추상화를 유보한 점, Invariant SE7 신설 제안 등 기술적 판단이 적절합니다.

수정이 필요한 항목은 1건(MEDIUM)이며, 나머지는 보충 사항입니다.

## 리뷰 항목

| ID | 심각도 | 위치 | 내용 |
| --- | --- | --- | --- |
| R-1 | MEDIUM | §6 테스트 매트릭스, 도메인별 추가 테스트 케이스 (챌린지) | §3.1에서 "PARTIAL_REFUNDED는 챌린지 스코프 아님"으로 명시했고, §4.1에서도 챌린지는 `{COMPLETED_PAYMENT, COMPLETED_CANCEL_PAYMENT}`만 적용한다고 설계했습니다. 그런데 §6의 도메인별 추가 테스트 케이스에서 "챌린지: `PARTIAL_REFUNDED` Order가 `count=1`로 잡히는지"를 검증하라고 되어 있어 설계 의도와 모순됩니다. 이 테스트 케이스는 **소셜링**에 해당하는 내용으로 보이며, 챌린지에 대해서는 "`PARTIAL_REFUNDED` Order가 필터에서 **제외**되는지"를 확인하는 음성 테스트(negative test)로 수정해 주시면 좋겠습니다. |
| R-2 | LOW | §3.1 챌린지 갭 / 전체 설계 전제 | 본 OnePager는 챌린지에 부분환불이 없다는 전제(챌린지 결제/환불 Invariant 3)를 기반으로 설계되었습니다. WEBB-1321 v2 리뷰에서 지적한 바와 같이, 현재 챌린지 결제/환불 규칙 명세서 §6 정합성 표에 얼리버드 `PARTIAL_REFUNDED` 행이 아직 남아 있고 §5에도 얼리버드 언급이 있어 Invariant 3과 모순되는 상태입니다. 본 OnePager의 설계 자체에는 문제가 없으나, **규칙 명세서의 해당 모순이 해소된 후에 구현을 진행**하시면 혼동이 없을 것으로 보입니다. |
| R-3 | LOW | §3.1, §5.1 챌린지 | 챌린지 TOBE 필터에 `COMPLETED_CANCEL_PAYMENT`를 추가하셨는데, 이 상태에서 `remainPrice > 0`인 실제 시나리오(예: 본인 취소 중 당일/3일 이내 → 환불 없음)가 §3.1이나 §5.1에 1~2줄 정도 언급되면 리뷰어와 구현자가 의도를 더 빠르게 파악할 수 있을 것 같습니다. §4.1에 소셜링/클럽 예시는 있으나 챌린지 예시가 없어 보충해 주시면 좋겠습니다. |

## 잘 된 점

- **서비스 분리(§2)를 Day 1에 "순수 이동"으로 먼저 수행**: diff 경계를 분리해서 리뷰와 롤백이 용이합니다.
- **Rule of Three(§2.3)**: 공통 인터페이스 추상화를 3개 구현체 확보 후로 유보한 판단이 적절합니다.
- **before/after diff 검증(§7)**: 실 데이터 기반 영향 분석 후 배포 의사결정하는 프로세스가 잘 설계되었습니다.
- **범위 외 항목의 명확한 정의**: WEBB-1325, WEBB-1326과의 경계가 분명하고 후속 과제 연결이 체계적입니다.
- **Invariant SE7 신설 제안**: Order-Payment 구조적 정합성을 정산 쿼리에서 재검증하는 안전망은 비용 대비 효과가 큰 방어 수단입니다.
- **`improvements.md` §5 미선행 상태 대응**: 필터가 `COMPLETED_CANCEL_PAYMENT`를 이미 포함하여 §5 구현 후 자연 반영되도록 설계한 것이 좋습니다.
