## 1. 챌린지 결제 구조

```
ChallengeMember (1:1) Order (1:1) Payment
```

- `ChallengeMember.orderId IS NOT NULL @unique`
- 유료·무료 구분 없이 Order는 반드시 존재한다

### 결제 방식 분기

```
Challenge.price > 0  →  단건 결제 (Bootpay confirmReceipt)
Challenge.price = 0  →  무료 (PG 호출 없음, Order만 생성)
```

## 2. 결제 확정 플로우

```
[단계 A] 챌린지 가용 여부 재검증 (챌린지 도메인)
  - Challenge.status = RECRUITING
  - availCount > 0
  - startDate > NOW() + 1시간

[단계 B] 결제 처리
  유료 (price > 0):
    Bootpay.confirmReceipt(receiptId)
    → 성공: receipt 데이터 확보
    → 실패: 전체 플로우 중단, 유저에게 에러 반환

  무료 (price = 0):
    이 단계 건너뜀

[단계 C] DB 트랜잭션
  Order.orderStatus = WAITING_PAYMENT → COMPLETED_PAYMENT
  OrderHistory 생성
  유료: Payment 생성

  챌린지 도메인 후처리 (같은 트랜잭션):
    ChallengeMember 생성
      - COMMON: status = APPROVE → ChallengeRank 생성, availCount 차감
      - CHOICE: status = REQUEST
```

## 3. 취소/환불 플로우

챌린지 도메인(challenge-pseudocode.md)에서 취소를 트리거하면 결제 도메인에 위임된다.

### 3-1. 환불 금액 계산

```
무료 (OrderKind.FREE 또는 price = 0):
  → 환불 없음

Challenge.status ≠ RECRUITING:
  → 환불 없음 (CONFIRM_PLAYING 이후 취소)

Challenge.status = RECRUITING인 경우:
  주문 생성 후 30분 이내:
    → 전액 환불 (얼리버드 취소 보호)
  startDate까지 4일 이상 남은 경우:
    → 전액 환불 (여유 취소 허용)
  그 외 (30분 초과 + 4일 미만):
    → 환불 없음
```

### 3-2. 환불 실행

```
유료:
  Bootpay.cancelReceipt(receiptId, refundPrice)
  → 성공: Order.orderStatus → COMPLETED_CANCEL_PAYMENT
  → 실패: Order.orderStatus → FAILED_CANCEL_PAYMENT
          Slack 알림 (수동 처리 채널)
  OrderHistory 기록 (성공/실패 모두)

무료:
  PG 호출 없이 Order.orderStatus → COMPLETED_CANCEL_PAYMENT
  OrderHistory 기록
```

### 3-3. CLOSE_BY_SYSTEM (인원 미달 자동 폐강)

```
챌린지 모집 마감 스케줄러(금요일 23시)가 CLOSE_BY_SYSTEM으로 전환 시:
  전체 APPROVE/REQUEST 멤버 → CANCEL
  → 전액 환불 (Challenge 귀책)
  OrderHistory 기록
  SMS_062 발송 (인원 미달 안내)
```

### 3-4. 호스트 폐강 (CLOSE)

```
호스트가 수동 폐강 시:
  전체 REQUEST/APPROVE 멤버 → CANCEL
  → 환불 없음 (OrderClaim = COMPLETED_CANCEL_HOST)
  OrderHistory 기록
```

## 4. 얼리버드 부분 환불

챌린지 모집 마감(RECRUITING → CONFIRM_PLAYING) 시점에 얼리버드 할인이 적용된 멤버에게 할인 금액을 환불한다.

```
대상:
  ChallengeMember.status = APPROVE
  AND Order.orderKind = CARD
  AND 얼리버드 할인 적용 (Order.refundPrice > 0)

처리:
  Bootpay.cancelReceipt(receiptId, refundPrice)   ← 할인액만 부분 환불
  cancel_id 형식: cancel_earlybird_{orderId}       ← 중복 요청 방지
  → 성공: Order.orderStatus → PARTIAL_REFUNDED
          Order.refundPrice, cancelledPrice, remainPrice 갱신
          OrderHistory 기록
  → 실패: Order.orderStatus → PARTIAL_REFUND_FAILED
          Slack 알림
          OrderHistory 기록

재시도:
  cancel_id 형식: cancel_earlybird_retry_{failedHistoryId}
  스케줄러 재시도 또는 관리자 수동 재시도 (order-common.md 섹션 4-3 참조)
```

## 5. 미처리 주문 스케줄러 (챌린지 한정 고려사항)

공통 스케줄러 규칙(order-common.md 섹션 4)에 추가되는 챌린지 특화 주의사항.

### WAITING_PAYMENT → confirm 미완료 처리 시

```
confirm 미완료 건을 스케줄러가 환불 처리할 때:
  - ChallengeMember가 생성되지 않은 상태이므로 도메인 후처리 불필요
  - Order만 FAILED_PAYMENT로 전환
  - ChallengeRank 없음, availCount 복구 불필요
```

### FAILED_CANCEL_PAYMENT 재처리 시

```
환불 실패 재처리 스케줄러가 실행될 때:
  - ChallengeMember는 이미 CANCEL/REJECT 상태
  - availCount는 이미 복구됨 (취소 트랜잭션 시점에 처리)
  - Order 상태만 COMPLETED_CANCEL_PAYMENT로 전환
  - 도메인 후처리 없음
```

## 6. 주문 상태와 멤버 상태 정합성

| ChallengeMember.status | Challenge.status | Order.orderStatus |
| --- | --- | --- |
| APPROVE | RECRUITING / CONFIRM_PLAYING / COMPLETE | COMPLETED_PAYMENT |
| REQUEST | RECRUITING | COMPLETED_PAYMENT |
| CANCEL (환불 성공) | - | COMPLETED_CANCEL_PAYMENT |
| CANCEL (환불 실패) | - | FAILED_CANCEL_PAYMENT |
| CANCEL (환불 없음) | - | COMPLETED_CANCEL_PAYMENT |
| APPROVE (얼리버드 환불 성공) | CONFIRM_PLAYING 이후 | PARTIAL_REFUNDED |
| APPROVE (얼리버드 환불 실패) | CONFIRM_PLAYING 이후 | PARTIAL_REFUND_FAILED |

`FAILED_CANCEL_PAYMENT`, `PARTIAL_REFUND_FAILED` 상태는 수동 처리 대기 상태다.
스케줄러 재시도 후에도 해소되지 않으면 수동 환불 후 각각 `COMPLETED_CANCEL_PAYMENT`, `PARTIAL_REFUNDED`로 전환한다.
