## 1. 소셜링 결제 구조

```
SocialingMember (1:1) Order (1:1) Payment
```

- 소셜링 참여권은 Order로 표현된다
- `SocialingMember.orderId`는 `NOT NULL @unique`
- 유료·무료 구분 없이 Order는 반드시 존재한다

### 결제 방식 분기

```
Socialing.price > 0  →  단건 결제 (Bootpay confirmReceipt)
Socialing.price = 0  →  무료 (PG 호출 없음, Order만 생성)
```

## 2. 결제 확정 플로우

소셜링 도메인의 confirm 흐름에서 호출되는 결제 처리 순서.

```
[단계 A] 소셜링 가용 여부 재검증 (소셜링 도메인)
  - Socialing.status = RECRUITING
  - Socialing.availCount > 0

[단계 B] 결제 처리
  유료 (price > 0):
    Bootpay.confirmReceipt(receiptId)
    → 성공: receipt 데이터 확보
    → 실패: 전체 플로우 중단, 유저에게 에러 반환

  무료 (price = 0):
    이 단계 건너뜀

[단계 C] DB 트랜잭션
  Order.orderStatus = WAITING_PAYMENT → COMPLETED_PAYMENT
  Order.orderPrice, remainPrice 갱신
  OrderHistory 생성

  유료인 경우 추가:
    Payment 생성 또는 기존 Payment 재사용 (receiptId 기준)

  소셜링 도메인 후처리 (같은 트랜잭션):
    SocialingMember 생성/UPSERT
    Socialing.availCount 차감 (APPROVE인 경우)
```

## 3. 취소/환불 플로우

소셜링 멤버 취소(본인·호스트 킥·폐강)는 소셜링 도메인에서 트리거된다.

Order 상태 변경은 아래 규칙을 따른다.

### 3-1. 일반 취소 (본인 취소, 킥, 거절, 폐강)

```
[소셜링 도메인 → 결제 도메인 위임]

유료이고 APPROVE 또는 REQUEST 상태였던 경우:
  PG 환불 실행 (Bootpay cancelReceipt)
  → 성공: Order.orderStatus → COMPLETED_CANCEL_PAYMENT
  → 실패: Order.orderStatus → FAILED_CANCEL_PAYMENT
          Slack 알림 (수동 처리 채널)
  OrderHistory 기록 (성공/실패 모두)

무료인 경우:
  PG 호출 없이 Order.orderStatus → COMPLETED_CANCEL_PAYMENT
  OrderHistory 기록
```

### 3-2. 부분 환불 (얼리버드 소셜링)

```
얼리버드 멤버가 취소되는 경우:
  할인 적용 금액만큼 부분 환불 (전액 환불 아님)

  → 성공: Order.orderStatus → PARTIAL_REFUNDED
          Order.refundPrice, cancelledPrice, remainPrice 갱신
  → 실패: Order.orderStatus → PARTIAL_REFUND_FAILED
          Slack 알림
  OrderHistory 기록
```

### 3-3. 스태프 환불 (COMPLETE 전이 시)

```
스태프(grade = STAFF)는 결제 후 참여하지만 소셜링이 완료되면 전액 환불된다.
grade 변경(MEMBER → STAFF) 시점에 즉각 환불하지 않고, COMPLETE 전이 시 일괄 처리한다.

소셜링이 COMPLETE 전이될 때:
  대상: SocialingMember WHERE grade = STAFF AND status = APPROVE

  유료인 경우:
    PG 전액 환불 실행 (cancelReason = SYSTEM_REFUND_BY_STAFF)
    → 성공: Order.orderStatus → COMPLETED_CANCEL_PAYMENT
    → 실패: Order.orderStatus → FAILED_CANCEL_PAYMENT + Slack 알림
    OrderHistory 기록

  무료인 경우:
    PG 호출 없이 Order.orderStatus → COMPLETED_CANCEL_PAYMENT
    OrderHistory 기록
```

## 4. 미처리 주문 스케줄러 (소셜링 한정 고려사항)

공통 스케줄러 규칙(order-common.md 섹션 4)에 추가되는 소셜링 특화 주의사항.

### WAITING_PAYMENT → confirm 미호출 처리 시

```
confirm 미호출 건을 스케줄러가 환불 처리할 때:
  - SocialingMember 생성이 완료되지 않은 상태이므로 도메인 후처리 불필요
  - Order만 COMPLETED_CANCEL_PAYMENT로 전환하면 됨
  - (SocialingMember가 없으므로 availCount 복구 불필요)
```

### FAILED_CANCEL_PAYMENT 재처리 시

```
환불 실패 재처리 스케줄러가 실행될 때:
  - SocialingMember는 이미 CANCEL/KICK/REJECT 상태
  - Socialing.availCount는 이미 복구됨 (취소 트랜잭션 시점에 처리)
  - Order 상태만 COMPLETED_CANCEL_PAYMENT로 전환
  - 도메인 후처리 없음
```

## 5. 주문 상태와 멤버 상태 정합성

소셜링 도메인 Invariant 3과 연계되는 구체적인 매핑.

| SocialingMember.status | Order.orderStatus |
| --- | --- |
| APPROVE | COMPLETED_PAYMENT |
| REQUEST | COMPLETED_PAYMENT |
| CANCEL (유료) | COMPLETED_CANCEL_PAYMENT 또는 FAILED_CANCEL_PAYMENT |
| CANCEL (무료) | COMPLETED_CANCEL_PAYMENT |
| KICK (유료) | COMPLETED_CANCEL_PAYMENT 또는 FAILED_CANCEL_PAYMENT |
| REJECT (유료) | COMPLETED_CANCEL_PAYMENT 또는 FAILED_CANCEL_PAYMENT |
| CANCEL (얼리버드 부분환불) | PARTIAL_REFUNDED 또는 PARTIAL_REFUND_FAILED |

`FAILED_CANCEL_PAYMENT`, `PARTIAL_REFUND_FAILED` 상태는 수동 처리 대기 상태다.

스케줄러 재시도 후에도 해소되지 않으면 수동 환불 후 `COMPLETED_CANCEL_PAYMENT`로 전환한다.
