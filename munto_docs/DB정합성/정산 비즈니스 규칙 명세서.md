## 1. 핵심 개념

### 1-1. 관련 테이블

| 테이블 | 역할 |
| --- | --- |
| `Settlement` | 정산 레코드. 호스트에게 지급할 금액과 산출 근거를 기록한다 |
| `Order` | 정산의 원천 데이터. 결제 금액(orderPrice)과 잔여 금액(remainPrice)을 참조한다 |
| `UserSettlement` | 호스트의 정산 계좌 정보 (은행, 계좌번호, 사업자 정보 등) |
| `SettlementChangeHistory` | 정산 상태 변경 이력 |

### 1-2. SettlementStatus 정의

```
PENDING              정산 생성 직후. 지급 대기 상태
COMPLETED            호스트에게 지급 완료
ONHOLD               지급 보류 (운영 판단)
LONG_TERM_UNSETTLED  paidOutAmount = 0인 정산. 생성 후 1주 뒤 자동 전환
                     ※ 네이밍 부적절: 실제 의미는 "정산 금액 없음"이므로 NO_PAYOUT 등으로 변경 검토
```

### 1-3. GatheringType 정의

```
SOCIALING   소셜링 (단건 모임)
CHALLENGE   챌린지 (기간제 모임)
CLUB        클럽 (구독형 모임)
COURSE      코스 (온라인 강좌)
```

## 2. 공통 스케줄 규칙

```
실행 시각: 매주 화요일 04:00 KST (production)
         매 10분 (dev)

실행 순서:
  1. createChallengeSettlement()
  2. createClubSettlement()
  3. createSocialingSettlement()
  4. createCourseSettlement()

중복 방지:
  각 도메인 아이템에 이미 활성(deletedAt IS NULL) Settlement이 있으면 생성하지 않는다.
  클럽은 추가로 이번 달에 이미 Settlement이 있으면 건너뛴다.

미정산 자동 전환:
  매주 화요일 05:00 KST에 이번 주 생성된 Settlement 중
  paidOutAmount = 0인 건을 LONG_TERM_UNSETTLED로 변경한다.
```

## 3. 도메인별 정산 규칙

### 3-1. 소셜링

```
[대상 선정]
  조건:
    - Socialing.status = COMPLETE (종료된 모임)
    - Socialing.startDate가 직전 1주(월~일) 범위에 포함
    - Socialing.price >= 10,000원
    - 해당 소셜링에 활성 Settlement이 없음

[Order 집계]
  해당 소셜링의 모든 Order를 순회하며:

    CASE OrderStatus = PARTIAL_REFUNDED (얼리버드 환불):
      paymentAmount += remainPrice    (실제 결제 유지 금액)
      discountAmount += refundPrice   (환불된 할인 금액)
      count += 1

    CASE OrderStatus = PARTIAL_REFUND_FAILED (환불 실패):
      paymentAmount += remainPrice
      discountAmount += refundPrice
      count += 1

    CASE remainPrice > 0 (기타 유효 주문):
      paymentAmount += remainPrice
      count += 1

  ⚠️ 주의: orderStatus를 명시적으로 필터링하지 않는다.
     COMPLETED_CANCEL_PAYMENT이어도 remainPrice > 0이면 정산에 포함될 수 있다.

[정산 금액 계산]
  staffCount = STAFF 등급 + APPROVE 상태 멤버 수
  settlementTargetCount = MIN(집계된 주문 수, maximumPerson - staffCount)
  settlementTargetAmount = MIN(paymentAmount, price × settlementTargetCount - discountAmount)

  paidOutAmount = FLOOR(settlementTargetAmount × 0.8)  // 수수료 20%

[정산 생성]
  Settlement 레코드를 생성한다.
  type = Socialing.clubId가 있으면 '클럽', 없으면 '일반'
```

### 3-2. 챌린지

```
[대상 선정]
  조건:
    - Challenge.status = COMPLETE
    - Challenge.endDate가 직전 1주(월~일) 범위에 포함
    - Challenge.price >= 10,000원
    - 해당 챌린지에 활성 Settlement이 없음

[Order 집계]
  조건: orderStatus = COMPLETED_PAYMENT만 포함

  ✅ 소셜링과 달리 orderStatus를 명시적으로 필터링한다.
     COMPLETED_CANCEL_PAYMENT인 주문은 제외된다.

  totalCount = 필터된 Order 수
  totalPaymentAmount = SUM(orderPrice)

[정산 금액 계산]
  settlementTargetAmount = MIN(totalPaymentAmount, price × maximumMembers)
  settlementTargetCount = settlementTargetAmount / price

  paidOutAmount = settlementTargetAmount × 0.8  // 수수료 20%
```

### 3-3. 클럽

```
[대상 선정]
  조건:
    - Club.price > 0
    - 2개월 전 ~ 1개월 전 사이에 생성된 Order 중 remainPrice > 0인 건이 있음
    - 이번 달에 해당 클럽의 활성 Settlement이 없음
    - 매월 1~7일에만 실행됨 (production)

[Order 집계]
  조건: remainPrice > 0인 Order만 포함 (createdAt 2개월전~1개월전)

  ⚠️ 주의: orderStatus를 체크하지 않는다.
     COMPLETED_CANCEL_PAYMENT이어도 remainPrice > 0이면 정산에 포함된다.
     이중결제로 Order가 2건 생기면 둘 다 정산될 수 있다.

  totalCount = 필터된 Order 수
  totalPaymentAmount = SUM(remainPrice)

[정산 금액 계산]
  settlementTargetAmount = totalPaymentAmount  (상한 없음)
  settlementTargetCount = totalCount

  paidOutAmount = settlementTargetAmount × 0.8  // 수수료 20%
```

### 3-4. 코스

```
[대상 선정]
  조건:
    - Course.price >= 50,000원
    - 2주 전(월~일)에 생성된 Order가 있음
    - 해당 코스에 활성 Settlement이 없음

[Order 집계]
  조건: remainPrice > 0인 Order만 포함 (createdAt 2주전)

  ⚠️ 주의: orderStatus를 체크하지 않는다.
     클럽과 동일한 문제가 발생할 수 있다.

  totalCount = 전체 Order 수 (필터 전)
  settlementTargetCount = remainPrice > 0인 Order 수
  totalPaymentAmount = price × settlementTargetCount
  settlementTargetAmount = SUM(remainPrice)

[정산 금액 계산]
  discountAmount = totalPaymentAmount - settlementTargetAmount

  paidOutAmount = CEIL(settlementTargetAmount × 0.85 × 0.7)  // 수수료 40.5%
```

## 4. 수수료율 요약

| 도메인 | 호스트 지급률 | 플랫폼 수수료 | 반올림 |
| --- | --- | --- | --- |
| 소셜링 | 80% | 20% | FLOOR (내림) |
| 챌린지 | 80% | 20% | 없음 |
| 클럽 | 80% | 20% | 없음 |
| 코스 | 59.5% (85% × 70%) | 40.5% | CEIL (올림) |

## 6. 알려진 문제점

### 6-1. orderStatus 필터링 불일치

```
챌린지만 orderStatus = COMPLETED_PAYMENT를 명시적으로 필터링한다.
소셜링, 클럽, 코스는 remainPrice > 0만 체크한다.

영향:
  - 취소된 주문(COMPLETED_CANCEL_PAYMENT)이라도 remainPrice가 0으로 업데이트되지 않았으면
    정산에 포함된다.
  - 이중결제로 Order가 2건 생기면 둘 다 정산될 수 있다.

개선 방향:
  모든 도메인에서 orderStatus = COMPLETED_PAYMENT (또는 PARTIAL_REFUNDED)만 정산 대상으로
  필터링하도록 통일해야 한다.
```

### 6-2. remainPrice 업데이트 누락 가능성

```
환불 시 Order.remainPrice = 원래금액 - 환불금액으로 업데이트되어야 한다.
전액 환불 시 remainPrice = 0이 되어 정산에서 자연스럽게 제외된다.

그러나 FAILED_CANCEL_PAYMENT(환불 실패) 상태에서는 PG 환불이 안 됐으므로
remainPrice가 그대로 남아있어 정산에 포함될 수 있다.

이 경우 호스트에게 지급된 후 수동 환불 시 정산 차감이 필요하다.
```

### 6-3. 소셜링 STAFF 정산 제외 로직

```
소셜링은 STAFF 등급 멤버를 정산 인원에서 차감한다.
  settlementTargetCount = MIN(주문수, maximumPerson - staffCount)

STAFF는 호스트를 보조하는 역할이므로 정산 대상이 아니다.
정산 가능 정원(settlementMaxCount)에서 STAFF 수를 빼는 것은 정상 동작이다.
```

## 7. Settlement 테이블 주요 컬럼

| 컬럼 | 설명 |
| --- | --- |
| gatheringType | SOCIALING / CHALLENGE / CLUB / COURSE |
| price | 1인당 가격 |
| maximumCount | 최대 정원 |
| staffCount | STAFF 멤버 수 (소셜링만 사용) |
| totalCount | 전체 주문 수 |
| confirmCount | 확정 참여자 수 |
| settlementMaxCount | 정산 가능 최대 인원 |
| settlementTargetCount | 실제 정산 대상 인원 |
| totalPaymentAmount | 전체 결제 금액 합계 |
| settlementTargetAmount | 정산 대상 금액 (수수료 차감 전) |
| discountAmount | 할인/환불 금액 합계 |
| paidOutAmount | 호스트 지급 금액 (수수료 차감 후) |
| paidOutDate | 실제 지급일 |
| status | PENDING → COMPLETED / ONHOLD / LONG_TERM_UNSETTLED |
| type | '클럽' 또는 '일반' (소셜링만 사용) |
