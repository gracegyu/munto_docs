## v3에서 발견된 문제

마무리를 위해 사소한 표기 정리(섹션 번호 중복 등)는 제외하고, 실제 동작 정합성과 산출물 완전성에 영향을 주는 항목만 정리합니다.

### Critical (3건)

#### C-001 [HIGH] 정산 §4-1 소셜링 식이 Invariant 3과 모순 — HOST 1슬롯 차감 누락

**위치**: 정산 §4-1 소셜링 [정산 금액 계산] / Invariant 3

Invariant 3은 소셜링 정원 상한을 `maximumPerson - 1(HOST) - staffCount`로 정의했지만, §4-1 [정산 금액 계산]은 `MIN(집계된 주문 수, maximumPerson - staffCount)`로 적혀 있어 `-1(HOST)`이 누락되어 있습니다. 챌린지 §4-2와 클럽 §4-3은 `-1(HOST)`이 반영되었으나 소셜링만 빠져, 정책 변경(2) "모임 정원에 호스트 포함"의 효과가 소셜링 정산에서는 발생하지 않습니다.

WEBB-1377 OnePager가 지적한 핵심 문제(소셜링 정산 캡에 HOST 1슬롯 미차감)와 직결되므로 §4-1을 `MIN(집계된 주문 수, maximumPerson - 1 - staffCount)`로 수정 부탁드립니다. Invariant 6 예시("정원 10명 소셜링 13건 중 9건 정산 반영")는 이미 -1(HOST) 차감을 가정한 9건으로 정정되어 있어 식 수정 후 자연 일치합니다.

#### C-002 [HIGH] 소셜링·챌린지 결제/환불 명세서 2개가 산출물 목록에서 누락 — 환불 금액 규칙의 정의 위치 부재

**위치**: WEBB-1321 코멘트 line 131-136 산출물 목록 / 소셜링 §1 헤더 / 챌린지 §1 헤더·Invariant 7·§5-5·§6·§7

`소셜링 결제/환불 규칙 명세서`와 `챌린지 결제/환불 규칙 명세서` 2개가 산출물 목록에서 빠졌습니다. 그러나 본문은 두 문서를 여전히 참조합니다. 소셜링 §1 헤더의 "주문/결제 세부 규칙" 링크, 챌린지 §1 헤더 링크, 챌린지 Invariant 7의 "환불 규칙은 [챌린지 결제/환불 규칙 명세서]에서 정의한다" 위임, §5-5/§6/§7-1/§7-2의 결제/환불 명세서 §X 참조 등입니다.

결과적으로 본인 취소 시간 컷오프(당일/30분/4일), 소셜링 STAFF 완료 시 환불, 챌린지 호스트 폐강 시 전액 환불, 결제 확정 시 금액 필드 세팅 규칙 같은 핵심 환불 규칙이 산출물 어디에서도 정의되지 않습니다.

다음 중 하나로 정리 부탁드립니다. (1) 두 문서가 Notion에 존재하면 산출물 목록에 추가, (2) 의도적 통합이라면 통합 위치(예: 참여/취소 문서의 신설 섹션)와 통합된 내용 인벤토리 명시 + 참여/취소 문서의 링크·위임 문구를 통합 위치로 정정. 본 항목 미해소 시 현 산출물만으로는 환불 금액 산출 코드·QA가 불가능합니다.

#### C-003 [HIGH] ASIS/TOBE 구분 누락 — 정원에 HOST 포함 정책이 현행 코드와 다른 도메인 존재

**위치**: 소셜링 §1-4·§2 Invariant 1 / 챌린지 §2 Invariant 1 / 클럽 §1-3·§2 Invariant 1

본 명세서는 정책 변경(2)을 TOBE 기준으로 반영했지만 현행 코드와의 차이가 명세서 어디에도 명시되지 않습니다. WEBB-1377 OnePager에 따르면 클럽은 `ClubMember`에 `grade=OWNER` row 존재로 ASIS-TOBE가 이미 정합이지만, 챌린지는 `ChallengeMember`에 grade 필드도 OWNER row도 없고, 소셜링은 `SocialingMemberGrade` enum에 HOST 값 자체가 없으며 HOST는 `SocialingMember`에 row 없이 `Socialing.userId`로만 식별됩니다.

특히 소셜링 Invariant 1 식 `availCount = maximumPerson - COUNT(SocialingMember WHERE status = APPROVE)`는 HOST가 `SocialingMember`에 row로 존재해야 성립하는데 현행은 HOST row가 없어 식이 깨집니다. 본 명세서를 그대로 코드 작성 기준으로 사용하면 WEBB-1377 마이그레이션 완료 전에는 정합이 무너집니다.

다음 중 하나로 보강 부탁드립니다. (1) 본 명세서가 WEBB-1377 적용 후 TOBE 기준임을 §1 헤더 또는 Invariant 1 주석에 명시 (예: "본 명세서는 WEBB-1377 적용 후 기준이며, 현행 코드와의 차이는 [WEBB-1377 OnePager] 참조"), (2) 도메인별로 ASIS와 TOBE를 분리 표기. 본 항목은 WEBB-1321 명세서와 WEBB-1377 마이그레이션의 적용 순서·범위 합의와 함께 결정되어야 합니다.

### Warning (3건)

#### W-001 [MEDIUM] 클럽 결제/환불 Invariant 5 vs §3-1 단계 B — UserBillingKey ASIS/TOBE 모호

**위치**: 클럽 결제/환불 Invariant 5 / §3-1 단계 B

Invariant 5는 "현재 구현: `ClubMembership.billingKey` / 개선 방향: `UserBillingKey`"로 ASIS와 TOBE를 모두 명시했는데, §3-1 단계 B는 `UserBillingKey 테이블 UPSERT (유저당 1개 활성 키 유지)`로 TOBE를 이미 사용하는 것처럼 적었습니다.

다음 중 하나로 정리 부탁드립니다. (1) §3-1을 ASIS 기준 `ClubMembership.billingKey` 저장으로 수정, (2) 본 명세서가 TOBE 기준이면 Invariant 5에서 "현재 구현"을 제거하고 `UserBillingKey — 유저당 1개 활성 키` 한 줄로 통일. C-003과 함께 명세서 전체의 ASIS/TOBE 입장을 한 번에 정리하면 자연스럽게 해소됩니다.

**위치**: 소셜링 §1 line 8 / 챌린지 §1 line 8·Invariant 7·§5-5·§6·§7-1·§7-2

C-002가 결정되면 함께 정리되는 항목입니다. 현재 다음 위치에서 산출물 목록에 없는 결제/환불 명세서를 참조하고 있습니다. 소셜링 §1 헤더의 "[소셜링 결제/환불 규칙 명세서]" 링크, 챌린지 §1 헤더의 "[챌린지 결제/환불 규칙 명세서]" 링크, 챌린지 Invariant 7의 "환불 규칙은 [챌린지 결제/환불 규칙 명세서]에서 정의한다" 위임, 챌린지 §5-5의 `order-challenge.md §4-1 호스트 주도 취소 참조` (해당 파일 부재), 그리고 §6·§7-1·§7-2의 `챌린지 결제/환불 규칙 명세서 §X` 참조입니다. #### W-003 [MEDIUM] 클럽 참여/취소 §5-2/§5-3/§6 — 환불 위임 표기가 다른 도메인과 비대칭

**위치**: 클럽 참여/취소 §5-2 / §5-3 / §6

클럽 참여/취소는 `클럽 결제/환불 규칙 명세서`가 산출물에 존재함에도 §5-2(멤버 강퇴), §5-3(멤버 거절), §6(본인 취소)에서 "결제 취소/환불 처리 (`order-club.md` 참조)"로 표기되어 있습니다. 다른 도메인 명세서가 사용하는 `클럽 결제/환불 규칙 명세서 §X` 표기와 일치하지 않고, 위임 대상 섹션도 명시되지 않아 단계별 추적이 어렵습니다. 위임 문장을 `클럽 결제/환불 규칙 명세서 §4-2 환불 금액 계산` 형태로 명시 부탁드립니다.

## 총평

v3는 정책 변경(환불 없는 취소 정산 제외 / 정원에 호스트 포함 / ONETIME 제거)을 정산 명세서·클럽 결제/환불 명세서·각 도메인 Invariant에 대체로 반영했습니다. v2 리뷰의 6건 미해소·신규 항목 중 5건이 해소되었습니다(ST-Q1만 LOW로 잔존).

마무리에 앞서 다음 두 영역만 확인되면 운영 적용이 가능한 수준입니다.

| 우선순위 | 항목 | 조치 |
| --- | --- | --- |
| 1 | C-001 정산 §4-1 소셜링 -1(HOST) 누락 | `MIN(집계된 주문 수, maximumPerson - 1 - staffCount)`로 수정 |
| 2 | C-002 결제/환불 명세서 2개 누락 | Notion 존재 시 산출물 목록에 추가 / 통합했으면 통합 위치 명시 |
| 3 | C-003 ASIS/TOBE 구분 | WEBB-1377 적용 후 기준임을 헤더에 명시 또는 도메인별 ASIS/TOBE 분리 |

C-001은 코드 한 줄 수정 수준이라 WEBB-1325·WEBB-1326 진행 중에도 빠르게 해소 가능합니다. C-002는 산출물 외부의 문서 두 개의 존재 여부 확인이 우선이며, 존재한다면 단순 목록 갱신만으로 마무리됩니다. C-003은 WEBB-1377 OnePager와 본 명세서의 적용 순서·범위 합의 후 한 문장 추가로 정리 가능합니다.

Warning 3건은 위 Critical 해소 시 자연스럽게 함께 정리되는 항목이므로 별도 진행은 불필요합니다.
