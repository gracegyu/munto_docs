## 1. 핵심 개념

### 1-1. 관련 테이블

| 테이블 | 소셜링 도메인에서의 역할 |
| --- | --- |
| `Socialing` | 소셜링 메인. 정원·상태·모집 조건 소유 |
| `SocialingMember` | 참가자. 참여 상태와 참여권(Order) 1:1 보유 |
| `Order` | 참여권의 존재 증명. 유료·무료 구분 없이 항상 존재 |

> `SocialingMember.orderId`는 `NOT NULL @unique` — 참가자는 반드시 하나의 Order를 가진다.
Order 내부 동작(결제 방식, PG 연동, 환불 실행)은 주문 도메인 문서에서 다룬다.
> 

### 1-2. SocialingRecruitType

```
COMMON  선착순: 결제 확정 즉시 APPROVE
CHOICE  승인제: 결제 확정 시 REQUEST → 호스트 승인 후 APPROVE
```

### 1-3. SocialingMemberStatus

```
REQUEST   승인 대기 (승인제 신청 직후)
APPROVE   참여 확정
REJECT    거절 (호스트가 REQUEST 거절)
CANCEL    취소 (본인 취소 / 자동 취소)
KICK      강퇴
DELETE    소프트 삭제 (CANCEL/REJECT/KICK 이후 목록에서 숨김 처리. deletedAt 기록)
```

### 1-4. SocialingMemberGrade

```
HOST    호스트. 소셜링 생성자. 1명. 결제 없이 참여
STAFF   스태프. 호스트가 지정. 최대 5명. 결제 후 COMPLETE 시 전액 환불
MEMBER  일반 참여자
```

### 1-5. Socialing.status (ItemStatus)

```
PLANNING         준비 중 (비공개)
RECRUITING       모집 중
CONFIRM_PLAYING  진행 확정 (시작 1시간 전, 최소 인원 충족)
CLOSE_BY_SYSTEM  자동 폐강 (시작 1시간 전, 최소 인원 미달)
CLOSE            호스트 직접 폐강
COMPLETE         진행 완료
```

## 2. 핵심 불변식 (Invariants)

모든 쓰기 작업은 아래 불변식을 보존해야 한다.

### **Invariant 1 — 정원 정합성**

```sql
Socialing.availCount
  = Socialing.maximumPerson
  - COUNT(SocialingMember WHERE socialingId = this.id AND status = 'APPROVE')
```

- `availCount < 0` 불가
- `availCount > maximumPerson` 불가

### Invariant 2 — 성별 정원 정합성

```
maleMaximumCount > 0인 경우:
  COUNT(APPROVE WHERE User.sex = MALE) ≤ maleMaximumCount

femaleMaximumCount > 0인 경우:
  COUNT(APPROVE WHERE User.sex = FEMALE) ≤ femaleMaximumCount
```

### Invariant 3 — 참여권 정합성

```
SocialingMember는 반드시 하나의 Order를 가진다 (유료·무료 무관)

Member.status = APPROVE  →  Order 존재, 참여 유효
Member.status = REQUEST  →  Order 존재, 참여 보류
Member.status ∈ {CANCEL, KICK, REJECT}
                         →  Order 존재, 참여 종료 (환불 처리됨 또는 예정)
```

Order의 상태(orderStatus) 전이 규칙은 주문 도메인 문서에서 정의한다.

### Invariant 4 — 소셜링 상태별 멤버 정합성

```
Socialing.status = CONFIRM_PLAYING  →  SocialingMember.status ≠ REQUEST
Socialing.status = CLOSE_BY_SYSTEM  →  SocialingMember.status ∉ {REQUEST, APPROVE}
Socialing.status = CLOSE            →  SocialingMember.status ∉ {REQUEST, APPROVE}
Socialing.status = COMPLETE         →  SocialingMember.status ≠ REQUEST
```

### Invariant 5 — 미승인 타임아웃

```
아래 조건 중 하나라도 해당하면 즉시 CANCEL 대상:
  A. status = REQUEST AND createdAt < NOW() - 24h AND Socialing.status = RECRUITING
  B. status = REQUEST AND Socialing.status ∉ {PLANNING, RECRUITING}
```

## 3. 소셜링 상태 전이 규칙

```
PLANNING
  └→ RECRUITING        (호스트 공개)

RECRUITING
  ├→ CLOSE             (호스트 직접 폐강)
  ├→ CONFIRM_PLAYING   (스케줄러: startDate 1시간 전, approvedCount >= minimumPerson)
  └→ CLOSE_BY_SYSTEM   (스케줄러: startDate 1시간 전, approvedCount < minimumPerson)

CONFIRM_PLAYING
  └→ COMPLETE          (스케줄러: startDate 도달)
```

상태 전이는 단방향이며 역방향 전이는 불가하다.

## 4. 참여 신청 플로우

### 4-1. 사전 검증 규칙

결제 시작 전 다음 조건을 순서대로 검증한다.

```
1. 소셜링이 존재해야 함
2. 신청자 ≠ 호스트
3. 동일 소셜링에 status ∈ {REQUEST, APPROVE}인 기존 멤버 없어야 함
   (CANCEL·REJECT 후 재신청은 허용)
4. Socialing.status = RECRUITING
5. Socialing.availCount > 0
6. Socialing.startDate > NOW()
7. 성별 상한 조건 충족 (Invariant 2)
8. recruitScope = CLUB → 신청자가 해당 클럽 멤버여야 함
9. 신청자의 정지 기간이 소셜링 기간과 겹치지 않아야 함
```

### 4-2. 주문 생성 규칙

```
- 4-1 검증 재실행
- 금액 일치 검증: payload.orderPrice = 최종 결제 금액
- 동일 userId + socialingId + status=WAITING_PAYMENT 주문이 이미 있으면 재사용
- 없으면 새 Order 생성
- 무료 소셜링(price = 0)도 동일하게 Order 생성 (결제 실행 여부만 다름)
```

### 4-3. 결제 확정 규칙

```
[단계 A] 재검증:
  - Socialing.status = RECRUITING
  - Socialing.availCount > 0

[단계 B] 결제 처리 (주문 도메인):
  - 유료: PG 결제 승인 실행
  - 무료: PG 호출 없이 바로 다음 단계

[단계 C] DB 트랜잭션:
  - Order.status → COMPLETED_PAYMENT
  - createSocialingMember() 실행 (4-4 참고)
```

A와 C 사이의 동시 요청을 막기 위해 availCount 차감은 반드시 C 트랜잭션 내에서 원자적으로 처리한다.

### 4-4. 멤버 생성 및 정원 차감 규칙

트랜잭션 내에서 실행된다.

```
FUNCTION createSocialingMember(userId, socialingId, orderId, recruitType, tx):

  [멤버 상태 결정]
  COMMON(선착순)  →  status = APPROVE, approvedAt = NOW()
  CHOICE(승인제)  →  status = REQUEST,  approvedAt = null

  [SocialingMember UPSERT]
  ON CONFLICT (socialingId, userId):
		UPDATE: { orderId, status, approvedAt, grade: MEMBER }  ← 재신청 케이스
    ※ 재신청 시 grade는 항상 MEMBER로 초기화한다.
    ON CREATE:
    INSERT: { socialingId, userId, orderId, status, approvedAt, grade: MEMBER }

  [정원 차감 — status = APPROVE인 경우만]
  updated = tx.Socialing.updateMany(
    WHERE id = socialingId AND availCount > 0,
    DATA { availCount: decrement(1) }
  )
  IF updated.count ≠ 1 → TX ROLLBACK  ← 정원 초과 방어

  [성별 정원 차감 검증]
  IF maleMaximumCount > 0 또는 femaleMaximumCount > 0:
    성별별 APPROVE 인원 재확인 → Invariant 2 위반 시 TX ROLLBACK
```

### 4-5. 재신청 규칙

```
- status ∈ {CANCEL, REJECT}인 멤버는 동일 소셜링에 재신청 가능
- (socialingId, userId) unique constraint → UPSERT로 처리
- 재신청 시 새 Order 생성 (기존 Order와 별개)
- SocialingMember.orderId는 새 orderId로 업데이트
- 4-1 사전 검증 전체 재실행 (availCount, 성별 조건 등)
```

## 5. 호스트 작업 규칙

### 5-1. 승인 (approve) — 승인제 전용

```
FUNCTION approve(socialingId, memberId):

  [선행 검증]
  - member.status = APPROVE → 이미 승인됨, 멱등 처리 후 종료
  - member.status ≠ REQUEST → 승인 불가

  [트랜잭션]
  BEGIN:
    updated = Socialing.updateMany(WHERE id=socialingId AND availCount > 0, DECREMENT 1)
    IF updated.count ≠ 1 → ROLLBACK  ← 정원 없음

    SocialingMember.status = REQUEST → APPROVE
    SocialingMember.approvedAt = NOW()

    IF 얼리버드 소셜링 && 혜택 조건 충족:
      DiscountAndSocialingMember 생성
  COMMIT
```

### 5-2. 거절 (reject) — 승인제 전용

```
FUNCTION reject(socialingId, memberId):

  [선행 검증]
  - member.status ≠ REQUEST → 거절 불가
  - member.status = REJECT  → 이미 거절됨, 멱등 처리 후 종료

  [트랜잭션]
  BEGIN:
    SocialingMember.status = REQUEST → REJECT
    ← availCount 변화 없음 (REQUEST는 정원을 차지하지 않음)
  COMMIT

  [취소 처리 위임 → 주문 도메인]
  유료 참여였다면 환불 처리 요청
  Order.status 전이는 주문 도메인에서 수행
```

### 5-3. 등급 변경 (grade update)

대상 등급 변경: MEMBER ↔ STAFF (양방향). HOST 등급은 변경 대상이 아니다.

```
FUNCTION updateSocialingMemberGrade(hostId, socialingId, members[{userId, grade}]):

  [공통 선행 검증]
  - 호출자 = 소셜링 호스트여야 함
  - Socialing.status = RECRUITING
  - 유료 소셜링만 적용 가능 (price > 0)
  - 변경하려는 grade가 현재 grade와 같으면 BadRequest

  [승격(MEMBER → STAFF) 추가 검증]
  - 소셜링 생성 후 24시간 이내일 것
    (`now <= createdAt + 24h`. 24h 초과 시 ForbiddenException)
  - 변경 후 STAFF 총 인원 ≤ 5명 (MAX_SOCIALING_STAFF_COUNT)

  [강등(STAFF → MEMBER) 추가 검증]
  - 별도 제약 없음 (시점·인원 가드 모두 미적용)

  [트랜잭션]
  FOR EACH member in payload:
    BEGIN:
      SocialingMember.grade = 새 grade

      IF 새 grade = STAFF AND member 얼리버드 보유:
        얼리버드 혜택을 다음 순번 멤버에게 이전 (handOverEarlyBirdBenefit)
      * STAFF는 완료 시 전액 환불 대상이므로 부분 환불 혜택을 보유할 수 없음

      IF 새 grade = MEMBER AND 얼리버드 슬롯 여유 있음
         (소셜링 얼리버드 활성 AND grantedCount < earlyBird.amount):
        해당 멤버에게 얼리버드 배정 (applyEarlyBirdBenefit)
        ← 슬롯이 이미 가득 찼다면(예: STAFF 승격 시 이전된 혜택을 다른 멤버가 보유 중) 배정하지 않음
    COMMIT

  [결제 처리]
  - grade 변경 시 즉각적인 환불은 없음
  - STAFF 환불은 소셜링 COMPLETE 시점에 일괄 처리
  - 얼리버드 혜택 이동에 따른 부분 환불 흐름은 §10 참조
```

### 5-4. 킥 (kick)

```
FUNCTION kick(socialingId, memberId):

  [선행 검증]
  - member.status ≠ APPROVE → 킥 불가 (APPROVE 상태만 킥 가능)
  - member.status = KICK    → 이미 킥됨, 멱등 처리 후 종료
  - member.grade = STAFF이고 호출자가 HOST가 아닌 경우 → 킥 불가
    (STAFF는 HOST만 킥 가능, 일반 STAFF 간 킥 불가)

  [트랜잭션]
  BEGIN:
    SocialingMember.status = APPROVE → KICK
    Socialing.availCount += 1

    IF member가 얼리버드 보유:
      다음 순번 멤버에게 얼리버드 혜택 이전
  COMMIT

  [취소 처리 위임 → 주문 도메인]
  유료 참여였다면 환불 처리 요청
```

## 6. 취소 규칙

### 6-1. 멤버 본인 취소

```
FUNCTION cancel(socialingId, userId, cancelReason?):

  [선행 검증]
  - member.status = CANCEL → 이미 취소됨, 멱등 처리 후 종료

  wasApprove = member.status = APPROVE

  [트랜잭션]
  BEGIN:
    SocialingMember.status → CANCEL
    SocialingMember.canceledAt = NOW()
    SocialingMember.cancelReason = cancelReason

    IF wasApprove:
      Socialing.availCount += 1

    IF member가 얼리버드 보유:
      다음 순번 멤버에게 얼리버드 혜택 이전
  COMMIT

  [취소 처리 위임 → 주문 도메인]
  wasApprove AND 유료 참여였다면 환불 처리 요청

  참고: REQUEST 취소 시 availCount 변화 없음
```

### 6-2. 호스트 직접 폐강 (CLOSE)

```
FUNCTION closeSocialingByHost(socialingId, hostUserId):

  [선행 검증]
  - Socialing.userId ≠ hostUserId → ForbiddenException
  - Socialing.status ≠ RECRUITING → 폐강 불가

  [멤버 일괄 처리]  [단일 트랜잭션 — 상태 전이 원자화]
  BEGIN TX:
    Socialing.status = CLOSE, Socialing.closedAt = NOW()

    FOR EACH member WHERE status ∈ {REQUEST, APPROVE}:
      wasApprove = (member.status = APPROVE)
      member.status → CANCEL, canceledAt = NOW()
      IF wasApprove: Socialing.availCount += 1
  COMMIT

  소셜링 상태 전이와 모든 멤버 상태 전이를 하나의 TX에 묶어
  부분 실패로 인한 Invariant 4 위반을 방지한다.

  [취소 처리 위임 → 주문 도메인] (TX 외부)
  FOR EACH 유료 멤버:
    PG 환불 실행 (Bootpay cancelReceipt)
    → 성공: Order.orderStatus → COMPLETED_CANCEL_PAYMENT
    → 실패: Order.orderStatus → FAILED_CANCEL_PAYMENT
            재시도 스케줄러가 복구 (주문/결제 공통 규칙 명세서 5-2)

  [예외 복구 — Sweeper 스케줄러]
  TX 커밋 후 환불 루프 크래시 등으로 "멤버 종료 상태 + Order=COMPLETED_PAYMENT"
  상태 불일치 Order가 발생한 경우, Sweeper 스케줄러가 이를 탐지하여
  FAILED_CANCEL_PAYMENT로 전환 후 기존 재시도 루프에 편입한다 (WEBB-1326).
```

## 7. 정원 변경 규칙

정원 변경 시 `availCount`는 반드시 `newMax - currentApprovedCount`로 **재계산**한다.
`oldMax → newMax` delta 방식은 실제 인원을 반영하지 못하므로 사용 금지.

### 7-1. 호스트 정원 변경

```
FUNCTION updateMaximumPerson(socialingId, newMax):

  currentApprovedCount = COUNT(SocialingMember WHERE socialingId=id AND status=APPROVE)

BEGIN TX:
    SELECT * FROM Socialing WHERE id = socialingId FOR UPDATE   ← 행 잠금

    currentApprovedCount = COUNT(SocialingMember
                                 WHERE socialingId = id AND status = APPROVE)

    IF newMax < currentApprovedCount → BadRequestException (ROLLBACK)

    Socialing.maximumPerson = newMax
    Socialing.availCount = newMax - currentApprovedCount
  COMMIT
```

### 7-2. 어드민 정원 변경

```
FUNCTION updateSocialingByAdmin(socialingId, newMax):

  BEGIN TX:
    SELECT * FROM Socialing WHERE id = socialingId FOR UPDATE   ← 행 잠금

    currentApprovedCount = COUNT(SocialingMember
                                 WHERE socialingId = id AND status = APPROVE)

    IF newMax < currentApprovedCount → BadRequestException (ROLLBACK)

    Socialing.maximumPerson = newMax
    Socialing.availCount = newMax - currentApprovedCount
  COMMIT
```

> 
> 
> - 7-1과 7-2의 구조는 동일하다.
>     - 호스트·어드민 모두 `newMax < currentApprovedCount`는 불가능하다.
>     - 이미 APPROVE된 인원을 강제로 내보낼 기준(누구를 내보낼지)이 없기 때문이다.
> - 정원 축소는 현재 APPROVE 인원 이상으로만 허용된다.

## 8. 스케줄러 처리 규칙

### 8-1. CONFIRM_PLAYING 전이

```
조건: status = RECRUITING AND startDate ≤ NOW() + 1h
      AND COUNT(APPROVE) >= minimumPerson

[단일 트랜잭션 — 상태 전이 원자화]
BEGIN TX:
  Socialing.status = CONFIRM_PLAYING

  FOR EACH member WHERE status = REQUEST:
    member.status → CANCEL, canceledAt = NOW()
    ← availCount 변화 없음 (REQUEST는 정원 미차지)
COMMIT

[취소 처리 위임 → 주문 도메인] (TX 외부)
환불 실행 및 실패 복구 흐름은 6-2와 동일.
크래시로 인한 상태 불일치 Order는 Sweeper 스케줄러가 복구 (WEBB-1326).
```

### 8-2. CLOSE_BY_SYSTEM 전이

```
조건: status = RECRUITING AND startDate ≤ NOW() + 1h
      AND COUNT(APPROVE) < minimumPerson

[단일 트랜잭션 — 상태 전이 원자화]
BEGIN TX:
  Socialing.status = CLOSE_BY_SYSTEM

  FOR EACH member WHERE status ∈ {REQUEST, APPROVE}:
    wasApprove = (member.status = APPROVE)
    member.status → CANCEL, canceledAt = NOW()
    IF wasApprove: Socialing.availCount += 1
COMMIT

[취소 처리 위임 → 주문 도메인] (TX 외부)
환불 실행 및 실패 복구 흐름은 6-2와 동일.
크래시로 인한 상태 불일치 Order는 Sweeper 스케줄러가 복구 (WEBB-1326).
```

### 8-3. COMPLETE 전이

```
조건: status = CONFIRM_PLAYING AND startDate ≤ NOW()

처리:
  Socialing.status = COMPLETE
  스태프(grade = STAFF) 멤버 환불 처리 위임
```

### 8-4. 24시간 미승인 자동취소

```
대상 (아래 중 하나):
  A. status = REQUEST AND createdAt < NOW() - 24h AND Socialing.status = RECRUITING
  B. status = REQUEST AND Socialing.status ∉ {PLANNING, RECRUITING}

처리:
  member.status → CANCEL
  ← availCount 변화 없음
  [취소 처리 위임 → 주문 도메인]
```

조건 B가 필요한 이유: 8-1/8-2 처리 중 일부 멤버가 누락될 경우,
소셜링이 이미 전이됐음에도 REQUEST 상태로 잔존할 수 있다.

## 9. availCount 변경 규칙

| 이벤트 | 변화 | 조건 |
| --- | --- | --- |
| 선착순 결제 확정 (COMMON) | -1 | TX 내부, `WHERE availCount > 0` 원자적 처리 |
| 호스트 승인 (REQUEST → APPROVE) | -1 | TX 내부, `WHERE availCount > 0` 원자적 처리 |
| 킥 (APPROVE → KICK) | +1 | TX 내부 |
| 취소 (was APPROVE) | +1 | TX 내부 |
| 취소 (was REQUEST) | 없음 | REQUEST는 정원 미차지 |
| 거절 (REQUEST → REJECT) | 없음 | REQUEST는 정원 미차지 |
| 승인제 신청 (→ REQUEST) | 없음 | — |
| 정원 변경 | `newMax - COUNT(APPROVE)` | 재계산 (delta 방식 금지) |
| 폐강/시스템 취소 (was APPROVE) | +1 | 각 멤버 처리 시 |
| 폐강/시스템 취소 (was REQUEST) | 없음 | — |
| 소셜링 생성 | `= maximumPerson` | 생성 시 APPROVE 0명 |

## 10. 얼리버드 (할인 혜택)

일부 유료 소셜링이 선택적으로 운영하는 선착순 할인 혜택. 결제 정가를 받고 할인액만큼 **부분 환불 형태로 지급**한다 — 멤버는 APPROVE를 유지하며 정원·status에 영향이 없다.

### 10-1. 정책 (노션 §3-5)

```
- 호스트가 소셜링 생성·수정 시 amount(슬롯 수)와 value(할인율)를 지정
- 할인율: 5% / 10% / 15% 중 선택
- amount ≤ minimumPerson
- grantedCount < amount 인 동안 선착순 부여
```

> 모델: `Discount(type=EARLY_BIRD)` ↔ `DiscountPolicy(kind=PERCENTAGE, value)` ↔ `DiscountAndSocialingMember`(부여된 멤버 매핑). 할인율 5/10/15% 화이트리스트 검증은 백엔드 미구현 — improvements.md §13 참조.
> 

### 10-2. 부여·이전

```
[부여]
트리거: MEMBER가 APPROVE로 진입 (§4-3 COMMON 결제 확정 / §5-1 CHOICE 승인)
조건:   Socialing.earlyBird != NULL
        AND member.grade = MEMBER  (HOST·STAFF 제외)
        AND grantedCount < amount
처리:   DiscountAndSocialingMember 생성
        PG 부분 환불 → Order.orderStatus = PARTIAL_REFUNDED
                       (실패 시 PARTIAL_REFUND_FAILED, order-common.md §5-3 재시도)

[이전 — handOverEarlyBirdBenefit]
트리거: 본인 취소 (§6-1) / 강퇴 (§5-4) / STAFF 승격 (§5-3)
처리:   기존 보유자의 DiscountAndSocialingMember 삭제
        다음 후보 1명 선정 → 부여 (조건은 위와 동일)

[다음 후보 선정]
같은 소셜링의 APPROVE + MEMBER + 미보유 멤버 중 approvedAt ASC 1명
```

### 10-3. 정산 영향

부분 환불된 Order는 `remainPrice`로 정산되고 할인액(`refundPrice`)은 호스트 부담이다. 상세 집계식은 [settlement.md §4-1](https://www.notion.so/order/settlement.md) 참조.

`PARTIAL_REFUND_FAILED`는 멤버 APPROVE 유지 + remainPrice 기준 정산 포함이며, PG 환불 실패 자체는 별도 재시도·Slack 알림으로 해소한다 (Invariant SE6).
