# Munto 앱이 내부 5종 표준 이벤트(회원가입·조회·좋아요·결제 시작·구매)를 문토 API로 전송하여 플랫폼·앱 버전별 정합성 검증 인프라를 구축한다

| 항목 | 내용 |
| --- | --- |
| 배경 | DEVT-134(Meta 광고 측정 인프라 원점 재설계) 트랙으로 정합성 검증 시트에서 내부 DB 집계값과 Meta 이벤트 매니저의 5종 표준 이벤트(회원가입·유료 소셜링 조회·좋아요·결제 시작·구매) 값을 일별로 비교하고 있다.<br><br>현재 내부 5종 이벤트 집계는 munto-mobile·munto-mobile-core 의 로컬 로거(LogOutputInterface)로 처리되며, Facebook App Events·Firebase Analytics 등 외부 분석 SDK 로만 송신되고 문토 백엔드 DB 에는 적재되지 않는다.<br><br>또한 내부 집계는 안드로이드와 iOS 를 구분하지 않고 합산되어 Meta 이벤트(플랫폼별 보고)와 1:1 매칭이 불가하다. |
| 현재 문제점 | Meta 이벤트와 내부 집계값 간에 차이가 관측되어도 원인(이벤트 트리거 시점 차이·이중 발화·플랫폼별 누락·앱 버전별 차이 등)을 진단할 데이터가 부족하다.<br><br>내부 이벤트가 백엔드 DB 에 적재되지 않아 SQL 기반 분석·재가공·과거 시점 비교가 모두 불가하다.<br><br>플랫폼(iOS·Android) 구분 집계가 없어 광고 매체별·캠페인별 성과 측정 정확도가 떨어진다. |
| 개선 방향 | munto-mobile·munto-mobile-core 의 `LogOutputInterface` 를 활용하여 내부 5종 표준 이벤트를 문토 API 로 전송하는 새 LogOutput(예: `MuntoApiEventsOutput`)을 추가한다. 기존 FacebookEventsOutput·Firebase Analytics Output 과 동일 인터페이스를 사용해 추가 부담을 최소화한다.<br><br>모바일 HTTP 클라이언트가 이미 헤더에 디바이스 정보(User-Agent·X-App-Platform·X-App-Version 등)를 포함하므로 백엔드는 별도 페이로드 없이 헤더만으로 플랫폼·앱 버전별 집계가 가능하다.<br><br>백엔드는 새 endpoint(예: `POST /v1/analytics/events`)를 추가하여 모바일이 전송하는 이벤트를 받아 DB 에 적재한다. 스키마는 일별·플랫폼별·앱 버전별 집계가 가능하도록 설계한다.<br><br>적재된 데이터를 munto-analyze-paid-socialing 스킬·정합성 검증 시트와 연동하여 Meta 이벤트와 1:1 비교 가능한 분석 흐름을 구축한다. |
| 고려요소 | 이벤트 전송 빈도와 배치 전략을 결정해야 한다. 사용자 액션 단위 즉시 전송과 N초·N건 배치 전송 사이의 trade-off(배터리·트래픽·정확성)를 평가한다.<br><br>네트워크 실패 시 이벤트 손실 허용 여부와 로컬 큐·재시도 정책을 정의한다. 적어도 한 번 전송(at-least-once) 시 이벤트 중복 제거(client-side event_id)도 함께 설계한다.<br><br>개인정보 측면에서 userId·event parameter 에 포함될 수 있는 PII 와 분석 목적 최소 수집 원칙을 검토한다.<br><br>백엔드 부하 — 전체 사용자 액션 이벤트를 받으면 트래픽이 크게 늘 수 있어 capacity planning·rate limiting·sampling 옵션을 사전 평가한다.<br><br>DEVT-134(Meta 측정 인프라 원점 재설계) 트랙과 정합되도록 설계 단계에서 트랙 담당자와 sync 한다. |
| 기대효과 | Meta 이벤트와 내부 집계값의 차이를 플랫폼·앱 버전·시점별로 원인 분석할 수 있어 측정 정확성·신뢰도가 향상된다.<br><br>안드로이드·iOS 구분 집계가 가능해져 광고 매체별 성과 측정과 캠페인 최적화 의사결정 정밀도가 올라간다.<br><br>향후 Meta 외 다른 광고 매체(Google Ads·TikTok Ads 등)와의 attribution 비교도 동일 인프라로 확장 가능하다. |

그리고 본문에서 큐 표기가 "Bull"(Day 2, 2장 ⑦, 3장 D6 둘째 셀, 5장 500 응답), "BullMQ"(3장 D6 첫째 셀, 5장 처리 정책), "Bull 큐"(5장 202 응답) 세 가지로 혼재되어 있습니다. 문맥상 7군데 모두 같은 비동기 큐 컴포넌트를 가리키는 것 같은데, 세 표기 모두 같은 것을 의도하신 게 맞는지 확인 부탁드립니다 — `bull`과 `bullmq`는 같은 저자의 별개 npm 패키지(`bullmq`가 후속작)라 단순 별칭은 아니라서요. 채택하신 패키지 기준으로 본문 표기 통일시켜 주세요.

마지막으로 D5와 D6 사이의 정합성 한 가지 짚어드립니다. D5에서 "dedup 없음, append-only"를 채택한 근거가 "ms까지 같은 (user·device·event·socialing/order) 조합은 사실상 발생 불가 → 자연 unique"였는데, D6에서 채택한 Bull/BullMQ는 기본적으로 at-least-once 보장이라 worker ack 실패·재시작 시 동일 job이 한 번 더 처리될 수 있습니다. 이 경우 사용자 행위가 두 번 일어난 게 아니라 시스템 내부 재처리로 동일 (user·event·ms) row가 두 번 적재되는 시나리오라, D5의 자연 unique 전제와는 결이 다릅니다. 정합성 검증을 위해 만든 시스템 자체가 중복으로 카운트를 부풀릴 수 있어 본 프로젝트 목적에 직접 영향이 갈 수 있으니, worker jobId 기반 dedup이나 자연 키 합성 unique constraint 같은 안전장치 도입을 검토해주시면 좋겠습니다.
