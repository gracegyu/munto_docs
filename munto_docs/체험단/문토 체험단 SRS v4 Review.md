## 1. 이전 리뷰(v1.0.0) 의견 반영 현황

## 2. v1.0.2 신규 리뷰 의견

### R-01 (P1). 7.6.2 이중 환불 방지 — 조건부 UPDATE 성공 후 Bootpay API 실패 시의 상태 처리

v1.0.2에서 추가된 이중 환불 방지 로직:

> `TrialRefund.status`를 `PENDING → REFUNDED`로 전환할 때 조건부 UPDATE를 사용하여 한쪽만 성공하도록 방어한다

이 로직의 실행 순서가 명확하지 않습니다. 두 가지 구현 방식이 가능한데:

**방식 A:** UPDATE(PENDING→REFUNDED) 먼저 → Bootpay API 호출
- Bootpay 실패 시 이미 REFUNDED 상태이므로, 재시도가 불가해짐 (이미 REFUNDED로 바뀌었으니 WHERE status='PENDING' 조건에 안 걸림)
- REFUNDED인데 실제 환불은 안 된 불일치 상태 발생

**방식 B:** Bootpay API 호출 먼저 → UPDATE(PENDING→REFUNDED)
- Bootpay는 성공했는데 DB UPDATE 전에 크래시하면, 재시도 시 Bootpay에 이중 환불 요청 가능
- 단, Bootpay cancel_id 멱등성 키로 방어 가능

**권장 사항:** 실행 순서를 SRS에 명시하고, 방식 A를 채택할 경우 Bootpay 실패 시 REFUNDED→PENDING 롤백 또는 REFUND_FAILED 상태 추가가 필요합니다. 방식 B를 채택할 경우 Bootpay cancel_id 멱등성 키를 활용한다는 점을 명시하면 됩니다. 이전 리뷰에서도 "REFUND_FAILED 상태 추가"를 제안했는데, v1.0.2에서 이 상태가 추가되지 않은 이유가 있다면 그 근거를 기재해 주시면 좋겠습니다.

### R-02 (P1). 6.5 초대권 환불 정책 테이블이 두 개 존재하며 내용이 중복됨

6.5 비즈니스 규칙에서 초대권 환불 정책 테이블(사용 전/후 + 결제 5일 기준)이 이미 상단에 있는데, 체험단 유저 취소/환불 정책 테이블 바로 아래에 동일한 테이블이 한번 더 나옵니다. 중복 기재로 보이므로 아래쪽 테이블을 삭제하면 될 것 같습니다.

### R-03 (P1). 7.2.4 초대권 소진 규칙 — "모임 3일 전부터는 모집을 취소해도 초대권은 소진된다"의 정의 불충분

> 모임 3일 전부터는 모집을 취소해도 초대권은 소진된다

여기서 "모집 취소"와 "초대권 소진"의 관계가 불명확합니다:

- "모집 취소"는 호스트가 체험단 모집 자체를 철회하는 것인지, 특정 멤버를 내보내는 것인지
- "초대권 소진"이 배정된 전체 초대권을 의미하는지(allocatedCount 전부), 실제 사용된 수량(usedCount)만 의미하는지
- 예: 5장 배정, 2장 소진(2명 신청) 상태에서 모임 3일 전에 모집을 취소하면 → 5장 모두 소진? 2장만 소진?

또한 이 규칙과 7.2.3 "1명이라도 신청한 이후에는 모집 조건 변경이 불가"의 관계도 모호합니다. 1명이라도 신청하면 모집 변경이 불가한데, "모집 취소"는 가능한 것인지, "모집 취소"도 "모집 변경"에 포함되는지 명확히 해 주시기 바랍니다.

**권장 사항:** "모집 취소"의 정의와, 취소 시 배정 초대권 중 어디까지가 소진 처리되는지 명확하게 기술

### R-04 (P1). 7.5.1 호스트가 "미참석" 선택 시 유저 보호 장치 부재

이전 리뷰(SRS 7.6.3 P1)에서 "호스트가 의도적으로 참여 확인을 하지 않는 경우 유저 보호 장치가 없다"고 지적했고, v1.0.1에서 24시간 암묵적 확정으로 해결했습니다. 그런데 새로운 문제가 생깁니다:

- 호스트가 24시간 내에 **의도적으로 "미참석"을 선택**하면, 실제 참석한 유저도 환급 대상에서 제외됩니다
- 현재 SRS에는 호스트의 "미참석" 선택에 대한 유저 이의 제기 절차가 없습니다
- 유저 입장에서는 모임에 참석하고 리뷰도 작성했는데 환급을 못 받는 상황이 발생할 수 있습니다

7.6.3에서 "운영팀이 필요 시 백오피스에서 수동 환급 가능"이라고 했지만, 유저가 이 상황을 인지하고 CS를 통해 이의를 제기하는 흐름이 SRS에 정의되어 있지 않습니다.

**권장 사항:** 호스트가 "미참석" 선택 시 유저에게 알림을 발송하고, 유저가 이의 제기할 수 있는 경로(CS 안내 또는 앱 내 이의 제기 버튼)를 정의하면 됩니다. 구현 범위가 부담되면 "미참석 선택 시 유저에게 알림 발송 + CS 안내"를 P1으로, "앱 내 이의 제기 기능"을 P3으로 분리하는 것도 방법입니다.

### R-05 (P1). 7.4.2 "동시에 하나의 체험단 모임만 참여 가능" 제약의 해제 시점

> 한 유저는 동시에 하나의 체험단 모임만 신청할 수 있다. 이미 다른 체험단에 신청(승인 대기 포함)한 유저는 추가 신청이 불가하다

이 제약이 해제되는 시점이 정의되어 있지 않습니다:

- 모임 종료 후 해제되는지?
- 모임 시작 후 해제되는지?
- 환급 완료 후 해제되는지?
- 승인제에서 거절당한 후 바로 다른 체험단에 신청할 수 있는지?
- 유저가 직접 취소(7.4.2 환불 정책에 따라)한 후 다른 체험단에 신청할 수 있는지?

체험단 대상 유저 조건이 "가입 30일 이내"이므로, 이 제약이 너무 오래 유지되면 체험단 참여 기회 자체를 잃게 됩니다.

**권장 사항:** "동시 참여 1건" 제약의 해제 조건(모임 종료, 취소/거절, 환급 완료 등)을 명시

### R-06 (P1). 2.3 스케줄러 — "참여 암묵적 확정"과 "환급 기한 처리"의 기준 시각 정의 필요

스케줄러에서 두 가지 시간 기반 처리가 있습니다:

- 참여 암묵적 확정: 모임 시작 시각 기준 24시간 후
- 환급 기한 처리: 모임 종료 7일 경과 시

그런데 스케줄러의 실행 주기가 정의되어 있지 않습니다. 매분? 매시간? 매일?

특히 "암묵적 확정 → 리뷰가 이미 작성되어 있으면 즉시 환급"이 가능한 구조에서, 스케줄러 실행 주기에 따라 환급 타이밍이 달라집니다. 매일 1회 실행이면 최대 24시간 지연, 매분이면 거의 즉시입니다.

**권장 사항:** 참여 암묵적 확정/환급 기한 처리 스케줄러의 cron 주기를 명시하고, 암묵적 확정 시 리뷰가 이미 작성되어 있으면 환급을 함께 트리거한다는 점을 기술

### R-07 (P2). 7.1.5 초대권 만료 — 배정 중인 초대권의 만료 처리

> 만료 시 미사용 초대권은 자동 소멸하며 환불이 불가하다

초대권이 소셜링에 배정된 상태(TrialRecruitment에 allocate됨)에서 만료일이 도래하면 어떻게 처리하는지가 정의되어 있지 않습니다:

- 배정 중인 초대권도 만료 소멸되는지?
- 만료 소멸되면 해당 TrialRecruitment의 allocatedCount를 줄이는지?
- 이미 모집 중인 소셜링의 체험단 모집이 중간에 축소되는지?

**권장 사항:** 배정 중인 초대권은 만료 대상에서 제외하고, 모임 종료 후 반환(RETURN) 시 이미 만료된 건은 반환 대신 만료 처리하는 방식이 자연스럽습니다. 어느 쪽이든 정책을 명시해 주시기 바랍니다.

## 4. 결론

v1.0.0 리뷰에서 제기한 P0 항목 4건이 모두 해결되었고, P1 항목도 대부분 반영되어 **문서 완성도가 크게 향상**되었습니다. 특히 본인인증 조건 통일, 체험단 유저 취소/환불 정책 추가, TrialTicketTransaction 원장 도입, 정산 규칙 구체화가 잘 이루어졌습니다.

v1.0.2에서 추가된 이중 환불 방지 로직(조건부 UPDATE)은 올바른 방향이지만, Bootpay API 실패 시의 상태 처리를 보완해야 합니다. 나머지 리뷰 의견도 정책 명확화 수준이므로, 피드백을 반영하면 **개발 착수 가능한 수준**입니다.
