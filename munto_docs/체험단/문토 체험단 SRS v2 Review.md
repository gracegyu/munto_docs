## 기존 리뷰 의견 반영 확인

v1.0.0 리뷰의 P0/P1 항목이 대부분 잘 반영되었습니다.

- **참여 확인 방식 (P0, Critical)**: QR → 호스트 수동 선택 + 24시간 암묵적 확정으로 변경. 호스트 미선택에 의한 유저 피해 방지까지 커버됨. **해결**
- **얼리버드+체험단 정책 충돌 (P0)**: 6.5와 7.2.1 문구가 통일됨 — "체험단 멤버에게 얼리버드 할인 미적용". **해결**
- **Order 필드 정합성 (P0)**: `orderType` 제거, `trialProductId IS NOT NULL + grade=TRIAL 조인`으로 일관성 확보. **해결**
- **TrialTicket 만료 관리 (P0)**: `TrialTicketTransaction` 원장 도입으로 구매 건별 만료일(expireAt) 추적 가능. **해결**
- **체험단 유저 취소/환불 (P0)**: 6.5에 상세 정책 테이블 추가. **해결**
- **모임 폐강 시 처리 (P0)**: 전액 환불 + 초대권 전액 반환 명시. **해결**
- **본인인증 조건 (P0)**: "노출 조건 아님, 신청 시점 확인" 확정 및 전체 통일. **해결**
- **환급 비율 스냅샷 (P1)**: `TrialRefund.refundRate`에 결제 시점 스냅샷 저장. **해결**
- **정산 분리 (P1)**: `grade != TRIAL` 조건 추가, 별도 스케줄러 불필요로 명확화. **해결**
- **우선순위 재분류 (P2)**: 리뷰 독려, 통계 대시보드 등이 P2로 분류됨. **해결**

## v1.0.1 변경 사항 리뷰

### Comment 1: 참여 확인 → 환급 병렬 조건 설계가 좋습니다

호스트 참석 확정과 리뷰 작성을 병렬 조건으로 변경한 것이 좋습니다. 기존에는 "참여 확인 → 리뷰 작성"으로 선행 관계였는데, 이제 둘 중 나중에 충족되는 시점에 환급이 실행되므로 유저가 리뷰를 먼저 쓸 수 있고, 호스트 지연으로 인한 유저 불편이 줄어듭니다. 24시간 암묵적 확정도 호스트 의도적 미선택에 대한 합리적인 보호 장치입니다.

---

### Comment 2: 동시성 제어 — 조건부 UPDATE만으로 충분한지 확인 필요

2.6에서 "기존 소셜링 정원 관리와 동일한 조건부 UPDATE 패턴"으로 oversell을 방지한다고 했는데, 초대권은 정원과 달리 **잔액 차감(TrialTicket) + 소진 카운트 증가(TrialRecruitment.usedCount)** 두 곳에 동시에 영향을 줍니다. 단순 조건부 UPDATE만으로 두 테이블 간 정합성이 보장되는지, 트랜잭션 범위를 명시해 주세요. 기존 소셜링 정원 관리 코드의 동시성 패턴이 구체적으로 어떤 것인지도 참조 코드 위치를 남겨두면 구현자에게 도움이 됩니다.

---

### Comment 3: 체험단 멤버가 정렬 알고리즘 멤버 수에 포함되는 것이 맞는지

2.8에서 "체험단 멤버(grade=TRIAL)는 기존 알고리즘의 멤버 수 계산에 포함된다"고 했는데, 이러면 체험단 모집이 활성화된 소셜링이 일반 목록에서 "인원이 차고 있는 인기 모임"으로 보일 수 있습니다. 이것이 의도된 것인지, 기획에서 일반 유저에게 체험단 멤버를 포함한 인원을 보여주는 것이 맞는지 확인해 주세요.

---

### Comment 4: 환급 조건 동시 충족 시 이중 실행 방어

7.6.1에서 "호스트 확정이 먼저이면 리뷰 작성 시 환급, 리뷰가 먼저이면 호스트 확정 시 환급"이라고 했는데, 두 이벤트가 **거의 동시에** 발생하면 환급이 이중 실행될 수 있습니다. `TrialRefund` 상태를 `PENDING → REFUNDED`로 원자적으로 전환하는 방어 로직(조건부 UPDATE 또는 낙관적 잠금)이 필요합니다. 간단하게라도 명시해 주세요.

---

### Comment 5: 변경 이력에 "ERD 수정(초대권 트랜잭션 테이블 추가)" 기재되어 있으나 ERD 문서 확인 필요

Jira comment에 "ERD 수정"이 변경 사항에 포함되어 있는데, ERD 문서(`erd.md`)도 함께 업데이트된 것인지 확인해 주세요. SRS에는 `TrialTicketTransaction` 원장이 잘 기술되어 있으나, ERD에도 동일하게 반영되어야 합니다.

## 종합

v1.0.0 대비 큰 폭으로 개선되었습니다. 참여 확인 방식 변경, 환급 조건 병렬화, 정산 규칙 명확화, TrialTicketTransaction 원장 도입 모두 좋은 설계입니다. 기존 P0 이슈 7건이 전부 해결되었고, P1도 대부분 반영됐습니다.

위 4건(Comment 2~5)을 보완하면 구현 착수 가능합니다.
