### Comment 1: 7.2.1 채팅방 생성 - 생성 프로세스

**위치:** `7.2.1 채팅방 생성` → 각 채팅방 생성 프로세스 부분

**Comment 내용:**

```
⚠️ 채팅방 생성 시 서버 선택 로직 명시 필요

채팅방 생성 시 DB에 서버 정보가 저장되는 것으로 이해되지만,
서버 선택 로직이 명시되지 않았습니다.

현재 상황:
- 클라이언트는 채팅방 조회 시 DB에 저장된 서버 정보를 받아 연결 (문제 없음)
- 하지만 채팅방 생성 시 어떤 서버를 선택하는지 불명확

권장사항:
- 채팅방 생성 시 서버 선택 로직 명시
  - Consistent Hashing 권장 (서버 확장 시 최소한의 재매핑)
  - 해시 함수 명시 (예: SHA-256)
  - 또는 고정값 반반 분배 (서버 2대 기준, 예: roomId % 2) - 단, 서버 확장 시 많은 재매핑 발생
- DB 스키마에 서버 정보 필드 포함 여부 확인
- 채팅방 조회 API 응답에 서버 정보 포함 여부 명시

참고: 부하 기반 동적 분배는 2.7장 향후 요구사항에 포함 예정
```

### Comment 3: 7.1.2 WebSocket 연결 관리 - 연결 수립

**위치:** `7.1.2 WebSocket 연결 관리` → "연결 수립" 부분

```

- 다중 채팅방 지원: 사용자가 여러 서버의 채팅방에 참여할 경우, 각 서버별로 독립적인 WebSocket 연결 관리
- 예: 채팅방 1(서버 A), 채팅방 2(서버 B)에 참여 시 → 서버 A와 서버 B에 각각 연결

명시해주시면 개발 시 혼란이 없을 것 같습니다.
```

서버 확장성을 고려하여 선택한 알고리즘을 명시해주세요.

참고: 부하 기반 동적 분배는 2.7장 향후 요구사항에 포함 예정

### Comment 5: 6.1.2 부하 분산 및 확장성

**위치:** `6.1.2 부하 분산 및 확장성` 부분

**Comment 내용:**

```
💡 서버 확장/축소 시 마이그레이션 전략 명시 권장

"향후 서버 확장 예정"이라고 했으므로, 서버 추가/제거 시 기존 채팅방 처리 방법을 명시하면 좋겠습니다.

Consistent Hashing 사용 시:
- 서버 추가/제거 시 최소한의 채팅방만 재매핑 (약 1/N, N=서버 수)
- 마이그레이션 대상 채팅방 식별 방법
- 마이그레이션 프로세스 (연결 유지 vs 재연결)
- 메시지 손실 방지 방법

고정값 분배 사용 시:
- 서버 추가 시 대부분의 채팅방이 재매핑됨
- 서버 확장 시점 전략: 새 채팅방만 새 서버에 할당, 기존 채팅방은 원래 서버 유지

현재는 필수는 아니지만, 향후 개선 계획에 포함되면 좋겠습니다.
```

## 🟢 P2: 개선 권장 사항

### Comment 2: 6.3.1 Availability - Failover Flow

**위치:** `6.3.1 Availability` → "Failover Flow" 부분

**Comment 내용:**

```
💡 서버 장애 시 DB 서버 정보 업데이트 프로세스 명시 권장

Failover Flow는 이미 잘 정의되어 있지만, DB의 서버 정보 업데이트 프로세스가 명시되지 않았습니다.

현재 Failover Flow:
- 서버 B로 자동 재연결
- 서버 B에서 데이터베이스를 통해 모든 채팅 데이터 조회

추가 명시 권장:
- 서버 장애 감지 시 DB의 채팅방 서버 정보를 다른 서버로 업데이트하는 프로세스
- 또는 클라이언트 재연결 시 DB에서 업데이트된 서버 정보를 조회하여 연결하는 방식
- 서버 복구 후 원래 서버로 복귀 여부 결정 (복귀 시 DB 업데이트 필요)

명시해주시면 Failover 프로세스가 더 명확해질 것 같습니다.
```
