# 채팅 서비스 SRS 리뷰 보고서 v2

## 1. 전체 평가

Redis 제거 및 채팅방별 고정 서버 연결 방식이 반영된 SRS를 재검토한 결과, 전반적으로 설계가 명확해졌습니다. 다만 몇 가지 보완이 필요한 부분이 있습니다.

**전체 점수: 88/100**

## 2. Redis 제거 및 채팅방별 고정 서버 연결 방식 반영 여부

### 2.1 ✅ 잘 반영된 부분

1. **Redis 미사용 명시**: 6.1.2에서 "Redis 등 외부 캐시 서버는 사용하지 않음" 명확히 기술
2. **채팅방 해시 기반 서버 분배**: 7.1.2에서 "채팅방 번호(`roomId`)를 해시하여 채팅 서버 중 적절한 서버에 분배" 명시
3. **동일 채팅방 동일 서버 처리**: "동일한 채팅방의 모든 메시지는 항상 같은 서버에서 처리" 명시

### 2.2 ⚠️ 보완이 필요한 부분

#### 🟡 **클라이언트 서버 선택 메커니즘 불명확**

**현황:**
- 2.3에서 "클라이언트가 채팅 서버에 채널 URL 요청 (REST API)" → "채팅 서버에서 채널 URL 및 WebSocket 연결 정보 제공"
- 하지만 채널 URL 요청 시 어떤 서버로 요청하는지, 그리고 그 서버가 어떻게 올바른 서버를 결정하는지 불명확

**시나리오:**
```
사용자가 채팅방 1에 접속하려고 함
→ 채팅방 1의 해시값으로 서버 A가 결정됨
→ 클라이언트는 어떻게 서버 A의 URL을 알 수 있는가?
```

**권장사항:**
1. **옵션 1: 로드밸런서가 해시 기반 라우팅**
   - 클라이언트는 항상 로드밸런서로 요청
   - 로드밸런서가 `roomId`를 기반으로 해시하여 적절한 서버로 라우팅
   - 이 경우 로드밸런서 설정 및 해시 알고리즘 명시 필요

2. **옵션 2: REST API가 서버 정보 제공**
   - 클라이언트가 채팅방 ID와 함께 채널 URL 요청
   - REST API 서버가 `roomId` 해시하여 적절한 서버 결정
   - 해당 서버의 WebSocket URL 반환
   - 이 경우 REST API 서버가 모든 서버 정보를 알고 있어야 함

**추가 필요 사항:**
- 채널 URL 요청 API 명세에 `roomId` 파라미터 및 서버 선택 로직 명시
- 해시 알고리즘 명시 (예: consistent hashing, modulo 등)

#### 🟡 **서버 장애 시 채팅방 재연결 전략 불명확**

**현황:**
- 6.3.1에서 Failover Flow 설명
- "기존 서버 A에 연결된 클라이언트는 연결 끊김 감지 후 서버 B로 자동 재연결"
- 하지만 채팅방별 고정 서버 방식에서는 문제 발생 가능

**문제 시나리오:**
```
채팅방 1 → 서버 A (해시 결과)
서버 A 장애 발생
→ 채팅방 1의 참여자들이 서버 B로 재연결 시도
→ 하지만 해시 알고리즘에 따르면 채팅방 1은 여전히 서버 A에 매핑됨
→ 서버 A가 복구되기 전까지 채팅방 1은 사용 불가?
```

**권장사항:**
1. **임시 Failover 전략 명시**
   - 서버 장애 시 해당 서버의 채팅방들을 다른 서버로 임시 재매핑
   - 해시 테이블 업데이트 또는 서버 목록에서 장애 서버 제외
   - 서버 복구 후 원래 서버로 복귀 여부 결정 (복귀 시 마이그레이션 필요)

2. **또는 Graceful Degradation**
   - 서버 장애 시 해당 채팅방은 읽기 전용 모드
   - 새 메시지 전송은 불가하지만 기존 메시지 조회는 가능
   - 서버 복구 후 정상 동작

3. **Health Check 및 자동 Failover 로직 명시**
   - 서버 장애 감지 방법
   - 채팅방 재매핑 트리거 조건
   - 재매핑 시 기존 연결 처리 방법

#### 🟡 **서버 확장/축소 시 채팅방 마이그레이션 전략 미명시**

**현황:**
- "향후 서버 확장 예정" 언급
- 하지만 서버 추가/제거 시 기존 채팅방 처리 방법 미명시

**문제 시나리오:**
```
현재: 서버 2대 (서버 A, 서버 B)
채팅방 1 → 서버 A (해시 결과)

서버 추가: 서버 3대 (서버 A, 서버 B, 서버 C)
→ 해시 알고리즘 변경 시 채팅방 1이 서버 C로 재매핑될 수 있음
→ 기존 서버 A에 연결된 사용자들은 어떻게 처리?
```

**권장사항:**
1. **Consistent Hashing 사용 권장**
   - 서버 추가/제거 시 최소한의 채팅방만 재매핑
   - 마이그레이션 대상 채팅방 식별 방법
   - 마이그레이션 프로세스 (연결 유지 vs 재연결)

2. **또는 서버 확장 시점 전략**
   - 새 채팅방만 새 서버에 할당
   - 기존 채팅방은 원래 서버 유지
   - 점진적 마이그레이션 (비활성 채팅방부터)

3. **서버 축소 시 전략**
   - 제거될 서버의 채팅방을 다른 서버로 마이그레이션
   - 마이그레이션 중 메시지 손실 방지 방법
   - 다운타임 최소화 전략

## 3. 아키텍처 설계 검토

### 3.1 ✅ 잘 설계된 부분

1. **단순성**: Redis 제거로 아키텍처 단순화, 운영 복잡도 감소
2. **일관성**: 같은 채팅방의 모든 참여자가 같은 서버에 연결되어 메시지 동기화 문제 없음
3. **확장성**: 해시 기반 분배로 서버 확장 가능

### 3.2 ⚠️ 잠재적 문제점

#### 🟡 **서버 부하 불균형 가능성**

**문제:**
- 채팅방별로 인원 수가 다름 (1:1 DM vs 그룹 채팅)
- 특정 서버에 대규모 채팅방이 집중될 경우 부하 불균형

**예시:**
```
서버 A: 100명 채팅방 1개, 10명 채팅방 5개 = 총 150명
서버 B: 2명 DM 채팅방 50개 = 총 100명
→ 서버 A의 부하가 더 높음
```

**권장사항:**
- 해시 알고리즘 선택 시 부하 분산 고려
- 또는 채팅방 인원 수를 고려한 가중치 적용
- 모니터링을 통한 부하 불균형 감지 및 대응 전략

#### 🟡 **서버별 독립적인 메모리 상태**

**현황:**
- 각 서버가 독립적으로 Socket.IO Room 관리
- 서버 간 상태 공유 없음 (Redis 미사용)

**잠재적 문제:**
- 서버 장애 시 해당 서버의 모든 채팅방 연결 끊김
- 서버 재시작 시 메모리 상태 손실 (DB에서 복구 가능하지만)

**권장사항:**
- 서버 재시작 시 DB에서 채팅방 정보 및 참여자 정보 복구 프로세스 명시
- 연결 상태는 DB에 저장하지 않으므로 재연결 필요 (이미 명시됨)

## 4. 기능 요구사항 검토

### 4.1 ✅ 잘 정의된 부분

기존 리뷰와 동일하게 기능 요구사항은 매우 상세하게 정의되어 있습니다.

### 4.2 ⚠️ 추가 보완 필요 사항

#### 🟡 **다중 채팅방 참여 시 연결 관리**

**시나리오:**
```
사용자가 여러 채팅방에 참여 중
- 채팅방 1 → 서버 A
- 채팅방 2 → 서버 B
- 채팅방 3 → 서버 A

사용자는 어떻게 두 서버에 동시에 연결?
```

**현황:**
- SRS에서 다중 채팅방 참여 시나리오 명시 없음
- "다중 기기 지원: 각 기기별로 독립적인 Socket.IO 연결 관리" (7.1.2)만 언급

**권장사항:**
1. **옵션 1: 채팅방별 독립 연결**
   - 각 채팅방마다 해당 서버에 별도 WebSocket 연결
   - 복잡도 증가하지만 서버 장애 시 영향 최소화

2. **옵션 2: 주 서버 하나만 연결**
   - 사용자가 참여한 채팅방 중 하나의 서버에만 연결
   - 다른 서버의 채팅방은 REST API로만 조회
   - 실시간 메시지는 해당 서버의 채팅방만 수신

3. **옵션 3: 게이트웨이 서버**
   - 클라이언트는 게이트웨이 서버 하나에만 연결
   - 게이트웨이가 각 채팅방 서버와 연결하여 메시지 중계
   - 이 경우 게이트웨이가 단일 장애점이 될 수 있음

**권장: 옵션 1 또는 2 중 선택하여 명시**

## 5. 성능 및 확장성

### 5.1 ✅ 장점

1. **단순한 아키텍처**: Redis 제거로 운영 복잡도 감소
2. **메시지 동기화 불필요**: 같은 채팅방은 같은 서버에서 처리하므로 서버 간 동기화 불필요
3. **확장 가능**: 해시 기반으로 서버 추가 가능

### 5.2 ⚠️ 고려 사항

#### 🟡 **서버당 최대 연결 수 제한**

**현황:**
- 한 채팅방 최대 인원: 100명
- 하지만 서버당 총 연결 수 제한 미명시

**권장사항:**
- 서버당 최대 WebSocket 연결 수 제한 명시
- 제한 초과 시 처리 방법 (새 서버 추가, 연결 거부 등)

#### 🟡 **해시 알고리즘 선택**

**현황:**
- "채팅방 번호 해시"만 언급
- 구체적인 해시 알고리즘 미명시

**권장사항:**
- Consistent Hashing 권장 (서버 추가/제거 시 최소 재매핑)
- 또는 Modulo 연산 (단순하지만 서버 추가 시 많은 재매핑)
- 해시 함수 명시 (예: SHA-256, MD5 등)

## 7. 결론

Redis 제거 및 채팅방별 고정 서버 연결 방식은 아키텍처를 단순화하고 메시지 동기화 문제를 해결하는 좋은 설계입니다. 다만, **클라이언트가 올바른 서버를 찾는 방법**, **서버 장애 시 처리 방법**, **다중 채팅방 참여 시 연결 관리**가 명확하지 않아 개발 시 혼란이 발생할 수 있습니다.
