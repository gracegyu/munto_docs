# 데이팅 매칭 알고리즘 v3 — AI/ML 전환 제안서

- **작성일:** 2026-03-26
- **대상 독자:** 대표, 개발팀
- **관련 문서:** 데이팅 알고리즘 개선 SRS (v2), 데이팅 노출 알고리즘 개선 기획

---

## 1. 요약(Executive Summary)

데이팅 서비스의 핵심은 **"이 사람에게 누구를 보여줄 것인가"** 입니다. 이 한 가지 결정이 매칭 성사율, 유저 만족도, 재방문율, 그리고 매출을 직접 좌우합니다.

현재 우리는 이 결정을 **사람이 머리로 만든 수식**에 맡기고 있습니다. v1에서는 가중합 4요소, v2에서는 베이지안 점수로 바꿨지만, 본질은 같습니다. 개발자가 "이 공식이 맞을 것 같다"고 추측한 규칙의 집합입니다. 어떤 규칙을 만들든, **수만 명의 개인별 취향을 몇 개의 수식으로 표현하는 것 자체가 불가능**합니다.

Tinder, Bumble, Hinge 등 글로벌 데이팅 앱들은 이미 AI/ML(인공지능/머신러닝) 기반 추천으로 전환했습니다. 규칙 기반 알고리즘으로 경쟁하는 것은, 계산기로 슈퍼컴퓨터와 경쟁하는 것과 같습니다.

**v3에서 AI 전환은 선택이 아니라 생존의 문제입니다.** 다행히 v1부터 AI 학습을 대비한 로그 수집 구조(누구에게 누구를 보여줬고, 그때 양쪽의 프로필/사진, 이후 행동 결과)를 이미 구축해 두었습니다. v2에서도 이 로그 수집을 유지하고 v2에서 새로 추가되는 요소까지 확장해야, v3 진입 시 충분한 학습 데이터가 준비됩니다.

## 2. 현재 방식의 구조적 한계

### 2.1 사람이 만든 수식의 한계

v1에서 v2로 전환하면서 점수 체계를 개선했지만, 근본적으로 달라진 것은 없습니다.

| 항목 | v1 | v2 | 본질 |
|------|----|----|------|
| 점수 수식 | 가중합 4요소 | 베이지안 단일 점수 | 사람이 만든 수식 |
| 신규 유저 | 7일 보너스 | Prior 고정값 (0.30/0.35) | 사람이 정한 초기값 |
| 노출 규칙 | 성별 구분 없음 | 7유형 분기 | 사람이 정한 규칙 |
| 파라미터 조정 | 수동 | system-config 수동 | 사람이 최적값 추측 |

v2의 베이지안 점수에서 남성 초기값을 0.30으로 정한 근거는 무엇입니까? 나이차 +3살이 피크인 근거는? 신규/정착중 여성 비율 75%의 근거는? 모두 **"이 정도가 적절할 것 같다"** 는 추측입니다. A/B 테스트로 검증한다고 해도, 테스트할 수 있는 조합의 수는 제한적이고, 30개 이상의 파라미터를 동시에 최적화하는 것은 사람의 능력으로는 불가능합니다.

### 2.2 개인화 불가

현재 시스템은 모든 유저를 7개 유형으로 분류하고, 같은 유형에는 같은 규칙을 적용합니다. 그러나 같은 "ESTABLISHED 남성"이라도 한 사람은 연상을 선호하고, 다른 사람은 같은 취미를 가진 상대를 원합니다. 7개 유형으로는 이런 개인별 차이를 표현할 수 없습니다.

### 2.3 반복되는 개발 비용

알고리즘을 바꿀 때마다 다음이 반복됩니다:

- SRS 재작성 (현재 v2 SRS: 1,727줄)
- 백오피스 테스트 페이지 전면 재작성 (기존 가중합 UI 전부 무효화)
- 새 점수 체계에 맞는 DB 스키마 재설계
- 파라미터 튜닝을 위한 수개월간의 A/B 테스트

이 비용은 v3, v4로 갈수록 계속 증가합니다. 규칙이 복잡해질수록 SRS도 길어지고, 수정 범위도 넓어집니다.

### 2.4 신규 유저 cold start 문제

현재 방식에서 신규 유저는 성별에 따라 고정 초기값(남성 0.30, 여성 0.35)을 받습니다. 프로필 사진이 매력적이든, 자기소개가 성실하든, 관심사가 풍부하든 **모두 같은 점수에서 시작**합니다. 신규 유저에게 가장 중요한 첫 경험을 "데이터가 없으니 일단 기본값"으로 처리하는 것입니다.

## 3. AI 기반 추천은 무엇이 다른가

### 3.1 비기술적 비유

**현재 방식**: 식당 직원이 메뉴판을 보고 "남성이니까 스테이크, 여성이니까 샐러드"처럼 정해진 규칙으로 추천

**AI 방식**: 경험 많은 소믈리에가 손님의 표정, 대화, 과거 방문 이력, 비슷한 취향의 다른 손님 데이터를 종합하여 "이분에게는 이 와인이 맞겠다"고 판단

### 3.2 기술적 비교

| 항목 | 규칙 기반 (v1, v2) | AI/ML 기반 (v3) |
|------|-------------------|-----------------|
| 최적값 탐색 | 사람이 추측 후 A/B 테스트 | 데이터에서 자동 학습 |
| 신규 유저 | 고정 초기값 | 프로필 사진, 자기소개, 유사 유저 패턴으로 예측 |
| 개인화 | 7유형 일괄 적용 | 유저별 개인화 추천 |
| 알고리즘 개선 | 수식 변경 → SRS → 개발 → 배포 | 모델 재학습 (코드 변경 최소) |
| 고려 변수 | 개발자가 정의한 수 개 | 수백 개 피처(특성)를 동시에 고려 |
| 파라미터 최적화 | 30개를 사람이 하나씩 조정 | 수천 개를 자동 최적화 |

### 3.3 신규 유저 cold start 해결

AI 방식에서는 신규 유저가 가입하는 순간, 다음 정보를 활용하여 매칭 예측이 가능합니다:

- **프로필 사진**: 이미지 임베딩(사진을 숫자 벡터로 변환)으로 비슷한 외형의 기존 유저가 어떤 반응을 받았는지 참조
- **자기소개 텍스트**: 텍스트 임베딩으로 관심사, 성격, 라이프스타일 유사도 분석
- **프로필 정보**: 나이, 지역, 학력, MBTI, 관심사 등을 종합한 유사 유저군의 과거 매칭 패턴
- **행동 패턴**: 가입 후 첫 몇 번의 좋아요/패스만으로도 빠르게 선호도 학습

고정 초기값 0.30이 아니라, **"이 유저와 비슷한 프로필의 기존 유저들은 이런 상대에게 좋아요를 받았다"** 는 데이터 기반 예측이 가능합니다.

## 4. 이미 시작된 AI 준비 — v1의 로그 수집

v1부터 AI 학습을 대비한 로그 수집 구조를 구축해 두었습니다:

- **Input(추천 시점)**: 누구에게 누구를 보여줬는지, 그때 양쪽 유저의 모든 프로필 정보와 사진 URL
- **Output(이후 행동)**: 상세 프로필 조회 여부, 좋아요/슈퍼좋아요/싫어요, 채팅방 생성 여부 등 이후 행동 전체

이 로그는 그 자체로 **"A를 B에게 보여줬을 때 어떤 결과가 나왔는가"** 라는 학습 데이터셋입니다. v2에서 알고리즘 구조가 바뀌더라도, 이 로그 수집 구조는 반드시 유지되어야 합니다.

### v2에서 추가로 수집해야 할 항목

v2에서는 v1에 없던 새로운 개념과 데이터가 도입됩니다. AI 학습 데이터의 풍부함을 위해 다음 항목을 로그에 추가해야 합니다:

| 추가 항목 | v2에서 새로 도입된 이유 | AI 학습에서의 활용 |
|-----------|----------------------|------------------|
| 유저 분류 상태 (NEW/SETTLING/ESTABLISHED) | v2 신규 개념 | 유저 성숙도에 따른 반응 패턴 차이 학습 |
| 뷰어 유형 (7유형 중 어느 것) | v2 신규 개념 | 노출 맥락(context)에 따른 반응 차이 학습 |
| 버킷 ID, percentile | v2 신규 개념 | 상대적 매력도 위치가 반응에 미치는 영향 학습 |
| 가입 경과일 (D+N) | 분류 전환 기준 | 서비스 이용 기간에 따른 선호도 변화 학습 |
| 노출 제한 여부 및 트리거 유형 | v2 신규 개념 | 제한 해제 후 반응 패턴 학습 |
| impression → 행동까지의 소요 시간 | v2에서 impression 기반으로 변경 | 관심도 강도 학습 (빠른 좋아요 vs 오래 본 후 좋아요) |
| 사이클 번호 및 슬롯 유형 (추천/발견) | v2에서 통합 사이클로 변경 | 추천/발견 탭별 반응 차이 학습 |
| 프로필 변경 이력 (변경 전/후 시점) | v2에서 완전 초기화로 변경 | 프로필 변경 전후 반응 변화 학습 |

나중에 AI로 전환하겠다고 결정한 뒤 로그를 보완하면, **과거 데이터에는 빠진 항목이 있어서 학습에 쓸 수 없습니다**. v2 개발 시점에 위 항목을 로그에 포함해 두면, v3 진입 시 이미 수개월치의 풍부한 학습 데이터가 준비되어 있습니다.

## 5. v3 로드맵

### 단계별 전환 계획

| 단계 | 시점 | 내용 | 핵심 목표 |
|------|------|------|----------|
| **v2 (현재)** | 진행 중 | 베이지안 점수 + 규칙 기반 파이프라인 | 서비스 품질 개선 + **데이터 축적 시작** |
| **v2.5 (과도기)** | v2 안정화 후 | 규칙 기반 베이스라인 유지 + ML 모델이 일부 요소 대체 | 규칙 vs ML 비교 검증 |
| **v3 (목표)** | v2.5 검증 완료 후 | ML 모델이 점수 산출의 중심 | 실시간 개인화 추천 |

### v2에서 선행 준비 사항 (추가 비용 최소)

v1에서 구축한 AI 학습용 로그 기반 위에, v2 개발 과정에서 다음을 함께 설계하면 큰 추가 비용 없이 v3를 준비할 수 있습니다:

1. **기존 AI 학습 로그 유지 + v2 신규 항목 확장**: v1의 로그 수집 구조(프로필/사진/행동 결과)를 그대로 유지하면서, v2에서 새로 도입되는 유저 분류, 뷰어 유형, 버킷, percentile, 가입 경과일, 노출 제한 정보, 사이클/슬롯 정보를 추가 기록 (4장 표 참조)
2. **피처 스토어 설계**: 인터랙션 데이터를 ML 학습에 바로 쓸 수 있는 형태로 저장하는 구조 설계
3. **A/B 테스트 인프라 확장**: 현재 7.10에서 설계한 실험 인프라를 ML 모델 비교가 가능한 수준으로 확장

### v2.5에서 검증할 것

규칙 기반을 한 번에 ML로 교체하는 것은 위험합니다. v2.5에서는 다음과 같이 **부분 교체 + A/B 비교**로 검증합니다:

| 교체 대상 | 규칙 기반 (현재) | ML 대체 | 비교 지표 |
|-----------|-----------------|---------|----------|
| 나이차 점수 | 고정 점수표 | 학습된 선호 모델 | 좋아요 비율 |
| 가점 항목 | MBTI/키워드 고정 가점 | 유사도 임베딩 | 매칭 성사율 |
| 신규 유저 초기 점수 | Prior 고정값 | 프로필 임베딩 예측 | 신규 유저 1주일 리텐션 |

### v3 최종 목표

- 추천 모델이 점수 산출의 중심 (규칙은 안전장치로만 유지)
- 유저별 실시간 개인화 추천
- 프로필 임베딩 기반 cold start 해결
- 모델 재학습으로 알고리즘 자동 개선 (SRS 재작성 불필요)

## 6. 예상 리소스 및 비용

### 인력 및 인프라

| 항목 | 내용 | 시점 |
|------|------|------|
| ML 엔지니어 | 1명 (채용 또는 외부 자문) | v2.5 진입 시 |
| 학습 인프라 | 클라우드 GPU 인스턴스 (AWS SageMaker, GCP Vertex AI 등) | v2.5 진입 시 |
| v2 선행 준비 | 기존 개발팀이 데이터 파이프라인 설계 (추가 인력 불필요) | v2 진행 중 |
| 이미지/텍스트 임베딩 | 사전 학습된 모델 활용 (자체 학습 불필요) | v2.5 |

### 학습 비용 — "AI는 비싸다"는 오해

"AI 학습에 수억 원이 든다"는 이야기는 ChatGPT 같은 대규모 언어모델(파라미터 수천억 개)을 처음부터 학습하는 경우입니다. 우리가 하려는 **추천 모델 학습**은 그것과 규모가 완전히 다릅니다.

| 항목 | LLM (ChatGPT 등) | 추천 모델 (우리 규모) |
|------|------------------|---------------------|
| 학습 데이터 | 인터넷 전체 텍스트 | 우리 서비스 로그 수백만 건 |
| 파라미터 수 | 수십억~수천억 개 | 수만~수십만 개 |
| 학습 시간 | 수주~수개월 | 수시간~하루 |
| GPU 비용 | 수억~수십억 원 | **월 수십만 원 수준** |

- 클라우드 GPU 인스턴스(예: AWS `ml.g4dn.xlarge`)를 시간당 약 $0.7~1.0에 사용 가능
- ACTIVE 유저 ~9,500명 규모의 인터랙션 데이터 학습에 수 시간이면 충분
- 매일 재학습하더라도 **월 7~30만 원** 수준
- GPU 서버를 직접 운영할 필요 없음 — 학습할 때만 클라우드 인스턴스를 켜고, 끝나면 종료
- 서비스 중 예측(추론)은 CPU로 처리 가능하므로 기존 서버 인프라에서 운영 가능
- 프로필 사진/텍스트 임베딩은 사전 학습된 모델(OpenAI API, HuggingFace 등)을 호출하는 방식이므로 자체 GPU 학습 불필요

### 규칙 기반 유지 비용 vs AI 전환 비용

오히려 **규칙 기반 알고리즘을 계속 수동으로 개선해 나가는 것이 더 비싸고, 효과는 더 낮습니다.**

| 비용 항목 | 규칙 기반 (v2, v3, v4... 계속) | AI 기반 (v3 전환 후) |
|-----------|-------------------------------|---------------------|
| 알고리즘 개선 시 | SRS 재작성 + 개발 + QA + 배포 (매번 수개월) | 모델 재학습 (수시간, 코드 변경 최소) |
| 파라미터 튜닝 | 30개 이상을 사람이 하나씩 A/B 테스트 (수개월) | 자동 최적화 (학습 1회로 수천 개 동시 조정) |
| 백오피스 테스트 페이지 | 알고리즘 변경 시 매번 전면 재작성 | 모델이 바뀌어도 입출력 인터페이스 유지 |
| 개발자 공수 | 수식 설계 + SRS + 구현 + 튜닝에 전담 투입 | 초기 구축 후 유지보수 최소 |
| 인프라 비용 | 추가 없음 | 월 7~30만 원 (클라우드 GPU) |
| **추천 품질** | **개인화 불가, 7유형 일괄 적용** | **유저별 개인화, 자동 개선** |

규칙 기반을 계속 유지하면, v3, v4를 만들 때마다 SRS 재작성(v2 기준 1,727줄), 테스트 페이지 재작성, 수개월간의 A/B 테스트가 반복됩니다. 개발자 인건비와 기회비용을 감안하면, **매 버전 수천만 원 이상의 비용이 반복적으로 발생**합니다. 반면 AI는 초기 구축 비용 이후 모델 재학습만으로 알고리즘이 자동 개선되며, 인프라 비용은 월 수십만 원에 불과합니다.

**비용은 AI가 더 낮고, 추천 품질은 AI가 압도적으로 높습니다.** 규칙 기반을 계속 유지하는 것은 비싼 비용을 지불하면서 낮은 품질에 머무르는 것입니다.

---

## 7. 결론

규칙 기반 알고리즘은 **v2가 마지막**이어야 합니다. v3부터는 AI/ML 기반으로 전환하지 않으면, 매번 수식을 바꾸고 SRS를 재작성하고 테스트 페이지를 다시 만드는 순환이 반복됩니다. 그리고 그 순환의 끝에서도 **개인화된 추천은 불가능**합니다.

v2는 그 자체로 서비스 품질을 개선하는 의미가 있지만, 동시에 **v3을 위한 데이터 축적 기간**이기도 합니다. 지금 데이터 파이프라인을 설계해 두면, v3 진입 시 이미 학습 재료가 준비되어 있습니다. 이 준비를 하지 않으면, v3를 시작할 때 또다시 처음부터 데이터를 모아야 합니다.

**v2를 잘 만들되, v3(AI)를 준비하는 것. 이것이 이 제안서의 핵심입니다.**
