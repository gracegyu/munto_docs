# 데이팅 매칭 알고리즘 v3 — AI Modular Pipeline 초기 설계서

- **작성일:** 2026-03-26
- **문서 성격:** 아이디어 수준 초안 (구현 스펙 아님)
- **관련 문서:** 데이팅 알고리즘 v3 AI 전환 제안서, 데이팅 알고리즘 개선 SRS (v2)

---

## 1. 목표

두 유저(A, B)를 지정하면 **서로 매칭될 확률**을 예측하는 AI 엔진을 설계한다.

- A→B 방향: A가 B를 봤을 때 좋아요를 보낼 확률
- B→A 방향: B가 A를 봤을 때 좋아요를 보낼 확률
- **상호 매칭 확률** = P(A→B) × P(B→A)

이 예측값을 기반으로 매칭 성사 가능성이 높은 상대를 우선 노출하는 것이 최종 목표이다.

---

## 2. 전체 파이프라인 개요

```
[Stage 1]          [Stage 2]          [Stage 3]         [Stage 4]         [Stage 5]
하드 필터    →    피처 추출     →    매칭 확률 예측  →   랭킹/최적화   →   사이클 배정
(후보 축소)       (임베딩 생성)       (AI 모델)         (비즈니스 룰)      (최종 노출)
```

각 단계는 독립된 모듈로, 입력과 출력이 명확히 정의된다. 모듈을 교체하거나 개선해도 다른 모듈에 영향을 주지 않는다.

---

## 3. Stage 1 — 하드 필터 (후보 축소)

**목적:** AI 모델이 평가할 필요가 없는 후보를 사전에 제거하여 연산량을 줄인다.

**입력:**
- 뷰어 유저 ID
- 전체 활성 유저 풀

**필터 조건 (v1, v2 공통으로 이미 검증된 항목):**

| 필터 | 설명 | 출처 |
|------|------|------|
| 성별 | 이성만 | v1, v2 공통 |
| 지역 | 등록 지역 기반 | v1, v2 공통 |
| 나이 | 유저 선호 나이 범위 (`UserPreferredAge`) | v1, v2 공통 |
| 이미 본 유저 | impression 기록 기준 영구 제외 | v2 (7.5.5) |
| 차단/신고 | 양방향 차단 제외 | v1, v2 공통 |
| 비활성 | 최근 N일 미접속 유저 제외 | v1, v2 공통 |
| 이미 매칭 | 상호 좋아요 성사된 유저 제외 | v1, v2 공통 |

**출력:** 후보 유저 ID 리스트 (수백~수천 명 수준으로 축소)

**설계 원칙:**
- 하드 필터는 규칙 기반을 유지한다. AI로 대체할 영역이 아니다.
- 필터 조건은 config로 관리하여 운영 중 조정 가능하게 한다.

## 4. Stage 2 — 피처 추출 (Feature Extraction)

**목적:** 뷰어와 각 후보의 프로필/행동 데이터를 AI 모델이 이해할 수 있는 숫자 벡터(피처)로 변환한다.

### 4.1 유저 피처 (양쪽 공통)

| 피처 그룹 | 항목 | 표현 방식 | 비고 |
|-----------|------|-----------|------|
| **프로필 사진** | 사진 1~N장 | **이미지 임베딩 벡터** (사전학습 모델) | 외형 유사도, cold start 해결의 핵심 |
| **자기소개** | 텍스트 | **텍스트 임베딩 벡터** (사전학습 모델) | 관심사, 성격, 라이프스타일 |
| **기본 정보** | 나이, 성별, 지역, 학력, 직업 | 수치/범주형 인코딩 | |
| **성향 정보** | MBTI, 흡연, 음주, 종교 | 범주형 인코딩 | |
| **관심사/키워드** | `UserKeyword`, `UserProfileInterest`, `togetherActivity` | 멀티핫 인코딩 또는 임베딩 | |
| **행동 통계** | 좋아요 보낸 비율, 받은 비율, 접속빈도, 평균 체류시간 | 수치형 정규화 | v1 활동성 박스의 확장 |
| **서비스 이용 기간** | 가입 경과일 (D+N) | 수치형 | v2 분류 전환 기준, 성숙도 |
| **과거 매칭 이력 통계** | 매칭 성사율, 채팅 전환율, 평균 응답 시간 | 수치형 정규화 | v1 수익유발 박스의 확장 |

### 4.2 페어 피처 (A-B 조합에서만 생성)

| 피처 | 설명 | 비고 |
|------|------|------|
| 나이차 | `B.age - A.age` (부호 포함) | v2 나이차 점수의 AI 버전 |
| 지역 거리 | 두 유저의 지역 간 거리 | |
| 관심사 겹침 수 | 키워드/관심사 교집합 크기 | v2 가점 체계의 AI 버전 |
| MBTI 궁합 | MBTI 조합별 피처 | v2 가점 체계의 AI 버전 |
| 사진 유사도 | 이미지 임베딩 간 코사인 유사도 | 외형 선호 패턴 학습 |
| 텍스트 유사도 | 자기소개 임베딩 간 코사인 유사도 | 가치관/라이프스타일 유사도 |

### 4.3 컨텍스트 피처

| 피처 | 설명 | 비고 |
|------|------|------|
| 시간대 | 노출 시점의 시간대 | 시간대별 반응 패턴 |
| 슬롯 유형 | 추천 / 발견 | 탭별 반응 차이 |
| 사이클 내 순서 | 8카드 중 몇 번째로 노출 | 순서 효과 |

**출력:** 피처 벡터 (유저 단독 피처 + 페어 피처 + 컨텍스트 피처)

**설계 원칙:**
- 이미지/텍스트 임베딩은 **사전 학습된 모델**(CLIP, OpenAI Embedding 등)을 호출하여 생성한다. 자체 학습 불필요.
- 임베딩은 프로필 변경 시에만 재생성하고, 캐싱하여 실시간 부하를 줄인다.
- 유저 단독 피처는 주기적으로(일 1회) 배치 갱신하고, 페어 피처는 Stage 3 진입 시 실시간으로 계산한다.

## 5. Stage 3 — 매칭 확률 예측 (AI 모델)

**목적:** 두 유저의 피처 벡터를 받아서 매칭 확률을 예측한다.

### 5.1 모델 구조 (초안)

```
[뷰어 피처 벡터]  →  [유저 인코더]  →  [뷰어 표현]
                                              ↘
                                        [인터랙션 레이어]  →  P(A→B)
                                              ↗
[후보 피처 벡터]  →  [유저 인코더]  →  [후보 표현]
                    (가중치 공유)

                  + [페어 피처] + [컨텍스트 피처]
```

- **유저 인코더**: 프로필 피처를 고정 크기의 유저 표현 벡터로 압축. 뷰어와 후보가 같은 인코더를 공유(Siamese 구조)
- **인터랙션 레이어**: 두 유저 표현 + 페어 피처 + 컨텍스트 피처를 결합하여 매칭 확률 출력
- **출력**: P(A→B) — A가 B를 봤을 때 긍정 반응(좋아요/슈퍼좋아요)할 확률 (0~1)

### 5.2 학습 데이터

v1부터 수집해 온 AI 학습 로그를 그대로 활용한다.

| 구분 | 내용 | 출처 |
|------|------|------|
| **Input** | 추천 시점의 양쪽 유저 프로필, 사진 URL | v1 AI 학습 로그 |
| **Label (긍정)** | LIKE, SUPER_LIKE, 친구신청, 채팅 시작 | 인터랙션 로그 |
| **Label (부정)** | DISLIKE, 빠른 패스(3초 이내), 미조회 | 인터랙션 로그, impression 로그 |
| **Label (강한 부정)** | 신고, 차단 | 안전 로그 |

**라벨 가중치 (v2 인터랙션 가중치 참고):**

| 행동 | 라벨 | 가중치 | 비고 |
|------|------|--------|------|
| SUPER_LIKE | 긍정 | 1.0 | 가장 강한 긍정 |
| 친구신청/수락, 채팅시작 | 긍정 | 1.0 | 매칭 이후 행동 |
| LIKE | 긍정 | 0.5 | 기본 긍정 |
| 프로필 상세 조회 | 약한 긍정 | 0.2 | 관심은 있으나 행동 미확정 |
| DISLIKE | 부정 | -0.1 | 기본 부정 |
| 빠른 패스 | 강한 부정 | -0.5 | 3초 이내 판단 |
| 미조회 | 약한 부정 | 0.0 | 노출되었으나 무관심 |
| 신고/차단 | 강한 부정 | -1.0 | 안전 시그널 |

### 5.3 모델 유형 후보

| 모델 유형 | 장점 | 단점 | 적합도 |
|-----------|------|------|--------|
| **Two-Tower (쌍탑)** | 유저 벡터 사전 계산 가능, 빠른 추론 | 페어 피처 활용 제한 | 1차 후보 |
| **Wide & Deep** | 규칙(Wide)과 학습(Deep) 병행 | 구조 복잡 | 2차 후보 |
| **LightGBM + 임베딩** | 구현 간단, 해석 가능 | 딥러닝 대비 표현력 제한 | PoC용 |

**권장 접근:**
1. **PoC**: LightGBM + 임베딩 피처로 빠르게 검증
2. **v2.5**: Two-Tower 모델로 전환, A/B 테스트
3. **v3**: Wide & Deep 또는 Transformer 기반으로 고도화

## 6. Stage 4 — 랭킹 및 비즈니스 룰 적용

**목적:** AI가 예측한 매칭 확률에 비즈니스 정책을 적용하여 최종 노출 순위를 결정한다.

### 6.1 랭킹 점수 산출

```
ranking_score = match_probability × business_multiplier
```

**business_multiplier 구성 (v2 규칙의 안전장치 역할):**

| 비즈니스 룰 | 적용 방식 | v2 대응 |
|-------------|-----------|---------|
| 신규 유저 부스트 | 가입 D+N에 따라 노출 확률 상향 | v1 신규보너스, v2 뷰어 유형 4/5 |
| 다양성 보장 | 같은 유형 연속 노출 방지 | v2 슬롯 배분 |
| 과금 유저 우대 | 결제 유저에게 약간 높은 노출 | v1 지출 박스 |
| 노출 제한 | 하위 품질 유저 격리 | v2 품질 게이트 (7.4) |
| 상호성 보정 | P(A→B)만 높고 P(B→A)가 낮으면 하향 | 신규 — v2에 없음 |

### 6.2 v2 규칙 기반과의 차이

| 항목 | v2 (규칙 기반) | v3 (AI + 비즈니스 룰) |
|------|---------------|----------------------|
| 나이차 점수 | 고정 점수표 (0.01~1.0) | AI가 자동 학습 |
| MBTI/키워드 가점 | 고정 배수 (1+가점) | AI가 자동 학습 |
| 뷰어 7유형 후보 풀 | 사람이 정의한 매칭 규칙 | AI 예측 + 다양성 룰로 대체 |
| percentile 기반 노출 | 버킷 내 상대 위치 | AI 예측 자체가 상대 위치를 반영 |

**핵심:** v2에서 사람이 만든 나이차 점수, MBTI 가점, 뷰어 유형 매칭 규칙은 **AI가 데이터에서 자동으로 학습**한다. Stage 4에는 AI가 학습할 수 없는 비즈니스 정책(신규 부스트, 다양성, 과금 우대, 안전 격리)만 남긴다.

## 7. Stage 5 — 사이클 배정 (최종 노출)

**목적:** 랭킹 결과를 사이클 단위(하루 3회, 8카드)로 배정한다.

| 항목 | 내용 |
|------|------|
| 실행 주기 | 하루 3회, 8시간 간격 (v2 동일) |
| 카드 수 | 추천 4 + 발견 4 = 8장 |
| 추천 슬롯 | ranking_score 상위 기반 |
| 발견 슬롯 | 탐색적 추천 (Exploration) — 확률적 다양성 부여 |
| 큐 관리 | 사이클마다 대체 (v2 동일) |

**Exploration vs Exploitation:**
- **추천 4장**: AI가 가장 높다고 예측한 매칭 확률 기반 (Exploitation)
- **발견 4장**: 의도적으로 예측 확률이 중간인 후보도 포함 (Exploration) — 모델이 학습하지 못한 새로운 패턴을 발견하기 위함

## 8. 학습 파이프라인

### 8.1 데이터 흐름

```
[서비스 DB]                    [로그 스토리지]               [학습 파이프라인]
    |                               |                            |
    ├─ 프로필 데이터          →     S3/DW에 스냅샷 적재          |
    ├─ 인터랙션 로그          →     이벤트 스트림 적재     →    학습 데이터셋 생성
    ├─ impression 로그        →     이벤트 스트림 적재     →    (페어 단위 라벨링)
    ├─ AI 학습 로그 (v1~)     →     S3 parquet                  |
    |                               |                            ↓
    |                               |                     [모델 학습]
    |                               |                            |
    |                               |                     [모델 배포]
    |                               |                            |
    |                               |                     [A/B 테스트]
```

### 8.2 학습 데이터셋 구조 (페어 단위)

각 학습 샘플은 **하나의 노출 이벤트**이다:

```
{
  "viewer": {
    "profile_features": [...],      // 기본 정보, 성향, 행동 통계
    "photo_embedding": [...],       // 사진 임베딩 벡터
    "bio_embedding": [...],         // 자기소개 임베딩 벡터
    "days_since_join": 45,          // 가입 경과일
    "classification": "ESTABLISHED" // v2 분류 상태 (로그 확장 항목)
  },
  "candidate": {
    "profile_features": [...],
    "photo_embedding": [...],
    "bio_embedding": [...],
    "days_since_join": 12,
    "classification": "SETTLING"
  },
  "pair_features": {
    "age_diff": -3,
    "region_distance": 15.2,
    "interest_overlap": 4,
    "photo_similarity": 0.72,
    "bio_similarity": 0.45
  },
  "context": {
    "slot_type": "recommendation",   // 추천/발견
    "cycle_position": 2,             // 8카드 중 순서
    "hour_of_day": 14,               // 노출 시간대
    "viewer_type": 1                 // v2 뷰어 유형 (로그 확장 항목)
  },
  "label": {
    "action": "LIKE",                // 실제 행동
    "time_to_action_sec": 8.3,       // impression → 행동까지 소요 시간
    "weight": 0.5                    // 라벨 가중치
  }
}
```

### 8.3 학습 주기

| 항목 | 내용 |
|------|------|
| 전체 재학습 | 주 1회 (전체 누적 데이터) |
| 증분 학습 | 일 1회 (최근 데이터로 미세 조정) |
| 임베딩 갱신 | 프로필 변경 시에만 (이벤트 기반) |
| 모델 평가 | 학습 시마다 오프라인 지표 검증 후 배포 |

## 9. v1/v2에서 AI로의 매핑

### 9.1 v1 4박스 → AI 피처

| v1 박스 | AI에서의 대체 |
|---------|-------------|
| 매력도 (받은 호감) | AI 모델이 사진/프로필 임베딩 + 과거 인터랙션 패턴으로 직접 예측 |
| 활동성 (접속/참여) | 행동 통계 피처로 입력 (모델이 가중치 자동 학습) |
| 수익 유발 (상대 과금 유도) | 과거 매칭 후 행동 통계 피처로 입력 |
| 지출 (직접 과금) | Stage 4 비즈니스 룰에서 처리 (모델 학습에는 미포함) |

### 9.2 v2 파이프라인 → AI 파이프라인

| v2 단계 | AI 대체 방식 |
|---------|-------------|
| 유저 분류 (NEW/SETTLING/ESTABLISHED) | 피처로 입력 (가입 경과일, 인터랙션 수) — 모델이 자동 분류 효과 학습 |
| 베이지안 점수 (raw_score) | AI 모델의 매칭 확률 예측이 대체 |
| 버킷/percentile | AI 예측이 자체적으로 상대 위치를 반영하므로 불필요 |
| 뷰어 7유형 매칭 규칙 | AI 예측 + Stage 4 다양성/부스트 룰로 대체 |
| 나이차 점수 (고정 점수표) | AI가 나이차 피처에서 자동 학습 |
| MBTI/키워드 가점 (고정 배수) | AI가 페어 피처에서 자동 학습 |
| 품질 게이트 (하위 15% 제한) | Stage 4 안전 룰로 유지 (규칙 기반) |
| 사이클 배정 (3회/일, 8카드) | Stage 5에서 그대로 유지 |

## 10. Cold Start 처리

AI 모델에서도 신규 유저의 cold start 문제가 있다. 단, v2보다 훨씬 효과적으로 해결된다.

### 10.1 데이터가 없는 신규 유저

| 가용 데이터 | 활용 방법 |
|------------|-----------|
| 프로필 사진 | 이미지 임베딩 → **"이 외형과 비슷한 기존 유저가 어떤 반응을 받았는가"** 참조 |
| 자기소개 | 텍스트 임베딩 → 관심사/성격 유사 유저 매핑 |
| 기본 정보 (나이, 지역, MBTI 등) | 유사 프로필 유저군의 평균 매칭 패턴 적용 |

### 10.2 v2와의 비교

| 항목 | v2 (규칙) | v3 (AI) |
|------|-----------|---------|
| 신규 남성 초기값 | 0.30 (고정) | 프로필 기반 예측값 (유저마다 다름) |
| 신규 여성 초기값 | 0.35 (고정) | 프로필 기반 예측값 (유저마다 다름) |
| 데이터 축적까지 | 15회 인터랙션 후 점수 안정화 | 3~5회 인터랙션만으로 빠르게 개인화 |

## 11. 단계적 전환 계획

### Phase 1 — PoC (v2 운영 중)

- LightGBM + 임베딩 피처로 오프라인 검증
- v1/v2 학습 로그를 사용한 모델 학습
- 오프라인 지표: AUC, NDCG, 매칭 성사율 예측 정확도
- **v2 운영에는 영향 없음** — 별도 환경에서 검증만

### Phase 2 — Shadow Mode (v2.5)

- v2 규칙 기반이 실제 추천을 수행하되, AI 모델도 동시에 예측
- AI 예측과 실제 결과를 비교하여 모델 품질 검증
- A/B 테스트: 일부 유저에게 AI 추천 적용, 규칙 기반과 비교
- 교체 대상 (순차적):
  1. 나이차 점수 → AI 학습 모델
  2. MBTI/키워드 가점 → 임베딩 유사도
  3. 신규 유저 초기 점수 → 프로필 임베딩 예측

### Phase 3 — AI 중심 (v3)

- AI 모델이 추천 점수의 중심
- 규칙 기반은 안전장치(노출 제한, 신고/차단)로만 유지
- 모델 재학습으로 알고리즘 자동 개선

## 12. v2에서 반드시 준비해야 할 것

이 파이프라인이 동작하려면, v2에서 다음 데이터가 **빠짐없이** 로그로 남아야 한다.

### 12.1 기존 유지 (v1에서 구축 완료)

- 추천 시점 양쪽 유저의 전체 프로필 스냅샷
- 사진 URL
- 이후 행동 결과 (좋아요/싫어요/슈퍼좋아요/채팅 등)

### 12.2 v2에서 신규 추가 필요

- 유저 분류 상태 (NEW/SETTLING/ESTABLISHED)
- 뷰어 유형 (7유형)
- 버킷 ID, percentile
- 가입 경과일 (D+N)
- 노출 제한 여부 및 트리거 유형
- impression → 행동까지 소요 시간
- 사이클 번호 및 슬롯 유형 (추천/발견)
- 프로필 변경 이력

## 13. 예상 모델 성능과 기대 효과

### 13.1 오프라인 검증 목표

| 지표 | v2 베이스라인 (규칙 기반) | v3 목표 (AI) |
|------|-------------------------|-------------|
| 좋아요 전환율 예측 AUC | 0.55~0.60 (추정) | 0.75+ |
| 매칭 성사율 | 현재 수준 | +20~30% 개선 목표 |
| 신규 유저 1주일 리텐션 | 현재 수준 | +15% 개선 목표 |

- 파라미터 튜닝이 자동화되어 운영 공수 대폭 감소
- 유저별 개인화 추천으로 서비스 만족도 향상
- 데이터가 쌓일수록 모델이 자동 개선되는 선순환 구조
