# 데이팅 알고리즘 개선 SRS Review

- **대상 문서:** `데이팅 알고리즘 개선 SRS.md` (v1.0.0, 26.03.25)
- **리뷰 관점:** 알고리즘·정책의 취지 자체는 기획 영역이므로 다루지 않습니다. SRS의 **내용 정합성, 일관성, 구현 가능성, DB/API 스펙 수준**에 집중합니다.
- **리뷰일:** 2026-03-26

---

## 1. 전체 문서에 해당하는 리뷰 의견

### R-01. DB 스키마가 확정 스펙이 아닌 개념 수준

4.4.1에서 신규 테이블·컬럼을 표로 정리하고 있으나, 실제 Prisma 모델이나 DBML이 없습니다. `UserAlgorithmScore`, `CycleQueue` 등 신규 테이블의 **컬럼명, 타입, 제약조건, 인덱스, relation**이 확정되어 있지 않으면 다른 개발자가 바로 마이그레이션을 작성할 수 없습니다. 기존 `User` 테이블에 추가하는 컬럼(`classification`, `isRestricted`, `fastPassCount` 등)도 마찬가지로 타입·기본값·nullable 여부가 누락되어 있습니다.

`AlgorithmTestUser`(7.10.2)만 유일하게 Prisma 모델이 있는데, 나머지 신규 테이블도 동일 수준으로 정의해 주셔야 합니다.

### R-02. API 스펙이 확정되어 있지 않음

4.4.2~4.4.3에서 기존/신규 API를 표로 나열하고 있으나, **요청·응답 스키마(필드명, 타입, 상태 코드, 에러 코드)** 가 Swagger/OpenAPI 수준으로 정의되어 있지 않습니다. 특히 신규 Impression API는 구현 완료(WEBB-1143)라고 하면서도, 응답 스키마·에러 케이스가 문서에 없습니다.

SRS에서 API를 다룬다면, 최소한 request/response JSON 스키마와 에러 코드 목록이 있어야 구현·QA가 가능합니다.

### R-03. [TBD] 항목이 구현에 직접 영향

여성 노출 점수 가중치(7.2.5), 가점 수치(7.6.5.2), 직업 카테고리 매칭(7.6.5.2)이 [TBD]로 남아 있습니다. 이 값들이 확정되지 않으면 점수 산출 로직을 구현할 수 없고, 단위 테스트도 작성할 수 없습니다. [TBD] 항목별로 **확정 일정 또는 임시 기본값**을 명시해 주시기 바랍니다.

### R-04. 기획 문서와 SRS 사이에 실제 불일치 존재

기획 문서(`데이팅 노출 알고리즘 개선 기획.md`, v2.3)와 SRS 사이에 다음 두 가지 불일치가 확인됩니다.

1. **순점수 방식**: 기획 문서에서는 신규 남성(15회 미만)에게 순점수 방식을 여전히 적용하고 있으나, SRS 7.2.2에서는 이를 명시적으로 폐기하고 전 분류 베이지안 통합으로 변경했습니다. 기획 문서에는 이 변경이 반영되어 있지 않습니다.
2. **프로필 변경 시 처리**: 기획 문서에서는 일관되게 "최근 10회 평가만 유지"로 기술하고 있으나, SRS 7.9.2에서는 "전체 삭제 + Prior 초기화"로 변경했습니다. SRS 본문에 "기획서에서 수정"이라고 언급하고 있지만, 실제 기획 문서(v2.3, 2026-02-12 최종 갱신)에는 이 변경이 반영되어 있지 않습니다.

기획과 SRS 중 어느 쪽이 최종 진실(SOT)인지 명시하고, SOT가 아닌 쪽 문서도 같이 갱신해 주시기 바랍니다. 그렇지 않으면 구현·QA 시 혼선이 발생합니다.

### R-05. system-config 파라미터 관리 UI(백오피스 콘솔)에 대한 언급 없음

SRS 전반에 걸쳐 30개 이상의 system-config 파라미터가 정의되어 있고, "코드 배포 없이 동적 조정 가능"하다고 기술하고 있습니다. 그러나 **이 파라미터를 누가, 어떤 화면에서, 어떻게 변경하는지**에 대한 언급이 없습니다. 운영팀이 직접 DB를 수정하는 것인지, 백오피스에 관리 UI가 필요한 것인지, 개발/테스트 환경에서는 어떻게 설정하는 것인지 명시해 주시기 바랍니다.

파라미터 수가 많고 상호 의존성이 있으므로(예: Prior 값 변경 → 노출 제한 임계값에 영향), 관리 UI가 필요하다면 SRS 범위에 포함하거나 별도 이슈로 분리해 주시기 바랍니다.

### R-06. 백오피스 알고리즘 테스트 페이지 — 기존 페이지와의 호환성 및 구조적 대책 누락

기존 백오피스 알고리즘 테스트 페이지(`백오피스 데이팅 알고리즘 테스트 페이지 구축 Onepager.md`)는 **가중합 4요소(attractiveness, activity, revenueTrigger, spending)** 기반으로 설계되어 있습니다. 알고리즘이 베이지안 단일 점수 + percentile + 뷰어 7유형 체계로 전면 변경되면, 기존 테스트 페이지의 점수 항목, 가중치 입력 UI, IndexedDB 스키마, 재계산 로직이 **전부 무효화**됩니다. 그러나 SRS에서는 이 영향에 대한 언급이 거의 없습니다.

이와 관련하여 두 가지 사항을 확인해 주시기 바랍니다.

**1) 테스트 페이지 개편 범위**: 기존 테스트 페이지를 새 알고리즘에 맞게 개편해야 하는데, 이 개편이 본 SRS 범위에 포함되는 것인지, 별도 이슈로 관리하는 것인지 명시해 주시기 바랍니다.

**2) 알고리즘 변경에 독립적인 테스트 구조 검토**: 현재 구조는 알고리즘이 바뀔 때마다 테스트 페이지의 점수 항목·계산 로직·UI를 모두 수정해야 합니다. 향후에도 같은 비용이 반복된다면, 알고리즘 변경에 독립적인 테스트 구조(예: 파이프라인 정의를 config 기반으로 추상화하여 테스트 페이지가 동적으로 입력 항목을 생성하는 방식 등)를 SRS 또는 별도 이슈에서 검토해 주시기 바랍니다.

### R-07. v1에서 구축한 AI 학습용 로그 수집 구조의 v2 계승 및 확장에 대한 언급 없음

v1에서는 추후 AI/ML 전환을 대비하여, 추천 시점의 **양쪽 유저 프로필·사진 URL(Input)**과 이후 행동(상세 프로필 조회, 좋아요/슈퍼좋아요/싫어요, 채팅방 생성 등의 Output)을 모두 로그로 기록하는 구조를 이미 구축해 두었습니다. 그런데 SRS에서는 이 AI 학습용 로그 수집에 대한 언급이 없습니다. 다음 두 가지를 확인해 주시기 바랍니다.

**1) 기존 로그 수집 유지 여부**: v2에서 알고리즘 구조가 전면 변경되면서 기존 AI 학습 로그 수집 구조가 그대로 동작하는지, 아니면 v2 구조에 맞게 수정이 필요한지 확인이 필요합니다. 최소한 "v1의 AI 학습 로그 수집 구조를 v2에서도 유지한다"는 명시가 있어야 합니다.

**2) v2 신규 항목 확장**: v2에서 새로 도입되는 개념 중 AI 학습에 필수적인 항목이 있으며, 로그에 추가 기록해야 합니다. 다음 항목을 로그 수집 범위에 포함하는지 확인해 주시기 바랍니다:

- **유저 분류 상태** (NEW/SETTLING/ESTABLISHED): 추천 시점의 양쪽 유저 분류 — 유저 성숙도에 따른 반응 패턴 차이를 학습하는 데 필수
- **뷰어 유형** (7유형 중 어느 것으로 노출되었는지): 노출 맥락(context)에 따른 반응 차이 학습
- **버킷 ID 및 percentile**: 상대적 매력도 위치가 반응에 미치는 영향 학습
- **가입 경과일 (D+N)**: 서비스 이용 기간에 따른 선호도 변화 학습
- **노출 제한 여부 및 트리거 유형**: 제한 적용/해제 전후 반응 패턴 학습
- **impression → 행동까지의 소요 시간**: 관심도 강도 학습 (빠른 좋아요 vs 오래 본 후 좋아요)
- **사이클 번호 및 슬롯 유형** (추천/발견): 추천 탭과 발견 탭에서의 반응 차이 학습
- **프로필 변경 이력** (변경 전/후 시점): 프로필 변경이 받는 반응에 미치는 영향 학습

나중에 AI로 전환하려고 할 때 과거 데이터에 이 항목들이 빠져 있으면, 해당 기간의 데이터를 학습에 활용할 수 없습니다. v2 설계 시점에 포함해 주시기 바랍니다.

## 2. 특정 섹션/문장에 해당하는 리뷰 의견

### R-08. 7.1.4 여성 ESTABLISHED 전환 조건 — 인터랙션 횟수 무관?

> 7.1.4 표: `ESTABLISHED | 가입 D+7 이후 (인터랙션 횟수 무관)`

이 정의대로면 **인터랙션 0회인 여성도 가입 8일차에 자동으로 ESTABLISHED**가 됩니다. 그런데 7.5.2에서 ESTABLISHED 여성(유형 3)은 "기존+신규/정착중 남성(자기 구간)"을 보게 되어 있습니다. 인터랙션이 전혀 없는 유저가 ESTABLISHED로 분류되면 베이지안 점수가 Prior(0.35) 그대로인 채로 자기 구간 매칭이 적용되는데, 이것이 의도된 동작인지 확인이 필요합니다.

### R-09. 7.2.5 여성 노출 점수 — 남성 노출 점수와의 비대칭이 버킷 percentile과 충돌

> 7.2.5: `남성 노출 점수 = 매력도` / `여성 노출 점수 = W1×매력도 + W2×좋아요보낸비율 + W3×접속빈도`

남성은 매력도(raw_score) 하나로 percentile을 계산하고, 그 percentile로 최종 노출 순위를 정하므로 흐름이 명확합니다. 그런데 여성은 다음과 같이 흐름이 끊깁니다:

1. 7.2.5에서 여성 노출 점수를 `W1×매력도 + W2×좋아요보낸비율 + W3×접속빈도`로 정의합니다.
2. 그런데 7.3.3에서 percentile은 `raw_score`(베이지안 매력도)만으로 계산합니다. 좋아요보낸비율, 접속빈도는 포함되지 않습니다.
3. 최종 후보 선택(7.6.5 Step 6)에서는 `weight = percentile × 나이차점수 × 가점`으로 확률을 산출합니다.

결과적으로, 7.2.5에서 정의한 **좋아요보낸비율과 접속빈도가 실제 파이프라인 어디에서도 반영되지 않습니다**. percentile을 노출 점수(매력도+좋아요비율+접속빈도) 기준으로 계산하는 것인지, 아니면 percentile은 raw_score 기준이고 좋아요비율/접속빈도는 Step 6에서 별도 가점으로 곱하는 것인지 명확히 해 주시기 바랍니다.

### R-10. 7.1.5 프로필 변경 시 "인터랙션 전체 삭제" — 물리 삭제인지, 논리 삭제인지

> 7.1.5: `인터랙션 데이터 | 전체 삭제`
> 7.9.2: `인터랙션 데이터 | 전체 삭제`

`UserCrushExpression` 등 인터랙션 테이블의 기존 데이터를 **물리 DELETE**하는 것인지, **소프트 삭제(플래그)**인지에 따라 구현이 완전히 달라집니다. 물리 삭제면 감사 추적이 불가능하고, 소프트 삭제면 베이지안 점수 집계 쿼리에서 삭제 플래그를 반드시 필터해야 합니다. 어느 쪽인지 명시해 주시기 바랍니다.

### R-11. 7.9.2와 7.1.5의 점수 초기화 서술 불일치

> 7.9.2 표: `매력도 점수 | 최근 10회 인터랙션으로 베이지안 재계산`

7.1.5와 7.9.2 하단에서 "기획서 원안(최근 10회) → 변경안(전체 삭제/Prior 초기화)"으로 바꿨다고 설명하면서도, 7.9.2 표에는 **기획서 원안의 문구("최근 10회")가 그대로** 남아 있습니다. 같은 섹션 안에서 표와 본문이 반대 내용을 말하고 있으므로, 표를 변경안에 맞게 수정해 주시기 바랍니다.

### R-12. 7.4.3 "첫 노출 전원 미조회/패스" — "첫 노출"의 기산점 불명확

> 7.4.3: `첫 노출 3명 전원이 미조회 또는 패스`

이 "첫 노출 3명"이 **계정 생성 이후 전체 기간의 최초 3명**인지, **프로필 변경 후 리셋된 시점 이후의 3명**인지가 불명확합니다. 프로필 변경 시 시그널 카운트를 초기화한다(7.4.4)고 했으므로, 프로필 변경 후 "첫 노출"이 다시 리셋되는 것이 맞는지 확인이 필요합니다.

### R-13. 7.6.5 가중치 산출 수식에서 "percentile" 단위 문제

> 7.6.5.1: `weight = percentile × 나이차 점수 (0.01~1.0)`
> 7.6.5.2: `weight = percentile × 나이차점수 × (1 + MBTI가점) × ...`

percentile은 0~100 범위(7.3.3)이고, 나이차 점수는 0.01~1.0입니다. 단순 곱셈이면 **weight 범위가 0~100**이 되는데, 이 값이 그대로 확률 산출(`weight_i / Σweight`)에 들어가면 수학적으로는 동작하지만, **percentile을 0~1로 정규화해서 쓰는 것인지 0~100 그대로 쓰는 것인지** 명시하지 않으면 구현자마다 다르게 해석할 수 있습니다.

### R-14. 7.5.2 유형 5 "30% → 자기구간 점진 이동" — 이동 함수 미정의

> 7.5.2: `정착중 여성 | 기존 남성만 (30%→자기구간 점진)`

D+0에서 상위 30%, D+7 이후 자기 구간으로 이동한다고만 되어 있고, **이동 속도·함수(선형? 비선형?)·중간 단계별 범위**가 정의되어 있지 않습니다. 7.1.5에서 프로필 변경 후 "D+4 간주"라는 언급도 있는데, D+4일 때 범위가 몇 %인지도 없습니다. 구현하려면 최소한 일별(또는 단계별) 범위 값이 필요합니다.
