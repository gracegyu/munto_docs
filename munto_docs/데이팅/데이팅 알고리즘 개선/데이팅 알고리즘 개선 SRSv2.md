## 1 Introduction (개요)

### 1.1 Purpose (목표)

본 문서는 데이팅 서비스의 매칭 알고리즘 및 점수 계산 모듈 전면 개편에 대한 소프트웨어 요구사항 명세서(SRS)이다.

- **대상 시스템**: `dating-backend`의 `score-calculation` 모듈 및 `matching-algorithm` 모듈
- **관련 이슈**: [DEVT-85](https://munto.atlassian.net/browse/DEVT-85) — 데이팅 매칭 알고리즘 개선
- **개정 대상 버전**: 현행 v1.x → 목표 v2.0
- **문서 대상 독자**: 백엔드 개발팀, PM

### 1.2 Product Scope (범위)

- 현행 매칭 알고리즘은 가중합 방식으로 운영되고 있으나, 매력도 점수와 실제 좋아요 비율 간 상관관계가 낮아 알고리즘 효율이 충분하지 않은 상태이다.
    - 상세 분석 내용은 [알고리즘 분석 리포트 2026-01-22](https://github.com/Munto-dev/dating-document/blob/main/algorithm/analysis/reports/2026-01-22_report.md)를 참조한다.
- 본 프로젝트는 매칭 엔진을 베이지안 추론 기반으로 전환하여, 매력도 점수의 변별력을 높이고 매칭 만족도와 리텐션을 개선하는 것을 목표로 한다.
    - 비즈니스 배경 및 기대효과 상세는 [DEVT-85](https://munto.atlassian.net/browse/DEVT-85) 및 [데이팅 노출 알고리즘 개선 기획](https://www.notion.so/2fde2bc7639d80578721e7fc1378b1d5?pvs=21)을 참조한다.
- 소프트웨어의 범위는 `dating-backend` 내 매칭 알고리즘 엔진(유저 분류, 점수 계산, 후보 선택 파이프라인, 사이클 운영)의 재설계이며, 상세 기능 요구사항은 7장에서 기술한다.
- 프론트엔드 UI/UX, 채팅, 결제/구독, 푸시 알림은 범위 외이다. 단, 클라이언트에서 impression API 호출(WEBB-1143, 구현 완료)과 A/B 테스트를 위한 `ExperimentServiceInterface` 연동이 필요하다.
- 아래 항목은 본 SRS 범위 외이며, **별도 이슈로 관리**한다:
    - **백오피스 알고리즘 테스트 페이지 개편**: 기존 테스트 페이지는 가중합 4요소(attractiveness, activity, revenueTrigger, spending) 기반이므로, 새 알고리즘(베이지안 + percentile + 뷰어 7유형)에 맞게 전면 개편이 필요하다. 알고리즘 변경에 독립적인 테스트 구조(config 기반 동적 항목 생성 등)도 함께 검토한다.
    - **system-config 관리 UI**: 본 SRS에서 정의한 30개 이상의 동적 파라미터(7.11)를 운영팀이 코드 배포 없이 변경할 수 있는 백오피스 관리 화면이 필요하다. 파라미터 간 상호 의존성(예: Prior 변경 → 노출 제한 임계값 영향)에 대한 가이드도 포함한다.

### 1.3 Document Conventions (문서규칙)

| 규칙 | 설명 |
| --- | --- |
| 수식 표기 | LaTeX 수식 사용 (예: `score = (α₀ + k) / (α₀ + β₀ + n)`) |
| 코드 표기 | TypeScript 코드 블록 사용 |
| 점수 범위 | 모든 점수는 0~1 범위로 정규화 |
| 가중치 합 | 동일 카테고리 내 가중치 합은 항상 1.0 |
| 미확정 항목 | `[TBD]`로 표기 |
| 우선순위 | 상위 요구사항의 우선순위는 하위 요구사항에 상속됨 |
- **우선순위 정의 (P1 ~ P3)**
    - **P1**: 프로젝트에 **반드시 포함되어야 하는 핵심 기능**
        - 프로젝트 목표 달성에 필수이며, 제외 시 릴리스 불가능
    - **P2**: **중요하지만 P1보다는 우선순위가 낮은 기능**
        - 프로젝트 일정 내 구현이 권장되며, 일정에 따라 조정 가능
    - **P3**: **추가되면 좋지만 필수는 아닌 기능**
        - 여유가 있을 때 구현하며, 추후 릴리스 대상이 될 수 있음

### 1.4 Terms and Abbreviations (정의 및 약어)

| 용어 | 정의 |
| --- | --- |
| **NEW (신규 유저)** | 남성: 인터랙션 0~4회, 여성: 받은 인터랙션 15회 미만 |
| **SETTLING (정착중 유저)** | 남성: 인터랙션 5~14회, 여성: 받은 인터랙션 15회 이상 AND 가입 D+7 이내 |
| **ESTABLISHED (기존 유저)** | 남성: 인터랙션 15회 이상, 여성: 받은 인터랙션 15회 이상 AND 가입 D+7 이후 |
| **인터랙션** | 프로필 조회, 좋아요, 슈퍼좋아요, 친구신청, 채팅시작, 패스 등 유저 간 행동 |
| **매력도 (raw_score)** | 다른 유저로부터 받은 인터랙션을 기반으로 베이지안 추론으로 산출한 점수 (0~1) |
| **베이지안 점수** | 사전확률(Prior)과 관측 데이터를 결합한 점수. `score = (α₀ + k) / (α₀ + β₀ + n)`. 전 분류 공통 적용 |
| **α₀ (Prior α)** | 베이지안 사전확률의 긍정 파라미터. 남성 3.0, 여성 3.5 |
| **β₀ (Prior β)** | 베이지안 사전확률의 부정 파라미터. 남성 7.0, 여성 6.5 |
| **k** | 관측된 긍정 인터랙션 가중치 합. 긍정 인터랙션 발생 시 해당 가중치만큼 증가 |
| **n** | 관측된 전체 인터랙션 가중치 합. 긍정·부정 모두 해당 가중치만큼 증가 (부정은 n만 증가, k 불변) |
| **사전확률 (Prior)** | α₀/(α₀+β₀)로 결정되는 초기 점수. 남성 0.30, 여성 0.35 |
| **버킷** | 연령대 × 지역 × 성별로 유저를 그룹핑한 단위. percentile 정규화에 사용 |
| **percentile (상대 점수)** | 버킷 내에서의 상대적 위치 (0~1). 사이클 직전 계산·저장 |
| **하드 필터** | 후보를 포함하거나 제외하는 필터. 기본 제외, 교차 노출 규칙, 유저 설정 필터 (나이, 학력, 흡연 등) |
| **소프트 필터** | 후보에 가점 또는 감점을 부여하는 필터. 나이차 점수, MBTI 가산점 등. 후보를 제외하지 않음 |
| **노출 제한** | 점수 기반(하위 15%) 또는 부정 시그널 기반으로 유저의 노출 범위를 격리 풀로 제한하는 정책. 성별 비대칭 적용 |
| **부정 시그널** | 신고/차단, 빠른 패스(3초 이내 DISLIKE), 첫 노출 전원 미조회/패스. 노출 제한 트리거 |
| **사이클** | 추천/발견 카드 생성 단위. 하루 3회, 8시간 간격 크론잡 실행. 추천 4명 + 발견 4명 |
| **뷰어 유형** | 성별 + 분류 + 노출 제한 여부 조합으로 결정되는 7개 파이프라인 변형 |
| **확률 샘플링** | 가중 랜덤 샘플링(Weighted Random Sampling without Replacement). 결정론적 top-N 대신 확률적 추출 |

## 2 Overall Description (전체 설명)

### 2.1 Product Perspective (제품 조망)

`dating-backend` 서비스의 핵심 매칭 엔진을 전면 개편하는 프로젝트이다.

**현행 시스템 (AS-IS)**:

- 단순 가중합 기반 점수 계산: `baseScore = W₁×매력도 + W₂×활동성 + W₃×수익유발 + W₄×지출`
- 성별 구분 없이 동일 로직 적용 (가중치만 상이)
- 신규유저 보너스: 7일간 감소하는 보너스 적용 (0.35 → 0.05)
- 추천/발견 별도 사이클 운영

**변경 시스템 (TO-BE)**:

- 베이지안 추론 기반 매력도 계산 (전 분류 통합): `score = (α₀ + k) / (α₀ + β₀ + n)`
- 성별·분류별 완전히 다른 노출 규칙 (뷰어 7유형 분기)
- 신규유저 보너스 폐지 → 베이지안 Prior로 대체
- 추천/발견 동일 사이클 통합 (하루 3회, 추천 4명 + 발견 4명)
- 노출 제한 정책: 점수 기반(하위 15%) + 부정 시그널 기반, 성별 비대칭 적용
- 버킷 기반 병렬 처리 + 확률 함수 기반 샘플링

**시스템 위치**: `dating-backend` 모노레포 내 `core/business/` 하위 모듈

```
dating-backend/
├── core/business/
│   ├── score-calculation/     ← 개편 대상
│   ├── matching-algorithm/    ← 개편 대상
│   ├── preference-matrix/     ← 일부 수정
│   ├── user-classification/   ← 신규 모듈
│   └── bucket-system/         ← 신규 모듈
├── core/system-config/        ← 설정값 관리
└── ...
```

### 2.2 Overall System Configuration (전체 시스템 구성)

```
┌─────────────────────────────────────────────────────────┐
│                  Cycle Scheduler (7.5)                    │
│              하루 3회, 8시간 간격 크론잡 실행                │
└──────────────────────┬──────────────────────────────────┘
                       │ trigger
                       ▼
┌─────────────────────────────────────────────────────────┐
│  Phase 1: 버킷 로드 + Percentile 계산 (7.3, 7.6.2)       │
│  16개 버킷 병렬 로드 → 3개 서브 풀 분리                      │
│  (established / new_settling / restricted)               │
│  + 노출 제한 판정 (7.4)                                   │
└──────────────────────┬──────────────────────────────────┘
                       │ bucketed & tagged users
                       ▼
┌─────────────────────────────────────────────────────────┐
│  Phase 2: 유저별 단일 잡 — Lambda (7.6)                  │
│                                                         │
│  1. 뷰어 유형 판별 (7개 유형)                              │
│  2. 하드 필터 (기본 제외 + 교차 노출 + 유저 설정)            │
│  3. 가중치 산출 (나이차 점수 × percentile × 가점)            │
│  4. 슬롯 기반 확률 샘플링 → 추천 4명 + 발견 4명              │
└─────────────────────────────────────────────────────────┘
```

### 2.3 Overall Operation (전체 동작방식)

**사이클 실행 파이프라인**:

```
[Cycle Trigger]
    │
    ├─ Phase 1: 버킷 로드 + percentile + 노출 제한 판정
    │
    └─ Phase 2: 유저별 단일 잡 (Lambda)
        ├─ 1. 뷰어 유형 판별 (7개 유형)
        ├─ 2. 하드 필터 (기본 제외 + 교차 노출 + 유저 설정)
        ├─ 3. 가중치 산출 (나이차 점수 × percentile × 가점)
        └─ 4. 슬롯 기반 확률 샘플링 → 추천 4명 + 발견 4명
```

**단계별 상세**:

| 단계 | 처리 내용 | 핵심 로직 |
| --- | --- | --- |
| Phase 1 | 버킷 로드 + percentile | 16개 버킷 병렬, 서브 풀 3분할 (established/new_settling/restricted) |
| Phase 1 | 노출 제한 판정 | 하위 점수 + 부정 시그널 → restricted 태그 |
| 뷰어 유형 판별 | 유형 분류 | 성별 + 분류 + 노출 제한 여부 → 7개 유형 중 1개 |
| 하드 필터 | 후보 제외 | 기본 제외, 교차 노출 규칙, 유저 설정 필터 (나이, 학력, 흡연 등) |
| 가중치 산출 | 확률 산출 | 나이차 점수(곱셈) × percentile × 가점(MBTI, 키워드, 관심사 등) |
| 확률 샘플링 | 카드 추출 | 분류별 슬롯 배분에 따라 가중 랜덤 추출 → 추천 4명 + 발견 4명 |

### 2.4 Product Functions (제품 주요 기능)

> 각 기능의 상세 스펙은 7장에서 기술한다.
> 

| # | 기능 | 설명 | 7장 참조 |
| --- | --- | --- | --- |
| 1 | 유저 분류 체계 | 성별별 NEW/SETTLING/ESTABLISHED 분류 | 7.1 |

### 2.6 Assumptions and Dependencies (가정과 종속 관계)

### 2.6.1 유저 풀 규모

본 알고리즘은 프로덕션 DB 기준 다음 유저 규모를 기반으로 설계하였다.

- 전체 유저: 약 27,900명 (남성 13,400 / 여성 6,800 / 성별 미지정 7,700)
- ACTIVE 유저: 약 9,500명 (남성 7,300 / 여성 2,200)
- 7일 접속 기준: 약 3,200명 (남성 2,500 / 여성 700)
- 남녀 비율: 7일 접속 기준 남:여 ≈ 3.5:1

버킷 시스템(7.3)은 연령대×지역×성별 = 최대 16개 버킷으로 분할하므로, 버킷당 평균 수십 ~ 수백 명 수준이다. 알고리즘 개선 후 유저 규모가 1.5 ~ 2배 확장될 수 있다는 가정 하에 성능 요구사항(5장)을 설계하였다.

한번 본 유저는 영구 제외하는 정책(7.5.5)을 적용하였다. 병목 구간인 남성→여성 기준으로 1일 소비량 24명(3사이클 × 8카드), 필터 후 여성 풀 약 350명으로 산출하면 약 2주에 풀이 자연 소진된다. 서비스 수명을 약 2주로 가정하면 영구 제외해도 풀 고갈 문제가 없으며, 신규 유저 유입 및 프로필 변경 시 제외 목록 초기화로 풀이 자연 순환된다.

버킷 별로 속한 유저 수에 따라 다음 전략이 변경될 수 있다:

- 특정 버킷(예: 41세+ 비수도권 여성)의 유저 수가 극소인 경우 percentile 정규화의 정밀도가 낮아지며, 버킷 경계 조정이 필요할 수 있다
- 여성 풀이 현재보다 축소될 경우 확률 샘플링의 다양성이 제한되며, 풀 부족 대응(7.8.2)이 빈번하게 발생할 수 있다
- 반대로 유저 규모가 2배 이상 성장할 경우, 버킷 기반 병렬 처리(7.6.2)의 메모리 사용량 모니터링이 필요하다

## 7 Functional Requirements (기능 요구사항)

### 7.1 유저 분류 체계

### 7.1.1 개요

모든 유저는 **받은 인터랙션 횟수**를 기준으로 3단계로 분류한다. 남성과 여성은 동일한 용어 체계(NEW / SETTLING / ESTABLISHED)를 사용하되, 분류 기준이 다르다.

| 분류 | 영문 코드 | 설명 | 점수 산출 방식 |
| --- | --- | --- | --- |
| 신규 | `NEW` | 평가 데이터가 거의 없는 유저 | 베이지안 (Prior 지배) |
| 정착 중 | `SETTLING` | 평가 데이터는 충족되었으나 아직 초기 단계 | 베이지안 (균형) |
| 기존 | `ESTABLISHED` | 충분한 평가 데이터를 가진 정착 유저 | 베이지안 (관측 지배) |
- 분류에 따라 점수 산출 방식(7.2)과 노출 전략(7.5)이 달라진다.
- 남성과 여성 모두 동일한 용어 체계를 사용하되, 분류 기준이 다르다.

### 7.1.2 인터랙션 정의

유저 분류 및 매력도 점수 계산에 사용되는 인터랙션 종류와 가중치는 다음과 같다. 인터랙션 가중치는 `system-config`를 통해 동적 조정 가능하다 (7.11 참조).

| 인터랙션 유형 | 설명 | 가중치 | API 매핑 |
| --- | --- | --- | --- |
| 프로필 조회 | 상대방이 내 프로필 상세 조회 | 0.2 | `POST /matching/impression` (WEBB-1143) |
| 좋아요 (LIKE) | 상대방이 나에게 좋아요 | 0.5 | `POST /matching/crush` (LIKE) |
| 슈퍼좋아요 (SUPER_LIKE) | 상대방이 나에게 슈퍼좋아요 | 1.0 | `POST /matching/crush` (SUPER_LIKE) |
| 친구 신청 받음 | 상대방이 나에게 친구 신청 | 1.0 |  |
| 친구 신청 수락됨 | 내가 보낸 친구 신청이 수락됨 | 1.0 |  |
| 채팅 시작 | 상대방이 채팅 시작 | 1.0 |  |
| 패스 (DISLIKE) | 상대방이 패스 | 0.1 | `POST /matching/crush` (DISLIKE) |
| 프로필 미조회 | 카드 노출되었으나 프로필 미조회 | 0.0 | impression 있으나 프로필 조회 없음 |

> **전 분류 공통**: 베이지안 점수 적용 (7.2 참조). 부정 인터랙션은 분모(n)만 증가시켜 긍정 비율을 희석한다.
> 
> 
> **빠른 필터링**: NEW 유저의 부정 시그널 빠른 걸러냄은 점수가 아닌 카운트 기반 규칙(7.7)이 담당한다.
> 

### 7.1.3 남성 유저 분류

**받은 인터랙션 횟수**만을 기준으로 분류한다.

```
받은 인터랙션 횟수
0회        5회           15회                    ∞
├──────────┼─────────────┼────────────────────────┤
│   NEW    │  SETTLING   │     ESTABLISHED        │
│  (0~4)   │   (5~14)    │       (15+)            │
└──────────┴─────────────┴────────────────────────┘
```

| 분류 | 조건 | 설명 |
| --- | --- | --- |
| NEW | 받은 인터랙션 0~4회 | 평가 데이터가 없는 신규 남성 |
| SETTLING | 받은 인터랙션 5~14회 | 평가 데이터 부족, 정착 진행 중인 남성 |
| ESTABLISHED | 받은 인터랙션 15회 이상 | 충분한 평가 데이터를 가진 기존 남성 |

### 7.1.4 여성 유저 분류

**받은 인터랙션 횟수**와 **가입 후 경과 일수**를 복합적으로 고려한다.

```
인터랙션 < 15회           인터랙션 ≥ 15회
┌──────────────┐      ┌──────────────────────────┐
│              │      │                          │
│     NEW      │      │  D+7 이내   │  D+7 이후   │
│              │      │ (SETTLING)  │(ESTABLISHED)│
│              │      │             │             │
└──────────────┘      └──────────────────────────┘
```

| 분류 | 조건 | 설명 |
| --- | --- | --- |
| NEW | 받은 인터랙션 15회 미만 | 평가 데이터 부족한 신규 여성 |
| SETTLING | 받은 인터랙션 15회+ **AND** 가입 D+7 이내 | 가입 초기, 상위 남성 우선 노출 |
| ESTABLISHED | 가입 D+7 이후 (인터랙션 횟수 무관) | 완전히 정착한 기존 여성 |

### 7.1.5 분류 전환 규칙

분류 전환은 **일반 전환**과 **프로필 변경 시 역전환** 두 가지 경로가 있다.

**일반 전환 (순방향)**

- 인터랙션 누적에 따라 `NEW → SETTLING → ESTABLISHED` 순서로 전환
- 인터랙션 발생 시점에 **실시간** 재평가
- 여성의 경우 D+7 경과 시점에 SETTLING → ESTABLISHED 자동 전환

**프로필 변경 시 역전환 (남녀 공통)**

프로필을 변경하면 이전 평가를 **전체 초기화**하고 **SETTLING으로 전환**된다. 프로필을 변경한 시점에서 이전 유저 정보와 연속성을 가져야 할 이유가 없기 때문이다 (기획서에서 수정).

| 항목 | 처리 | 설명 |
| --- | --- | --- |
| 인터랙션 데이터 | **전체 소프트 삭제(점수 계산에서 제외)** | DB에서 물리 삭제하지 않고 비활성 플래그 처리. 이전 데이터는 분석·학습용으로 보존하되, 베이지안 점수 집계 쿼리에서 제외 |
| 매력도 점수 | **Prior로 초기화** | 남성 0.30, 여성 0.35 |
| 유저 분류 | **SETTLING으로 전환** | 새 출발 |
| 노출 제한 | **무조건 해제** | 시그널 카운트 + 점수 기반 모두 초기화 |

적용 대상별 상세:

| 유저 유형 | 적용 여부 | 비고 |
| --- | --- | --- |
| 남성 (모든 유형) | O | 인터랙션 전체 삭제, Prior 0.30, **유형 2**로 전환 |
| 여성 (모든 유형) | O | 인터랙션 전체 삭제, Prior 0.35, **유형 5**로 전환 (D+4 간주) |

> **여성 특칙**: 프로필 변경 후 D+4 유저로 간주되어, 3일간 점진적으로 자기 점수 구간으로 이동한다.
> 

> **제한사항**: 프로필 변경 후 재평가는 **7일 내 1회만** 적용된다. (빈번한 프로필 변경을 통한 점수 리셋 악용 방지)
> 

**비활성화/정지 해제**

- 기존 매력도 점수 및 분류 **그대로 유지**
- 설계 의도: 일시적 조치이므로 기존 평가 보존

**탈퇴 후 재가입**

- **NEW로 완전 초기화** — 새로운 유저로 간주, 처음부터 평가 시작
- 설계 의도: 탈퇴는 영구적 선택이므로 기존 데이터 폐기

### 7.1.6 엣지 케이스

| 케이스 | 처리 방식 | 근거 |
| --- | --- | --- |
| 프로필 변경 7일 이내 재변경 | 재평가 **미적용** | 7일 내 1회 제한 — 기존 SETTLING 상태 유지 |
| 노출 제외 남성이 프로필 변경 | SETTLING으로 **복귀** | 재평가 기회 부여 — 노출 제외에서 탈출하는 유일한 경로 |
| 프로필 변경 후 인터랙션 15회 도달 | ESTABLISHED로 **재전환** | 일반 전환 규칙 동일 적용 |

### 7.2 매력도 점수 계산

### 7.2.1 개요

매력도 점수는 **다른 유저로부터 받은 인터랙션**을 기반으로 산출하는 핵심 지표이다. **모든 분류(NEW/SETTLING/ESTABLISHED)에 동일한 베이지안 점수 체계**를 적용한다.

| 유저 분류 | 점수 산출 방식 | 점수 범위 | Prior 강도 |
| --- | --- | --- | --- |
| NEW | 베이지안 | 0~1 | 강함 (Prior 지배) |
| SETTLING | 베이지안 | 0~1 | 중간 (Prior ↔ 관측 균형) |
| ESTABLISHED | 베이지안 | 0~1 | 약함 (관측 데이터 지배) |

```
매력도 = f(프로필조회, 좋아요, 슈퍼좋아요, 친구신청받음, 친구신청수락됨, 채팅시작, 패스)
```

### 7.2.2 설계 변경: 순점수 → 베이지안 통합

**기획서 원안 (A안)**

기획서에서는 NEW 유저에게 별도의 순점수 방식(`Σ긍정 - Σ부정`)을 적용하도록 설계하였다. 

의도는 다음과 같다

- 부정 시그널에 빠르게 반응하여 매력 없는 NEW 유저를 빠르게 걸러냄
- 패스(-0.5), 프로필 미조회(-1.0) 등으로 점수가 음수까지 떨어질 수 있음
- 점수가 낮아진 NEW 유저는 즉시 노출 범위 축소

**변경안 (B안) — 채택**

순점수 방식을 폐기하고, **전 분류 베이지안 통합 + 카운트 기반 규칙 분리**로 변경한다.

| 기능 | A안 (순점수) | B안 (베이지안 통합) |
| --- | --- | --- |
| NEW 유저 점수 | 순점수 (별도 산식) | 베이지안 (동일 산식) |
| 부정 시그널 빠른 필터링 | 점수로 처리 (음수 → 노출 축소) | **카운트 규칙으로 분리** (7.7) |
| 점수 체계 | 2개 (순점수 + 베이지안) | **1개** (베이지안만) |
| NEW→SETTLING 전환 | 점수 시스템 교체 필요 | **점수 연속성 유지** |

**변경 사유**:

1. **책임 분리**: "빠른 필터링"은 점수가 아닌 규칙(7.7)이 담당한다. 신고/차단 즉시 미노출, 빠른 패스 누적 시 미노출 — 이 규칙들은 점수 산식과 무관하게 작동한다.
2. **단일 점수 체계**: 두 개의 점수 체계를 관리하지 않아 구현·유지보수가 단순해진다.
3. **전환 연속성**: NEW→SETTLING 전환 시 점수가 그대로 이어진다. A안에서는 순점수 → 베이지안 전환 시 점수 스케일이 달라지는 문제가 있었다.

**A안 의도의 보완 방식**:

| A안의 의도 | B안에서의 보완 |
| --- | --- |
| 부정 시그널에 빠르게 반응 | 7.7 카운트 기반 미노출 규칙: 신고/차단 즉시 + 빠른 패스 N회 누적 시 미노출 |
| 매력 없는 NEW 유저 빠르게 걸러냄 | 7.7 규칙 + 티어 시스템(7.3)에서 하위 구간 격리 |
| 부정 시그널 가중치 차등 | 베이지안에서도 부정은 분모만 증가 → 긍정 비율 희석 효과. 추가로 Prior 강도 조절로 NEW 단계에서 소수의 부정이 점수에 큰 영향 |

### 7.2.3 베이지안 점수 (전 분류 공통)

Beta 분포 기반 점수 산출 방식이다. **NEW/SETTLING/ESTABLISHED 모두 동일한 수식**을 사용하며, Prior 파라미터로 분류별 차이를 표현한다.

**수식**

```
score = (α₀ + k) / (α₀ + β₀ + n)

  α₀ : 사전확률 α 파라미터 (긍정 사전)
  β₀ : 사전확률 β 파라미터 (부정 사전)
  k  : 관측된 긍정 인터랙션 가중치 합
  n  : 관측된 전체 인터랙션 가중치 합
```

**성별별 사전확률 (Prior)**

| 성별 | Beta 분포 | 초기값 (α₀/(α₀+β₀)) | 설명 |
| --- | --- | --- | --- |
| 남성 | Beta(3.0, 7.0) | **0.30** | 보수적 시작 — 남성 풀이 크므로 낮게 출발 |
| 여성 | Beta(3.5, 6.5) | **0.35** | 약간 높게 시작 — 여성 풀 희소성 반영 |

> **Prior 강도와 분류의 관계**: Prior 강도(α₀+β₀)는 "가상의 관측 N회"와 동일한 효과를 갖는다. 남성 α₀+β₀=10은 "10회 가상 관측"을 의미하므로, 실제 인터랙션 5회(SETTLING 진입)까지는 Prior가 점수를 지배하고, 15회(ESTABLISHED 진입)에서는 관측 비중이 60%로 증가한다.
> 

> 사전확률 파라미터는 `system-config`를 통해 동적 조정 가능하다 (7.11 참조).
> 

**인터랙션별 점수 반영 방식**

| 인터랙션 | 분류 | 가중치 | k 가산 | n 가산 | 비고 |
| --- | --- | --- | --- | --- | --- |
| 프로필 조회 | 긍정 | 0.2 | +0.2 | +0.2 |  |
| 좋아요 | 긍정 | 0.5 | +0.5 | +0.5 |  |
| 슈퍼좋아요 | 긍정 | 1.0 | +1.0 | +1.0 |  |
| 친구 신청 받음 | 긍정 | 1.0 | +1.0 | +1.0 |  |
| 친구 신청 수락됨 | 긍정 | 1.0 | +1.0 | +1.0 |  |
| 채팅 시작 | 긍정 | 1.0 | +1.0 | +1.0 |  |
| 패스 | 부정 | 0.1 | 0 | +0.1 | n만 증가 → 긍정 비율 희석 |
| 프로필 미조회 | 무시 | 0.0 | 0 | 0 | 점수에 미반영 |

> 부정 인터랙션은 **점수를 직접 깎지 않고**, 전체 분모(n)만 증가시켜 긍정 비율을 희석하는 방식으로 작동한다. NEW 유저 중 빠르게 걸러내야 하는 경우는 점수가 아닌 **카운트 기반 규칙(7.7)**이 담당한다.
> 

### 7.2.4 사전 점수 부여

평가 데이터가 없는 NEW 유저에게는 **사전 점수(Prior)**가 초기 노출 순위를 결정한다. 기획서에서 정의한 "사전 평가 수"는 베이지안 Prior의 α₀+β₀에 대응된다.

| 분류 | Prior 강도 (α₀+β₀) | 초기 점수 | 설명 |
| --- | --- | --- | --- |
| 신규 남성 (0~4회) | 10 | **0.30** (4티어 수준, 중간 이하) | 보수적 시작 |
| 신규 여성 (0~14회) | 10 | **0.35** (중간값) | 여성 풀 희소성 반영 |
| 정착 중 (5~14회) | - | **기존 점수 유지** | 관측 데이터가 누적되어 Prior 희석 중 |

> **설계 의도**: Prior가 사전 점수 역할을 겸한다. 신규 남성은 0.30(중간 이하)으로 시작하여 긍정 시그널을 받으면 빠르게 상승하고, 신규 여성은 0.35(중간값)으로 시작한다. 관측 데이터가 쌓이면 Prior는 자연스럽게 희석된다.
> 

### 7.2.5 성별별 노출 점수

매력도 점수를 기반으로 **최종 노출 순위**를 결정하는 점수 체계이다. 성별에 따라 산식이 다르다.

**남성 노출 점수 (여성에게 노출 시 순위 결정)**

```
남성 노출 점수 = 매력도
```

- 단순하게 매력도만으로 순위 결정
- 기존 가중합 방식의 지출, 수익유발 등 요소 제거

**여성 노출 점수 (남성에게 노출 시 순위 결정)**

```
여성 노출 점수 = W1 × 매력도 + W2 × 좋아요보낸비율 + W3 × 접속빈도
```

| 요소 | 설명 | 가중치 |
| --- | --- | --- |
| 매력도 | 남성들에게 받은 인터랙션 기반 점수 | 0.6 (임시) |
| 좋아요 보낸 비율 | 노출 대비 좋아요 보낸 비율 | 0.25 (임시) |
| 접속 빈도 | 최근 7일 접속 횟수 | 0.15 (임시) |

> W1 + W2 + W3 = 1.0 (가중치 합 제약)
> 

> **설계 의도**: 여성의 경우 매력도뿐 아니라, 남성에게 인터랙션을 유발하는 **활동성**도 고려한다. 활발하게 좋아요를 보내고 자주 접속하는 여성이 더 높은 노출 순위를 받는다.
> 

> 가중치는 `system-config`를 통해 동적 조정 가능하다 (7.11 참조).
> 

### 7.3 버킷 시스템

### 7.3.1 개요

버킷은 노출 경계가 아니라, 점수 계산 시 비교 기준(Reference Group)으로 사용한다. 유저를 연령대·지역으로 그룹핑하여 동일 버킷 내 상대적 위치(percentile)를 산출하는 데 사용된다.

버킷 활용 방식 (Reference Group)

- 후보 생성: 유저 필터(나이/거리/기타) 우선 적용 → 교차 버킷 노출 가능
- 점수 계산: 동일 버킷 내 percentile로 상대적 위치 산출
    - 예: 25세 남성 점수 = "20~26세 남성들 중 상위 몇 %"
- 설계 의도: 연령대별 점수 스케일 차이를 보정해 공정한 랭킹

> **핵심 원칙**: 버킷은 노출을 막는 기준이 아니라, **동일 연령대 내 상대적 위치**를 계산하는 용도로만 사용한다. 유저의 나이/거리 필터 설정이 노출 범위를 결정하며, 버킷과 무관하게 교차 노출이 가능하다.
> 

### 7.3.2 버킷 분류 기준

| 버킷 유형 | 분류 | 설명 |
| --- | --- | --- |
| 연령대 | 20~26세 / 27~33세 / 34~40세 / 41세+ | 7년 단위 구분 |
| 지역 | 수도권 / 비수도권 | 수도권 = 서울, 경기, 인천 |
- 버킷 조합 = 연령대(4) × 지역(2) × 성별(2) = **최대 16개 버킷**
- 유저는 생년월일과 등록 지역 기준으로 **하나의 버킷**에 소속

### 7.3.3 점수 계산 흐름

매력도 점수(7.2)의 절대값은 버킷마다 스케일이 다를 수 있다 (예: 20대 남성과 30대 남성의 평균 점수 차이). 이를 보정하기 위해 **동일 버킷 내 percentile**로 변환하여 교차 버킷 비교를 공정하게 한다.

**실시간: 절대 점수(raw_score) 갱신**

인터랙션 발생 시 해당 유저의 베이지안 점수를 즉시 갱신한다.

```
인터랙션 발생 → raw_score = (α₀ + k) / (α₀ + β₀ + n) 재계산 → DB 저장
```

**사이클 생성 시: 상대 점수(percentile) 계산 → 추출**

```
[사이클 생성 직전]
  1. 버킷별 정렬: 16개 버킷 각각 노출 점수 기준 정렬
     남성 버킷: 노출 점수 = raw_score (매력도)
     여성 버킷: 노출 점수 = W1×매력도 + W2×좋아요보낸비율 + W3×접속빈도 (7.2.5)
  2. 상대 점수 산출: 버킷 내 percentile 계산 → DB 저장
     예: 버킷 "M_20_METRO" 482명 중 153등 → percentile 68.3

[사이클 생성]
  3. 필터 적용: 요청 유저의 나이/거리/기타 필터 → 후보 풀 생성
  4. 추출: 후보 풀에서 percentile(상대 점수) 기준 정렬 → 상위 8명 선택
```

**교차 버킷 비교 예시**

```
후보 풀 (필터 통과, 버킷 혼재)
┌──────────────────────────────────────────┐
│ 유저A (M_20_METRO) percentile: 85  ← 1위 │
│ 유저B (M_27_METRO) percentile: 78  ← 2위 │
│ 유저C (M_20_LOCAL) percentile: 72  ← 3위 │
│ 유저D (M_27_LOCAL) percentile: 70  ← 4위 │
│ ...                                      │
│ → 상위 8명 추출                           │
└──────────────────────────────────────────┘

20대 raw 0.45와 30대 raw 0.40을 직접 비교하면 불공정하지만,
각각 버킷 내 percentile로 변환하면 공정하게 줄 세울 수 있다.
```

**저장 구조**

| 필드 | 갱신 시점 | 설명 |
| --- | --- | --- |
| `raw_score` | 실시간 | 베이지안 매력도 절대 점수 (0~1) |
| `exposure_score` | 사이클 직전 | 노출 점수
남성 = 매력도(raw_score)
여성 = W1×매력도 + W2×좋아요비율 + W3×접속빈도 |
| `bucket_percentile` | 사이클 직전 | 버킷 내 `exposure_score` 기준 상대 위치 (0~1) |
| `bucket_id` | 프로필 변경 시 | 소속 버킷 식별자 (예: M_20_METRO) |

percentile은 사이클 직전에 계산·저장되므로 **항상 최신**이며, 백오피스/분석 대시보드에서도 조회 가능하다.

**설계 의도**: 유저 선호도(필터)는 그대로 반영하고, 점수 정규화 목적만 버킷으로 분리한다. 초기 서비스 규모에서 버킷을 하드 경계로 쓰면 풀이 과도하게 축소되어 평가 축적 속도가 느려질 수 있다.

버킷 연령대 경계 및 지역 분류는 `system-config`를 통해 동적 조정 가능하다 (7.11 참조).

### 7.4 노출 제한

### 7.4.1 개요

특정 조건을 충족하는 유저를 **노출 제한** 상태로 지정한다. 노출 제한의 영향은 성별에 따라 비대칭으로 적용된다.

```
[트리거 1] 베이지안 점수 하위 기준 이하     ──┐
[트리거 2-a] 신고/차단 1건              ──┤
[트리거 2-b] 빠른 패스 3회 누적          ──┼→ "노출 제한" 상태
[트리거 2-c] 첫 노출 3명 전원 미조회/패스  ──┘
```

**성별 비대칭 설계:**

| 성별 | 내 피드 (내가 보는 사람) | 상대 피드 (나를 보는 사람) |
| --- | --- | --- |
| 제한 남성 | **제한 여성만** (하드 격리) | 정상 풀에서 제외 (하위 구간 여성에게 자연 노출 가능) |
| 제한 여성 | **정상과 동일** (자기 구간) | 정상 남성 풀에서 제외, 제한 남성의 후보로만 명시적 등장 |

> **설계 의도**: 여성 유저에게 불쾌한 경험을 방지하는 것이 핵심. 제한 여성의 피드까지 격리하면 여성 풀 희소성(남:여 ≈ 3.5:1)으로 인해 서비스 경험이 과도하게 나빠진다. 따라서 제한 여성은 정상 피드를 유지하되, 정상 남성의 후보 풀에서만 제외한다.
> 

### 7.4.2 트리거 1 — 하위 점수 기반

| 항목 | 규칙 |
| --- | --- |
| 조건 | 버킷 내 percentile이 **하위 기준** 이하 |
| 판정 시점 | 사이클 생성 시 percentile 계산 후 자동 판정 |
| 해제 | percentile이 하위 기준 초과로 회복 시 자동 해제 |

> 하위 기준은 **성별별로 독립 설정**한다. `system-config`를 통해 동적 조정 가능하다 (7.11 참조).
> 

| 파라미터 | 기본값 | 설명 |
| --- | --- | --- |
| `score_restrict_threshold_male` | 15% | 남성 점수 기반 노출 제한 기준 |
| `score_restrict_threshold_female` | 15% | 여성 점수 기반 노출 제한 기준 |

### 7.4.3 트리거 2 — 부정 시그널 기반

| 부정 시그널 | 판정 기준 | 노출 제한 적용 |
| --- | --- | --- |
| **신고/차단** | 1건 접수 | **즉시** |
| **빠른 패스** | impression 후 3초 이내 DISLIKE **3회 누적** | 3회 도달 시 |
| **첫 노출 전원 미조회/패스** | 첫 노출 3명 전원이 미조회 또는 패스 | **즉시** |

**첫 노출 전원 미조회/패스 판정 로직:**

```
IF 해당 유저가 노출된 첫 3명 전원이 미조회 또는 패스:
  → 즉시 노출 제한 적용

  미조회: impression 기록 있으나 프로필 조회 없음
  패스: DISLIKE (시간 무관)
  
"첫 3명" 기준: 계정 생성 또는 프로필 변경 이후 시점의 최초 3명.
  프로필 변경 시 첫 노출 카운트가 리셋되므로, 변경 후 새로 노출된 3명을 기준으로 재판정한다.
```

**빠른 패스 판정 로직:**

```
빠른 패스 여부 = (DISLIKE.createdAt - impression.createdAt) ≤ 3초

  impression.createdAt : POST /matching/impression 서버 수신 시각
  DISLIKE.createdAt    : POST /matching/crush (DISLIKE) 서버 수신 시각
```

> 첫 노출 대상 수(3명), 빠른 패스 누적 횟수(3회), 시간 임계값(3초) 모두 `system-config`를 통해 동적 조정 가능하다.
> 

### 7.4.4 해제 및 복귀

| 트리거 | 해제 조건 | 복귀 상태 |
| --- | --- | --- |
| 하위 점수 | percentile 하위 기준 초과 회복 | 자동 해제 → 정상 노출 |
| 빠른 패스 | **프로필 변경** | SETTLING으로 전환 (7.1.5), 빠른 패스 카운트 초기화 |
| 신고/차단 | **프로필 변경** | SETTLING으로 전환 (7.1.5), 신고 카운트 초기화 |
| 첫 노출 전원 미조회/패스 | **프로필 변경** | SETTLING으로 전환 (7.1.5) |

> 프로필 변경이 유일한 능동적 복귀 경로이다 (7일 내 1회 제한 적용).
> 

### 7.5 사이클 운영

### 7.5.1 개요

추천과 발견을 **동일 사이클**로 통합 운영한다. 

하루 3회, 8시간 간격으로 크론잡이 실행되며, 매 사이클마다 추천 4명 + 발견 4명 = 총 8명을 새로 생성한다.

```
┌──────────┬──────────┬──────────┐
│ 사이클 1  │ 사이클 2  │ 사이클 3  │
│ 00:00    │ 08:00    │ 16:00    │
│ 추천 4명  │ 추천 4명  │ 추천 4명  │
│ 발견 4명  │ 발견 4명  │ 발견 4명  │
└──────────┴──────────┴──────────┘
```

> 사이클 실행 시각은 `system-config`로 조정 가능하다 (7.11 참조).
> 

### 7.5.2 뷰어 유형별 파이프라인 분류

사이클 실행 시 각 유저는 아래 7개 유형 중 하나로 분류되며, 유형에 따라 후보 선택 규칙이 달라진다.

| # | 뷰어 | 내가 보는 후보 | 나를 보는 후보 |
| --- | --- | --- | --- |
| 1 | 기존 남성 (정상) | 기존+신규/정착중 여성 (자기 구간) | 기존+신규/정착중 여성 (자기 구간) |
| 2 | 신규/정착중 남성 (정상) | 기존 여성만 (자기 구간) | 기존 여성 (자기 구간) |
| 3 | 기존 여성 (정상) | 기존+신규/정착중 남성 (자기 구간) | 기존+신규/정착중 남성 (자기 구간) |
| 4 | 신규 여성 (정상) | 기존 남성만 (상위 30% 고정) | 기존 남성 (자기 구간) |
| 5 | 정착중 여성 (정상) | 기존 남성만 (30%→자기구간 점진) | 기존 남성 (자기 구간) |
| 6 | 노출 제한 남성 | **노출 제한 여성만** | 정상 풀 제외 (하위 여성에게 자연 노출 가능) |
| 7 | 노출 제한 여성 | **3번과 동일** (자기 구간) | 정상 남성 풀 제외, 6번의 후보로만 명시적 등장 |

> **자기 구간**: 뷰어의 percentile ±N% 범위. 명시적 범위가 없으면 자기 구간이 기본값이다.
**6번과 7번의 비대칭**: 제한 남성은 하드 격리(제한 여성만), 제한 여성은 정상 피드 유지 (7.4.1 참조).
> 

### 7.5.3 사이클 실행 흐름

```
[크론잡 트리거] (8시간 간격)
    │
    ├─ Phase 1: 버킷 데이터 로드 + percentile 계산
    │   └─ 16개 버킷, 각각 3개 서브 풀 구성:
    │       established_pool / new_settling_pool / restricted_pool
    │
    ├─ Phase 2: 노출 제한 판정
    │   └─ 트리거 1(하위 점수) + 트리거 2(부정 시그널) → restricted 태그 갱신
    │
    ├─ Phase 3: 정상 유저 피드 생성 (유형 1~5, 7)
    │   └─ 유저별 단일 잡 → 추천 4명 + 발견 4명
    │
    ├─ Phase 4: 제한 남성 피드 생성 (유형 6)
    │   └─ restricted_pool(여성)에서 추천 4명 + 발견 4명
    │
    └─ Phase 5: 결과 저장
        ├─ 추천 큐: 유저별 4명
        └─ 발견 큐: 유저별 4명
```

> **Phase 3과 4 분리 이유**: 유형 6(제한 남성)은 후보 풀 자체가 다르므로(restricted_pool) 정상 파이프라인과 분리한다. 유형 7(제한 여성)은 정상 파이프라인(3번)과 동일하므로 Phase 3에서 함께 처리한다.
> 

### 7.5.4 기존 대비 변경

### 7.5.4.1 추천

| 항목 | 기존 | 변경 |
| --- | --- | --- |
| 생성 방식 | 크론잡 (오후 2시 생성, 오후 8시 노출) | **크론잡, 8시간 간격** |
| 주기 | 1일 1회 | **1일 3회** |
| 인원 | 4명 | **4명** (변경 없음) |
| 큐 관리 | 1일 1회 대체 | **사이클마다 대체** |

### 7.5.4.2 발견

| 항목 | 기존 | 변경 |
| --- | --- | --- |
| 생성 방식 | 접속 시 비동기 (Fire-and-forget) | **추천과 동일 크론잡** |
| 주기 | 8시간마다 추가 | **8시간마다 새 큐 생성** |
| 인원 | 가변 (누적) | **4명 고정** |
| 큐 관리 | **누적** (이전 큐 유지 + 새 큐 추가) | **대체** (이전 큐 소멸, 새 큐만 유효) |
| 동작 | Fire-and-forget 비동기 | **추천과 동일한 크론잡** |

> **핵심 변경**: 발견이 접속 기반 누적에서 **크론잡 기반 대체**로 변경되어, 추천과 동일한 사이클에서 동일한 파이프라인(7.6)으로 생성된다.
> 

### 7.5.5 큐 갱신 규칙

| 규칙 | 설명 |
| --- | --- |
| 새 사이클 = 새 큐 | 이전 사이클 큐를 **대체** (누적 아님) |
| 미조회 카드 | 이전 사이클에서 미조회 카드는 소멸. 큐에만 있고 안 본 유저는 **다음 사이클에 다시 나올 수 있음** |
| 중복 기준 | **impression(카운팅) 기준** — 큐 배정이 아닌, 실제 카드를 본(impression 기록된) 유저만 제외 |
| 제외 기간 | **영구** — 한번 본 유저는 다시 나오지 않음 |
| 프로필 변경 시 | 상대방이 프로필을 변경하면 **제외 목록에서 초기화** — 새 프로필이므로 다시 나올 수 있음 |
| 풀 부족 시 대체 | 신규/정착중 슬롯에 후보 부족 시, 기존 유저로 대체 |
| 카드 축소 | 전체 후보 부족 시 카드 수 축소 (4→3→2→1명) |

### 7.6 후보 선택 파이프라인

### 7.6.1 개요

사이클 생성 시 각 유저에게 노출할 **추천 4명 + 발견 4명 = 총 8명**을 선택하는 전체 파이프라인이다. 뷰어 유형(7.5.2)에 따라 후보 풀과 슬롯 규칙이 달라진다.

```
[Phase 1 — 버킷별 워커 16개 병렬]
  Step 1. 버킷 데이터 로드 + percentile 계산 + 서브 풀 3분할
         (노출 제한 판정 포함 — 7.4 기준으로 restricted 태그 갱신)

[Phase 2 — 유저별 단일 잡 (Lambda)]
  Step 2. 뷰어 유형 판별 (7.5.2의 7개 유형 중 1개)
  Step 3. 기본 제외 필터 (하드)
  Step 4. 교차 노출 규칙 (하드)
  Step 5. 유저 설정 필터 (하드)
  Step 6. 슬롯 기반 할당 샘플링 → 추천 4명 + 발견 4명
```

**실행 환경**: 사이클 파이프라인은 **AWS Lambda**(단일 호출)로 실행된다. 상시 API 서버와 분리된 독립 프로세스이므로, 다른 서버와 상태를 공유할 필요가 없다. 따라서 Redis 등 외부 인메모리 스토어 없이 **Lambda 프로세스 메모리**에서 처리하며, 실행 완료 시 메모리가 해제된다.

> **단일 잡 설계**: 한 유저에 대해 1회만 파이프라인을 실행하고, 추천·발견 슬롯을 순차 할당한다. 뽑힌 후보는 즉시 풀에서 제거되므로 추천↔발견 간 중복이 자동 방지된다.
> 

> Phase 1에서 DB의 유저 데이터를 Lambda 메모리에 한 번 적재하고, Phase 2에서는 DB에 재접근하지 않고 메모리 내 데이터만으로 처리한다. 유저별 개별 DB 쿼리를 실행하지 않으므로 DB 부하가 최소화된다. 현재 규모(~8K rows, ~50MB)에서는 단일 Lambda(최대 10GB 메모리)로 충분하다. 유저 규모 확장 시에는 Lambda를 버킷 단위로 분산 호출하면 되며, 각 Lambda가 독립적으로 DB에서 데이터를 로드하므로 별도 공유 스토어 없이 수평 확장이 가능하다
> 

### 7.6.2 [Step 1] 버킷 데이터 로드 + 서브 풀 구성

16개 버킷 각각에서 활성 유저를 로드하고, percentile 계산 후 서브 풀로 분리한다. percentile 계산은 노출 제한 유저를 포함한 **버킷 전체** 기준으로 수행한다 (하위 15% 기준이 버킷 전체 대비여야 의미가 있으므로).

**여성 버킷 (서브 풀 3개)**

```
Worker "F_20_METRO" (여성 버킷 예시, 80명):
  1. 활성 유저 80명 전체 로드 → 서버 메모리
  2. 노출 점수 계산: W1×매력도 + W2×좋아요보낸비율 + W3×접속빈도 (7.2.5)
  3. 80명 전체 기준 노출 점수 정렬 → percentile 계산·DB 저장
  4. 서브 풀 분리:
     established_pool (45명)  ← 정상 ESTABLISHED 유저
     new_settling_pool (25명)  ← 정상 NEW + SETTLING 유저
     restricted_pool (10명)    ← 노출 제한 유저 (트리거 1 or 2 충족)
```

> 여성 버킷은 3개 서브 풀이 필요하다 — 노출 제한 남성(유형 6)이 restricted_pool에서 후보를 뽑아야 하기 때문이다.
> 

**남성 버킷 (서브 풀 2개 + 제외 태그)**

```
Worker "M_20_METRO" (남성 버킷 예시, 80명):
  1. 활성 유저 80명 전체 로드 → 서버 메모리
  2. 노출 점수 = raw_score (매력도만, 7.2.5)
  3. 80명 전체 기준 노출 점수 정렬 → percentile 계산·DB 저장
     (노출 제한 남성도 포함하여 계산)
  4. 서브 풀 분리:
     established_pool (50명)  ← 정상 ESTABLISHED 유저
     new_settling_pool (20명)  ← 정상 NEW + SETTLING 유저
     (제한 남성 10명 → 서브 풀에 넣지 않고 제외 태그만 부여)
```

> 남성 버킷은 restricted_pool이 불필요하다 — 어떤 여성 뷰어도 "제한 남성 풀"에서 후보를 뽑지 않기 때문이다. 제한 남성은 정상 여성의 피드 생성 시 제외 태그로 걸러낸다.
> 

**전체 그룹 수**: 여성 버킷 8개 × 3풀 + 남성 버킷 8개 × 2풀 = **40개 그룹**

| Phase | DB 부하 | 메모리 부하 | 비고 |
| --- | --- | --- | --- |
| Phase 1 | 16 쿼리 (~8K rows) | 버킷당 ~500명 | 전체 1초 미만 |
| Phase 2 | **0** | 9.5K × Lambda 메모리 | DB 접근 없음 |

### 7.6.3 [Step 2] 뷰어 유형 판별

각 유저를 7.5.2의 7개 유형으로 분류한다. 유형에 따라 Step 4(교차 노출), Step 6(슬롯 배분)의 규칙이 달라진다.

| 유형 | 참조할 후보 서브 풀 | 탭당 슬롯 배분 | 노출 범위 |
| --- | --- | --- | --- |
| 1 (기존 남성) | 여성 established + new_settling | 기존 1 + 신규/정착 3 | 자기 구간 |
| 2 (신규/정착중 남성) | 여성 established만 | 기존 4 | 자기 구간 |
| 3 (기존 여성) | 남성 established + new_settling | 기존 3 + 신규/정착 1 | 자기 구간 |
| 4 (신규 여성) | 남성 established만 | 기존 4 | 상위 30% 고정 |
| 5 (정착중 여성) | 남성 established만 | 기존 4 | 30%→자기구간 점진 |
| 6 (제한 남성) | **여성 restricted_pool만** | 제한 4 | restricted_pool 전체 |
| 7 (제한 여성) | **유형 3과 동일** | 기존 3 + 신규/정착 1 | 자기 구간 |

### 7.6.4 [Step 3~5] 하드 필터

하드 필터는 후보를 **포함하거나 제외**하는 필터이다. 조건에 맞지 않는 후보는 후보 풀에서 완전히 제거되며, 이후 단계에서 고려되지 않는다. Step 3~5는 모두 하드 필터이며, 순서대로 적용된다.

**Step 3. 기본 제외 필터 — 시스템이 자동 적용**

모든 뷰어에게 공통으로 적용되는 시스템 규칙이다. 유저가 설정할 수 없다.

| 제외 조건 | 설명 |
| --- | --- |
| 이미 본 유저 | **impression 기록 기준** 영구 제외. 큐에만 있고 안 봤으면 제외 안 함 |
| 프로필 변경 유저 | 상대방이 프로필 변경 시 **제외 목록에서 초기화** → 다시 후보로 등장 가능 |
| 차단/신고한 유저 | 양방향 제외 (내가 차단한 사람 + 나를 차단한 사람 모두) |
| 매칭 성사된 유저 | 이미 매칭된 상대 |
| 비활성 유저 | `inactive_threshold_days`(기본 7일) 이상 미접속 유저 제외 (7.8 참조) |
| 노출 제한 유저 | 정상 뷰어(유형 1~5, 7)의 풀에서 **제한 남성 제외**. 제한 여성은 유형 6의 restricted_pool에서만 등장 |

**Step 4. 교차 노출 규칙 — 뷰어·후보의 분류 조합으로 결정**

뷰어의 분류(NEW/SETTLING/ESTABLISHED)와 후보의 분류 조합에 따라 노출 가능 여부를 판단한다. 신규 유저끼리는 서로 볼 수 없고, 기존 유저를 통해서만 노출된다.

```
                    남성 (후보)
             신규/정착중    기존
          ┌────────────┬────────────┐
 신규/    │     ✕      │     ✓      │
 정착중   │  서로 못봄  │ 여성→남성   │
여 ───────┼────────────┼────────────┤
성 기존   │     ✓      │     ✓      │
(뷰어)    │ 남성→여성   │  서로 봄    │
          └────────────┴────────────┘
```

> 설계 의도: 신규 유저는 검증된 기존 유저에게만 노출되어 좋은 첫인상을 갖게 한다. 기존 유저가 신규 유저를 평가하는 역할을 담당한다.
> 

**Step 5. 유저 설정 필터 — 유저가 직접 설정**

유저가 추천/발견 탭에서 직접 설정한 조건이다. 설정하지 않은 항목은 적용되지 않는다 (전체 허용).

| 필터 항목 | 선택 방식 | DB 컬럼 | 예시 |
| --- | --- | --- | --- |
| 나이 | 구간 선택 | `UserPreferredAge` (minAge/maxAge) | 25~32세만 보기 |
| 학력 | 다중 선택 | `UserProfile.education` | 대학교, 대학원만 보기 |
| 흡연 | 다중 선택 | `UserProfile.smoking` | 비흡연만 보기 |
| 종교 | 다중 선택 | `UserProfile.religion` | 무교, 기독교만 보기 |
| 음주 | 다중 선택 | `UserProfile.drinking` | 음주 안 함만 보기 |

> Step 3~5를 거치면 후보 풀에는 **모든 하드 조건을 통과한 유저만** 남는다. 이 풀에 소프트 필터를 적용하여 최종 선택 확률을 산출한다.
> 

### 7.6.5 [Step 6] 가중치 산출 + 슬롯 기반 할당 샘플링

하드 필터를 통과한 후보 풀에 가중치를 부여하여 최종 선택 확률을 산출한 뒤, 슬롯 규칙에 따라 확률적으로 추천·발견 카드를 뽑는다.

### 7.6.5.1 나이차 점수

뷰어와 후보의 나이차에 따라 **곱셈 방식**으로 가중치를 조정한다. 나이차가 선호 범위를 벗어나면 선택 확률이 대폭 감소한다. 나이 필터 설정 여부와 무관하게 **항상 적용**된다 (필터 범위 내에서도 나이차에 따른 선호도 차이가 있으므로).

```
weight = percentile × 나이차 점수

  percentile: 0~1 범위 (버킷 내 상대 위치, 0.0=최하위 ~ 1.0=최상위)
  나이차 점수: 0.01~1.0 범위
```

| 항목 | 설명 |
| --- | --- |
| DB 소스 | `User.birthDate` — 뷰어·후보 나이차 계산 → 점수표 조회 |
| 여성 피크 | **+3살 연상**이 1.0 (최대), 연하는 급격히 하락 |
| 남성 피크 | **-3살 연하**가 1.0 (최대), 연상은 급격히 하락 |
| 범위 | 0.01 (거의 0) ~ 1.0 (최대) |

> 전체 점수표는 부록 참조. 점수표는 `system-config`를 통해 동적 조정 가능하다 (7.11 참조).
> 

> **나이차 점수를 곱셈으로 분리하는 이유**: 나이는 매칭에서 민감한 요소로, 선호 범위를 크게 벗어나면 다른 조건이 아무리 좋아도 매칭 만족도가 낮다. 곱셈 방식으로 강하게 반영하여 나이 미스매칭을 사전에 방지한다.
> 

### 7.6.5.2 가점 항목

후보에 **가점**을 부여하는 소프트 필터이다. 조건이 맞으면 선택 확률이 올라가고, 맞지 않아도 **불이익은 없다** (×1.0 유지).

```
weight = percentile × 나이차점수 × (1 + MBTI가점) × (1 + 키워드가점) × (1 + 활동가점) × (1 + 관심사가점) × (1 + 직업가점)
```

| 가점 항목 | DB 소스 | 매칭 방식 | 설명 |
| --- | --- | --- | --- |
| MBTI 호환 | `UserProfile.mbti` | 뷰어 MBTI ↔ 후보 MBTI 호환성 | 호환 유형 매칭 시 가점 |
| 취향 키워드 | `UserPreferredKeyword` ↔ `UserKeyword` | 내가 원하는 키워드 = 상대방의 키워드 | 일치 키워드 수에 비례하여 가점 |
| 함께하고 싶은 활동 | `UserProfile.togetherActivity` | 뷰어-후보 활동 일치 | 일치 시 가점 |
| 관심사 | `UserProfileInterest` | 뷰어-후보 관심사 겹침 | 겹치는 관심사 수에 비례하여 가점 |
| 직업 카테고리 | `UserProfile.jobCategoryId` |  동일 카테고리 매칭 시 가점 (임시) | 선호 직업군 매칭 시 가점 |

> 가점 수치는 `system-config`를 통해 동적 조정 가능하다 (7.11 참조). 아래는 임시 기본값이며, A/B 테스트(7.10)를 통해 최적값을 탐색한다.
> 

| 가점 항목 | 임시 기본값 | 설명 |
| --- | --- | --- |
| MBTI 호환 | +0.15 | 호환 유형 매칭 시 |
| 취향 키워드 | 일치 1개당 +0.05 | 최대 3개 = +0.15 |
| 함께하고 싶은 활동 | +0.10 | 일치 시 |
| 관심사 | 겹침 1개당 +0.03 | 최대 5개 = +0.15 |
| 직업 카테고리 | +0.05 | 동일 카테고리 매칭 시 |

**확률 산출**

```
후보 i의 선택 확률 = weight_i / Σ(weight_all_in_sub_pool)

weight = percentile ×  나이차점수(0.01~1) × (1 + 가산점)
```

**샘플링 방식**: 가중 랜덤 샘플링 (Weighted Random Sampling without Replacement)

- 서브 풀에서 가중치 기반으로 1명 추출 → 풀에서 제거 → 반복
- 추천 4명 → 발견 4명 순서로 할당 (중복 자동 방지)

**유형 1: 기존 남성이 보는 여성**

| 탭 | 기존 여성 | 신규/정착중 여성 | 비고 |
| --- | --- | --- | --- |
| 추천 (4명) | 1명 | 1~2명 + 1~2명 | SETTLING 또는 NEW 중 택1 |
| 발견 (4명) | 1명 | 1~2명 + 1~2명 | SETTLING 또는 NEW 중 택1 |

> 신규/정착중 여성 **최대 3명 (75%)**. 전체 풀 평균 75% 목표.
> 

**유형 2: 신규/정착중 남성이 보는 여성**

| 탭 | 기존 여성 |
| --- | --- |
| 추천 (4명) | 4명 |
| 발견 (4명) | 4명 |

**유형 3, 7: 기존 여성 / 제한 여성이 보는 남성**

| 탭 | 기존 남성 | 신규/정착중 남성 | 비고 |
| --- | --- | --- | --- |
| 추천 (4명) | 3명 | 0~1명 | SETTLING 또는 NEW 중 택1 |
| 발견 (4명) | 3명 | 0~1명 | SETTLING 또는 NEW 중 택1 |

> 신규/정착중 남성 **최대 1명 (25%)**. 전체 풀 평균 25% 목표.
> 

**유형 4, 5: 신규/정착중 여성이 보는 남성**

| 탭 | 기존 남성 |
| --- | --- |
| 추천 (4명) | 4명 |
| 발견 (4명) | 4명 |

> 유형 4는 상위 30% 고정, 유형 5는 30%→자기구간 점진 이동 (Step 7 참조).
> 

**유형 6: 제한 남성이 보는 여성**

| 탭 | 제한 여성 (restricted_pool) |
| --- | --- |
| 추천 (4명) | 4명 |
| 발견 (4명) | 4명 |

> 격리 풀 부족 시 카드 수 축소 (4→3→2→1명).
> 

**실행 흐름 예시 (유형 3: 기존 여성 뷰어 A)**

```
후보 풀 (Step 7 통과):
  기존 남성 풀: 80명 (각각 weight 산출됨)
  신규/정착중 남성 풀: 25명

[추천 슬롯 할당]
  1. 기존 풀에서 3명 확률 추출 → 풀에서 제거
  2. 신규/정착중 풀에서 1명 확률 추출 → 풀에서 제거
  → 추천 4명 확정

[발견 슬롯 할당]
  3. 남은 기존 풀(77명)에서 3명 확률 추출
  4. 남은 신규/정착중 풀(24명)에서 1명 확률 추출
  → 발견 4명 확정

결과: 추천 4 + 발견 4 = 8명 (중복 없음)
```

> **풀 부족 시**: 해당 슬롯의 가용 인원만큼만 채우고, 전체 카드 수가 8명 미만이 될 수 있다 (축소).
> 

### 7.7 노출 카운팅

### 7.7.1 카운팅 기준

노출은 **유저가 실제로 프로필 카드를 본 시점**(impression)을 기준으로 카운트한다. 사이클에 배정되었지만 보지 않은 카드는 노출로 집계하지 않는다.

| 기준 | 설명 |
| --- | --- |
| 노출 (카드) | **유저가 프로필 카드를 본 시점** (클라이언트에서 `POST /matching/impression` 호출) |
| 노출 (상세) | **유저가 프로필 상세 페이지를 본 시점** (동일하게 `POST /matching/impression` 호출) |
| 개별 카운팅 | 통으로 묶지 않고 **개별 프로필 단위**로 카운트 |
| 발견 탭 기준 | 화면에 실제로 표시되어야 노출로 인정 |

> **변경 사항**: 기존에는 "프로필이 표시됨"이었으나, 실제로 유저가 카드를 본 시점(impression API 호출)을 기준으로 변경.
> 

> 중복 집계 기준: `viewerId + targetUserId + DATE(createdAt)` — 동일 날짜 내 동일 대상은 1회로 집계 (WEBB-1143 참조).
>

### 7.8 활동 필터 규칙

### 7.8.1 미접속 유저 제외

| 미접속 기간 | 처리 |
| --- | --- |
| 기본 기간 초과 | **노출에서 제외** |
| 기본 기간 이내 | 정상 노출 |

| 파라미터 | 기본값 | 설명 |
| --- | --- | --- |
| `inactive_threshold_days` | 7일 | 미접속 유저 제외 기준 |
| `inactive_threshold_days_extended` | 15일 | 풀 부족 시 확장 기준 |

### 7.8.2 풀 부족 시 대응

풀이 부족할 경우, **활동 기간 확장 → 카드 축소** 순서로 단계적 대응한다.

```
IF 노출 대상 풀 < 필요 노출 수 THEN
  1. 활동 기간 확장: inactive_threshold_days → inactive_threshold_days_extended로 미접속 기준 완화
  2. 여전히 부족 시: 운영팀에 알림 발송 (버킷별 풀 부족 알림)
  3. 여전히 부족 시: 노출 카드 수 축소 (4명 → 3명 → 2명 → 1명)
  4. 최후에도 부족하면 미노출 처리
END IF
```

**절대 하지 않는 것:**

| 금지 사항 | 이유 |
| --- | --- |
| 점수 범위 확대 | 신규 유저에게 전체 유저가 랜덤 노출 → 득보다 실 |
| 자동 버킷 확장 | 매칭 품질 저하 |

### 7.9 유저 상태 변경 처리

### 7.9.1 개요

유저 상태가 변경되었을 때 매력도 점수, 분류, 노출 제한 등 알고리즘 전반에 미치는 영향을 정의한다.

| 상태 변경 | 분류 | 점수 | 노출 제한 | 인터랙션 데이터 |
| --- | --- | --- | --- | --- |
| 비활성화 후 해제 | 유지 | 유지 | 유지 | 유지 |
| 탈퇴 후 재가입 | **NEW 초기화** | **Prior로 초기화** | **해제** | **소프트 삭제** |
| 프로필 변경 | **SETTLING 전환** | **Prior로 초기화** | **무조건 해제** | **소프트 삭제** |

> 설계 의도: 비활성화/해제는 일시적 조치이므로 기존 평가를 보존한다. 탈퇴 후 재가입은 완전히 새로운 시작으로 처리한다.
> 

### 7.9.2 프로필 변경 시 처리

프로필 변경 시 분류 전환 및 데이터 처리 상세는 7.1.5를 참조한다. 알고리즘 전반에 미치는 영향을 정리하면:

| 영향 범위 | 처리 | 참조 |
| --- | --- | --- |
| 유저 분류 | SETTLING으로 전환 | 7.1.5 |
| 매력도 점수 | 인터렉션 전체 초기화 | 7.2.3 |
| 버킷 percentile | 다음 사이클에서 재계산 | 7.3.3 |
| 노출 제한 (시그널) | 빠른 패스/신고 카운트 초기화 → 해제 | 7.4.4 |
| 노출 제한 (점수) | 재계산된 percentile 기준으로 재판정 | 7.4.2 |
| 후보 선택 | 다음 사이클부터 SETTLING 규칙 적용 | 7.6.3 |

**설계 변경: 최근 10회 유지 → 완전 초기화**

기획서 원안은 "최근 10회 인터랙션만 유지"였으나, **인터랙션 전체 초기화**로 변경한다.

| 항목 | 기획서 원안 | 변경안 |
| --- | --- | --- |
| 인터랙션 데이터 | 최근 10회 유지 | **전체 소프트 삭제(점수 계산에서 제외)** |
| 매력도 점수 | 최근 10회 기반 재계산 | **Prior로 초기화** (남성 0.30, 여성 0.35) |
| 노출 제한 | 재계산 결과에 따라 유지 가능 | **무조건 해제** (시그널 + 점수 모두) |

변경 사유:

- 프로필을 변경한 시점에서 이전 유저 정보와 연속성을 가져야 할 이유가 없다 — 새 프로필에 대한 평가가 이전 프로필에 대한 반응과 무관하기 때문 (기획서에서 수정)
- 기존 방식(최근 10회 유지)에서는 하위 유저가 프로필을 바꿔도 과거 부정 데이터 때문에 복귀 불가 → 양극화 심화
- 7일 내 1회 제한으로 악용 방지

**프로필 변경 후 뷰어 유형 (7.6.3 참조):**

| 변경 전 | 변경 후 | 설명 |
| --- | --- | --- |
| 남성 (모든 유형) | **유형 2** (신규/정착중 남성) | Prior 0.30으로 초기화, 기존 여성만 볼 수 있음 |
| 여성 (모든 유형) | **유형 5** (정착중 여성) | Prior 0.35로 초기화, D+4 점진 이동 |

> **제한사항**: 프로필 변경 후 재평가는 **7일 내 1회만** 적용된다 (빈번한 변경을 통한 점수 리셋 악용 방지).

### 7.10 실험 분류 (A/B 테스트)

### 7.10.1 개요

알고리즘 변경 시 **A/B 테스트**를 통해 효과를 검증할 수 있는 실험 인프라를 제공한다. 서버에서 유저별로 실험 그룹을 할당하고, 클라이언트는 `ExperimentServiceInterface`를 통해 실험 변수를 조회한다.

```
[서버]                              [클라이언트]
  유저별 실험 그룹 할당                ExperimentServiceInterface
  실험 변수 API 제공         ───→     initialize() → 변수 캐싱
  알고리즘 버전별 분기 실행             getVariable(id) → 변수 조회
```

> A/B 테스트는 Section 2(가정과 종속)에 포함하지 않으며, 이 섹션에서만 다룬다.
> 

### 7.10.2 서버 — 실험 그룹 할당

**기존 인프라 (임시):**

데이팅 DB에 `AlgorithmTestUser` 테이블이 존재한다 (약 10만 건).

```
model AlgorithmTestUser {
  id        Int      @id @default(autoincrement())
  versionId String   @db.VarChar(50)  // 샘플링 회차 식별자 (ex: 20260225_01)
  gender    Gender
  userId    Int
  createdAt DateTime @default(now())

  @@unique([versionId, gender, userId])
}
```

- `versionId`: 실험 회차 식별자 — 동일 versionId 내에서 유저를 그룹 분배
- 성별별로 독립 샘플링
- 임시 구조로, 범용 실험 인프라로 교체 필요

**TO-BE 설계:**

| 항목 | 설명 |
| --- | --- |
| 그룹 할당 | 유저별로 실험 그룹(A/B/C...) 할당, 사이클 생성 시 그룹에 따라 알고리즘 버전 분기 |
| 할당 기준 | 유저 ID 기반 해시 — 동일 유저는 항상 동일 그룹에 할당되어 재현·반복 가능 |
| 변수 제공 | 실험 그룹별 알고리즘 파라미터를 API로 제공 |
| 지속성 | 한번 할당된 그룹은 실험 기간 동안 유지 (사이클마다 바뀌지 않음) |

### 7.10.3 클라이언트 — ExperimentServiceInterface

문토 앱의 `ExperimentServiceInterface`를 데이팅에도 동일하게 적용한다.

```dart
abstract class ExperimentServiceInterface {
  Future<void> initialize();      // 서버에서 변수 fetch + 로컬 캐싱
  Future<void> refresh();          // 포그라운드 복귀 시 갱신
  T? getVariable<T>(String experimentId, {T? defaultValue});
  bool hasVariable(String experimentId);
  ExperimentVariablesMap getAllVariables();
  Future<void> clearCache();       // 로그아웃 시 캐시 제거
}
```

| 특성 | 설명 |
| --- | --- |
| 캐싱 | 로컬 캐시 우선, 네트워크 실패 시에도 캐시된 값으로 동작 |
| 갱신 | 앱 초기화 시 + 포그라운드 복귀 시 서버에서 최신 변수 fetch |
| 공유 | munto-mobile에서 API 호출 로직 구현, 데이팅은 코어 캐시 공유 |

### 7.10.4 실험 가능 파라미터

알고리즘의 `system-config` 파라미터를 실험 변수로 오버라이드할 수 있다.

| 카테고리 | 실험 가능 항목 | 예시 |
| --- | --- | --- |
| 점수 계산 (7.2) | Prior 파라미터, 인터랙션 가중치 | 남성 Prior Beta(3.0,7.0) vs Beta(2.5,7.5) |
| 버킷 (7.3) | 연령대 경계, 지역 분류 | 7년 단위 vs 5년 단위 |
| 필터 (7.6.6~7) | 나이차 점수표, 가산점 수치 | 나이차 피크 ±3 vs ±2 |
| 노출 (7.6.7~8) | 슬롯 배분 비율, 확률 함수 가중치 | 기존:신규 = 75:25 vs 60:40 |
| 사이클 (7.5) | 사이클 횟수, 실행 시각 | 3회/일 vs 4회/일 |
| 노출 제한 (7.4) | 하위 기준, 빠른 패스 임계값 | 하위 15% vs 10% |

### 7.12 AI 학습용 로그 수집

### 7.12.1 개요

v1에서는 추후 AI/ML 전환을 대비하여, `ActionLog` 테이블에 추천 시점의 유저 행동(Input/Output)을 기록하는 구조를 구축하였다. `metadata` 컬럼이 jsonb이므로 스키마 변경 없이 v2 신규 항목을 추가할 수 있다.

```
ActionLog (기존, 변경 없음):
  id, userId, targetUserId, actionType, metadata (jsonb),
  deviceType, appVersion, osVersion, deviceInfo, createdAt
```

> v1의 `ActionLog` 구조는 v2에서도 **그대로 유지**한다. v2 신규 항목은 `metadata` jsonb에 추가 기록한다.
> 

### 7.12.2 v2 신규 로그 항목

기존 로그(프로필, 사진 URL, 행동)에 추가로 다음 항목을 기록한다. 나중에 AI로 전환할 때 이 항목들이 없으면 해당 기간의 데이터를 학습에 활용할 수 없으므로, v2 설계 시점에 포함한다.

**추천 시점 기록 (Input)**

| 항목 | 설명 | 학습 용도 |
| --- | --- | --- |
| 뷰어 분류 상태 | 추천 시점의 뷰어 NEW/SETTLING/ESTABLISHED | 유저 성숙도에 따른 반응 패턴 학습 |
| 후보 분류 상태 | 추천 시점의 후보 NEW/SETTLING/ESTABLISHED | 유저 성숙도에 따른 반응 패턴 학습 |
| 뷰어 유형 | 7.5.2의 7개 유형 중 어느 것으로 노출되었는지 | 노출 맥락(context)에 따른 반응 차이 학습 |
| 뷰어 버킷 ID + percentile | 뷰어의 버킷 소속 및 상대적 위치 | 상대적 매력도 위치가 반응에 미치는 영향 학습 |
| 후보 버킷 ID + percentile | 후보의 버킷 소속 및 상대적 위치 | 동일 |
| 뷰어 가입 경과일 (D+N) | 추천 시점 기준 가입 후 N일 | 서비스 이용 기간에 따른 선호도 변화 학습 |
| 후보 가입 경과일 (D+N) | 추천 시점 기준 가입 후 N일 | 서비스 이용 기간에 따른 선호도 변화 학습 |
| 노출 제한 여부 + 트리거 | 뷰어·후보 각각의 제한 상태 및 트리거 유형 | 제한 적용/해제 전후 반응 패턴 학습 |
| 사이클 번호 | 당일 몇 번째 사이클인지 (1/2/3) | 시간대별 반응 차이 학습 |
| 슬롯 유형 | 추천 탭 / 발견 탭 | 추천과 발견에서의 반응 차이 학습 |

**행동 기록 (Output)**

| 항목 | 설명 | 학습 용도 |
| --- | --- | --- |
| impression → 행동까지 소요 시간 | impression 시점부터 LIKE/DISLIKE/프로필 조회까지 경과 시간 | 관심도 강도 학습 (빠른 좋아요 vs 오래 본 후 좋아요) |
| 프로필 변경 이력 | 변경 전/후 시점 기록 | 프로필 변경이 받는 반응에 미치는 영향 학습 |

> 위 항목들은 `ActionLog.metadata` jsonb에 추가 기록한다. 기존 v1 metadata 필드는 그대로 유지한다.
> 

**metadata 예시 (v2 확장)**

```json
{
  // v1 기존 항목 (유지)
  "profileData": { ... },
  "photoUrls": [ ... ],

  // v2 신규 항목 (추가)
  "v2": {
    "viewerClassification": "ESTABLISHED",
    "targetClassification": "NEW",
    "viewerType": 1,
    "viewerBucketId": "M_27_METRO",
    "viewerPercentile": 0.72,
    "targetBucketId": "F_20_METRO",
    "targetPercentile": 0.65,
    "viewerDaysSinceSignup": 45,
    "targetDaysSinceSignup": 3,
    "viewerIsRestricted": false,
    "targetIsRestricted": false,
    "cycleNumber": 2,
    "slotType": "recommendation",
    "impressionToActionSeconds": null
  }
}
```
