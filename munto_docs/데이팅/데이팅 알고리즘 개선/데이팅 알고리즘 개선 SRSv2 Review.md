## 2. 미반영 / 부분 반영 항목

### R-01. DB 스키마 — **반영 확인 완료**

ERD 별도 문서(`데이팅 알고리즘 개선 SRSv2 ERD.md`)가 작성되어 있으며, R-01에서 요구한 수준을 충족한다.

**잘 된 점:**
- `UserAlgorithmScore`, `UserAlgorithmState`, `CycleQueue` 3개 신규 테이블이 **CREATE TABLE SQL**로 정의 (컬럼명, 타입, 기본값, nullable, FK 모두 명시)
- ENUM 타입 (`Classification`, `RestrictionTrigger`, `TabType`) 정의 포함
- 인덱스 설계 적절 (`bucketId+exposureScore`, `bucketId+bucketPercentile`, `userId+cycleDate+cycleNumber+tabType` 등)
- R-01 원본에서 우려했던 "`User` 테이블 직접 컬럼 추가" 대신 `UserAlgorithmState` 별도 테이블로 분리 — 관심사 분리 측면에서 더 나은 설계
- `interactionResetAt` 컬럼으로 소프트 삭제 기준 시점 관리 (R-10 반영과 연동)
- 폐기 예정 테이블 6개, 기존 유지 테이블 목록 정리

**경미한 확인 사항:**
- `firstExposurePassCount` / `firstExposureTotal` 두 컬럼의 판정 로직(`passCount >= total AND total >= first_exposure_count`)은 SRS 7.4.3에서 커버되므로 ERD에서 별도 명시 불필요하나, 두 컬럼의 관계가 ERD만 보면 직관적이지 않을 수 있음

### R-02. API 스펙 — **반영 확인 완료 (경미한 불일치 1건)**

API 별도 문서(`데이팅 알고리즘 개선 SRSv2 API.md`)가 **OpenAPI 3.0.3 포맷**으로 작성되어 있으며, R-02에서 요구한 수준을 충족한다.

**잘 된 점:**
- 모든 엔드포인트(6개)에 request/response JSON 스키마, 상태 코드, 에러 코드 포함
- Impression API의 에러 케이스 3종(400 INVALID_TARGET_USER, 401 UNAUTHORIZED, 404 USER_NOT_FOUND) 명시
- Crush API의 에러 케이스가 `examples`로 복수 케이스(invalidTarget, invalidType, alreadyExpressed) 제공
- 각 API description에 SRS 섹션 참조(7.4.3, 7.5.5, 7.7 등) 명시되어 구현 시 맥락 파악 용이
- `CardListResponse`에 `cycleNumber`, `expiresAt` 추가로 v2 사이클 구조 반영

**확인 필요 (1건) — 신고/차단 해제 조건 불일치:**

`/safety/report`와 `/safety/block-user` 모두 description에 **"프로필 변경으로 해제 불가, 운영팀 수동 해제만 가능"**이라고 되어 있다. 그런데 SRS 7.4.4에서는 신고/차단의 해제 조건이 **"프로필 변경 → SETTLING 전환, 신고 카운트 초기화"**로 정의되어 있어 **SRS와 API 문서 간 불일치**가 있다.

| 문서 | 신고/차단 해제 조건 |
|------|---------------------|
| SRS 7.4.4 | 프로필 변경 시 해제 (SETTLING 전환, 카운트 초기화) |
| API 문서 | 프로필 변경으로 해제 불가, 운영팀 수동 해제만 가능 |

→ 어느 쪽이 맞는지 확인하여 통일할 것. 정책적으로 중요한 차이이므로 구현 전 확정 필요.

### R-08. 여성 ESTABLISHED 전환 조건 — **확인 완료 (의도된 동작)**

v2에서도 7.1.4의 여성 ESTABLISHED 조건은 "가입 D+7 이후 (인터랙션 횟수 무관)"으로 동일하다. 개발자 확인 결과 **의도된 동작**이다. 기획 과정에서 여성 유저는 7일 기간 동안 15회 이상 인터랙션을 달성할 것이라고 전제하였으므로, 인터랙션 횟수를 별도 조건으로 걸지 않는다.

### R-14. 유형 5 점진 이동 함수 — 부분 반영 (구현 블로커 아님)

v2에서도 유형 5(정착중 여성)의 "30% → 자기구간 점진 이동" 구체 스펙(이동 함수, 일별 범위)이 명시되어 있지 않으나, **선형 보간(`range = 30% + (selfRange - 30%) × (D / 7)`)이 자연스러운 기본 해석**이므로 구현 진행에 문제 없다. 필요 시 system-config 파라미터로 조정 가능하게 구현하면 된다.

## 3. 총평

**v1 리뷰 의견 14건 중 13건이 반영/확인 완료**되었으며, 특히 다음 개선이 두드러진다:
- R-01(ERD): `UserAlgorithmState` 별도 테이블 분리 설계, 소프트 삭제용 `interactionResetAt` 컬럼 등 구현 즉시 착수 가능한 수준
- R-02(API): OpenAPI 3.0.3 포맷의 상세 스키마로 QA·프론트엔드 연동까지 커버
- R-09(여성 노출 점수 흐름)의 `exposure_score` 도입은 파이프라인 정합성을 크게 향상
- R-07(AI 학습 로그) 7.12 섹션의 상세도가 높아 구현에 충분
- R-10(소프트 삭제) 명시로 감사 추적과 점수 계산 로직의 책임이 명확해짐
- R-13(percentile 단위) 0~1 통일로 구현 혼선 방지

**구현 착수 전 확인이 필요한 사항** (1건):

| # | 항목 | 블로커 수준 | 설명 |
|---|------|------------|------|
| 1 | 신고/차단 해제 조건 불일치 (R-02) | **중간** | SRS 7.4.4에서는 "프로필 변경 시 해제"인데, API 문서에서는 "운영팀 수동 해제만 가능". 정책 확인 필요 |

R-14(유형 5 점진 이동 함수)는 선형 보간이 자연스러운 기본 해석이므로 블로커에서 제외. **구현 진행 가능**하며, R-02 정책 확인만 병행하면 된다.
