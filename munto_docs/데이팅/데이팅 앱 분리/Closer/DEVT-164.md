# [취향인연] 데이팅 앱 독립 분리

| 항목 | 내용 |
| --- | --- |
| 배경 | 데이팅 기능을 문토 앱에서 독립된 별도 앱으로 분리할 예정입니다. 기획 확정 전에 인프라 및 앱 등록 등 분리를 위한 사전 준비 작업을 진행합니다. |
| 현재 문제점 | 데이팅 앱이 문토 앱 내 패키지 형태로 동작하고 있어 Firebase 프로젝트, 스토어 앱 등이 분리되어 있지 않습니다.<BR>Analytics, FCM, Remote Config 데이터가 문토와 혼재될 수 있습니다. |
| 개선 방향 | 1. Firebase 프로젝트 분리 (FCM, Analytics, Remote Config) 및 google-services.json / GoogleService-Info.plist 교체<BR>2. App Store Connect 데이팅 앱 신규 등록 (번들 ID: kr.munto.closer)<BR>3. Google Play Console 데이팅 앱 신규 등록<BR>4. dating-backend Firebase 프로젝트 분리 (FCM 푸시 독립)<BR>5. Sendbird / Bootpay 웹훅 URL 데이팅 전용으로 변경 (munto-backend 경유 제거) |
| 고려요소 | 인증(JWT) 구조는 기존 문토 토큰 호환 방향으로 별도 이슈(WEBB-1196)에서 결정 예정이므로 본 이슈에서는 다루지 않습니다. <BR>데이팅 DB User.id = 문토 userId FK 통합 구조는 그대로 유지합니다. <BR>디자인 시스템(muntorial) 교체 및 신규 기능은 기획 확정 후 별도 이슈로 진행합니다. <BR>딥링크의 경우 문토 앱에 구현된 딥링크 구조를 참고하여 데이팅 앱에도 동일하게 적용 가능한지 검토가 필요합니다. (도메인, App Links / Universal Links, 푸시 알림 딥링크 등) |
| 기대효과 | 기획 확정 즉시 데이팅 앱 독립 출시가 가능한 상태로 준비됩니다. <BR>Firebase 데이터가 앱별로 분리되어 분석 정확도가 향상됩니다.<BR>스토어 앱 슬롯 선점으로 출시 일정 리스크가 감소합니다. |

**먼저 정리 (구현·일정 전)**

- closer v1.0이 통합 JWT(WEBB-1196, `SRS.md` §2.6 278행)와 자체채팅(채팅내제화 SRS, §2.6 279행) 위에 올라가는데 둘 다 "가정"으로만 잡혀 있습니다. 특히 JWT는 DEVT-164 고려요소에서 본 이슈 범위 밖으로 빼 둔 항목이라, 가정이 아니라 담당·마감이 박힌 선행 의존성으로 올리고 둘 중 하나라도 지연될 때의 대응을 명시해 주세요.
- §2.2 변경 성격 다이어그램(206행)이 임베드 데이팅을 자체채팅에 연결하면서 동시에 "변경 없음"으로 분류(193~195행)합니다. 현 데이팅 채팅은 Sendbird라 임베드도 전환이 필요하니, 임베드 채팅 전환·DM 마이그레이션의 소유(어느 프로젝트가·언제)를 한 줄 짚어 주세요.

**확인 권장**

- 캔디를 기본값 Bootpay 실결제로 팔고 심사 때만 `use_iap`로 인앱결제 토글하는 방식(§7.2.1 646행)은 신규 앱 첫 심사에서 Apple Guideline 3.1.1 위반으로 반려될 소지가 있습니다. 결제·사업 담당과 확인하고, 필요 시 첫 출시 시나리오를 §7.10.1에 명시해 주세요.
- (경미) 401 단일-flight refresh 구현에 필요한 refresh 엔드포인트 계약(TBD-1 861행)이 closer 착수(§7.1.4)보다 먼저 확정되도록 순서만 챙겨 주세요.
