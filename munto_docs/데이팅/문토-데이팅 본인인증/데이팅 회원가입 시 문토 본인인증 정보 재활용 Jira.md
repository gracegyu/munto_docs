# 데이팅 회원가입 시 문토 본인인증 정보 재활용 (서버-서버 API 연동)


| 항목 | 내용 |
| --- | --- |
| 배경 | 데이팅 회원가입 시 본인인증(Bootpay)을 요구하고 있으나, 문토 플랫폼에 이미 본인인증을 완료한 유저가 많음. 두 서비스는 DB가 분리(`munto` / `munto_dating`)되어 있어 직접 DB 조회는 불가하지만, 양쪽 모두 동일한 `hashedCI`(Bootpay unique 값의 SHA256 해시)를 저장하고 있어 연동 기반이 존재함 |
| 현재 문제점 | 문토에서 이미 본인인증을 완료한 유저도 데이팅 가입 시 동일한 본인인증을 처음부터 다시 수행해야 함. 이는 중복 작업으로 UX를 저하시키고, 유저 이탈 가능성을 높임 |
| 개선 방향 | 1. **문토 백엔드**에 `GET /v3/user/me/personal-authentication` 엔드포인트 추가 - JWT 인증 기반으로 해당 유저의 본인인증 정보(`name`, `birth`, `gender`, `hashedCI`) 반환 <br> 2. **데이팅 백엔드**에서 가입 플로우 중 문토 토큰으로 위 API를 서버-서버 호출하여 본인인증 정보를 받아옴 <br> 3. 문토에 본인인증 정보가 존재할 경우, 데이팅의 `IdentityVerification` 테이블에 `hashedCI` 자동 저장 및 `User` 테이블의 `name`, `birthDate`, `gender` 자동 완성 → 별도 Bootpay 인증 절차 스킵 <br> 4. 문토에 본인인증 정보가 없을 경우 기존 Bootpay 인증 플로우 유지 |
| 고려요소 | • **보안**: 서버-서버 통신 시 인증 방식 확인 필요 (문토 JWT 그대로 전달 vs 별도 서비스 토큰) <br> • **개인정보**: 실명, 생년월일 등 민감 정보를 서비스 간 전달하므로 전송 구간 암호화(HTTPS) 필수 확인 <br> • **중복 방지**: `hashedCI` 기반 중복 인증 체크 로직은 기존과 동일하게 유지 <br> • **미성년자 체크**: 자동 완성 시에도 만 19세 미만 여부 검증 필수 <br> • **롤백**: 문토 API 호출 실패 시 기존 Bootpay 플로우로 fallback 처리 |
| 기대효과 | • 문토에서 이미 본인인증을 완료한 유저는 데이팅 가입 시 인증 단계 스킵으로 가입 전환율 향상 <br> • 유저 경험 개선 (동일한 인증을 두 번 하는 불편함 제거) <br> • 향후 타 서비스 확장 시에도 재사용 가능한 문토 인증 조회 API 기반 마련 |

### 1. 근본 문제: 본인인증이 왜 두 곳에 있는가

userId 통합 때와 동일한 문제입니다. 본인인증(Identity Verification)은 결제(Payment)와 마찬가지로 **서비스 공통 도메인**인데, 문토에는 `PersonalAuthentication`, 데이팅에는 `IdentityVerification` + `User`로 각각 별도 관리되고 있습니다. 이번 OnePager는 이 이중 구조를 유지한 채 **스냅샷 복사 파이프라인을 추가**하는 방식입니다.

### 2. 이 방식으로 진행하면 생기는 리스크

- **데이터 정합성**: 한쪽에서 재인증(개명 등)하면 다른 쪽은 옛 데이터 유지 → 양방향 실시간 동기화가 없으면 불일치가 영구적
- **복잡도**: JWT/M2M 이중 인증 경로, 신뢰 게이트, 스키마 변환 매핑, 폴백 로직 등 Risk Assessment에 9가지 위험 시나리오가 나열됨 — 근본 설계 문제의 신호
- **확장성**: 서비스가 추가될 때마다 n(n-1) 양방향 연동 필요

### 3. 그렇다고 지금 전면 재설계는 현실적이지 않음

결제도 `munto-backend`, `dating-backend`에서 각각 Bootpay를 호출하는 구조(서버사이드 결제 OnePager 참고)이고, 본인인증만 먼저 통합하기는 어렵습니다. 현재 userId 통합 완료, 서버사이드 결제 전환 안정화 등 진행 중인 작업이 있는 상황에서 본인인증 도메인 통합까지 동시에 진행하면 변경 범위가 과도합니다. 따라서 **현 단계에서는 이 방식(스냅샷 복사)으로 진행하되**, 아래를 OnePager에 반영해주세요.

### 4. OnePager에 추가/보완 요청 사항

1. **과도기적 해결책임을 명시** — "본인인증 도메인 통합(공유 DB 또는 별도 서비스)이 근본 해결이며, 이번 연동은 과도기적 조치"라는 점을 문서 상단에 기재

## **문제 — 실패 케이스에서 사용자가 블로킹 대기**

스냅샷 GET은 서버 간 네트워크 호출이므로, Onepager에 추가 예정인 탄력성 정책**(재시도 1회, 전체 응답 타임아웃 5초)** 기준으로 최악의 경우 아래 시간이 소요됩니다.

| 시도 | 전체 응답 타임아웃 | 백오프 대기 |
| --- | --- | --- |
| 최초 시도 | 5초 | — |
| 재시도 1회 | 5초 | 200ms |
| **합계** | **최대 ~10초** |  |

5xx·타임아웃이 연속 발생하는 최악의 경우, 사용자는 본인인증 화면에서 **약 10초를 블로킹 상태로 대기**한 뒤 Bootpay 화면을 보게 됩니다.

일반적인 성공 케이스는 수백 ms~1~2초 이내이지만, 실패 케이스에서 사용자 경험이 크게 나빠지는 구조입니다.

## **대안 — OnboardScreen 진입 시 백그라운드 Prefetch**

코드 기준 데이팅 가입 플로우는 아래 순서입니다.

```
OnboardScreen (/dating/onboard)  → TermsAgreementModal (약관 동의 바텀시트) → 회원가입 API  → AuthenticationScreen (/dating/onboard/auth)  ← 현재 호출 시점
```

`AuthenticationScreen` 직전 풀스크린 라우트인 `OnboardScreen` **진입 시점**에 스냅샷 GET을 백그라운드로 먼저 호출해두면, 사용자가 약관 동의·회원가입 API를 거치는 동안 응답이 도착합니다.

```
[OnboardScreen 진입]    ↓ (동시에 백그라운드에서)스냅샷 GET 요청 시작    ↓CTA 탭 → TermsAgreementModal 약관 동의 → 회원가입 API 완료    ↓[AuthenticationScreen 진입]결과 이미 있음 → 즉시 분기 (대기 없음)결과 아직 없음 → 짧은 로딩 (대부분 이미 경과됨)결과 실패     → 즉시 Bootpay 표시
```

이 방식은 사용자가 UI를 조작하는 시간 동안 요청이 이미 진행되기 때문에, 본인인증 화면 진입 시 체감 대기 시간이 **대폭 줄어들게 됩니다.**

### 2. 방안 B를 선택하지 않는 이유

방안 B는 타임아웃과 재시도를 줄여서 실패 시 빠르게 폴백하자는 접근인데, 이는 **문제의 근본 원인(호출 시점이 늦음)을 해결하는 것이 아니라 실패 시 피해를 줄이는 것**에 불과합니다. 타임아웃을 줄이면 정상 응답도 놓칠 수 있어, 스냅샷 활용률 자체가 떨어질 수 있습니다. 반면 방안 A는 성공 케이스에서도 대기 없이 즉시 분기가 가능하므로 상위 호환입니다.

### 3. 방안 A 구현 시 유의사항

방안 A를 채택하더라도 아래 사항을 OnePager에 반영해 주시기 바랍니다.

1. **Prefetch 결과 캐싱 범위**: `OnboardScreen` 진입 시 시작한 요청의 결과(Promise 또는 상태)를 `AuthenticationScreen`까지 어떻게 전달할 것인지 — 상태 관리(예: React Query 캐시, 전역 상태 등) 방식을 명시해 주세요.
2. **Prefetch 실패 시 재시도 여부**: 백그라운드 Prefetch가 실패한 경우, `AuthenticationScreen` 진입 시점에 한 번 더 시도할 것인지, 바로 Bootpay 폴백으로 갈 것인지 정책을 정해 주세요. 저는 **재시도 없이 즉시 Bootpay 폴백**을 권장합니다 — 이미 충분한 시간이 경과했으므로 같은 요청을 반복해도 성공 가능성이 낮고, 사용자 대기만 늘어납니다.
3. **탄력성 정책은 여전히 필요**: 방안 A를 채택하더라도 Prefetch 요청 자체의 타임아웃·재시도 정책(R-08에서 언급한 탄력성 섹션)은 OnePager에 명시해야 합니다. Prefetch이므로 타임아웃을 여유 있게 잡을 수 있지만, 무한정 대기하면 안 되므로 상한선은 설정해 주세요.

### 4. 김범진님 의견에 대해

김범진님이 "최소 호출 원칙의 관점에서 B 방안도 괜찮을 것 같은데 실제 API 응답 시간을 확인해봐야 할 것 같습니다"라고 하셨는데, **실제 API 응답 시간과 무관하게 방안 A가 우위입니다.** API가 빠르더라도(수백 ms) Prefetch가 손해 볼 것이 없고, API가 느리면 Prefetch의 이점이 더 커집니다. 최소 호출 원칙도 방안 A에서 위배되지 않습니다 — `OnboardScreen`에 진입한 사용자는 데이팅 가입 의사가 있는 사용자이므로, 본인인증 확인이 필요한 대상입니다. 불필요한 호출이 아닙니다.

위 유의사항을 OnePager에 반영하여 업데이트해 주세요.
