# 문토-데이팅 본인인증 정보 연동 OnePager v1.1.0 리뷰

- **대상 문서:** `문토-데이팅 본인인증 정보 연동 OnePager.md` (v1.1.0, 2026-04-06)
- **리뷰 관점:** 아키텍처 전제, 구현 세부 확정 사항, API 스펙 완결성, 운영 고려사항
- **리뷰일:** 2026-04-07

---

## 1. 전체 평가

전반적으로 잘 작성된 OnePager입니다. 특히 양방향 시나리오(문토→데이팅, 데이팅→문토)를 모두 설계한 점, GET만 사용하는 원칙(선행 DB에 쓰기 없음), 폴백 전략의 명확함, Risk Assessment 9가지 시나리오 상세 작성, 스냅샷 필드 문서에서 양쪽 스키마 차이를 정확히 분석한 점이 좋습니다.

아래는 보완이 필요하다고 판단되는 항목입니다.

---

## 2. 전체 문서에 해당하는 리뷰 의견

### R-01 (P0). 과도기적 조치임을 문서에 명시

본인인증(Identity Verification)은 결제(Payment)와 마찬가지로 서비스 공통 도메인입니다. userId 통합 때와 동일한 구조적 문제로, 문토에는 `PersonalAuthentication`, 데이팅에는 `IdentityVerification` + `User`로 같은 데이터가 이중 관리되고 있습니다. 이번 OnePager는 이 이중 구조를 유지한 채 스냅샷 복사 파이프라인을 추가하는 방식입니다.

현 시점에서 본인인증 도메인 통합까지 진행하기는 어렵지만(서버사이드 결제 전환 안정화 등 진행 중인 작업 존재), **이번 연동이 과도기적 조치이며 근본 해결은 본인인증 도메인 통합(공유 DB 또는 별도 서비스)**이라는 점을 배경 섹션 상단에 명시해 주시기 바랍니다.

### R-02 (P0). 재인증 시 동기화 정책 누락

현재 문서에는 "해당 데이터베이스에 인증 레코드가 없을 때 한 차례 적재"라고만 기재되어 있습니다. 아래 시나리오에 대한 정책이 빠져 있습니다:

- 문토에서 인증 → 데이팅에 스냅샷 복사 완료 → 사용자가 개명 후 문토에서 재인증 → 데이팅은 옛 이름 유지

"무시한다"도 정책이지만 명시가 필요합니다. 이벤트 기반 갱신, 후행 재접속 시 재조회, 또는 재인증 시 동기화 미수행 중 하나를 선택하여 기재해 주시기 바랍니다.

### R-03 (P1). 스냅샷 GET 호출 시점이 특정되지 않음

데이팅 가입 플로우의 **어느 단계에서** 스냅샷 GET을 호출하는지 명시가 없습니다:
- 데이팅 회원가입 시작 시점?
- 본인인증 화면 진입 시점?
- Bootpay 화면을 띄우기 직전?

서버사이드 결제 OnePager에서 `onConfirm` 시점을 명확히 특정한 것처럼, 클라이언트/백엔드 플로우 상 정확한 호출 위치를 명시해 주시기 바랍니다. 호출 시점에 따라 클라이언트 구현과 UX 플로우가 달라집니다.

### R-04 (P2). 공수 재검토

클라이언트 10h, 문토 BE 10h, 데이팅 BE 10h (총 30h)로 산정되어 있습니다. Risk Assessment에 9가지 시나리오가 있고, 양방향 흐름 + 스키마 변환 매핑(스냅샷 필드 문서: `gender` 숫자→enum, `birth` 문자열→DateTime 등) + JWT/M2M 이중 인증 + 폴백 로직 + 테스트까지 고려하면 각 축 15~20h이 현실적으로 보입니다.

## 3. 특정 섹션/문장에 해당하는 리뷰 의견

### R-05 (P1). 시나리오 A — 호출 주체 혼재

> 데이팅: 동일 사용자가 데이팅에 진입할 때 문토 스냅샷 GET(Authorization: Bearer 문토 JWT)을 수행합니다.

> 문토 스냅샷 경로는 앱이 Bearer로 직접 호출할 수 있으며, 데이팅 측은 일반적으로 백엔드를 경유합니다.

앱이 직접 호출하는 건지, 데이팅 백엔드가 문토 JWT를 프록시하여 서버 간 호출하는 건지 **하나로 확정**해 주시기 바랍니다. API 명세에서는 "데이팅 백엔드는 클라이언트가 전달한 Munto 액세스 토큰을 Authorization 헤더에 그대로 실어 이 API를 호출한다"로 되어 있으므로, 후자(데이팅 BE 프록시)가 의도인 것으로 보입니다. OnePager 본문도 이에 맞게 통일이 필요합니다.

### R-06 (P1). 시나리오 B — `datingUserId` 확보 경로 불명확

> 문토 백엔드: 후행 진입 시 데이팅 internal 스냅샷 GET(M2M)을 호출합니다. datingUserId는 서버 연계로 확정된 값만 허용됩니다.

문토 백엔드가 `datingUserId`를 어떻게 알아내는지 구체적 경로가 없습니다. "OAuth·계정 연계 저장소에서만 확정"이라고 했는데, 이것이 현재 존재하는 저장소인지, 새로 구현이 필요한 건지 명시해 주시기 바랍니다.

userId가 이미 통합된 상태라면 문토 userId = 데이팅 userId이므로 별도 매핑 없이 동일 값을 사용할 수 있는 것 아닌지 확인이 필요합니다. 이에 따라 시나리오 B의 공수가 달라집니다.

### R-07 (P1). API 명세 — `verified: false` 응답 정책 미확정

> verified가 true면 인증 완료 사용자. false면 200으로 내릴 수 있음(팀 정책에 따라 404 대체).

`200 + verified: false`와 `404`는 후행 BE의 분기 로직이 완전히 다릅니다. 특히 폴백 트리거 조건에 직접 영향을 주므로, "정책에 따라"로 두지 말고 구현 전에 확정해 주시기 바랍니다.

### R-08 (P1). Risk Assessment — 탄력성 섹션 누락

> 재시도·서킷 브레이커·폴백은 아래 Technical 「탄력성」에 따릅니다.

참조하고 있는 Technical Description의 「탄력성」 섹션이 문서에 존재하지 않습니다. 서버사이드 결제 OnePager처럼 타임아웃 계층화(서버→선행 BE 몇 초, 클라이언트→서버 몇 초)와 재시도 정책(횟수, 4xx 시 재시도 여부)을 명시해 주시기 바랍니다.

## 5. 결론

기능 설계의 전체 구조(선행 POST → 후행 GET → 검증 → 저장/폴백)는 잘 정리되어 있습니다. P0 의견 2건(과도기적 조치 명시, 재인증 동기화 정책)과 P1 의견 5건(호출 시점·주체 확정, userId 매핑, 응답 정책, 탄력성)을 반영하면 **개발 진행 가능**합니다.
