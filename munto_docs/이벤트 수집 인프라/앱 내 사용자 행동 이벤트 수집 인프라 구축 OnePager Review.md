# 앱 내 사용자 행동 이벤트 수집 인프라 구축 OnePager Review

- **대상 문서:** `앱 내 사용자 행동 이벤트 수집 인프라 구축 OnePager.md` (v1.0.0, 26.03.26)
- **리뷰 관점:** Jira 요구사항 반영 여부, 구현 가능성, DB/API 스펙 수준, 운영 고려사항
- **리뷰일:** 2026-03-26

---

## 1. 전체 평가

전반적으로 잘 작성된 OnePager입니다. 특히 DB 설계 방식 비교(JSON 단일 vs 전용 테이블 vs 혼합), 공통 필드 처리 방식 비교(옵션 A vs B), Flutter 클라이언트 구조 등 **설계 결정 과정을 근거와 함께 명시한 점**이 좋습니다. 아키텍처 다이어그램도 전체 데이터 흐름을 한눈에 파악할 수 있도록 잘 구성되어 있습니다.

아래는 보완이 필요하다고 판단되는 항목입니다.

---

## 2. 전체 문서에 해당하는 리뷰 의견

### R-01. DB 스키마에 `source_route`, `source_widget`, `target_route`, `prev_route` 컬럼이 누락

API 요청 스키마에는 `source_route`, `source_widget`, `target_route`, `prev_route`가 이벤트별 상위 레벨 필드로 정의되어 있습니다. 그런데 DB 스키마(`app_events` 테이블)에는 이 4개 필드에 대한 컬럼이 없습니다. `parameters` JSONB 안에 넣는 것인지, 별도 컬럼으로 저장하는 것인지 명시해 주시기 바랍니다.

이 필드들은 모든 이벤트에 공통으로 포함되는 필드이고, 향후 "어느 화면에서 어떤 행동이 가장 많이 발생하는가" 등의 집계 쿼리에서 빈번하게 사용됩니다. JSONB 안에 넣으면 GIN 인덱스로 조회는 가능하지만 B-tree 인덱스 대비 집계 성능이 떨어지므로, **별도 컬럼으로 분리하는 것이 적합**합니다. 어느 쪽이든 DB 스키마에 명확히 반영해 주시기 바랍니다.

### R-02. API 응답 스키마 및 에러 처리가 정의되어 있지 않음

`POST /v1/events`의 요청 스키마는 JSON 예시로 잘 정의되어 있으나, **응답 스키마**(성공/실패 시 response body, HTTP 상태 코드)가 없습니다. 특히 다음 케이스에 대한 처리가 필요합니다:

- 배치 내 일부 이벤트만 유효하지 않을 때 (전체 실패? 부분 성공?)
- `user_id`가 null인 이벤트 (비로그인 유저) — 서버에서 허용하는지
- `events` 배열의 최대 크기 제한
- 중복 이벤트 수신 시 처리 (멱등성)
- 인증 실패 시 (401/403) 클라이언트의 재시도 정책

클라이언트(MuntoOutput)에서 `_safeWriteEvent`로 실패를 무시한다고 해도, 서버 측 동작은 명시되어야 합니다.

### R-03. 배치 전송의 구체적 동작이 불충분

"10개 단위 배치 전송"이라고 언급되어 있으나, 다음 사항이 정의되어 있지 않습니다:

- **시간 기반 플러시**: 이벤트가 10개 미만일 때 언제 전송하는지 (예: 5초마다 타이머 플러시)
- **앱 백그라운드/종료 시**: 버퍼에 남은 이벤트의 처리 방식 (즉시 플러시? 다음 실행 시 전송? 유실 허용?)
- **오프라인 시**: 네트워크 미연결 상태에서의 이벤트 큐잉 및 재전송 정책
- **버퍼 최대 크기**: 메모리 보호를 위한 상한

이벤트 유실이 허용되는 수준인지, 최소한의 전송 보장이 필요한지에 따라 설계가 달라지므로, 정책을 명시해 주시기 바랍니다.

### R-04. `session_id` 생성 규칙이 정의되어 있지 않음

`session_id`가 공통 필드로 포함되어 있으나, 이 값이 어떻게 생성되는지에 대한 정의가 없습니다:

- 앱 실행 단위인지, 로그인 단위인지
- UUID 형식인지, 다른 형식인지
- 앱이 백그라운드에서 돌아올 때 세션이 유지되는지, 새로 생성되는지
- 기존 Firebase의 `session_id`와 같은 값을 쓰는지, 별도 생성인지

세션 분석의 기준이 되는 필드이므로 정의가 필요합니다.

### R-05. 수집 대상 이벤트가 TBD

"수집 대상 이벤트 및 이력 관리" 섹션이 `TBD`로 남아 있습니다. 첫 번째 적용 대상인 배너 이벤트(`view_banner`, `click_banner`)는 명확하지만, 이 인프라가 **범용 이벤트 수집 인프라**로 설계되어 있으므로 향후 어떤 이벤트까지 확장할 계획인지 최소한의 로드맵이 있어야 합니다. 예를 들어:

- 화면 진입/이탈 이벤트
- 소셜링/데이팅 관련 주요 행동 이벤트
- 결제 관련 이벤트

이벤트 종류에 따라 `parameters` 구조가 달라지므로, 주요 이벤트별 `parameters` 스키마 예시가 있으면 구현 및 QA에 도움이 됩니다.

## 3. 특정 섹션/문장에 해당하는 리뷰 의견

### R-06. DB 스키마 — `app_events` 테이블의 파티셔닝 전략 부재

> `CREATE TABLE app_events (...)`

이 테이블은 모든 이벤트를 단일 테이블에 저장하는 구조입니다. 이벤트 수집이 활성화되면 데이터가 빠르게 증가합니다. Jira에서 언급한 배너만 해도 홈 화면 진입마다 노출 이벤트가 발생하고, 향후 화면 진입/이탈까지 확장하면 일일 수십만 건이 될 수 있습니다.

다음 사항을 검토해 주시기 바랍니다:
- **테이블 파티셔닝**: `occurred_at` 기준 월별/주별 Range 파티셔닝
- **데이터 보존 정책**: 원본 이벤트를 몇 개월간 보존할 것인지 (TTL)
- **아카이빙**: 오래된 데이터는 S3 등으로 이관하는 정책

### R-07. DB 스키마 — Materialized View 갱신 주기 미정의

> `CREATE MATERIALIZED VIEW banner_event_stats AS ...`

Materialized View는 자동으로 갱신되지 않습니다. 다음을 명시해 주시기 바랍니다:
- 갱신 주기 (예: 매시간, 매일)
- 갱신 방식 (`REFRESH MATERIALIZED VIEW CONCURRENTLY` 사용 여부)
- 갱신을 수행하는 주체 (크론잡, 별도 스케줄러)

실시간에 가까운 대시보드가 필요하다면 Materialized View 대신 일반 View나 실시간 집계 방식을 검토할 수도 있습니다.

### R-08. DB 스키마 — `user_id` 컬럼에 FK 제약 및 인덱스 타입 고려

> `user_id INT`

`user_id`가 `User` 테이블의 FK인지 명시되어 있지 않습니다. 이벤트 테이블의 성격상 FK를 걸면 INSERT 성능에 영향이 있으므로, FK 없이 논리적 참조만 유지하는 것이 일반적입니다. 어느 쪽인지 명시해 주시기 바랍니다.

또한 `idx_app_events_user_id`는 단독 인덱스인데, 실제 쿼리 패턴은 "특정 유저의 특정 기간 이벤트"일 가능성이 높으므로 `(user_id, occurred_at)` 복합 인덱스가 더 적합할 수 있습니다.

### R-09. API 설계 — `user_id`를 클라이언트가 직접 전송하는 구조의 보안 문제

> `"user_id": 12345`

현재 설계에서는 클라이언트가 `user_id`를 요청 body에 포함하여 전송합니다. 그런데 `Authorization: Bearer {token}`으로 인증하고 있으므로, **서버에서 토큰으로부터 `user_id`를 추출**하는 것이 보안상 안전합니다. 클라이언트가 `user_id`를 직접 지정하면 다른 유저의 ID를 넣어 위조할 수 있습니다.

`user_id`는 서버에서 토큰 기반으로 자동 설정하고, 클라이언트는 전송하지 않는 방식을 권장합니다. 비로그인 유저의 경우에만 `user_id: null`로 처리하면 됩니다.

### R-10. Flutter 클라이언트 — `MuntoOutput`을 앱별 별도 구현하는 것에 대한 검토

> `MuntoOutput`은 `munto-mobile`과 `dating-mobile` 각각에 별도로 구현한다. 두 앱이 바라보는 백엔드 엔드포인트나 인증 방식이 다를 수 있으므로 코드를 공유하지 않는다.

"다를 수 있으므로"는 현재 다른 것인지, 향후 다를 가능성을 대비하는 것인지 불명확합니다. 현재 두 앱의 백엔드 엔드포인트와 인증 방식이 동일하다면, 동일한 로직을 두 곳에 복제하는 것은 유지보수 부담이 됩니다. 다음 중 어느 상황인지 확인해 주시기 바랍니다:

1. 현재 엔드포인트/인증이 **실제로 다른 경우**: 별도 구현이 맞습니다.
2. 현재는 동일하지만 **향후 분리 가능성이 있는 경우**: 공통 로직은 `munto-mobile-core`에 두되, 엔드포인트/인증만 외부에서 주입하는 구조가 적합합니다.

### R-11. Resource and Scheduling — Jira에 명시된 `munto-mobile-core` 작업이 누락

Jira 이슈의 작업 범위에는 **`munto-mobile-core`: MuntoOutput 구현**이 포함되어 있으나, OnePager의 기술 설계에서는 `munto-mobile-core` 수정 없이 `munto-mobile`에서만 작업하는 것으로 변경되었습니다. Resource 테이블의 "`MuntoOutput` 구현" 대상 레포도 `munto-mobile`로 되어 있습니다.

이것은 설계 과정에서의 합리적인 결정(옵션 B 선택)이지만, **Jira 이슈의 작업 범위 항목과 불일치**합니다. Jira 이슈도 함께 업데이트하여 혼선을 방지해 주시기 바랍니다.
