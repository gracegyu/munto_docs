- **리뷰 관점:** Jira 요구사항 반영 여부, 아키텍처 결정 타당성, 운영 안정성, 구현 세부사항 완결성

## 1. 전체 평가

전반적으로 잘 작성된 OnePager입니다. 특히 아키텍처 3안 비교(Rate Limiter 내장 vs Bull Queue vs 마이크로서비스 분리)에서 기존 팀 운영 패턴을 근거로 B안을 채택한 판단이 명확하고, Worker 실행 앱 선택도 의존성과 장애 격리 관점에서 잘 분석되어 있습니다. PG 호출 유형별로 큐 대상과 동기 유지를 구분한 적용 범위 테이블, 기존 Bull Queue 사용 현황 정리, 동기 유형의 인라인 재시도 전략 별도 명시 등 실무적인 설계가 돋보입니다.

아래는 보완이 필요하다고 판단되는 항목입니다.

## 2. 전체 문서에 해당하는 리뷰 의견

### R-01 (P0). 멀티 인스턴스 환경에서 Bull limiter 글로벌 적용 검증이 "확인 필요" 상태로 남아 있음

Risk Assessment에서 직접 "API 서버 2개 인스턴스 운영 중 — Bull Worker가 2개 뜰 경우 limiter가 인스턴스 합산으로 글로벌 적용되는지 확인 필요"를 **높음** 리스크로 분류해 놓았으나, 대응 방안이 "프로덕션 환경에서 실제 동작 검증 필요"로 미확정 상태입니다. 그런데 Technical Description 전반에서는 "Bull limiter는 Redis 기반으로 글로벌하게 적용되므로 스케일 아웃 시에도 합산 제어됨"이라는 전제 위에 설계가 진행되어 있습니다.

이 문서의 핵심 전제(멀티 인스턴스 합산 제어 가능)가 미검증 상태이므로, **개발 착수 전에 스테이징 환경에서 2인스턴스 + Bull limiter 동작을 검증**하거나, 검증 전까지의 대안(limiter 값을 보수적으로 설정, 또는 Worker를 단일 인스턴스 앱으로 이동)을 확정해야 합니다. Risk Assessment의 대안 중 어느 쪽을 채택하는지 기재해 주시기 바랍니다.

### R-02 (P0). 큐 enqueue 후 환불 결과를 호출부에 어떻게 전달하는지 정의되지 않음

기존 동기 호출 구조에서는 환불 API를 호출한 서비스가 즉시 성공/실패를 알 수 있었습니다. Bull Queue 비동기로 전환하면 enqueue 시점에는 환불 결과를 모릅니다. 다음 사항이 정의되어 있지 않습니다:

- **enqueue 후 API 응답**: 호출부(스케줄러, 어드민 등)에 "접수 완료"를 반환하는 것인지, 환불 완료까지 대기하는 것인지
- **유저향 API(사용자 직접 환불 요청)**: 유저에게 "환불 접수됨"으로 안내하고 나중에 완료 알림을 보내는 것인지, 환불 완료까지 동기 대기하는 것인지
- **환불 완료/실패 콜백**: Worker 처리 완료 후 호출부에 결과를 전달하는 메커니즘(이벤트, 폴링, 푸시 알림 등)

특히 유저가 직접 환불을 요청하는 케이스는 UX에 직접적으로 영향을 주므로, 동기 유지인지 큐 대상인지 재검토가 필요합니다. 적용 범위 테이블에서 `refund`를 일괄 "Bull Queue 비동기"로 분류했는데, 유저 직접 요청 환불과 배치 환불을 구분해야 할 수 있습니다.

### R-03 (P0). 기존 호출 지점 전수 목록이 문서에 없음

Jira의 고려요소에 "기존 호출 지점(스케줄러, API, 콜백 등) 전수 조사 필요"가 명시되어 있고, 마일스톤 Week 1에도 "PG API 호출 지점 전수 목록화"가 있습니다. 그러나 OnePager 본문에는 "15개 이상의 진입점"이라고만 기재되어 있을 뿐, 실제 호출 지점 목록이 없습니다.

큐 전환 대상과 동기 유지 대상을 구분하려면 호출 지점별 현황(호출 서비스, PG 호출 유형, 호출 빈도, 동기/비동기 요구 여부)이 필요합니다. OnePager 단계에서 전수 목록까지는 어려울 수 있으나, 최소한 **주요 호출 경로와 예상 호출량**을 정리하여 적용 범위 결정의 근거를 보강해 주시기 바랍니다.

### R-04 (P1). 환불 큐 전환 시 트랜잭션 일관성 처리가 명시되지 않음

기존 동기 환불 흐름에서는 "PG 환불 → DB 상태 업데이트"가 하나의 트랜잭션(또는 순차 처리) 안에서 실행되었을 가능성이 높습니다. Bull Queue로 전환하면 enqueue 시점과 실제 환불 처리 시점이 분리됩니다.

- enqueue 전에 DB 상태를 "환불 진행 중"으로 변경하는지
- enqueue만 하고 Worker가 처리할 때 DB를 갱신하는지
- enqueue 성공했으나 Worker 처리 전에 서버가 재시작되면 DB 상태는 어떻게 되는지

환불 상태 전이(주문 상태 머신)에 대한 설계가 필요합니다.

### R-05 (P1). DLQ 처리 전략의 구체성 부족

"3회 모두 실패 시 DLQ 이관 + Slack 알림 + DB PARTIAL_REFUND_FAILED 기록"으로 기재되어 있으나, 다음 사항이 정의되어 있지 않습니다:

- **DLQ에서의 수동 재처리 절차**: 관리자가 어떤 도구(어드민 페이지, 스크립트, Bull Dashboard)로 DLQ 항목을 확인하고 재처리하는지
- **DLQ 적체 임계값 알림**: 단건 실패마다 Slack 알림인지, N건 이상 누적 시 알림인지
- **DLQ 보존 기간**: `removeOnFail: false`로 영구 보존 시 Redis 메모리 증가 문제. 오래된 실패 건의 정리 정책
- **수동 재처리 시에도 멱등성이 보장되는지**: DLQ에서 같은 건을 두 번 재처리하는 상황

마일스톤 Week 3에 "기존 실패 건 수동 재처리 스크립트 작성"이 있으므로, 이 스크립트의 대략적인 동작(DLQ 항목 조회 → 멱등성 키 확인 → 재 enqueue)도 기재하면 좋겠습니다.

## 3. 특정 섹션/문장에 해당하는 리뷰 의견

### R-07 (P1). 처리 흐름 — 초당 최대 25건 설정의 근거 보강 필요

> 초당 최대 25건 (`limiter: { max: 25, duration: 1000 }`) — PG 한도 30건 대비 5건 버퍼 확보

버퍼 5건은 동기 유지 유형(`confirmReceipt`, `validateReceipt`, `getReceipt`, `certificate`)의 호출량을 고려한 값이어야 합니다. 그런데 동기 유형의 **피크 시 초당 호출량 추정치**가 없으므로 5건 버퍼가 충분한지 판단이 불가합니다.

예를 들어 결제 확정(`confirmReceipt`)이 피크 시 초당 10건 발생하면, 큐 25건 + 동기 10건 = 35건으로 PG 한도를 초과합니다. 동기 유형의 예상 피크 호출량을 추정하고, 이를 기반으로 limiter 값을 산정하는 근거를 추가해 주시기 바랍니다.

### R-08 (P1). 적용 범위 — `requestSubscribeCardPayment`(정기결제) 큐 전환 시 결과 확인 방법

> `requestSubscribeCardPayment` (정기결제) — Bull Queue 비동기 — 스케줄러 배치 처리

정기결제는 스케줄러에서 배치로 처리하므로 비동기 큐가 적합합니다. 다만, 정기결제 성공/실패에 따라 멤버십 상태를 업데이트하는 후속 처리가 필요합니다. 현재 스케줄러의 동기 호출에서는 결과를 바로 받아 후속 처리를 했을 텐데, 큐 전환 후에는 Worker 완료 시 후속 처리(멤버십 갱신, 결제 알림 등)를 어떻게 트리거하는지 명시가 필요합니다.

### R-09 (P1). 인라인 재시도 — 재시도 대상 에러에 네트워크 타임아웃 포함 여부

> 재시도 대상 에러: RC_CANCEL_SERVER_ERROR(9003), PG 서버 5xx

네트워크 타임아웃(ETIMEDOUT, ECONNRESET 등)이나 PG 서버 무응답 상황은 재시도 대상에 포함되는지 명시가 없습니다. Rate Limit 에러만 재시도 대상으로 한정하는 것인지, PG 서버 일시 장애까지 포함하는 것인지 확정해 주시기 바랍니다. 또한 BootpayService에서 PG 호출 시 **타임아웃 값**이 현재 몇 초로 설정되어 있는지, 인라인 재시도와 합산한 최대 응답 시간이 클라이언트 타임아웃 이내인지 확인이 필요합니다.

### R-10 (P2). 관련 이슈 — 이슈 번호 불일치

> WEBB-1272 — 본 이슈

Jira 이슈 번호가 `WEEB-1227`인데, 관련 이슈에서 `WEBB-1272`로 기재되어 있습니다. 오타인지 별도 이슈인지 확인해 주시기 바랍니다.

### R-11 (P2). Risk Assessment — Bull Queue 장애 시 fallback 동기 호출 경로의 구현 범위

> Redis(Bull 기반) 장애 시 결제 큐 중단 — DLQ 설정, Redis 클러스터 구성, Bull 장애 시 fallback 동기 호출 경로 보존

"fallback 동기 호출 경로 보존"이 구체적으로 무엇을 의미하는지 불명확합니다. 기존 동기 호출 코드를 삭제하지 않고 남겨두는 것인지, Redis 장애 감지 시 자동으로 동기 경로로 전환하는 서킷 브레이커를 구현하는 것인지에 따라 공수가 크게 달라집니다. feature flag로 점진적 전환한다고 했는데, 이것이 곧 fallback 경로인지 명확히 연결해 주시기 바랍니다.

## 5. 결론

Bull Queue 기반 중앙 큐 관리라는 아키텍처 방향은 팀의 기존 운영 패턴과 인프라를 잘 활용한 합리적인 선택입니다. PG 호출 유형별 동기/비동기 구분, 인라인 재시도 전략 분리 등 실무적으로 세밀한 설계가 돋보입니다.

다만 P0 의견 3건(멀티 인스턴스 limiter 검증, 비동기 전환 후 결과 전달 방식, 호출 지점 목록화)은 설계의 핵심 전제 및 적용 범위에 직접 영향을 주므로 **개발 착수 전에 반드시 확정**이 필요합니다. P1 의견 6건(트랜잭션 일관성, DLQ 구체화, 일정, limiter 근거, 정기결제 후속 처리, 인라인 재시도 범위)을 함께 반영하면 **개발 진행 가능**합니다.
