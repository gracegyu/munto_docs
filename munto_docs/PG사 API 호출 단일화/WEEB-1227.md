# [개선] PG사 API 호출 단일화 및 Rate Limit 관리 체계 구축

| 항목 | 내용 |
|------|------|
| 배경 | 현재 PG사(나이스페이먼츠)의 초당 동시처리 한도가 30회로 제한되어 있으나, 백엔드에서 PG API를 호출하는 경로가 스케줄러, 함수 콜백, 사용자 직접 요청 등 여러 곳에 분산되어 있어 전체 호출량을 중앙에서 제어하지 못하는 상황 |
| 현재 문제점 | - PG API 호출 지점이 분산되어 있어 초당 한도 초과 시점을 예측할 수 없음<br>- 한도 초과 시 RC_CANCEL_SERVER_ERROR(에러코드 9003) 발생하여 환불 실패<br>- 실패 건에 대한 자동 재시도 로직이 없어 수동 처리 필요<br>- 2026년 기준 동일 에러로 약 10건 이상 발생, 대량 실패 케이스(WEBB-1132)도 존재 |
| 개선 방향 | 1. PG API 호출을 공통 서비스 또는 별도 결제 서버로 단일화하여 중앙 집중 관리<br>2. Rate Limiter 또는 Queue 기반으로 초당 호출 수를 PG사 한도(30회/초) 이내로 제어<br>3. 환불 실패 시 자동 재시도(exponential backoff) 로직 추가<br>4. 실패 건 모니터링 및 알림 체계 구축 |
| 고려요소 | - 결제 서버 분리 vs 공통 서비스 내 큐 관리 방식 비교 검토 필요<br>- 기존 호출 지점(스케줄러, API, 콜백 등) 전수 조사 필요<br>- 재시도 시 멱등성(idempotency) 보장 방안<br>- 장애 시 DLQ(Dead Letter Queue) 처리 전략 |
| 기대효과 | PG사 Rate Limit으로 인한 환불/결제 실패 근본적 해소, CS 인입 감소, 운영 안정성 향상 |
