# VOD 서비스 폐지 OnePager 리뷰

리뷰어: PM
리뷰 대상 문서: VOD 서비스 폐지 OnePager (v1.0.0, 김범진 작성)
리뷰 일자: 2026-04-07
참조 문서: VOD SRS, VOD 폐지 시나리오, VOD 폐지 프로세스 구축, VOD 서비스 폐지 Jira

---

## 1. 전체 평가

OnePager로서 프로젝트 개요, 리스크, 일정, 기술적 구현 방향을 한 문서에 담아 구조가 잘 잡혀 있습니다. 다만, 기존에 작성된 상위 기획 문서(VOD 폐지 시나리오, VOD 폐지 프로세스 구축)와 대조했을 때 **누락·불일치·보완이 필요한 사항**이 있어 아래에 정리합니다.

---

## 2. 일정 불일치 및 단계 정의 혼선

### 2-1. 단계 번호 불일치

| 구분 | OnePager | 폐지 시나리오 문서 |
| --- | --- | --- |
| 1단계 | 내부 검토 및 의사결정 (~4/8) | 내부 검토 및 의사결정 (~4/15 신규 결제 차단 배포) |
| 신규 결제 차단 | "신규 결제 차단 배포" (4/9~4/15) — 별도 행 | 1단계에 포함 |
| 2단계 | 서비스 종료 공지 (4/16) | 서비스 종료 공지 (4/16) |
| 3단계 | 신규 판매 중단 (4/16~5/16) | 신규 판매 중단 (4/16~5/16) |
| 4단계 | VOD 관련 페이지·기능 제거 (5/18~) | VOD 관련 페이지·기능 제거 (5/18~) |
| 5단계 | 사후 관리 및 기록 보존 (~2031) | 사후 관리 및 기록 보존 (~2031) |

- **문제**: OnePager의 Resource and Scheduling 테이블에서 "신규 결제 차단 배포"가 1단계와 별도 행으로 분리되어 있어 총 6행이지만 단계 번호는 5단계까지만 표기됩니다. Jira 문서의 개선 방향에서는 4단계로 구분하고 있습니다. **단계 체계를 통일**해주세요.

### 2-2. 신규 결제 차단 일정 상세

- 폐지 시나리오 문서에는 *"4/13(월)~4/15(수) 중에 신규 결제 차단 조치가 배포될 수 있을 것 같다는 내용을 전달 받음"*이라고 기록되어 있으나, OnePager에서는 **4/9(목)~4/15(수)**로 명시되어 있습니다.
- 4/9(목)이 개발 시작일이고 4/15(수)이 배포 완료 목표라면 그렇게 명확히 기재해주세요. 현재는 "4/9부터 차단"으로 읽힐 수 있습니다.

## 3. 환불 관련 내용 누락 (중요)

### 3-1. 환불 시나리오 미반영

- 폐지 시나리오 문서에서 **가장 큰 분량을 차지하는 핵심 내용**이 환불 처리입니다:
  - 전자상거래법 제17조·제18조에 따른 환불 의무
  - 비례 환불(시청 비율 기반) vs 전액 환불 기준
  - 환불 신청 후 3영업일 이내 처리 의무 (지연 시 연 40% 이내 지연배상금)
  - 일괄 환불 불가 → 개별 신청 + 선제 환불 이중 구조
- OnePager의 **Risk Assessment**에 환불 관련 리스크가 전혀 없고, **Technical Description**에도 환불 처리 시스템 언급이 없습니다.
- **권고**: Risk Assessment에 "환불 처리 지연 리스크" 항목을 추가하고, Technical Description에 "결제 취소 API 유지 기간" 및 "환불 처리 운영 체계"를 명시해주세요.

### 3-2. OnePager 본문과의 모순

- OnePager의 Technical Description에 *"서비스 종료 이후에도 환불 문의 처리를 위해 CS 운영 체계 및 결제 취소 API 일정 기간 유지 필요"*라고 언급하면서도, Risk Assessment와 Resource and Scheduling에는 환불 관련 일정이나 리스크가 빠져 있습니다. 문서 내 정합성을 맞춰주세요.

## 4. 법적 근거 보완 필요

### 4-1. 약관 조항 번호 오류 가능성

- OnePager에서 **"약관 제30조 제1항"**을 인용하고 있습니다. 폐지 시나리오 문서에서도 동일 조항을 인용하므로 정확한 것으로 보이나, 약관 개정 후 조문 번호가 변경될 수 있으므로 **현행 약관 기준임을 명시**해주세요.

### 4-2. 약관 개정 일정 누락

- 폐지 시나리오 문서 STEP 5에서 VOD 관련 약관 조항(제29조~제40조) 삭제 및 조문 번호 재정비가 필요하다고 명시되어 있습니다.
- 폐지 프로세스 구축 문서 Phase 1에서도 약관 처리를 **가장 먼저 착수해야 할 단계**로, 전체 일정의 크리티컬 패스로 지정하고 있습니다.
- OnePager의 Technical Description에 *"약관 개정 시 조문 번호 재정비 및 준용 조항 검토 필요"*라고 언급은 되어 있으나, **Resource and Scheduling 테이블에 약관 개정 일정이 빠져 있습니다**. 마일스톤에 추가해주세요.

## 5. 호스트 관련 처리 누락

### 5-1. 호스트 선공지

- 폐지 시나리오 문서에서 **"호스트에게 먼저 VOD 개편 안내 전달"**을 명시하고 있으며, 폐지 프로세스 구축 문서에서도 Phase 2(호스트 고지)를 별도 단계로 상세하게 다루고 있습니다.
- 호스트 유형별 차등 고지(제휴 호스트 / 활성 호스트 / 비활성 호스트)도 정의되어 있습니다.
- OnePager에서는 호스트 고지가 2단계 공지에 간략히 포함되어 있으나, **제휴 호스트 계약 종료 절차(Phase 3)**에 대한 언급이 전혀 없습니다.

### 5-2. 호스트 미정산 잔액

- Risk Assessment에 "호스트 미정산 잔액 처리 지연" 리스크가 있지만 발생 가능성이 **"상"**으로 표기되어 있습니다. 이 리스크가 "상"이라면 대응 방안이 *"서비스 종료 전 미정산 잔액 전액 정산 완료"*만으로는 부족합니다. **구체적인 정산 완료 목표일**을 기재해주세요.

---

## 6. 사용자 구제 방안 누락

- 폐지 프로세스 구축 문서 Phase 4에서 수강 중인 사용자에 대한 구제 방안을 상세히 정리하고 있습니다:
  - 콘텐츠 다운로드 기간 제공
  - 환불 처리 (미수강/미완료)
  - 대체 보상 (크레딧·이용권)
  - 유예 기간 운영
- 또한 이해관계자별 중요도 기반 차등 대응 전략도 수립되어 있습니다.
- OnePager에는 **기존 구매자 시청 권한 유지** 외에 구제 방안이 없습니다. 사업적 판단 항목이므로 최소한 Business and Marketing Justification 또는 Risk Assessment에 "수강 중 사용자 구제 방안"을 참조 링크로라도 포함해주세요.

## 7. 기술적 세부 사항 보완

### 7-1. API 차단 방식 구체화

- OnePager: *"VOD 결제 엔드포인트에 차단 로직 추가"*
- VOD SRS를 참조하면 관련 API는 다음과 같습니다:
  - `POST /api/course/{courseId}/validate` (결제 검증)
  - `POST /api/course/{courseId}` (수강 신청/결제)
  - `GET /api/courses/{courseId}/upload-url` (업로드 URL)
- **권고**: 차단 대상 API 엔드포인트 목록과 응답 방식(410 Gone vs 404 Not Found)을 명시해주세요. Jira 문서에서는 "410 Gone 또는 404 응답 처리"로 기재되어 있으나, SRS의 기존 에러 응답 체계와 일관성 있는 선택이 필요합니다.

### 7-2. Course 상태 활용

- VOD SRS에 이미 `CourseStatus`가 정의되어 있고 `PAUSED`(일시 중지), `CLOSED`(판매 종료) 상태가 있습니다.
- 신규 결제 차단 시 기존 Course의 상태를 `CLOSED`로 변경하는 방식을 활용할 수 있는지 검토해주세요. 별도 차단 로직을 추가하는 것보다 기존 상태 체계를 활용하는 것이 유지보수에 유리할 수 있습니다.

### 7-3. 마이페이지 시청 권한 로직

- OnePager: *"구매 이력 존재 여부 기준으로 분리 관리"*
- VOD SRS의 `CourseEnrollment` 모델을 보면 `status` 필드로 `APPROVE`, `CANCEL`, `DELETE` 등 상태가 관리됩니다.
- **주의**: 단순히 "구매 이력 존재 여부"가 아니라 `CourseEnrollment.status = APPROVE`이고 `이용 기간(availableDays)` 내인 경우만 시청이 가능해야 합니다. SRS의 비즈니스 규칙 6.5.1에 따르면 *"수강 상태가 CANCEL, DELETE, COMPLETED인 경우에는 영상 재생이 불가능"*합니다.
- 서비스 폐지 후에도 이 규칙을 그대로 유지할 것인지, 아니면 폐지 특례로 이용 기간을 무제한 연장할 것인지 명확히 정의해주세요.

### 7-4. 검색·추천 피드 제거 범위

- VOD SRS에서 Redis 캐시를 통한 VOD 리스트 캐싱이 사용되고 있습니다 (6.4.1절).
- 화면 제거 시 **Redis 캐시 무효화**도 필요합니다. 캐시에 VOD 데이터가 남아 있으면 검색·추천에서 여전히 노출될 수 있습니다.

## 8. 인프라 및 비용 관련

- VOD 서비스에 사용되는 AWS 리소스:
  - S3 (영상 원본/인코딩 저장)
  - MediaConvert (인코딩)
  - CloudFront (스트리밍 CDN)
  - Lambda (인코딩 트리거)
- 폐지 후에도 기존 구매자 시청 유지를 위해 **S3, CloudFront는 계속 운영**해야 합니다.
- **권고**: OnePager에 폐지 후 유지되는 인프라 vs 즉시 해제 가능한 인프라를 구분하여 기재해주세요. MediaConvert, Lambda는 신규 업로드가 없으므로 즉시 비활성화 가능하지만, S3와 CloudFront는 시청 기능 유지 기간 동안 비용이 계속 발생합니다.

## 9. Resource and Scheduling 보완

### 9-1. 누락 마일스톤

| 누락된 항목 | 참조 문서 | 권고 추가 일정 |
| --- | --- | --- |
| 약관 개정 (VOD 조항 삭제) | 폐지 시나리오 STEP 5, 프로세스 구축 Phase 1 | 2단계 공지 이전 또는 동시 |
| 호스트 선공지 | 폐지 시나리오 1단계, 프로세스 구축 Phase 2 | 2단계 전체 공지 이전 |
| 제휴 호스트 계약 정리 | 프로세스 구축 Phase 3 | 호스트 고지와 병행 |
| 환불 처리 운영 체계 구축 | 폐지 시나리오 STEP 3 | 1단계 결제 차단 이전 |
| 인프라 정리 (MediaConvert/Lambda 비활성화) | SRS 2.1절 | 4단계 이후 |
| 분석 도구·소통 채널 폐기 | 프로세스 구축 Phase 8 | 최종 검증 단계 |
| 검색 엔진 색인 제거 (robots.txt/sitemap) | 프로세스 구축 Phase 9 | 4단계와 병행 |

### 9-2. QA 범위 명시

- OnePager의 Risk Assessment에 "마이페이지 시청 기능 별도 QA, 회원 ID 기반 권한 관리 로직 세분화"가 있으나, Resource and Scheduling에 QA 일정이 **결제 차단 배포 내에만** 포함되어 있습니다.
- 3단계(화면 제거)에 대한 QA 일정도 별도로 잡아야 합니다. 폐지 프로세스 구축 문서에서는 "통합 QA: 1~2일"로 별도 산정하고 있습니다.

## 10. 문서 형식 관련

- **API: N/A, ERD: N/A** → 신규 API/ERD가 없다는 의미로 보이나, 기존 API 비활성화 목록을 정리하는 것이 개발 실행에 도움이 됩니다. 별도 문서로 분리하거나, 최소한 참조 링크를 넣어주세요.
- **Project Description**에 [VOD 폐지 시나리오] 링크가 있는데, 해당 문서의 히스토리 섹션을 보면 VOD 완전 롤백 가정 하에 작성된 시나리오와 현재 시나리오(마이페이지 유지)가 혼재되어 있습니다. OnePager에서 참조할 때 **현행 시나리오 기준임을 명시**해주세요.
