# 토큰 관리 및 라우트 시스템 개선 OnePager 리뷰

## 전체 평가

문서는 기술적 내용이 잘 정리되어 있으며, 문제점과 해결 방안이 명확하게 제시되어 있습니다. 다만 OnePager 템플릿의 필수 항목 일부가 누락되어 있고, 일정 및 리소스 관련 정보가 부족합니다.

---

## 누락된 필수 항목

### 1. OnePager 템플릿 필수 항목

- **Project Name**: 문서 제목이 있으나 명시적 프로젝트명 섹션 없음
- **Date**: 작성일 또는 제출일 없음
- **Submitter Info**: 작성자/제출자 정보 없음

### 2. Resource and Scheduling Details 부족

현재 "구현 범위"에 Phase만 나열되어 있으나 다음 정보가 필요합니다:

- 각 Phase별 예상 소요 기간 (일/주 단위)
- 필요한 인력 및 역할 (예: Flutter 개발자 1명, 백엔드 협업 필요 여부)
- Phase 간 의존성 및 병렬 작업 가능 여부
- 전체 프로젝트 예상 완료 시점

---

## 기술적 개선 사항

### 1. Phase 순서 재검토 필요

**현재 순서**: Phase 1 (라우팅) → Phase 2 (토큰 서비스) → Phase 3 (중요 플로우) → Phase 4 (레거시 정리)

**문제점**:

- Phase 1에서 라우트 가드를 구현하려면 TokenService가 필요함
- Phase 3의 "토큰 사전 갱신 통합"은 Phase 2의 TokenService가 완료되어야 가능

**권장 순서**:

- Phase 1: 토큰 서비스 구축 (기반 인프라)
- Phase 2: 라우팅 시스템 통합 (TokenService 활용)
- Phase 3: 중요 플로우 통합
- Phase 4: 레거시 정리 및 테스트

또는 Phase 1과 2를 병렬 진행 가능한 범위를 명시

### 2. 토큰 갱신 직렬화 구현 세부사항 부족

**현재 내용**: "Completer 패턴과 요청 큐를 활용한 병렬 요청 제어"

**추가 필요 사항**:

- 동시 요청이 많을 때 큐 크기 제한 및 타임아웃 정책
- 토큰 갱신 실패 시 대기 중인 요청 처리 방법 (재시도 vs 실패 전파)
- 서버 SSO 환경에서 클라이언트 측 직렬화가 충분한지 검증 필요

### 3. 토큰 자동 갱신 로직 명확화 필요

**현재 내용**: "토큰 만료 시간의 80% 경과 시 자동 갱신 (백그라운드)"

**추가 필요 사항**:

- 80% 기준의 근거 (예: 토큰 유효기간이 1시간이면 48분 경과 시)
- 백그라운드 갱신 실패 시 재시도 정책
- 앱이 백그라운드에 있을 때도 갱신하는지, 포그라운드 전환 시 갱신하는지
- 여러 앱 인스턴스가 동시에 실행될 때의 동기화 문제 (Flutter는 단일 인스턴스이지만 명시)

### 4. 기존 토큰 관리 방식과의 마이그레이션 전략 부족

**현재 문제점**:

- ApiToken(FlutterSecureStorage)
- SendbirdToken(SharedPreferences)
- KakaoHelper(자체 TokenManager)

**추가 필요 사항**:

- 각 토큰 타입별 마이그레이션 우선순위
- 마이그레이션 중 기존 코드와의 공존 전략
- SendbirdToken과 KakaoHelper도 FlutterSecureStorage로 통합하는지, 아니면 별도 관리하는지
- 기존 토큰 데이터 마이그레이션 스크립트/로직 필요 여부

### 5. 라우트 가드 구현 세부사항 부족

**현재 내용**: "화면 진입 전 `redirect` 콜백에서 인증 상태 및 토큰 유효시간 체크"

**추가 필요 사항**:

- GoRouter의 `redirect`는 동기 함수인데, 토큰 갱신은 비동기 작업임. 비동기 토큰 갱신을 어떻게 처리할지
- 토큰 갱신 중 사용자에게 보여줄 로딩 UI 처리 방법
- 여러 중요 플로우가 연속으로 진입할 때 중복 갱신 방지 방법

### 6. 중요 플로우 토큰 보장의 근본적 한계 (P0 - Critical)

**현재 내용**:

- 결제: 10분, VOD: 30분, 소셜링: 5분, 프로필 수정: 5분 토큰 유효시간 "보장"

**심각한 문제점**:

- **Access Token 유효시간만 체크하여 보장하려고 하지만, Refresh Token이 만료되면 Access Token을 갱신할 수 없음**
- 예시: Refresh Token이 5분 후 만료되는데, VOD 재생 플로우에 진입하여 30분 토큰 유효시간을 요구하면 보장 불가능
- 현재 문서는 "보장"이라고 명시하고 있으나, 실제로는 "Access Token 유효시간이 충분한 경우에만 보장"임

**수정 필요 사항**:

1. **용어 변경**: "보장" → "Access Token 유효시간 확인 및 필요 시 갱신 시도"
2. **Refresh Token 유효시간 체크 추가**:
   - 중요 플로우 진입 시 Access Token뿐만 아니라 Refresh Token 유효시간도 체크
   - Refresh Token이 부족하면 갱신 불가능하므로 사용자에게 재로그인 안내
3. **TokenService 인터페이스 보완 필요**:
   - 현재: `getTokenExpirationTime()` (Access Token만 체크하는 것으로 보임)
   - 추가 필요: `getRefreshTokenExpirationTime()` 또는 `isRefreshTokenValid(minRequiredMinutes)`
   - `ensureTokenValid()` 메서드도 Refresh Token 유효시간을 함께 체크하도록 수정
4. **실패 시나리오 명시**:
   - Refresh Token 만료 시: 재로그인 화면으로 리다이렉트
   - Access Token 갱신 실패 시: 재시도 또는 재로그인 안내
5. **각 시간 기준의 근거**:
   - 결제 플로우 평균 소요 시간이 8분이므로 10분 요구 등
   - 단, Refresh Token 유효시간이 요구 시간보다 짧으면 보장 불가능
6. **중요 플로우 목록 완전성 검토**: 채팅방 입장, 실시간 매칭 등 추가 필요 여부 확인

### 2. 모니터링 및 알림 전략 부족

**현재 내용**: "모니터링 시스템 구축" (리스크 완화 방안에만 언급)

- 토큰 갱신 실패율 모니터링

- 각 Phase별 롤백 트리거 조건

## 문서 구조 개선 제안

### 1. OnePager 템플릿 형식 준수

문서 상단에 다음 섹션 추가:

```
Project Name: 토큰 관리 및 라우트 시스템 개선
Date: YYYY-MM-DD
Submitter Info: [작성자 정보]
```

### 2. Resource and Scheduling Details 섹션 추가

"구현 범위" 섹션을 확장하거나 별도 섹션 추가:

- 각 Phase별 예상 기간
- 필요한 인력
- Phase 간 의존성 다이어그램
- 전체 타임라인

### 3. 테스트 전략 섹션 추가

현재 "테스트 및 검증"만 언급되어 있으나 구체적 전략 필요:

- 단위 테스트 범위 (TokenService, 라우트 가드 등)
- 통합 테스트 시나리오 (중요 플로우별)
- E2E 테스트 계획
- 성능 테스트 (토큰 갱신 직렬화 효율성)

## 긍정적인 부분

1. **문제점 분석이 명확함**: 현재 상태와 문제점이 구체적으로 정리되어 있음
2. **해결 방안이 체계적임**: 토큰 서비스, 라우팅, DI 통합으로 단계적 접근

4. **리스크 인식**: 잠재적 위험 요소를 파악하고 완화 방안 제시

## 추가 검토 필요 사항

1. **서버 측 변경 필요 여부**: 클라이언트에서 토큰 갱신 직렬화를 하더라도, 서버에서 동시 갱신 요청을 받을 수 있음. 서버 측 처리도 필요한지 확인 필요

2. **다른 플랫폼 영향**: 현재 문서는 Flutter 앱에 대한 내용인데, iOS/Android 네이티브 코드와의 연동이 필요한지 확인 필요

3. **딥링크 처리**: 라우팅 시스템 통합 시 기존 딥링크가 GoRouter와 호환되는지, 추가 작업이 필요한지 확인 필요
