## 1. TCL 분석 결과

### 1.1 현재 TCL의 결제 관련 테스트 케이스 현황

현재 TCL을 보면 기본적인 결제 플로우는 잘 커버하고 있습니다. 결제 성공/실패, UI 검증, 관리자 기능 등은 테스트 케이스가 있네요.

하지만 이번 사고의 핵심 경로인 웹훅 처리와 스케줄러 동작에 대한 테스트가 전혀 없습니다. 이 부분이 가장 큰 문제라고 봅니다.

**현재 있는 테스트 케이스**:

1. 결제 성공/실패 플로우

   - 일반 유료 결제 정상 흐름
   - 얼리버드 할인 결제
   - 결제 실패 예외 처리
   - 네트워크 오류 발생 시

2. 결제 UI/UX 검증

   - 결제 정보 화면 진입
   - 환불 규정 동의
   - 결제 완료 화면 이동

3. 관리자 결제 관리
   - 결제 내역 조회
   - 환불 처리
   - 결제 상세 정보 확인

**없는 테스트 케이스 (핵심 누락)**:

1. 부트페이 웹훅 처리 테스트

   - 웹훅 수신 및 처리 확인
   - PaymentRequest 생성 여부 확인
   - 웹훅 실패 시 재시도 로직

2. 스케줄러 동작 테스트

   - refundExpiredPaymentRequest() 스케줄러 실행 확인
   - PaymentRequest 생성 후 30분 경과 시 동작 확인
   - Payment 존재 여부 확인 로직

3. PaymentRequest → Payment 생성 시간 간격 테스트

   - 웹훅 수신 후 Payment 생성까지 소요 시간 측정
   - 30분 이내 Payment 생성 확인
   - Payment 생성 실패 시 스케줄러 동작 확인

4. 서비스 구분 로직 테스트

   - 데이팅 서비스 vs 기존 서비스 구분 확인
   - 서비스별 웹훅 처리 분기 확인
   - 스케줄러에서 서비스 구분 적용 여부

5. 결제 취소 모니터링 테스트
   - 결제 금액과 취소 금액 대조 확인
   - 이상치 발생 시 알림 전송 확인
   - 일일 결제/취소 통계 정확성

### 1.2 TCL의 문제점

**Critical - 즉시 보완 필요**:

1. 웹훅 처리 검증 부재

   - 부트페이 웹훅 수신 및 처리에 대한 테스트 케이스가 전혀 없습니다
   - 실제 사고의 핵심 경로인 웹훅 → PaymentRequest → 스케줄러 플로우가 검증되지 않았습니다

2. 스케줄러 동작 미검증

   - 자동 환불 스케줄러의 동작이 테스트되지 않았습니다
   - 30분 경과 후 자동 취소 로직이 정상 동작하는지 확인할 방법이 없습니다

3. 서비스 구분 로직 미검증
   - 데이팅 서비스와 기존 서비스의 결제 처리 분기 테스트가 없습니다
   - 서비스별 스케줄러 적용 여부를 확인할 수 없습니다

**High - 우선 보완 권장**:

4. 시간 기반 시나리오 부재

   - PaymentRequest 생성 후 30분 경과 시나리오 테스트가 없습니다
   - 실제 운영 환경에서 발생할 수 있는 시간 지연 케이스가 검증되지 않았습니다

5. 모니터링 및 알림 검증 부재
   - 결제/취소 금액 모니터링 동작 확인 테스트가 없습니다
   - 이상치 발생 시 알림 전송 확인 테스트가 없습니다

**Medium - 점진적 보완**:

6. 통합 테스트 부족
   - 웹훅 → PaymentRequest → Payment → 스케줄러 전체 플로우 통합 테스트가 없습니다
   - 실제 운영 환경과 유사한 E2E 테스트가 부족합니다

## 2. 재발 방지 대책 OnePager 분석

### 2.1 현재 방안의 충분성 평가

재발 방지 대책을 보니 웹훅 진입점에서 서비스 구분하는 건 좋은 방향인 것 같습니다. 이게 근본 원인을 해결하는 방법이니까요.

모니터링도 있긴 한데, 하루 2회만 하는 건 좀 부족할 수 있을 것 같습니다. 이번 사고처럼 빠르게 진행되는 경우에는 하루 2회로는 늦을 수 있어요.

완전 분리는 TBD 상태인데, 이건 장기 과제로 보이고 당장은 구체적 계획이 필요해 보입니다.

**적절한 방안**:

1. 웹훅 진입점에서 서비스 구분

   - 근본 원인 해결 방안으로 보입니다
   - 데이팅 서비스와 기존 서비스의 결제 처리 분리가 핵심입니다
   - 이 방안은 적절하다고 판단됩니다

2. 결제/취소 금액 모니터링
   - 하루 2회 (12시 기준) 모니터링
   - 이상치 발생 시 Slack 알림
   - 방향은 좋은데 빈도는 좀 더 검토가 필요해 보입니다

**보완이 필요한 방안**:

3. 완전 분리 (TBD)
   - 현재 TBD 상태로 구체적 일정이 없습니다
   - 장기적으로는 필요하지만 당장은 구체적 계획이 필요합니다

### 2.2 재발 방지 대책의 부족한 부분

**Critical - 즉시 보완 필요**:

1. 스케줄러 비활성화 상태의 영구적 해결책 부재

   - 현재는 스케줄러 비활성화로 임시 조치한 상태입니다
   - 데이팅 서비스에 대한 스케줄러 적용 제외 로직이 구체화되지 않았습니다
   - 스케줄러에서 서비스 타입 확인 로직을 추가하는 게 필요해 보입니다

2. 모니터링 빈도 및 임계값 미정의

   - 하루 2회 모니터링의 적절성을 검증해야 합니다
   - 이상치 판단 기준이 명확하지 않습니다 (예: 취소율 몇 % 초과 시 알림?)
   - 모니터링 빈도는 실시간 또는 1시간 단위로 검토하는 게 좋을 것 같고, 임계값은 취소율 5% 초과 시 알림 정도가 적절할 것 같습니다

3. 테스트 및 검증 절차 부재
   - 방지 대책 적용 후 검증 방법이 정의되지 않았습니다
   - 스테이징 환경에서 웹훅 → 스케줄러 플로우 테스트가 필요합니다
   - 데이팅 서비스 결제 후 30분 경과 시나리오 테스트도 필요합니다

**High - 우선 보완 권장**:

4. 롤백 계획 부재

   - 방지 대책 적용 후 문제 발생 시 롤백 절차가 정의되지 않았습니다
   - 롤백 절차 및 조건을 명시하는 게 좋겠습니다

5. 장기적 해결책 일정 부재
   - 완전 분리 (TBD)의 구체적 일정 및 마일스톤이 없습니다
   - Phase 1: 웹훅 진입점 구분 (1주), Phase 2: 스케줄러 로직 수정 (1주), Phase 3: 완전 분리 (1개월) 정도로 계획하면 좋을 것 같습니다

### 3.2 테스트 자동화 권장 사항

**웹훅 테스트**: 부트페이 웹훅 Mock 서버를 구축하고 CI/CD 파이프라인에 추가하는 게 좋을 것 같습니다.

**스케줄러 테스트 (시간 기반)**:

- "30분 경과" 테스트는 실제로 매우 어렵습니다. 스테이징 환경에서 30분을 실제로 기다리는 건 비현실적입니다.
- **대안 1**: 시간 조작 유틸리티 사용 (예: Jest의 `jest.useFakeTimers()`, Python의 `freezegun`) - PaymentRequest 생성 시간을 과거로 설정하고, 현재 시간을 30분 후로 이동시켜 테스트
- **대안 2**: 스케줄러 함수를 직접 호출하여 테스트 - 30분 경과 조건을 만족하는 PaymentRequest를 생성하고, 스케줄러 함수를 수동 실행
- **대안 3**: 모니터링으로 대체 - 실제 30분 경과는 운영 환경에서 모니터링으로 확인하고, 스케줄러 로직 자체는 단위 테스트로 검증

**모니터링 테스트**: 스크립트 단위 테스트를 만들고, 이상치 발생 시나리오를 자동 생성해서 검증하는 방식으로 하면 될 것 같습니다.

## 4. 재발 방지 대책 보완 제안

### 4.1 즉시 보완 필요 사항

#### 1. 스케줄러 로직 수정 (필수)

현재는 스케줄러를 비활성화한 상태인데, 이건 임시 조치입니다. 영구적인 해결책이 필요합니다.

스케줄러에서 서비스 타입을 확인해서 데이팅 서비스는 제외하도록 수정하는 게 좋을 것 같습니다. 대략 이런 식으로요:

```javascript
// 스케줄러에서 서비스 타입 확인 로직 추가
refundExpiredPaymentRequest() {
  const expiredRequests = getExpiredPaymentRequests(); // 30분 이상

  for (const request of expiredRequests) {
    // 데이팅 서비스는 스케줄러에서 제외
    if (request.serviceType === 'DATING') {
      continue;
    }

    // 기존 로직: Payment 존재 여부 확인 후 환불
    if (!hasPayment(request.id)) {
      refundViaBootpay(request);
    }
  }
}
```

검증은 데이팅 서비스 PaymentRequest 생성 후 30분 경과 시나리오를 테스트하고, 기존 서비스는 정상 동작하는지 확인하면 됩니다.

#### 2. 모니터링 강화

현재는 하루 2회 (12시 기준)만 모니터링하는데, 이번 사고를 보면 좀 더 자주 확인하는 게 좋을 것 같습니다.

빈도는 1시간 단위 정도가 적절할 것 같고, 임계값은 다음과 같이 설정하면 좋을 것 같습니다:

- 취소율 5% 초과 시 경고 알림
- 취소율 10% 초과 시 긴급 알림
- 일일 취소 금액 100만원 초과 시 알림

구현은 대략 이런 식으로 할 수 있습니다:

```python
def monitor_payment_cancellation():
    today_payments = get_today_payments()
    today_cancellations = get_today_cancellations()

    cancellation_rate = today_cancellations / today_payments * 100

    if cancellation_rate > 10:
        send_urgent_alert("취소율 10% 초과!")
    elif cancellation_rate > 5:
        send_warning_alert("취소율 5% 초과")
```

## 5. 종합 평가 및 권장 사항

### 5.1 TCL 평가

현재 TCL은 기본적인 결제 성공/실패 플로우는 있지만, 실제 사고 원인인 웹훅/스케줄러 관련 테스트가 전혀 없습니다. 즉시 강화가 필요합니다.

### 5.2 재발 방지 대책 평가

재발 방지 대책은 웹훅 진입점 구분은 적절하지만, 모니터링 빈도 및 임계값이 미정의되어 있고, 스케줄러 로직 수정이 구체화되지 않았습니다. 보완이 필요합니다.

### 5.3 최종 권장 사항

**즉시 조치 (1주 이내)**:

1. TCL에 Priority 1 테스트 케이스 7개 추가

3. 모니터링 강화: 빈도 및 임계값 정의
4. OnePager 보완: 테스트 및 검증 절차 추가

**단기 조치 (1개월 이내)**:

5. TCL에 Priority 2 테스트 케이스 추가
6. 테스트 자동화 구축
7. 롤백 계획 수립

**장기 조치 (3개월 이내)**:

8. 완전 분리 로드맵 수립 및 실행
9. E2E 테스트 환경 구축

## 6. 개발자 제안 vs 분석자 제안 비교

| 구분 | 개발자 제안 (OnePager) | 분석자 제안 (본 보고서) | 차이점 및 부족한 이유 |
| --- | --- | --- | --- |
| **웹훅 진입점 구분** | 웹훅 진입점에서 문토↔데이팅 서비스 구분 | 웹훅 진입점에서 서비스 구분 | 동일 - 적절한 방안 |
| **스케줄러 처리** | 명시 없음 (현재 비활성화 상태 유지) | 스케줄러에서 서비스 타입 확인하여 데이팅 서비스 제외 로직 추가 | **부족**: 개발자는 스케줄러 비활성화를 임시 조치로만 인식, 영구적 해결책 미제시 |
| **모니터링** | 하루 2회 (12시 기준), 이상치 발생 시 Slack 알림 | 1시간 단위 모니터링, 취소율 5%/10% 임계값 정의, 일일 취소 금액 100만원 초과 시 알림 | **부족**: 빈도가 너무 낮음 (하루 2회는 사고 발견이 늦을 수 있음), 임계값 미정의로 실제 알림 기준 불명확 |
| **테스트 및 검증** | 명시 없음 | 스테이징 환경 테스트, 배포 전 체크리스트, 배포 후 모니터링 절차 | **부족**: 개발자는 코드 수정에 집중, 배포 후 검증 방법 미고려 |
| **TCL 강화** | 명시 없음 | 웹훅/스케줄러 관련 테스트 케이스 7개 추가 (Priority 1), 통합 테스트 케이스 추가 | **부족**: 개발자는 TCL을 검토하지 않았거나, TCL이 사고와 연관이 있다고 인식하지 못함 |
| **롤백 계획** | 명시 없음 | 롤백 절차 및 조건 명시 | **부족**: 문제 발생 시 대응 절차 미정의 |
| **장기 계획** | 완전 분리 (TBD) | Phase별 구체적 일정 및 마일스톤 제시 (Phase 1: 1주, Phase 2: 1주, Phase 3: 1개월) | **부족**: TBD 상태로 구체적 실행 계획 없음 |

### 개발자가 TCL 개선을 생각하지 못한 이유 분석

1. **역할 인식 차이**: 개발자는 코드 수정과 시스템 개선에 집중하고, 테스트 케이스 작성은 QA/테스터 영역으로 인식하는 경향이 있음
2. **사고 원인 분석 범위**: 개발자는 재발 방지에 초점을 맞춰 코드 수정 방안만 제시했고, "왜 이 문제를 사전에 발견하지 못했는가"라는 근본 원인 분석까지는 하지 않음
3. **TCL 검토 누락**: 기존 TCL을 검토하지 않았거나, TCL에 웹훅/스케줄러 테스트가 없다는 사실을 인지하지 못함
4. **시간 기반 테스트의 어려움**: "30분 경과 후 스케줄러 동작" 같은 시간 기반 테스트는 실제로 구현하기 매우 어렵습니다. 스테이징 환경에서도 30분을 실제로 기다리기는 비현실적이고, 시간 조작 유틸리티를 사용하더라도 복잡합니다. 개발자는 이런 어려운 테스트보다는 코드 수정과 모니터링으로 문제를 해결하려 했을 가능성이 높습니다. 다만, 이런 테스트가 어렵다는 이유로 아예 고려하지 않은 것은 한계로 볼 수 있습니다. (대안: 시간 조작 유틸리티 사용, 또는 모니터링으로 대체)

## 7. 결론

현재 TCL은 결제 취소 사고를 방지하기에 부족합니다. 특히 웹훅 처리, 스케줄러 동작, 서비스 구분 로직에 대한 테스트가 전혀 없어 동일한 사고가 재발할 수 있습니다.

재발 방지 대책은 방향성은 적절하나 구체적 실행 계획이 부족합니다. 스케줄러 로직 수정, 모니터링 강화, 테스트 절차가 구체화되어야 합니다.

개발자 제안은 코드 수정 중심이었고, 테스트 케이스 강화나 검증 절차는 누락되었습니다. 특히 TCL 개선은 개발자가 생각하지 못한 부분인데, 이는 사고의 근본 원인(테스트 부재)을 해결하는 중요한 요소입니다.

즉시 TCL 강화 및 재발 방지 대책 보완이 필요합니다.
