## 1. 총평 — 이전 지적은 대부분 닫혔고, 같은 "부분 수정 잔재"가 일부 재발했다

이번 차수는 만족스럽다. 이전 리뷰에서 [상]으로 올린 블로커(읽음 모델, 보관 정책, 정렬 방향, AdminAuth, 소켓 계약 문서)가 **실제로 닫혔고**, 신고·차단을 채팅 서버에서 떼어내 각 앱 백엔드로 이관한 결정은 "차단은 채팅을 넘어 매칭·프로필·소셜링까지 영향을 준다"는 본질을 정확히 짚은 좋은 판단이다. Kafka(크로스 서비스) ↔ Redis(채팅 내부)의 역할 분리, 제재 enforcement(Kafka → denylist → Redis fanout → 타깃 disconnect)의 Decision Log, REST/WebSocket 배포 분리까지 — 단순 문구 수정이 아니라 **아키텍처 수준의 재설계를 일관되게 반영**했다. socket-events.md·kafka-events.md 신설로 V1 계약 산출물도 갖춰졌다.

다만 이전에 가장 강하게 지적했던 **"앞쪽(전략)은 새로 쓰고 뒤쪽(상세 절)은 옛 문장이 남는" 부분 수정 패턴이 일부 재발**했다. 대표가 두 가지다. (1) 푸시 경로 — 7.5.1 정본(3313~3315)은 "V1·V2 직접 호출 / V3~ Redis Streams"로 단계화됐는데, 같은 7.5.1 하위 bullet(3329·3340)과 7.4.3(3230)은 여전히 "오프라인 → Redis Streams"로 단정한다. 지난번 푸시 경로 지적과 동일한 유형이다. (2) 차단 소유권 — 7.6.7(3576)은 "차단 SOT = 앱 백엔드, 채팅 `blocks` 테이블 없음"으로 깔끔하게 정리됐는데, 정작 그걸 참조하라고 가리키는 7.8.4 각주(3906)는 "1:1 DM 차단은 채팅 서버가 소유"라고 정반대로 적혀 있다.

또 이번 수정본에서 **자동신고 발행의 Outbox 패턴이 통째로 빠졌다**(이전 수정본에는 있었음). 채팅 메시지 저장과 Kafka 발행의 원자성을 보장하던 장치라, 빠지면 자동신고가 유실될 수 있다(§3 N10). 정리하면 **전체 골격·결정은 구현 착수 가능한 수준**이고, 남은 건 새 결정이 미처 닿지 못한 하위 절·다이어그램·보조 문서의 정합을 맞추는 일이다. 아래 [상] 2건(차단 소유권 모순·푸시 단계화 잔재)은 서로를 부정하거나 V1에 없는 경로를 쓰는 서술이라 구현 전에 닫는 게 좋고, [중] 항목은 코드 생성·SSOT 혼동·유실을 막기 위해 함께 정리하면 된다.

## 3. 이번 차수 신규·잔여 이슈 (PR 코멘트형 — 그대로 붙여넣기용)

> 심각도: **[상]** 서로를 부정/구현 불가 · **[중]** 코드 생성·SSOT 혼동·유실 유발 · **[하]** 표기·완전성 보강. 각 제목 옆 *(달 위치)* 는 라인 코멘트 앵커. (라인은 이번 수정본 기준.)

### N1 [상] 1:1 DM 차단 소유권이 7.8.4와 7.6.7에서 정반대입니다 *(달 위치: `srs.md` 7.8.4 끝 각주 3906행, 대조: 7.6.7 3576행)*

7.6.7은 "차단 SOT = 각 앱 백엔드(데이팅/문토)이고, 채팅 서버에는 `blocks` 테이블이 없으며, 채팅 서버는 차단을 이벤트 기반으로 강제만 한다"로 깔끔하게 정리돼 있습니다. 그런데 7.8.4 마지막 각주(3906)는 "사용자 간 1:1 DM 차단(상대 메시지 안 보이게)은 별개로 채팅 서버가 소유 — 7.6.7 참조"라고 적혀 있어, 가리키는 7.6.7과 정면으로 충돌합니다. "소유"의 의미가 모호해 구현 팀이 채팅 서버에 차단 저장소를 다시 만들 수 있습니다.

7.8.4 각주를 7.6.7과 같은 표현으로 맞춰 주세요. 차단의 소유·저장은 앱 백엔드이고 채팅 서버는 "방 종료(EXPIRED)·강제 퇴장으로 강제 실행만 담당"이라는 점이 드러나도록 "소유" 표현을 "강제 실행"으로 바꾸면 됩니다.

### N2 [상] 푸시 경로가 7.4.3·7.5.1 하위에서 다시 "무조건 Redis Streams"로 적혀 있습니다 *(달 위치: `srs.md` 7.4.3 3230행 / 7.5.1 3329·3340행, 정본: 7.5.1 3313~3315행)*

이전에 지적했던 항목인데 절반만 고쳐졌습니다. 7.5.1 본문(3313~3315)은 "V1·V2(1:1 DM)는 chat-notification 직접 호출 / V3~는 Redis Streams"로 정확히 단계화됐습니다. 그런데 같은 7.5.1의 "푸시 알림 발송 조건"(3329)과 "온라인/오프라인 분기 처리"(3340), 그리고 7.4.3(3230)은 버전 구분 없이 "오프라인 → Redis Streams XADD → Push Worker"로 단정합니다. 이대로면 V1·V2에서 존재하지도 않는 경로를 쓴다고 읽힙니다.

7.4.3·7.5.1 하위 bullet의 "Offline → Redis Streams" 서술을 7.5.1 정본(2.7.5)과 동일하게 "V1·V2 직접 호출 / V3~ Redis Streams"로 통일하거나, "(V3~ 그룹 기준, V1·V2는 2.7.5)" 한 줄을 붙여 주세요.

### N3 [중] JWT 인증 설명이 7.1.1에서 모호합니다 (발급 주체·토큰이 둘로 읽힘) *(달 위치: `srs.md` 7.1.1 2297·2301행, 대조: 4.1.2 1190·1204행)*

4.1.2(1190·1204)는 WEBB-1196 통합 설계를 명확히 적었습니다 — 발급은 **munto-backend 단일**, 데이팅 백엔드는 같은 `JWT_KEY`로 **검증만**(2단계 교환 없음), **단일 통합 JWT 하나**. 그런데 7.1.1(2297)은 "문토 백엔드와 데이팅 백엔드에서 **발급한** JWT 토큰을 통합하여 검증"이라 적혀 발급 주체가 둘인 것처럼 읽히고, 바로 아래 bullet(2301)의 "문토와 데이팅 서비스의 **토큰을 모두 검증**"도 토큰이 둘이라는 옛 구도로 읽혀 4.1.2의 '단일 통합 JWT'와 어긋나 보입니다.

의도(예전엔 문토·데이팅이 각각 토큰을 발급·교환하던 것을, 이제 munto-backend가 발급하는 단일 통합 JWT로 합친다)가 드러나도록 7.1.1을 4.1.2와 같은 표현으로 다듬어 주세요 — 예: "발급은 munto-backend 단일, 데이팅 백엔드는 동일 `JWT_KEY`로 검증만 하며, 문토·데이팅을 아우르는 **단일 통합 JWT**를 검증한다."

### N4 [중] 메시지 보관 기간이 6.1.3에서 "6개월"로 적혀 6.5.6(30/90일)과 어긋납니다 *(달 위치: `srs.md` 6.1.3 1596·1641행, 대조: 6.5.6 2177~2179행)*

6.5.6과 ERD는 30일(사용자 조회) → 90일(백오피스 운영 DB) → 1년(S3) 3단계로 정리돼 있는데, 6.1.3의 Missed Event 복구 범위에는 "보존 범위 … (6개월, 6.5.6 참조)"(1596)와 "오랜 기간 미접속(6개월 이상)"(1641)이 남아 있습니다. 6.5.6에는 6개월 단계가 없어 재연결 갭 채우기 한계가 정책과 어긋납니다.

6.1.3의 "6개월"을 실제 정책 값으로 맞춰 주세요(사용자 조회는 30일, 운영 DB 실데이터는 90일까지 — 재연결 복구는 운영 DB 기준 90일이 자연스럽습니다).

### N5 [중] 관리자 "삭제된 메시지도 조회 가능"인데 ERD엔 soft delete가 없습니다 *(달 위치: `api/swagger.yaml` 910~911행, 대조: `database/erd.md` messages 정의 197~216행 · 90일 DELETE 528~530행)*

`GET /admin/rooms/{roomId}/messages` 설명(910~911)은 "블라인드 처리된 메시지도 원문 조회 가능 / 삭제된 메시지도 조회 가능"이라고 합니다. 그런데 ERD `messages`에는 `is_deleted`·`deleted_at` 같은 soft-delete 컬럼이 없고, 정리 쿼리(528~530)와 관리자 삭제(swagger 983~985)는 물리 삭제(hard delete)입니다. 메시지 삭제가 물리 삭제라면 "삭제된 메시지 admin 조회"가 성립하지 않습니다.

메시지 삭제를 soft delete(예: `deleted_at`/`is_deleted`)로 둘지, 아니면 admin 조회 대상은 "블라인드"만인지 확정하고 ERD·설명을 맞춰 주세요. (블라인드만 원문 보존이라면 swagger의 "삭제된 메시지도 조회 가능" 문구를 정리.)

### N6 [중] int64 표기가 보조 스키마 일부에 누락됐습니다 *(달 위치: `api/swagger.yaml` ImageItem.messageId 1618~1620행 · Announcement.messageId 1646~1647행 · AdminMessageSearchItem.id 2063~2065행)*

핵심인 `Message.id`(1560~1562)와 커서 파라미터(1253~1270)는 `format: int64`로 잘 반영됐습니다. 다만 메시지 id를 담는 보조 스키마들(`ImageItem.messageId`, `Announcement.messageId`, `AdminMessageSearchItem.id`)은 아직 `type: integer`(format 없음)이라, 코드 생성 시 32비트로 해석될 수 있습니다. 동일하게 `format: int64`를 붙여 주세요.

### N7 [중] 감사 로그 조회 API가 없고, action enum이 SRS와 ERD에서 다릅니다 *(달 위치: `database/erd.md` audit_action_type 68~72행·audit_log 384~392행, 대조: `srs.md` 7.8.1 3797행 / `api/swagger.yaml` paths)*

ERD에 `audit_log` 테이블과 `audit_action_type`(`DELETE_MESSAGE`/`FORCE_LEAVE`/`UPDATE_PROFANITY`)이 추가된 건 좋습니다. 두 가지만 맞춰 주세요. (1) SRS 7.8.1(3797)은 action을 "DELETE_MESSAGE, BLOCK_USER, UPDATE_PROFANITY"로 적었는데, 차단·제재가 앱 백엔드로 이관됐으니 `BLOCK_USER`는 ERD의 `FORCE_LEAVE`로 통일하는 게 맞습니다. (2) swagger V1 기능 목록(24행)에는 "감사 로그"가 있는데 `/admin/audit*` 조회 엔드포인트가 없습니다 — 백오피스에서 감사 로그를 보려면 조회 API가 필요하니, V1 산출물로 추가하거나 "내부 기록 전용(조회 V2)"임을 명시해 주세요.

### N8 [중] 신규 보조 문서들의 계약 정합을 맞춰 주세요 *(달 위치: 아래 각 파일·라인)*

socket-events.md·kafka-events.md·client-architecture.md가 새로 들어오면서 문서 간 계약이 일부 어긋납니다. 한 번에 정리 부탁드립니다.

- **재연결 동기화 SSOT 단일화** — `srs.md` 2.7.5(823행)는 재연결을 REST `GET /rooms/{id}/messages?after=`로, 6.1.3(1621~1624행)과 socket-events는 `join_room` + `missedMessages`로 설명합니다. 두 경로 중 V1 정본을 하나로 못 박아 주세요.
- **LastMessage 필드 불일치** — socket-events의 `chat_list` lastMessage에는 `messageId`가 있는데(239행) swagger `LastMessage`(1449~1464행)에는 없습니다. 커서 동기화에 쓰려면 swagger에도 추가가 필요합니다.
- **SDK 시그니처 vs 소켓 계약** — `client-architecture.md`의 `markRead(roomId)`(52행)는 socket-events `mark_read`(167~168행)가 **`messageId` 필수**인 점과 어긋납니다. 또 `sendMessage`는 `tempId`를 반환값으로만 표기했는데 소켓 `send_message`(128~131행)는 `tempId`가 **입력 필수**입니다. (메시지 본문 필드는 양쪽 `content`로 일치합니다.) SDK 공개 API와 소켓 페이로드의 필수 필드를 한쪽 기준으로 정렬해 주세요.
- **WSS URL 불일치** — socket-events(32행)·swagger(45행)는 `wss://chat-server.munto.kr`, client-architecture(233행)는 `wss://chat.munto.kr`.

### N9 [중] 크로스서비스 전송으로 Kafka(MSK)를 택한 비교 근거를 SRS에 남겨 주세요 *(달 위치: `srs.md` 4.4 1229행 / 7.8.4 Decision Log 3882행)*

자동신고(S1)·계정 제재(S2)에 Kafka(MSK)를 쓰기로 한 건 "이미 보유한 MSK 활용"이라는 점에서 합리적인 선택입니다. 다만 4.4에는 "Kafka로 한다"는 결론만 있고 **어떤 대안과 비교해 이 결론에 이르렀는지**가 없습니다(7.8.4 Decision Log도 enforcement 방식만 다루고 전송 수단 비교는 없음). 채팅 푸시는 과거 Kafka→Redis Streams로 바꾼 이력이 있어서, 같은 서비스 안에서 "푸시는 Redis / 크로스서비스는 Kafka"가 왜 갈리는지 비교 근거 없이 보면 일관성 의문이 생깁니다. (이건 "여러 대안을 비교 검토했다면 그 비교 과정도 SRS에 남긴다"는 우리 원칙이기도 합니다.)

제재 enforcement Decision Log를 잘 작성하셨으니, 전송 방식 선택도 같은 형식으로 비교표를 남겨 주세요. 최소한 (1) Kafka(MSK 재사용) (2) REST S2S — 차단 강제(S2b)가 이미 쓰는 방식 (3) 공유 Redis denylist + Pub/Sub — 이미 fanout 계층으로 구축 중 이 세 안을 두고, 물량(제재·자동신고는 동접과 무관하게 저빈도)·내구성/리플레이·팀 간 결합도·기존 인프라 관점에서 왜 Kafka인지 정리하면 됩니다. 특히 **같은 앱 백엔드를 향하는데 S2b는 REST, S1은 Kafka로 갈린 이유**를 한 줄 적어주시면 일관성 의문이 해소됩니다. (결론이 Kafka로 유지되더라도, 조직 차원 근거 — 기존 MSK·상대 팀 Kafka 생태계 — 를 명시하면 충분합니다.)

### N10 [중] 자동신고를 Kafka에 직접 produce하면 Outbox(원자성)가 빠져 유실 위험이 있습니다 *(달 위치: `srs.md` 4.4 S1 1233행, 대조: `database/erd.md` 테이블 목록)*

이전 수정본에는 "Kafka 발행 원칙 — Outbox Pattern"이 있어서, 채팅 메시지 저장/블라인드 처리와 **같은 DB 트랜잭션**에서 `outbox_events`에 기록한 뒤 Outbox Publisher가 Kafka로 발행하는 구조였습니다. 이번 갱신본에서는 이 서술이 통째로 빠지고(SRS·ERD 전체에서 `outbox` 0건, ERD `outbox_events` 테이블도 삭제), 4.4 S1(1233)이 "웹소켓 서비스가 자동신고 Kafka 토픽에 produce"로만 적혀 있습니다. 메시지 저장 성공 후 Kafka 발행이 실패하면(또는 그 반대) 자동신고가 유실되는 dual-write 문제가 생깁니다.

의도적으로 단순화한 것인지, 정리 중 빠진 것인지 확인 부탁드립니다. 자동신고 유실을 막으려면 Outbox 패턴(또는 동등한 at-least-once 보장)을 다시 명시하는 게 안전합니다.

### N11 [하] 마무리 메모 (지금 막지 않음)

- `kafka-events.md`의 `user-sanction-event`(38~72행)에는 DLQ·소비 실패 재시도 정책이 없습니다(`chat-auto-report-*`엔 DLQ 명시, 82행). 제재 누락은 보안 영향이 있으니 추후 보강 권장.
