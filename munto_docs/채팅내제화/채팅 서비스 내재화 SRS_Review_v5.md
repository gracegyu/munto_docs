## 1. 총평 — 골격은 합격, 그러나 "부분만 고친 흔적"이 곳곳에 남았다

**결론부터: 본격 리뷰를 진행할 가치가 있다(게이트 통과). 단, 무결성·문서 간 정합은 아직 미완이다.**

먼저 칭찬할 부분은 분명하다. V4 재작성 가이드에서 요구한 **상위 전략(Top-down)이 이번엔 실제로 문서를 끌고 간다.** Time-to-Market 선언(1.2), V1~V4 단계 정의(2.7.1), Sendbird 단계별 대체표(2.7.2), 오프라인 2층위 단계화(2.7.5), 컷오버·강업·flip·롤백 메커니즘(2.8.4), 용량 실측 재산정(5.1/5.2), 클라이언트 로컬 DB 확정(4.6.1 + client-local-db.md), 백오피스 단계화(7.8)까지 — 지난 차수에서 비어 있던 골격이 대부분 채워졌다. 특히 아래 셋은 단순 AI 출력이라고 보기 어려운, **본인이 고민한 흔적이 보이는** 부분이다.

- **문토 DM 마이그레이션 Decision Log(2.8.2)**: A/B/C안을 비교하고 B안(자연 소멸) 채택 근거·재검토 트리거까지 명시. 가이드가 시킨 적 없는 형식인데 스스로 의사결정을 기록했다.
- **강업과 백엔드 flip의 분리(2.8.4)** + **Closer는 OTA라 강업 불필요** 같은 예외 처리. "한 방 안에서 백엔드 혼재 불가"라는 본질을 정확히 짚었다.
- **재연결 갭 채우기를 '선택'이 아니라 '정합성 요구사항'으로 V1 필수 처리(2.7.5)** — 커서 단조 증가 키, 멱등 머지까지 언급. 채팅 내재화의 진짜 함정을 이해하고 있다.

**그런데 사용자의 의심("몇 시간 만에 다시 썼다 → 빠짐없이 자가검토했는가")은 절반은 맞았다.** 새로 쓴 상위 섹션(1.2, 2.7, 2.8)은 훌륭한데, **기존 7장·6장 상세 본문이 새 결정(ERD 변경·푸시 단계화·용량 재산정)에 맞춰 함께 갱신되지 않았다.** 즉 "앞쪽 전략은 새로 쓰고, 뒤쪽 상세는 옛날 문장이 그대로 남은" 부분 수정의 전형적 흔적이 보인다. 아래 3장에 실제 충돌을 라인까지 적었는데, 그중 **같은 7장 안에서조차 7.5.1은 푸시를 단계화했는데 7.3.1/7.3.2는 옛 서술 그대로**인 사례가 결정적 증거다. 전체를 한 번 더 통독하며 정합을 맞췄다면 나오기 어려운 모순이다.

정리하면: **전략·구조는 합격선을 넘었다.** 게이트를 통과시키고 **본격 리뷰까지 이어서 수행했다**(3장 = 부분 수정 잔재로 인한 무결성 충돌, 4장 = API·ERD·정책의 설계 누락/충돌, 6장 = 추가로 잘한 점). 결과적으로 V5는 "구현 착수 직전" 단계이며, 남은 일은 **새로 쓴 전략과 기존 상세(7장·6장·ERD·API)의 정합을 맞추고 V1 필수 산출물의 빈칸을 메우는 것**이다. 3장 [상] 3건 + 4장 [상] 4건은 **구현이 불가능하거나 문서가 서로를 부정하는** 상태라 착수 전에 먼저 닫아야 한다.

## 3. 무결성·문서 간 충돌 (실측 확인 — 본격 리뷰의 1차 지적)

> 모든 항목은 실제 파일 라인을 직접 확인한 것이다. 심각도: **[상]** 구현 불가/상호 부정 · **[중]** 명백한 불일치 · **[하]** 표기·참조 정리.

### 3.1 [상] 읽음 처리 모델이 SRS 본문 ↔ ERD에서 정반대

- **ERD / Jira 변경 이력**: `message_reads`·`unread_message_counts` 테이블을 **제거**하고, `chat_room_participants.last_read_message_id`(BIGINT) 한 컬럼 + `COUNT(*) WHERE id > last_read_message_id` 방식으로 단순화했다고 명시(DEVT-31 99~104행, ERD 147~196행).
- **그런데 SRS 7.4.2(3082~3088행)**는 여전히 **"카운터 방식 사용", `unreadCount + 1`, 저장 위치 = `ChatRoomMember` 테이블의 `unreadCount` 컬럼"** 으로 서술 → ERD가 지운 방식이 SRS 본문에 그대로 살아 있다.
- 게다가 테이블명 **`ChatRoomMember`(7.4.2) ↔ `chat_room_participants`(ERD)** 도 불일치.
- **요청**: 7.4.2를 ERD의 `last_read_message_id` 모델로 통일(또는 ERD를 카운터 방식으로 되돌리는 결정). 현재는 두 문서가 서로를 부정해 어느 쪽으로 구현해야 할지 알 수 없다.

### 3.2 [상] 오프라인 푸시 경로가 7장 안에서 자기모순 (단계화 vs 무조건 Redis Streams)

- **2.7.5(780~783행)·6.1.3·7.5.1(3278~3280, 3294, 3305행)**: V1·V2(1:1 DM)는 **chat-notification 직접 호출**(Redis Streams·Push Worker 생략), V3~(그룹)부터 Redis Streams + Push Worker — 정확히 단계화됨. ✅
- **그런데 같은 7장의 7.3.1 텍스트 전송(2819행)·7.3.2 이미지 전송(2865행)**은 단계 구분 없이 **"Redis Streams를 통해 푸시 이벤트 발행"** 으로 단정 → V1·V2에선 존재하지도 않는 경로를 쓴다고 서술.
- **요청**: 7.3.1/7.3.2의 푸시 발행 단계를 7.5.1·2.7.5와 동일하게 "V1·V2 직접 호출 / V3~ Redis Streams"로 맞출 것. (같은 장 안에서 갈리는 것이 가장 명백한 부분 수정 흔적이다.)

### 3.3 [상] client-local-db가 존재하지 않는 SRS 섹션(§2.9.2)을 참조

- **client-local-db.md 200행**: 델타 동기화를 "SRS §2.9.2"로 참조. 그러나 **SRS v5에 2.9 섹션은 없다**(2장은 2.8.5에서 끝나고 3장으로 넘어감). 델타 동기화 `sync()`는 실제로 **2.7.5(798·809행)**, 저장 대상은 4.6.1에 있다.
- 같은 파일 176행의 "SRS §2.8.3 하위호환"도 부정확(2.8.3 제목은 "마이그레이션 단계", 하위호환은 2.8절 전체).
- **요청**: client-local-db.md의 SRS 참조 번호를 실존 섹션으로 정정.

### 3.4 [중] messages.id 타입 표기 불일치 (SERIAL vs BIGSERIAL)

- **SRS 7.3.5(2981행)**: 정렬 기준 `messageId (PostgreSQL SERIAL)`.
- **ERD(209행)·client-local-db(66행)**: `BIGSERIAL` / `BIGINT`. (DEVT-31 108행에서 "SERIAL→BIGSERIAL, 21억 초과 방지"로 변경했다고 직접 명시.)
- **요청**: 7.3.5를 `BIGSERIAL`로 정정. (API `Message.id`도 `format: int64` 반영 검토.)

### 3.5 [중] PENDING 메시지 저장소 충돌 (SharedPreferences vs SQLite)

- **SRS 6.1.1(1448·1470행)**: PENDING/FAILED 메시지 저장 위치 = **SharedPreferences**.
- **client-local-db.md(32~46행)**: 전송 실패 메시지는 **SQLite `pending_messages` 테이블**에 영구 저장하고, 손상 복구 시에도 "캐시와 분리 저장해 보존·우선 플러시"하는 핵심 자산으로 설계.
- 두 서술이 기술적으로 다르다(KV 저장소 vs 관계형 테이블). 특히 손상 복구 전략(client-local-db 185~191행)은 `pending_messages`가 DB 안에 있다는 전제라, SharedPreferences로 적으면 복구 전략과도 어긋난다.
- **요청**: 저장소를 한쪽으로 통일(권장: client-local-db의 `pending_messages` 기준). 6.1.1 본문 수정.

### 3.6 [중] 용량 수치 — 6.1.3에 옛 "1만 동접" 전제 잔존

- **5.1/5.2**: Sendbird 실측 PCC(피크 340 / 최근 189) 기반으로 "1만 동접 전제 과설계"를 명시적으로 폐기하고 1vCPU/2GB×2 task로 재산정. ✅
- **그런데 6.1.3(1622행)**: "동시 접속 **10,000명** 기준 Redis 부하·비용 문제 없음" 이 그대로 남음 — 바로 8행 위(1614행)에서 "실측 피크 PCC ~340"을 말해 놓고 모순.
- 5.2의 연결 임계치 "예: 4,500개(90%)"(1343행)도 구 5,000 기반 잔재로 보임.
- **요청**: 6.1.3·5.2의 구 용량 수치를 5.1 실측 기준으로 정리하거나 "(레거시 참고)"로 명확히 분리.

### 3.7 [중] V2+ 전제인 `sync()` 서버 API가 API 스펙에 없음

- SRS 2.7.5(798·809행)·client-local-db(200행)는 V2+ 델타 동기화를 **서버 `sync` API** 전제로 기술하나, **채팅API.yaml에 `/sync` 경로가 없다.**
- V2 산출물이므로 V1 본격 리뷰의 차단 사유는 아니지만, "전제만 있고 인터페이스가 없는" 상태는 명시 필요.
- **요청**: API 스펙에 sync 엔드포인트를 (V2 태그로) 추가하거나, 본문에 "V2에서 정의 예정"으로 표시.

### 3.8 [중] API `LastMessage.type` enum 축소

- API의 `Message.type`은 6값(TEXT/IMAGE/SYSTEM/CHALLENGE_CHECKIN/FEED/POLL)인데, `LastMessage.type`(1635행)은 `[TEXT, IMAGE, SYSTEM]`만 → 채팅방 목록의 "마지막 메시지"가 챌린지·투표일 때 표현 불가.
- **요청**: `LastMessage.type`을 `Message.type`과 동일 enum으로 통일(또는 축소 의도면 본문에 명시).

### 3.9 [하] 버전 태깅 — 7장은 이미 태깅 완료, DB·API는 "미리 전체 설계"가 원칙

- (정정) 본격 리뷰에서 확인하니 **7장 각 절 헤더에 `[V1]/[V2+]/[V3]/[V4]` 태그가 이미 달려 있다**(예: 7.2.1.1 `[V3 소셜링 / V4 클럽·챌린지]`, 7.8.3 `[V2+]`). V4 가이드의 핵심 요구는 충족됐다 — **칭찬할 부분.**
- 다만 **2.7.4(760행)는 여전히 "단계 태깅의 구체 작업은 후속 차수에서 정리"라고 적혀 있어** 실제 결과(이미 태깅 완료)와 어긋난다 → 2.7.4 문구를 "7장에 단계 태그 반영 완료, P1~P3 세분류만 후속"으로 갱신.
- **DB(ERD)·API는 단계별로 쪼개 만드는 게 아니라, V1~V4 전체를 미리 설계(forward-compatible)해 두는 것이 원칙이다.** 스키마 설계는 싸지만 운영 중 파괴적 마이그레이션은 비싸고 위험하기 때문이다(특히 hard-cutover·강업 구조). 그리고 실제로 **이 ERD는 이미 V1~V4를 포괄하게 설계돼 있다** — `chat_rooms`의 `socialing_id`/`club_id`/`challenge_id`(nullable), `message_type`의 `CHALLENGE_CHECKIN`/`FEED`/`POLL` 등 V3/V4용 컬럼을 V1 시점에 미리 둠. **잘한 점이다.** (실제 물리 배포·마이그레이션만 단계별 additive로 진행하면 된다.)
- 다만 **API(swagger)·ERD에는 단계 태그가 없어** 어느 엔드포인트/테이블이 어느 버전에 필수·활성화되는지가 안 보인다. 이는 **빌드를 쪼개라는 뜻이 아니라**, 일정·검수에서 "V1 개통 필수 항목"을 빠르게 가려내기 위한 **추적 메타데이터**일 뿐이므로, 7장처럼 태그만 달아두기를 권장.
- 경계: "미리 다 설계"가 **남의 소유 테이블까지 채팅 ERD로 끌어오라는 뜻은 아니다.** 클럽 투표·피드·채널 등은 문토 백엔드 소유(7.7)이므로, 채팅 ERD는 자기 소유 범위(방·메시지·참여자·신고·차단·알림설정·금칙어 + 누락된 audit_log)만 전체 설계한다.

### 4.1 데이터 보관 정책이 문서마다 다릅니다 *(달 위치: `api/erd.md` "데이터 보관 정책" 섹션, 528행 부근)*

여기 ERD 정리 쿼리는 메시지를 `sent_at` 기준 30일이 지나면 운영 DB에서 바로 삭제하는 한 단계로만 돼 있습니다. 그런데 SRS 6.5.6은 30일(사용자 조회) → 90일(백오피스 검색, 운영 DB) → 1년(S3 콜드스토리지)의 3단계로 정의돼 있고, 7.8.7 백오피스 메시지 검색도 "최대 90일"을 전제합니다. 지금 ERD대로면 30일에 삭제돼서 백오피스 90일 검색과 1년 보관이 동작하지 않습니다. 그리고 2.8.2에서 클럽은 "완전 마이그레이션(대화 내역 보존)"이라고 했는데, "30일 후 사용자 조회 불가"와도 부딪힙니다.

두 가지를 정리해 주세요. (1) 보관 단계(운영 DB 90일 → S3 1년 아카이빙)를 ERD 정리 쿼리와 아키텍처에 반영해 주세요. (2) "사용자 조회 30일"이 만료가 있는 데이팅 DM만 해당인지, 문토 DM·소셜링·클럽까지 공통으로 적용할 건지 확정이 필요합니다. 채팅 종류별로 보관 기간을 다르게 둘지는 PM과 함께 정해야 할 것 같습니다.

### 4.2 메시지 정렬 방향이 swagger와 SRS/ERD에서 반대입니다 *(달 위치: `api/swagger.yaml` 460행 `GET /rooms/{roomId}/messages` — 정렬 설명 468행 부근)*

메시지 목록 조회 설명에는 "최신부터 내림차순"으로 적혀 있는데, SRS 7.3.5와 ERD는 `messages.id` 오름차순을 정렬 기준(SSOT)으로 잡고 있습니다. 커서 페이지네이션은 정렬 방향이 계약의 핵심이라, 클라이언트가 이 방향에 맞춰 메시지를 이어 붙이고 화면에 표시합니다. 방향이 문서마다 다르면 클라·서버 구현이 갈립니다.

정렬 방향과 `after`/`before`가 각각 어느 방향으로 페이지를 확장하는지를 한 곳(swagger description 권장)에 명확히 적고, SRS 7.3.5·ERD와 같은 표현으로 통일해 주세요.

### 4.3 `messages.id`가 BIGINT인데 API 타입에 반영돼 있지 않습니다 *(달 위치: `api/swagger.yaml` `Message` 스키마 1721행 / `MessageId` 파라미터 1424행, SRS 7.3.5 2972행)*

ERD에서 `messages.id`는 BIGSERIAL이고 클라이언트 로컬 DB도 BIGINT인데(이미 "21억 초과 방지"로 BIGSERIAL로 바꾸셨죠), swagger에서는 `Message.id`와 커서 파라미터가 `format` 없는 `integer`로 돼 있습니다. 이러면 코드 생성 시 32비트로 해석돼 id가 잘릴 수 있습니다.

`Message.id`와 커서 파라미터에 `format: int64`를 명시하고, 커서가 순수 id 기반인지(아니면 복합 키 인코딩인지)도 한 줄 적어 주세요. (SRS 7.3.5 본문에도 `SERIAL`로 적힌 곳이 남아 있으니 `BIGSERIAL`로 함께 맞춰 주세요.)

### 4.4 "V1 운영 필수"로 적은 기능이 정작 API·테이블에 없습니다 *(달 위치: SRS 7.8 3752행·7.8.1 3773행·7.8.4 3864행 / `api/swagger.yaml` `/admin/*` 880행~ / `api/erd.md` 테이블 정의 85행~)*

7.8에서 관리자 사용자 차단(7.8.4)·감사 로그(7.8.1)를 V1 운영 필수로 못 박으셨는데, 산출물이 비어 있습니다. 우선 관리자 차단 API가 swagger `/admin/*`에 없습니다(`/blocks`는 사용자 자가 차단용입니다). 7.8.4의 차단 효과인 "차단된 사용자의 모든 채팅방 나가기"는 채팅 서버가 수행하는 동작이라 채팅 서버 API가 필요한데, 본문은 "채팅 서버 또는 문토 백엔드에 차단 요청"으로 주체가 모호합니다. 또 감사 로그(DELETE_MESSAGE·BLOCK_USER·UPDATE_PROFANITY·검색 로그)를 V1 필수로 적었는데 ERD에 `audit_log` 테이블이 없고, 7.8.4의 제재 상태(7일/30일/영구 정지)를 어느 테이블/서비스가 보유하는지도 정해져 있지 않습니다.

"V1 필수"라고 적은 항목은 API와 ERD 테이블까지 V1 산출물로 닫아 주세요. 구체적으로 (1) 관리자 차단/제재 API를 `/admin/*`에 추가, (2) `audit_log` 테이블을 ERD에 추가, (3) 제재 상태를 채팅 서버가 가질지 문토 백엔드가 가질지 경계를 명시해 주세요.

### 4.5 관리자 API가 일반 사용자와 권한이 구분되지 않습니다 *(달 위치: `api/swagger.yaml` `securitySchemes` 1408행 / `/admin/*` 880행~)*

`/admin/*` 엔드포인트들이 일반 사용자 API와 똑같이 `BearerAuth`(JWT)만 요구하고 있어, SRS 7.8.1에서 말한 `UserGrade.ADMIN` 권한 검증이 스펙에는 드러나 있지 않습니다. 이대로면 코드 생성이나 게이트웨이 설정에서 관리자 보호가 누락될 수 있습니다.

관리자 전용 스코프를 스펙에 반영해 주세요. 별도 securityScheme를 두거나 ADMIN 클레임을 요구하도록 표기하고, 권한 없는 호출에 대한 403 응답도 `/admin/*`에 추가해 주세요.

### 4.6 소켓 이벤트 계약 문서가 없습니다 *(달 위치: SRS 1.5 87행 / `api/swagger.yaml` 10·33행, 또는 PR 상단 일반 코멘트)*

SRS 1.5는 `./api/socket-events.md`를, swagger는 `./websocket-events.md`를 가리키는데 둘 다 레포에 없습니다. 채팅 서버는 REST보다 소켓이 1차 인터페이스인데, 핵심 이벤트(`send_message`/`new_message`/`mark_read`/`join_room`/`typing`/`chat_list_updated` 등)가 SRS 본문(7.1~7.5)에 흩어져 있을 뿐 한 곳에 모은 계약 문서가 없습니다.

소켓 이벤트 명세 문서를 산출물로 추가해 주세요(이벤트명·payload·ack·에러·버전 포함). V1 구현에 선행되어야 하는 계약이고, 1.5의 링크도 실제 경로로 연결해 주세요.

### 4.7 신고·차단의 처리 주체와 멱등 규칙을 명확히 해주세요 *(달 위치: SRS 7.6.6 3514행·7.6.7 3562행·7.8.4 3864행 / `api/swagger.yaml` `CreateRoomRequest` 2129행)*

신고(7.6.6)·차단(7.6.7)·관리자 차단(7.8.4)이 모두 "채팅 서버 또는 문토 백엔드"로 적혀 있어 어느 쪽이 처리 주체(SOT)인지 모호합니다. 한쪽으로 확정해 주세요. 또 swagger에 백엔드 전용 API(방 생성·참여자·상태 변경)와 클라이언트용 API(목록·메시지·설정·신고)가 description으로만 구분돼 있어, SDK·BE 팀이 경계를 혼동할 수 있습니다. 태그나 `x-audience` 같은 표기로 호출 주체를 구분해 주세요. 그리고 멱등 방 생성 규칙(7.2.1.4: DM은 정렬한 `(userId1, userId2)` 조합)이 `CreateRoomRequest` 스키마 설명에 빠져 있으니 추가해 주세요. (200/201로 신규·기존을 구분한 부분은 잘 반영돼 있습니다.)

### 4.8 기본값·문서 참조 몇 가지를 맞춰 주세요 *(달 위치: `api/swagger.yaml` `Count` 1451행·검색 1142행 / SRS 1.5 88행·7.3.1 2800행 / `api/erd.md` 4행·52행·76행)*

메시지 목록 기본 건수가 swagger는 20, SRS 7.3.3·7.8.7은 50으로 다릅니다 — 한쪽으로 통일해 주세요. 문서 링크도 정리가 필요합니다: SRS 1.5의 ERD 경로가 `./database/erd.md`로 돼 있으나 실제 파일은 `./api/erd.md`이고, ERD 4행의 `[SRS](./srs.md)`도 실제 파일명과 다릅니다. ERD 타입 정의 주석의 SRS 참조도 일부 어긋나 있습니다(금칙어 일치타입은 7.8.4→7.8.5, report_type은 7.6.3→7.6.6). 마지막으로 Rate Limiting 수치(초당 3 / 분당 60)는 6.5.3에만 있고 7.3.1엔 수치 없이 언급만 돼 있으니, 7.3.1에서 6.5.3을 참조하도록 연결해 주세요.

### 4.9 단계·완전성 마무리 (낮은 우선순위) *(달 위치: SRS 7.8.8 3988행·7.7.3 3735행, 또는 PR 상단 일반 코멘트)*

다음은 지금 당장 막는 건 아니지만 메모로 남깁니다. 7.8.8의 `is_suspicious` 플래그(V2+)는 ERD `messages`에 컬럼이 없으니 추후 추가가 필요합니다. 7.7.3의 클럽/챌린지 연동 API(V4)는 아직 swagger에 없는데 단계상 정상이며 추후 V4로 추가하면 됩니다. 그리고 API·ERD에는 V1~V4 단계 태그가 없는데(7장 본문에는 있습니다), 어느 엔드포인트·테이블이 어느 버전에 필수인지 추적하기 위한 메타데이터이니 7장처럼 태그만 달아두면 좋겠습니다.

## 6. 추가로 잘한 점 (본격 리뷰에서 확인)

- **클럽/챌린지(7.7)**: 문토 백엔드를 SOT로 두고 채팅 서버는 시스템 메시지 표시/연동만 담당하는 경계가 깔끔. V4로 명확히 분리.
- **DM 어뷰징(7.6.8)**: 수치를 무리하게 확정하지 않고 `TBD + 담당(PM) + 결정 시점(V2 착수 전) + Jira 티켓 + 결정 기준(데이터 분석)`으로 **보류를 관리 가능한 형태로** 남김.
- **금칙어 단계화(7.6.3)**: "런타임 필터링은 V1(시드 목록) / 관리 UI는 V2+"로 기능을 쪼갠 판단이 합리적.
- **멱등 방 생성**: 신규 201 / 기존 200 구분(7.2.1.4)이 API에 반영됨.
- **한글 검색 인덱스**: 백오피스 검색용 `pg_bigm` GIN 인덱스를 ERD에 미리 반영.
- **무결성 제약**: `chat_rooms` CHECK(GROUP→socialing/club/challenge 중 하나, DATING_DM→match_id)로 타입별 필수 컬럼 강제.
- **알림 설정 소유권**: `notification_settings`를 채팅 서버 소유로 정리(발송 주체 일치, s2s 회피) — 7.5.2 결정과 ERD 정합.
