# 채팅 솔루션 대체하여 자체 솔루션 개발

# **센드버드에서 자체 채팅 서버로 전환 - 상세 분석**

## 배경

현재 문토와 데이팅 앱에서 센드버드(SendBird) 서드파티 서비스를 이용하여 채팅 기능을 제공하고 있으나, 사용자 증가에 따른 비용이 지속적으로 증가하고 있습니다. 특히 MAU(월간 활성 사용자) 기반 과금 구조로 인해 서비스 규모가 커질수록 비용 부담이 커지고 있어, 장기적인 비용 절감과 기술적 자립을 위해 자체 채팅 서버 구축이 필요합니다. 

## 현재 문제점

1. **높은 운영 비용**: 센드버드의 MAU 기반 과금으로 사용자 증가 시 비용이 선형적으로 증가
1. **서드파티 의존성**: 외부 서비스에 대한 의존도가 높아 커스터마이징 제약 및 장애 대응 어려움
2. **데이터 제어권 부족**: 메시지 데이터가 센드버드 서버에 저장되어 데이터 관리 및 백업 정책 수립의 어려움
3. **기능 확장 제약**: 센드버드에서 제공하지 않는 커스텀 기능 구현의 어려움
4. **MSA 구조와의 불일치**: 문토와 데이팅이 MSA로 개발되어 있으나 채팅 서비스는 외부 서비스로 분리되어 있어 아키텍처 일관성 부족 


## 개선 방향

1. **독립적인 채팅 마이크로서비스 구축**: NestJS + Socket.IO  기반의 자체 채팅 서버 개발
1. **실시간 통신 구현**: WebSocket을 통한 양방향 실시간 메시징 구현
2. **메시지 저장 시스템**: PostgreSQL을 통한 메시지 영구 저장 및 Redis를 통한 캐싱 및 Pub/Sub 메시지 브로커 활용
3. **확장 가능한 아키텍처**: 로드밸런서를 통한 수평 확장 및 Redis Pub/Sub 또는 Kafka를 통한 서버 간 메시지 동기화
4. **통합 인증 시스템**: 기존 JWT 토큰 시스템과 통합하여 SSO(Single Sign-On) 지원
5. **기존 기능 유지**: 현재 센드버드에서 제공하는 모든 기능(1:1 채팅, 그룹 채팅, 파일 전송, 읽음 상태, 타이핑 표시 등) 구현

## **1. 총평 — 핵심 전략은 Time-to-Market, 그런데 문서엔 Top-down이 없다**

**이 프로젝트의 핵심 전략은 Time-to-Market이다.** Sendbird는 MAU(월간 활성 사용자) 기반 과금이라 사용자가 늘수록 비용이 가파르게 오르고, 메시지 한도·기능 커스터마이징·사용량(limit) 제약까지 겹쳐 있다. 따라서 **가능한 한 빠르게, 단계별로 Sendbird를 걷어내는 것**이 목표다. 한 번에 전 기능을 자체 구축하는 게 아니라, 가장 독립적이고 빠르게 떼어낼 수 있는 데이팅 채팅(V1)을 먼저 출시하고 → 소셜링 채팅·문토 DM(V2) → 챌린지·클럽 채팅(V3) 순으로 확장하며, **각 단계가 끝날 때마다 해당 영역의 Sendbird 비용이 실제로 줄어드는** 구조로 가야 한다. 이 전략이 문서 전체를 끌고 가는 상위 명제여야 한다.

그런데 v4는 거꾸로다. Redis 명령어, 재시도 횟수, 상태 전이 규칙까지 매우 상세한데, **"무엇을 어느 순서로, 어디까지 만들 것인가"라는 상위 골격이 없다.** 상세 구현(7장, 6.1.3)은 1만 동접·풀 Offline 푸시 파이프라인을 전제로 깔아 두었는데, 정작 단계별 요구사항(2.7)은 "전부 P1, v1.0에 다 넣음" 한 단락으로 끝난다. 추상화 레벨이 거꾸로이고, Time-to-Market 전략과도 어긋난다 — 처음부터 풀 기능을 다 만들면 V1 출시가 늦어지고 Sendbird 비용 절감 시점도 그만큼 밀린다.

이번에 바꾼 전략(Time-to-Market 기반 단계적 Sendbird 제거 = V1 데이팅 채팅만, Offline 최소화, 알림 플랫폼 선택)이 문서에 **거의 반영돼 있지 않다.** 아래 세 가지를 상위 구조로 먼저 세우고, 그 다음 7장 상세가 각 버전에 매핑되도록 다시 써야 한다. 지금은 상세부터 쌓아 올린 bottom-up 문서라, 전략이 바뀌면 어디를 고쳐야 하는지조차 추적이 안 된다.

## **2. 이번에 반드시 반영시킬 핵심 (재작성 지시 3건)**

### **2.1 버전 전략: V1 = 데이팅 채팅, V2 = 소셜링 채팅·문토 DM, V3 = 챌린지·클럽 채팅**

**현재 문서 상태 (충돌 지점):**

- 2.7 단계별 요구사항: `v1.0`에 **소셜링 + 문토 DM + 데이팅 DM 모두** 포함이라고 적혀 있음 → **새 전략(V1=데이팅만)과 정면 충돌.**
- 2.8.3 마이그레이션 단계: Phase 1~5로 되어 있고 Phase 2가 데이팅 — Phase 개념과 새 V1/V2/V3 개념이 따로 논다.
- 2.8.4 클라이언트 전환: `v1.x / v2.0 / v3.0`을 **"Sendbird only → 자체+이전보기 → 자체 only"라는 앱 버전 의미로 이미 사용 중.** 여기에 새 전략의 V1/V2/V3(기능 범위 단계)가 겹치면 **같은 'V2'가 두 가지를 뜻하게 됨.** 용어 충돌을 먼저 정리해야 한다(예: 기능 단계는 `Release 1/2/3`, 앱 호환 버전은 `app v1.x/v2.x`로 명명 분리).

**요구할 재구성:**

- V1/V2/V3를 **2장 상위에 하나의 표로** 정의하고, 7장 각 기능에 `[V1]/[V2]/[V3]` 태그를 달아 어느 버전 산출물인지 추적되게 할 것.
- V1은 **데이팅 채팅만**이며, 클라이언트가 둘이라는 점을 분명히: **Closer(React Native, 신규·Sendbird 미적용)** + **문토앱 임베드 데이팅(Flutter, 기존 Sendbird 방 이관 필요).** 즉 V1에서 이미 `@munto/chat-sdk`와 `munto_chat_sdk` **둘 다** + 데이팅 DM 마이그레이션까지 나와야 한다(2.8.2의 임베드 데이팅 Export 이관이 V1 범위인지 명확화).
- **Sendbird를 단계별로 무엇을 대체하는지** 표로 명시. 예시 골격(값은 개발자가 채움):

| 단계 | 신규 구축 채팅 범위 | 이때 Sendbird가 담당하는 것 | Sendbird에서 떼어내는 것 |
| --- | --- | --- | --- |
| **V1** | 데이팅 채팅 — 데이팅 DM (Closer + 임베드 데이팅) | 소셜링·문토 DM·챌린지·클럽 전부 | 데이팅 DM |
| **V2** | 소셜링 채팅 + 문토 DM | 챌린지·클럽 채팅 | 소셜링·문토 DM |
| **V3** | 챌린지·클럽 채팅 (인증·인증률, 피드·채널·투표 등 특수 기능) | 없음 (계약 해지) | 챌린지·클럽 (전환 완료) |

→ 핵심은 "각 버전 종료 시점에 Sendbird 비용이 얼마나 줄어드는가"가 보이게 하는 것. 지금 2.8은 Phase는 있는데 **버전 경계와 Sendbird 잔존 범위가 매핑되지 않는다.**

**1.2 Product Scope에 넣을 문구 (개발자 가이드):**

이 단계 전략은 문서 맨 앞인 **1.2 Product Scope**에서 먼저 선언해야 한다. 현재 1.2는 Sendbird 의존·비용·커스터마이징 제약만 짧게 언급하고, **Time-to-Market 전략과 단계적 제거 방침이 빠져 있다.** 아래 ① Sendbird의 구체적 문제, ② Time-to-Market 전략, ③ 단계적 제거(V1→V2→V3)를 명시할 것. 다음 문구를 그대로 또는 다듬어 1.2에 넣으면 된다:

> 문토와 데이팅 서비스는 현재 채팅 기능을 외부 솔루션 Sendbird에 의존하고 있다. Sendbird는 MAU(월간 활성 사용자) 기반 과금 구조여서 사용자가 늘수록 비용이 가파르게 증가하고, 메시지 한도·기능 커스터마이징·사용량(limit) 제약으로 문토·데이팅 고유 정책을 자유롭게 반영하기 어렵다. 본 프로젝트는 이러한 비용·제약 문제를 해소하기 위해 자체 채팅 서버로 전환하되, **Time-to-Market을 최우선으로 단계적으로 진행**한다. 전 기능을 한 번에 자체 구축하지 않고, 가장 독립적이고 빠르게 전환 가능한 **데이팅 채팅(V1)을 먼저 출시**해 Sendbird 의존을 즉시 일부 제거한 뒤, **소셜링 채팅·문토 DM(V2)** → **챌린지·클럽 채팅(V3)** 순으로 확대한다. 각 단계가 완료될 때마다 해당 영역의 Sendbird 사용을 중단해 비용을 단계적으로 절감하고, 최종 단계에서 Sendbird 계약을 완전히 해지한다.
> 

→ 이렇게 1.2에 전략을 박아 두면, 2장(버전 범위표)·2.7(단계별 요구사항)·2.8(마이그레이션)이 모두 이 선언을 구체화하는 형태로 자연스럽게 정렬된다.

### **2.2 Offline 지원을 V1/V2/V3로 단계화 + 버전별 Overall Diagram 3개**

**현재 문서 상태:**

- 풀 Offline 푸시 파이프라인(Redis Streams → Push Worker → 노티 서버 → FCM/APNs, 3단계 Active/Idle/Offline 상태 모델)이 **v1.0에 통째로 박혀 있음**(2.3, 6.1.3, 7.4.3, 7.5). "Offline 초기 최소화" 전략이 전혀 반영 안 됨.
- Overall diagram은 2.1(Context), 2.2.1(SDK 내부), 2.2.2(서버 내부) **단일 버전 기준 하나씩**뿐. 버전별 구분 없음.

**요구할 재구성:**

- **Offline 지원 범위를 버전별로 표로 먼저 정의.** "Offline 지원"이 두 층위(① Offline 수신자에 대한 시스템 푸시 발송, ② 클라이언트 로컬 캐시/전송 큐)임을 구분해서, 각 버전이 어디까지 하는지 적어야 함.

**아래 표는 어디까지나 예시(초안)다. 실제 버전별 범위는 결정해서 채워야 한다:**

| 버전 | Offline 푸시(FCM/APNs) | 클라이언트 Offline 동작 | 비고 |
| --- | --- | --- | --- |
| **V1** | 최소화 — (예) 노티 서버 직접 호출 등 단순 경로, Redis Streams/Push Worker 생략 가능 여부 검토 | FAILED 메시지 로컬 저장 + 수동 재전송만 | 데이팅 DM 규모(동접 수십~수백)에 맞춰 과설계 배제 |
| **V2** | 3단계 상태 모델 + 비동기 큐 도입 | 미수신 메시지 동기화 등 확대 | 소셜링·문토 DM 전환으로 그룹·발송량 증가 대응 |
| **V3** | 고도화(중복/유실 보장 강화, DLQ 등) | 풀 Offline 캐시 | 챌린지·클럽 채팅 단계 |

> **"Offline 최소화"의 의미 (확정).** **V1에서도 오프라인 푸시는 필수다.** 앱이 꺼져 있거나 백그라운드인 상태에서 상대가 메시지를 보내면 오프라인 푸시로 알려야 하므로, 오프라인 푸시 기능 자체는 **V1부터 반드시 포함**한다. 따라서 "최소화"는 **푸시 기능을 빼는 것이 아니라, 푸시를 보내는 인프라(Redis Streams + Push Worker 비동기 큐, 3단계 Active/Idle/Offline 상태 모델)를 V1에서는 생략하고 노티 서버 동기 호출 같은 단순 경로로 처리한다**는 뜻이다. 비동기 큐·상태 모델 등 고도화 인프라는 발송량이 늘어나는 V2 이후 도입한다.
> 
- **Overall Diagram을 V1/V2/V3 3개로 분리해서 그릴 것.** 각 다이어그램이 담아야 하는 것:
    - **V1 다이어그램**: 클라이언트 = Closer(RN) + 임베드 데이팅(Flutter) 2종만. V1 수준의 (최소화된) Offline 경로 표시.
    - **V2 다이어그램**: **V1 클라이언트 + V2 클라이언트가 동시에 현장에 존재**. 둘의 Offline 지원 수준이 다르므로, 같은 채팅 서버가 **서로 다른 Offline 능력의 클라이언트를 동시에 상대**하는 관계를 표현(버전 협상/기능 플래그/하위호환 경로).
    - **V3 다이어그램**: V1 + V2 + V3 클라이언트 3종 공존. 각각의 Offline 지원 차이와 서버의 분기 처리를 모두 표현.
    - 이게 중요한 이유: V1 앱을 한 번 배포하면 사용자가 업데이트 안 하고 계속 쓴다. **V2/V3 서버는 구버전 클라이언트의 (낮은) Offline 능력을 계속 떠받쳐야 한다.** 단일 다이어그램으로는 이 하위호환 부담이 안 보인다.

### **2.3 알림(푸시) 아키텍처 — 전송 큐와 배포 플랫폼을 모두 비교 후 선택**

알림 아키텍처는 **서로 다른 두 축**의 결정이 필요한데, v4는 둘 다 비교 없이 단정해 두었다.

- **축 A. 전송/큐 방식**: Chat Server → (큐?) → 노티 서버 → FCM/APNs 사이에 큐를 둘지, 둔다면 무엇으로 둘지 (큐 없음 / Redis Streams / SQS / Kafka)
- **축 B. 노티 서버 배포 플랫폼**: 노티 서버를 어디서 돌릴지 (serverless / ECS / K8S)

**현재 문서 상태:**

- 축 A: 6.1.4에서 **Redis Streams vs Kafka만** 비교하고 Redis Streams로 단정. "큐가 정말 필요한가(직접 호출)"와 **SQS** 옵션, 규모별 전환 경로, Redis failover 시 이벤트 유실 한계는 빠져 있다.
- 축 B: `closer-notification`을 **"데이팅 AWS ECS 클러스터에 별도 ECS 서비스로 신규 배포"**로 **선택 과정 없이 ECS 단정**(1.4 용어, 6.1.3). 컴퓨팅 플랫폼(serverless/ECS/K8S) 비교가 없다.

**요구할 재구성 (축 A — 전송/큐 방식):** 아래를 규모별로 비교하고, V1/V2/V3 전환 경로를 적게 할 것.

| 방식 | 지연 | 내구성 | 확장성 | 운영/비용 | 적합 |
| --- | --- | --- | --- | --- | --- |
| **큐 없음(직접 비동기 호출)** | 가장 낮음 | 없음(프로세스 죽으면 유실) | 제한적 | 가장 단순 | **V1 저트래픽** — 채팅은 소켓이 1차, 푸시는 보조라 허용 가능 |
| **Redis Streams (현재 안)** | ~10ms | **약함 — Primary failover 시 미복제분 유실, MAXLEN 트리밍** | 좋음 | Redis 재사용(추가 인프라 0) | 중간 규모의 실용적 중간책 |
| **SQS (+DLQ)** | 수십 ms | 강함(디스크 기반, at-least-once, DLQ 기본) | 무한 | 관리형, 운영 부담 낮음 | **고트래픽·푸시 무손실 요구** |
| **Kafka** | ~수백 ms | 매우 강함 | 매우 높음 | 무겁다 | 초대용량·replay/순서까지 필요할 때만(현 규모 과잉) |
- **내 잠정 판단(축 A)**: **V1은 큐 없이 노티 서버 비동기 직접 호출로 충분**해 보인다(Time-to-Market + Offline 최소화). 발송량이 커지는 V2 이후 Redis Streams 도입, **"푸시 절대 무손실"이 요구되면 Redis Streams 대신 SQS(+DLQ)**를 고려. Kafka는 현 규모 과잉(앞 리뷰에서도 정리됨). Redis Streams를 쓰더라도 **failover 유실 한계**(v3 리뷰 Comment 2)를 문서에 명시하고 수용 여부를 정할 것.

**요구할 재구성 (축 B — 배포 플랫폼):** 아래 항목으로 비교표를 넣고 선택 근거를 적게 할 것. 내가 판단할 때의 대략적 결론도 같이 적어 둔다(개발자에겐 "비교해서 근거 쓰라"고만 전달):

| 항목 | Serverless (Lambda + SNS/FCM) | ECS Fargate (현재 안) | K8S (EKS) |
| --- | --- | --- | --- |
| **Latency** | Cold start 시 수백ms~초 지연 위험(채팅 푸시 3초 SLA에 불리). Provisioned Concurrency로 완화 가능하나 비용↑ | 상시 기동 → 안정적 저지연. 채팅 푸시 즉시성에 적합 | 상시 기동 → 저지연. ECS와 동등 |
| **확장성** | 트래픽에 자동·무한에 가까운 스케일. 버스트에 강함 | Auto Scaling으로 가능하나 스케일 반응이 분 단위 | HPA/Cluster Autoscaler로 유연하나 운영 복잡 |
| **가용성** | 관리형, AZ 분산 자동. 운영 부담 최소 | Multi-AZ 서비스로 확보. 검증된 패턴 | Multi-AZ 가능하나 컨트롤플레인·노드 관리 필요 |
| **비용** | 저트래픽 시 매우 저렴(사용량 과금), 고정 트래픽엔 비효율 | 상시 기동분 고정비. 현재 규모에 합리적 | 클러스터 유지비(컨트롤플레인+노드) 고정, 소규모엔 과함 |
| **운영 복잡도** | 가장 낮음 | 낮음(기존 데이팅 ECS 자산 재사용) | 가장 높음 |
- **비교는 실제 트래픽 데이터에 근거할 것** (전화로 전달한 내용): 현재 채팅량(일/시간당 메시지 수), **시간대별로 채팅이 몰리는지(피크/버스트 패턴)**, 그리고 이로부터 나오는 **채팅 Push 발송 요구 건수**를 측정해 비교 기준으로 삼아야 한다. 이 트래픽 형태가 플랫폼 선택을 좌우한다 — 평소 발송량은 낮고 특정 시간대에만 몰리는 **버스트형**이면 serverless의 자동 스케일·사용량 과금이 유리해질 수 있고, 발송이 하루 종일 비교적 **고르게 상시** 발생하면 ECS 상시 기동이 더 경제적이다. '느낌'이 아니라 **실측 건수**로 Latency·확장성·비용 축을 판단하게 할 것.
- **내 잠정 판단**: V1처럼 **발송량이 작고 즉시성이 중요한** 채팅 푸시에는 **K8S는 과투자**, serverless는 cold start로 3초 SLA가 흔들릴 수 있어 보조/저빈도 경로에 적합. **기존 데이팅 ECS 자산을 재사용하는 ECS Fargate가 현재 규모엔 가장 합리적**으로 보인다. 단, 이걸 **"비교 없이 ECS"로 적으면 안 되고**, 위 트래픽 실측 + 표로 트레이드오프를 명시한 뒤 ECS를 고른 근거(즉시성 SLA + 기존 클러스터 재사용 + 운영 단순성)를 쓰게 할 것.
- 두 축(전송/큐 + 배포 플랫폼)은 별개 결정이므로 **각각 비교·선택**하고, V1에서는 둘 다 가장 단순한 쪽(큐 없음 + ECS 재사용)으로 시작해 규모에 따라 올리는 전환 경로를 적을 것. 2.2의 Offline 최소화와 같은 맥락.

### **2.4 클라이언트 로컬 저장소(DB) — 고민이 부족하다**

**현재 문서 상태:**

- 로컬 DB 핵심이 **외부 문서로 빠져 있음.** 4.6.1은 상세를 `database/client-local-db.md`로 미루고, 본문엔 논리 원칙 몇 줄만 있다(그 외부 문서는 리뷰 대상에 없어 본문만으로는 검증 불가).
- **DB 기술이 미확정.** 4.6.1에 "Flutter → Drift/Isar, React Native(Closer) → SQLite/MMKV"로 **둘씩 나열만** 되어 있고 선택이 없다. 게다가 **MMKV는 key-value 저장소라 관계형(SQLite)과 성격이 다른데** `/`로 묶어 둬서, 무엇을 메시지 영속 저장에 쓰고 무엇을 설정/캐시에 쓸지 구분이 안 된다.
- 2.2.1 다이어그램에 로컬 테이블 4개(`pending_messages`, `cached_messages`, `cached_rooms`, `cached_thumbnails`)만 그려져 있고, **버전별 차이도, 스키마 진화 전략도 없다.**

**요구할 재구성:**

핵심은 두 가지다 — **① 클라이언트 DB 기술을 RN·Flutter 각각 확정**, **② 코드 공유는 불가하므로 각 플랫폼에서 따로 구현하되 논리 스키마만 공유.**

- **DB 기술 확정**(`/` 나열 금지, 각 플랫폼 하나로 결정 + 근거 한 줄). **개발자가 후보를 직접 비교해서 선택**하게 할 것 — 비교 기준: 대량 메시지 조회/정렬/페이지네이션, 반응형 갱신, 스키마 마이그레이션 지원, 번들 크기, 암호화, 유지보수/커뮤니티 활성도. 단순 설정/플래그는 key-value 저장소로 역할 분리.
- **플랫폼별 별도 구현이 전제.** 선택한 로컬 DB 라이브러리가 **그 자체로 각 플랫폼의 ORM** 역할을 하므로 별도 ORM을 얹을 필요는 없다. 단 RN·Flutter는 언어·엔진·쿼리 API가 모두 달라(JS/TS vs Dart) **코드와 ORM을 공유할 수 없다.** 모델·쿼리·마이그레이션은 RN·Flutter에서 각각 구현한다.
- **공유하는 것은 "논리 스키마" 1벌뿐.** 테이블/필드/타입/인덱스를 한 곳(SOT)에 정의하고, 양쪽이 동일하게 구현하도록 **필드 매핑표 + parity(정합) 테스트**로 어긋남을 막을 것.
- **로컬 DB 손상(corruption)을 전제로 복구 전략을 설계.** 로컬 DB는 파일 기반이라(SQLite든 자체 엔진이든 공통) 이론상 손상될 수 있다. "안 깨지게"가 아니라 **"깨져도 복구"**가 핵심이며, 클라이언트 DB는 캐시이고 서버가 SOT이므로 다음을 명문화할 것: ① 앱 시작/이상 감지 시 무결성 체크, ② 손상 시 **로컬 DB 폐기 후 서버에서 재동기화**(메시지는 서버에 영구 저장돼 손실 없음), ③ 단 **아직 서버로 못 보낸 PENDING/FAILED 전송 큐는 별도 보호**(분리 저장·우선 플러시)해 유실 방지. 이 복구 전략은 V1부터 포함.
- **V1/V2/V3별로 로컬에 무엇을 저장하는지 표로 구분.** 예시 골격(값은 결정 필요):

| 버전 | 로컬 저장 대상(예시) | 비고 |
| --- | --- | --- |
| **V1** | 진행 중 DM 메시지 캐시, 전송 실패(FAILED/PENDING) 큐, 채팅방 목록, `lastMessageId`, 썸네일 | 데이팅 DM 1:1 중심 |
| **V2** | + 그룹 채팅방/참여자, 읽지 않은 수, 공지 캐시 | 소셜링 채팅·문토 DM |
| **V3** | + 검색 인덱스, 첨부 캐시 eviction, 멀티기기 동기화 상태 등 | 챌린지·클럽 채팅(특수 기능) |
- **V3까지 견디는 forward-compatible 스키마를 V1 단계에서 미리 설계.** 핵심은 **로컬 DB 스키마 버전 컬럼 + 마이그레이션 정책**이다. V1 앱이 현장에 남은 채로 V2/V3 스키마가 나오므로(2.2의 하위호환과 동일 문제), **컬럼 추가는 nullable·기본값으로 흡수, 파괴적 변경 금지, 앱 시작 시 자동 마이그레이션** 같은 규칙을 명문화해야 한다. 지금처럼 V1만 보고 테이블을 짜면 V2/V3에서 로컬 스키마 깨짐·재설치 유발 위험.
- 상세 스키마 문서(`client-local-db.md`)를 **SRS와 함께 리뷰 가능한 형태로** 제출하도록 요청(테이블 정의·인덱스·삭제/보관 정책·스키마 버전). 본문 4.6.1에도 최소한 테이블 목록과 스키마 버전 전략은 직접 적어 둘 것.
- **V5 문서에는 서버 ERD와 클라이언트 DB ERD를 모두 포함(또는 명확히 링크)할 것.** ① 서버 ERD(PostgreSQL)는 V1/V2/V3 단계별 추가 테이블을 표기, ② 클라이언트 ERD는 **RN·Flutter 공통 "논리 ERD" 1벌**을 SOT로 그리고 플랫폼별(각자 선택한 DB 기준) 타입/구현 매핑을 주석으로 달 것(ORM·엔진이 달라 코드는 공유 못 하므로 ERD는 논리 1벌로 통일). 서버 ERD ↔ 클라이언트 ERD의 필드 타입·의미 정합도 ERD 차원에서 한 번 더 맞출 것.

### **2.5 백오피스(운영 관리 도구) — V1 범위·산출물·일정이 비어 있다**

**현재 문서 상태:**

- 7.8에 백오피스 **기능 요구사항은 8개 하위로 상세히** 있다(조회·모니터링·통계·차단·금칙어·메시지 삭제·검색). "기존 문토 백오피스 권한 체계(`UserGrade.ADMIN`)를 따름"이라 기존 백오피스에 얹는 뉘앙스다.
- 그런데 **산출물(1.7)에 백오피스가 없다** — Mobile SDK·TypeScript SDK·채팅 서버 백엔드만 있다. 백오피스 화면(프론트)을 신규로 만드는지, 채팅 서버는 API만 제공하고 기존 문토 백오피스가 소비하는지 **정의가 없다.** 별도 SRS로 분리한다는 전략도 없다.
- 2.7은 "v1.0에 백오피스 등 포함"이라 한 줄 적었지만, **2.8.3 Phase 표·일정·리소스에는 백오피스 작업·담당·공수가 빠져 있다.** V1과 동시에 진행한다는 명시도 없다.

**요구할 정리:**

- **백오피스 산출물·범위 확정.** (a) 신규 백오피스 화면 개발 / (b) 채팅 서버는 백오피스 API만 제공하고 기존 문토 백오피스 프론트가 소비 / (c) 별도 SRS로 분리 — 셋 중 무엇인지 명시. 7.8에 기능은 상세한데 산출물에 없으니 이 정합부터 맞춰야 한다. (b)면 프론트 공수를 크게 줄일 수 있어 일정에 직접 영향.
- **V1 개통 필수 범위를 분리.** 데이팅은 어뷰징·신고 리스크가 커서 **신고 처리·사용자 차단·채팅 모니터링 같은 운영 필수 최소 도구는 V1 Open과 동시에 있어야 한다**(없으면 사고 시 대응 불가). 반면 통계·금칙어 관리 UI·메시지 검색 등은 V2+로 미뤄도 되는지 버전 태깅. "백오피스 전체를 V1"인지 "운영 필수 최소만 V1"인지 결정.
- **일정·리소스에 반영.** V1이 1.5개월(3MM)로 짧은데 백오피스 공수가 Phase 표에 안 잡혀 있다. 채팅 서버·SDK·마이그레이션과 **동시 진행이 전제이므로 담당·공수를 일정에 명시**하고, 공수가 안 맞으면 V1 백오피스 범위를 운영 필수 최소로 줄일 것.
- **백오피스도 V1/V2/V3 단계별로 진행.** 백오피스는 채팅 본체와 별개로 한 번에 다 만드는 것이 아니라, **각 버전에서 자체 채팅으로 전환된 채팅 유형만 백오피스도 함께 지원**한다(아직 Sendbird에 남아 있는 채팅은 기존 Sendbird 관리 도구로 운영). 따라서 7.8을 통째로 V1에 넣지 말고, **7.8 각 기능에도** `[V1]/[V2]/[V3]` **태그**를 달아 채팅 범위와 같은 속도로 열리게 해야 한다. 단계별 골격(예시):

| 버전 | 백오피스 대상 채팅 | 단계별 백오피스 범위(예시) |
| --- | --- | --- |
| **V1** | 데이팅 채팅(DM) | 운영 필수 최소 — 신고 처리, 사용자 차단, 채팅 모니터링/조회 |
| **V2** | 소셜링 채팅·문토 DM | + 그룹 채팅 모니터링, 금칙어 관리, 채팅방/사용자 통계 |
| **V3** | 챌린지·클럽 채팅 | + 특수 기능 관리(인증·인증률, 투표 등), 메시지 검색·통계 고도화 |

→ 이렇게 하면 "백오피스 전체 = V1 필수"라는 과부담을 피하면서, **각 단계 Open에 꼭 필요한 운영 도구만 그때그때 확보**할 수 있다. 산출물·일정·버전 태깅이 모두 이 단계 원칙으로 정렬되어야 한다.

## **3. 그 외 발견 (정합성·충돌 — 재작성 시 같이 정리)**

- **[중] 2.7이 우선순위 의미를 무력화.** 1.3에서 P1~P3를 정의해 놓고 2.7에서 "모든 요구사항을 P1로 분류"한다고 단언 → 우선순위 체계가 죽는다. V1/V2/V3 도입과 함께 7장 기능을 실제로 P1/P2/P3 + 버전 태그로 나눠야 Top-down이 산다.
- **[중] 동접 수치 스케일이 들쭉날쭉.** 5.1은 현재 250명·데이팅 후 300~350명인데, 5.2·6.1.3은 **10,000 연결** 기준으로 설계(서버 2대, 5,000/대). V1이 데이팅 DM만이면 실제 동접은 수백 수준이다. **10,000 전제 인프라·Offline 큐 설계가 V1엔 과설계** 아닌지 — 버전별로 용량 목표를 분리해야 한다(2.2 Offline 최소화·2.3 플랫폼 선택과 직결).
- **[중] 버전 용어 3중 충돌.** 산출물 `v1.0.0`(1.7.2) / 앱 전환 `v1.x·v2.0·v3.0`(2.8.4) / 새 전략 `V1·V2·V3`. 세 개가 다른 의미인데 표기가 겹친다. 명명 규칙을 문서 앞에서 한 번 정리하지 않으면 7장 태깅이 전부 모호해진다.
- **[하] Heartbeat 수치 불일치.** 2.2.1 다이어그램·5.2 본문은 "Heartbeat 30초", 6.1.3은 "pingInterval 25초 / pingTimeout 20초, 별도 커스텀 ping 없음". 30s vs 25s가 다르고, "커스텀 없음"이라면서 SDK 다이어그램엔 Heartbeat 컴포넌트를 박아둠. 한쪽으로 통일.
- **[하] As-Is/To-Be 혼재.** 6.1.3 "현재 채팅 서버는 두 대로 운영" — 아직 구축 전인 신규 서버를 운영 중인 것처럼 기술. 신규 구축 SRS에서 "현재 운영 중" 표현은 오해를 부른다.
- **[하] 푸시 설정 저장 경로가 문토 편향.** 7.5.2는 "문토 백엔드에 설정 변경 요청"만 기술. Closer(데이팅) 사용자의 알림 설정은 dating-backend 경로인지 누락 — V1이 데이팅 우선이므로 V1에서 먼저 필요한 경로다.
- **[하] 메시지 송수신 응답시간 SLA 공백.** 5.3 Response Time이 "None(문토와 동일)". 푸시 3초 SLA는 있는데 정작 채팅 송수신 자체의 목표 응답시간이 비어 있다.

## **4. 재작성 요청 요지 (개발자 가이드 요약)**

1. **1.2 Product Scope에 Time-to-Market 전략과 Sendbird 단계적 제거 방침을 선언**할 것(문구는 2.1의 가이드 참조). Sendbird 문제(MAU 과금·비용 증가·기능/limit 제약)와 "V1→V2→V3로 빠르게 떼어낸다"는 큰 그림을 문서 맨 앞에서 명시.
2. 2장에 **V1/V2/V3 범위표 + Sendbird 단계별 대체표**를 세우고, 7장 기능에 버전 태그를 달 것. V1은 데이팅 채팅만(Closer RN + 임베드 데이팅 Flutter, 마이그레이션 포함).
3. **Offline 지원을 버전별로 단계화**하고, **Overall Diagram을 V1/V2/V3 3개**로 분리. V2엔 V1·V2 클라이언트, V3엔 V1·V2·V3 클라이언트를 모두 그려 Offline 능력 차이와 하위호환 관계를 표현.
4. **알림 배포 플랫폼(serverless/ECS/K8S)을 Latency·확장성·가용성·비용으로 비교**한 뒤 선택 근거를 명시(현재는 비교 없이 ECS 단정). 비교 시 현재 채팅량·시간대별 집중도·Push 발송 건수 실측을 감안.
5. **클라이언트 로컬 DB를 RN/Flutter 각각 확정**하고, V1/V2/V3별 저장 데이터를 구분하며, **V3까지 견디는 forward-compatible 스키마(스키마 버전 + 마이그레이션 정책)를 V1에서 미리 설계**할 것.
6. **백오피스 산출물·범위를 확정**(신규 화면 / API만 제공+기존 백오피스 재사용 / 별도 SRS 중 택1)하고, **V1 Open에 필요한 운영 필수 최소 도구(신고·차단·모니터링)를 V1으로 분리**해 일정·리소스에 공수를 명시할 것.
7. 위가 정리되면 동접 용량 목표·우선순위(P1~P3)·버전 용어도 버전별로 다시 정합 맞출 것.

- API/소켓 이벤트 스펙(swagger·socket-events) 및 서버 ERD 상세 정합 — 전략·구조 재작성 후 V5에서 확인 예정.
- 로컬 DB 상세 스키마 문서(`client-local-db.md`)는 리뷰 대상에 없어 본문 4.6.1 기준으로만 평가함(2.4 참조). 상세 문서 제출 시 V5에서 검증 예정.
- 백오피스 기능 명세(7.8) 세부 — 산출물·V1 범위·일정 이슈는 2.5에서 다뤘고, 개별 기능 스펙 검증은 범위·일정 확정 후 V5에서.
