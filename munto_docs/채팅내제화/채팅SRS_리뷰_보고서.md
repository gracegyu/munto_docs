# 채팅 서비스 SRS 리뷰 보고서

## 1. 전체 평가

SRS 문서는 전반적으로 매우 상세하고 체계적으로 작성되었습니다. Jira 이슈의 주요 요구사항을 대부분 반영하고 있으며, 기능 요구사항, 비기능 요구사항, 기술 스택 등이 명확하게 정의되어 있습니다.

**전체 점수: 85/100**

## 2. Jira 이슈 요구사항 반영 여부

### 2.1 ✅ 잘 반영된 부분

1. **기술 스택**: NestJS + Socket.IO, PostgreSQL, MSA 구조 모두 반영
2. **비용 절감 목표**: SendBird 대체 목적이 명확히 기술됨
3. **기능 범위**: 1:1 채팅, 그룹 채팅, 푸시 알림 등 핵심 기능 모두 포함
4. **통합 JWT 인증**: 통합 JWT Payload 기반 인증 체계 상세히 정의
5. **하위 호환성**: 기존 SendBird 채팅방과의 병행 운영 전략 명시

### 2.2 ⚠️ 불일치 또는 누락된 부분

#### 🔴 **중요: Redis 사용 여부 불일치**

**Jira 이슈 요구사항:**
- "PostgreSQL을 사용한 메시지 영구 저장 및 Redis를 사용한 캐싱 및 Pub/Sub 메시지 브로드캐스트 활용"
- "확장 가능한 아키텍처: 로드밸런서를 사용한 수평 확장 및 Redis Pub/Sub 또는 Kafka를 사용한 서버 간 메시지 동기화"

**SRS 현황:**
- 6.1.2 부하 분산 및 확장성: "Redis 등 외부 캐시 서버는 사용하지 않음 (데이터베이스 중심 아키텍처)"
- 3.1.2 소프트웨어 환경: Redis 언급 없음

**문제점:**
- Jira에서는 Redis Pub/Sub을 사용하여 다중 서버 간 메시지 브로드캐스트를 한다고 명시
- SRS에서는 Redis를 사용하지 않는다고 명시
- 현재 두 대의 서버로 운영한다고 했는데, 서버 간 메시지 동기화 방법이 불명확

**권장사항:**
1. Jira 요구사항에 맞춰 Redis Pub/Sub 사용 여부를 명확히 결정
2. 다중 서버 환경에서 메시지 브로드캐스트 방법을 구체적으로 정의
3. Redis 사용 시: Pub/Sub을 통한 서버 간 메시지 동기화 플로우 추가
4. Redis 미사용 시: Jira 이슈 업데이트 필요

#### 🟡 **기술 스택 상세 불일치**

**Jira 이슈:**
- 인프라: AWS ECS, 로드밸런서 (WebSocket 지원)
- 모니터링: CloudWatch, Prometheus + Grafana

**SRS:**
- 인프라: AWS ECS, 로드밸런서 언급 있음 (3.1.1)
- 모니터링: CloudWatch만 명시 (3.1.2), Prometheus + Grafana 언급 없음

**권장사항:**
- Prometheus + Grafana 사용 여부 명확히 결정 및 반영

## 3. 기능 요구사항 완전성

### 3.1 ✅ 잘 정의된 기능

1. **인증 및 연결 관리**: JWT 기반 인증, WebSocket 연결 관리, 재연결 메커니즘 모두 상세히 정의
2. **채팅방 관리**: 소셜링, DM, 데이팅 DM 채팅방 생성 및 관리 상세히 정의
3. **메시지 전송/수신**: 텍스트, 이미지 메시지 전송 프로세스 명확
4. **읽음 처리**: 읽음 상태 관리 및 실시간 업데이트 정의
5. **푸시 알림**: Kafka 기반 푸시 알림 발송 프로세스 정의
6. **안전 기능**: 금칙어 필터링, 부적절한 이미지 감지, 신고 기능 모두 포함

### 3.2 ⚠️ 보완이 필요한 부분

#### 🟡 **데이팅 DM 채팅방 활성화 프로세스 불명확**

**현황:**
- 7.2.1.3에서 `roomStatus: CREATED` → `ACTIVATED` 전환 언급
- 활성화 시점에 `activatedAt` 설정 언급
- 하지만 활성화 트리거(재화 차감)가 언제, 어디서 발생하는지 불명확

**권장사항:**
- 데이팅 백엔드에서 재화 차감 후 채팅 서버에 활성화 요청하는 API 플로우 추가
- 또는 채팅 서버에서 데이팅 백엔드 API를 호출하여 재화 차감 확인하는 플로우 명시

#### 🟡 **채팅방 만료 처리 자동화 미명시**

**현황:**
- 소셜링 채팅방: 모임 종료 후 30일 경과 시 자동 종료 (6.5.1)
- 데이팅 DM: `chatExpiresAt` 기반 만료 (7.2.1.3)
- 하지만 만료 처리 스케줄러나 배치 작업 언급 없음

**권장사항:**
- 채팅방 만료 처리 스케줄러 또는 배치 작업 명시
- 만료된 채팅방 처리 정책 (삭제 vs 보관) 명확화

#### 🟡 **이미지 업로드 프로세스 상세화 필요**

**현황:**
- 7.3.2에서 S3 Presigned URL 요청 및 업로드 프로세스 언급
- 하지만 Presigned URL 발급 API 엔드포인트 및 권한 검증 프로세스 불명확

**권장사항:**
- S3 Presigned URL 발급 API 명세 추가
- 이미지 업로드 실패 시 처리 프로세스 추가

**현황:**

**권장사항:**
- Jira 이슈의 비용 분석(현재 규모: 동시 접속 250명, 데이팅 추가 후 300~350명)을 기반으로 성능 요구사항 정의

#### 🟡 **확장성 전략 구체화 필요**

**현황:**

**권장사항:**
- 해시 알고리즘 명시 (예: consistent hashing)
- 서버 추가/제거 시 리밸런싱 전략 명시
- 채팅방 마이그레이션 전략 (필요 시)

## 5. 아키텍처 및 설계

### 5.1 ✅ 잘 설계된 부분

1. **MSA 구조**: 별도 서비스로 분리하여 독립 배포 가능
2. **통합 JWT**: 문토와 데이팅 서비스 통합 인증 체계
3. **하위 호환성**: 기존 SendBird 채팅방과 병행 운영 전략

### 5.2 ⚠️ 보완이 필요한 부분

#### 🔴 **다중 서버 환경 메시지 동기화 불명확**

**문제점:**
- 현재 두 대의 서버로 운영
- 채팅방 번호 해시 기반으로 서버 분배
- 하지만 같은 채팅방 참여자가 다른 서버에 연결된 경우 메시지 브로드캐스트 방법 불명확

**시나리오:**
```
서버 A: 사용자 1, 2 (채팅방 1)
서버 B: 사용자 3 (채팅방 1)
→ 사용자 1이 메시지 전송 시 사용자 3에게 어떻게 전달?
```

**권장사항:**
1. **Redis Pub/Sub 사용 시:**
   - 서버 A에서 메시지 수신 → DB 저장 → Redis Pub/Sub 발행
   - 서버 B가 Redis 구독하여 메시지 수신 → Socket.IO로 브로드캐스트
   - 플로우 다이어그램 추가

2. **Redis 미사용 시:**
   - 데이터베이스 폴링 방식 (비권장)
   - 또는 모든 서버가 모든 채팅방을 구독 (확장성 문제)
   - Jira 요구사항과 불일치

#### 🟡 **데이터베이스 스키마 미정의**

**현황:**
- 6.4.1에서 "PostgreSQL(RDB) 사용"만 언급
- 구체적인 테이블 구조, 관계, 인덱스 전략 미정의

**권장사항:**
- ERD 또는 주요 테이블 스키마 정의 추가
- 인덱스 전략 (채팅방 ID, 사용자 ID, 메시지 시간 등)
- 데이터 파티셔닝 전략 (필요 시)

## 8. 핵심 문제 해결 여부

### 8.1 ✅ 해결되는 문제

1. **비용 절감**: SendBird 대체로 월 4,500~11,000만원 비용 절감 (Jira 코멘트 기준)

**문제:**
- Jira에서 Redis Pub/Sub 사용 명시
- SRS에서 Redis 미사용 명시
- 이 불일치로 인해 다중 서버 환경에서 메시지 동기화가 제대로 작동하지 않을 수 있음

**우선순위: 높음**

## 10. 결론

SRS 문서는 전반적으로 매우 잘 작성되었으며, Jira 이슈의 핵심 요구사항을 대부분 반영하고 있습니다. 특히 기능 요구사항의 상세도와 비기능 요구사항의 포괄성이 우수합니다.
