# Consistent Hashing 설명

## 1. Consistent Hashing이란?

**Consistent Hashing**은 서버를 추가하거나 제거할 때 최소한의 데이터만 재분배하는 해시 알고리즘입니다.

## 2. 왜 필요한가?

### 2.1 일반적인 해시 방식의 문제점

**Modulo 연산 (나머지 연산) 예시:**
```
서버 2대: 서버 A, 서버 B
채팅방 1 → hash(roomId) % 2 = 0 → 서버 A
채팅방 2 → hash(roomId) % 2 = 1 → 서버 B
채팅방 3 → hash(roomId) % 2 = 0 → 서버 A
```

**서버 추가 시 문제:**
```
서버 3대로 증가: 서버 A, 서버 B, 서버 C
채팅방 1 → hash(roomId) % 3 = ? → 결과가 바뀔 수 있음!
채팅방 2 → hash(roomId) % 3 = ? → 결과가 바뀔 수 있음!
채팅방 3 → hash(roomId) % 3 = ? → 결과가 바뀔 수 있음!

→ 대부분의 채팅방이 다른 서버로 재매핑됨!
→ 기존 연결이 끊기고 재연결 필요
→ 서비스 중단 발생 가능
```

### 2.2 Consistent Hashing의 해결책

서버를 추가/제거해도 **대부분의 채팅방은 그대로 유지**되고, **변경이 필요한 채팅방만 최소한으로 재매핑**됩니다.

## 3. 동작 원리

### 3.1 기본 개념

1. **해시 링(Hash Ring)**: 0부터 2^32-1까지의 숫자를 원형으로 배치
2. **서버 배치**: 각 서버를 해시 링 위의 특정 위치에 배치
3. **채팅방 배치**: 각 채팅방을 해시 링 위의 특정 위치에 배치
4. **매핑 규칙**: 채팅방은 시계 방향으로 가장 가까운 서버에 할당

### 3.2 시각적 예시

```
해시 링 (0 ~ 2^32-1):

     서버 A (100)
        |
        |
서버 C (50)     서버 B (200)
        |
        |
    채팅방 1 (75) → 서버 C (가장 가까운 서버)
    채팅방 2 (150) → 서버 B (가장 가까운 서버)
    채팅방 3 (250) → 서버 A (가장 가까운 서버, 링을 돌아서)
```

### 3.3 서버 추가 시

**서버 추가 전:**
```
서버 A (100)
서버 B (200)

채팅방 1 (75) → 서버 A
채팅방 2 (150) → 서버 B
채팅방 3 (250) → 서버 A
```

**서버 C (120) 추가 후:**
```
서버 A (100)
서버 C (120) ← 새로 추가
서버 B (200)

채팅방 1 (75) → 서버 A (변경 없음)
채팅방 2 (150) → 서버 B (변경 없음)
채팅방 3 (250) → 서버 A (변경 없음)

→ 서버 C는 120~200 사이의 채팅방만 담당
→ 대부분의 채팅방은 그대로 유지!
```

**재매핑되는 채팅방:**
- 서버 C (120)와 서버 B (200) 사이에 있는 채팅방만 서버 C로 이동
- 전체 채팅방 중 일부만 재매핑 (예: 1/3 정도)

### 3.4 서버 제거 시

**서버 C (120) 제거 후:**
```
서버 A (100)
서버 B (200)

→ 서버 C가 담당하던 채팅방들만 서버 B로 재매핑
→ 나머지 채팅방은 그대로 유지
```

## 4. 채팅 서비스에서의 활용

### 4.1 적용 방법

```javascript
// 의사코드 예시

// 1. 서버 목록
const servers = [
  { id: 'server-a', hash: hashFunction('server-a') }, // 예: 100
  { id: 'server-b', hash: hashFunction('server-b') }, // 예: 200
];

// 2. 채팅방 서버 선택
function getServerForRoom(roomId) {
  const roomHash = hashFunction(roomId); // 예: 150
  
  // 해시 링에서 시계 방향으로 가장 가까운 서버 찾기
  const sortedServers = servers.sort((a, b) => a.hash - b.hash);
  
  for (const server of sortedServers) {
    if (server.hash >= roomHash) {
      return server.id; // 서버 B 반환
    }
  }
  
  // 링을 한 바퀴 돌았으면 첫 번째 서버
  return sortedServers[0].id; // 서버 A 반환
}

// 3. 서버 추가
function addServer(serverId) {
  const serverHash = hashFunction(serverId);
  servers.push({ id: serverId, hash: serverHash });
  
  // 서버 추가로 인해 재매핑이 필요한 채팅방만 식별
  // (구현 시 상세 로직 필요)
}
```

### 4.2 장점

1. **최소 재매핑**: 서버 추가/제거 시 일부 채팅방만 재매핑
2. **부하 분산**: 해시 링 위에 서버를 균등하게 배치하면 부하 분산 가능
3. **확장성**: 서버를 쉽게 추가/제거 가능

### 4.3 단점 및 해결책

**문제: 서버가 적을 때 부하 불균형**
```
서버 A (100)
서버 B (200)

→ 서버 A와 서버 B 사이의 간격이 큼
→ 이 구간의 채팅방이 많으면 부하 불균형
```

**해결책: Virtual Node (가상 노드)**
```
각 서버를 여러 개의 가상 노드로 분할

서버 A → 가상 노드 A1, A2, A3, A4
서버 B → 가상 노드 B1, B2, B3, B4

→ 해시 링 위에 더 많은 점이 생겨서 부하 분산 개선
```

## 5. 채팅 서비스 SRS에 반영할 내용

### 5.1 명시해야 할 사항

1. **해시 알고리즘**: Consistent Hashing 사용
2. **해시 함수**: SHA-256 또는 MD5 등 구체적 함수 명시
3. **Virtual Node**: 사용 여부 및 개수
4. **서버 추가/제거 프로세스**: 
   - 재매핑 대상 채팅방 식별 방법
   - 마이그레이션 프로세스 (연결 유지 vs 재연결)
   - 다운타임 최소화 전략

### 5.2 예시 문구

```
**서버 분배:**

- 해시 알고리즘: Consistent Hashing 사용
- 해시 함수: SHA-256
- Virtual Node: 각 서버당 4개의 가상 노드 사용 (부하 분산 개선)
- 채팅방 번호(`roomId`)를 해시하여 해시 링 위의 위치 결정
- 시계 방향으로 가장 가까운 서버에 할당
- 서버 추가/제거 시 최소한의 채팅방만 재매핑 (약 1/N, N=서버 수)
```

## 6. 다른 해시 방식과 비교

| 방식 | 서버 추가 시 재매핑 비율 | 구현 복잡도 | 부하 분산 |
|------|------------------------|------------|----------|
| **Modulo** | 거의 100% | 낮음 | 보통 |
| **Consistent Hashing** | 약 1/N (N=서버 수) | 중간 | 좋음 (Virtual Node 사용 시) |
| **Rendezvous Hashing** | 약 1/N | 높음 | 매우 좋음 |

**채팅 서비스 권장: Consistent Hashing**
- 구현 복잡도와 성능의 균형이 좋음
- 서버 확장 시 최소한의 재매핑으로 서비스 중단 최소화
