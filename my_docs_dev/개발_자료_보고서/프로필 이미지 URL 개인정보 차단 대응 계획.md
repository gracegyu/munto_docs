# 프로필 이미지 URL 개인정보 차단 대응 계획

## 1. 배경

- 일부 유저 프로필 이미지 URL에 전화번호가 그대로 포함된 사례 확인.
- 이는 개인정보 유출 위험에 해당하며, 보안사고 성격을 가짐.
- 추석 연휴 직전 발견되어 긴급 조치 필요성 제기.

---

## 2. AWS 인프라 현황

### (1) 도메인 / DNS

- **서비스 도메인**: `https://www.munto.kr`

  - 현재 DNS는 **ALB**로 연결: `dualstack.munto-web-prod-lb-1820412473.ap-northeast-2.elb.amazonaws.com`

- **이미지 CDN 도메인**: `https://images.munto.kr/` (CloudFront → S3)

### (2) 스토리지(S3)

- **버킷 ARN**: `arn:aws:s3:::munto-images`
- **프로필 이미지 경로**: `s3://munto-images/production-user/`
- **문제 파일 예시**: `1645594095388-image-010-2882-9403-image_cropper_60B1FDE4-D1D9-40FB-ABE2-2A86BC1F905B-36123-000030737A66A355.jpg`

  - **실제 문제 URL**: [https://www.munto.kr/\_next/image?url=https%3A%2F%2Fimages.munto.kr%2Fproduction-user%2F1645594095388-image-010-2882-9403-image_cropper_60B1FDE4-D1D9-40FB-ABE2-2A86BC1F905B-36123-000030737A66A355.jpg&w=384&q=75](https://www.munto.kr/_next/image?url=https%3A%2F%2Fimages.munto.kr%2Fproduction-user%2F1645594095388-image-010-2882-9403-image_cropper_60B1FDE4-D1D9-40FB-ABE2-2A86BC1F905B-36123-000030737A66A355.jpg&w=384&q=75)

### (3) CDN(CloudFront)

- **오리진**: S3 버킷 `munto-images`
- **역할**: 정적 이미지 원본 서빙(직접 접근 시 원본 노출 가능)

### (4) 애플리케이션 전달(Load Balancer)

- **ALB 이름**: `munto-web-prod-lb` (리전: ap-northeast-2, 서울)
- **리스너**:

  - `HTTP :80` → `HTTPS :443` **리다이렉트**
  - `HTTPS :443` → **대상 그룹 포워드**

- **대상 그룹**: `munto-web-prod-tg` (타깃 타입: **ip**)

  - 등록 타깃 예: `172.31.1.130:3000` (서브넷: **ap-northeast-2a / apne2-az1**)
  - VPC: `vpc-e22cfc89`

### (5) 컴퓨트(ECS)

- **클러스터**: `MUNTO-PROD`
- **서비스(웹 프런트)**: `munto-front-web-prod`

  - **로드 밸런싱**: ALB `munto-web-prod-lb` / **대상 그룹** `munto-web-prod-tg`
  - **컨테이너 포트**: `3000` (Next.js 애플리케이션, `/_next/image` 프록시 포함)
  - **스케줄링 전략**: `REPLICA` / 현재 **1 태스크 실행**(캡처 시점)

### (6) 애플리케이션 이미지 최적화(Next.js)

- **엔드포인트**: `/_next/image?url=<원본URL>&w=<width>&q=<quality>`

  - `url`: 퍼센트 인코딩된 **원본 이미지 URL**(예: `https://images.munto.kr/...`)
  - `w`: 리사이즈 가로 **width** (예: `384`)
  - `q`: JPEG/WebP **quality** (예: `75`)

## 3. 대응 방법 (근본 해결)

- **목적**: 개인정보(전화번호)가 포함된 잘못된 파일명을 제거하고 안전한 URL 구조로 교체.
- **방법**:

  1. `s3://munto-images/production-user/` 경로에 저장된 파일 중 민감정보 포함 객체를 스캔.
  2. 스크립트를 작성하여 해당 파일들을 안전한 규칙(유니크한 Unix Time 기반 문자열(예: `1619433233690`))으로 **Rename**.
  3. DB 내 URL을 일괄 수정하여 새 파일명으로 매핑.
  4. 샘플 파일 소수에 우선 적용 후, 문제 없을 시 전체 확대 적용.

- **특징**:

  - 단일 조치로 **근본 해결** 가능.
  - DB 및 S3 파일명을 동시에 정리하여 개인정보 포함 URL 제거.
  - 점진적 적용으로 서비스 장애 리스크 최소화.

## 4. 추가 고려사항

- **접근 로그 분석**: CloudFront/S3 Access Log 및 ALB 로그를 분석하여 과거 개인정보 포함 URL이 실제 외부 접근/노출되었는지 확인.
- **URL 생성 로직 개선**: 신규 업로드 시 절대 전화번호나 민감정보가 포함되지 않도록 BE 서버에서 강제 처리.
- **이상 탐지**: Request 로그를 정기적으로 스캔하여 개인정보가 포함된 패턴(전화번호 등)이 감지되면 즉시 알림.
- **장기 안정성 확보**: 필요 시 Presigned URL 또는 CloudFront Signed URL 도입.

## 6. 부록 – 원인 분석

1. **개발 초기 보안 인식 부족**: 최초 개발자가 개인정보라는 개념을 고려하지 않고, 이미지 파일명에 전화번호를 포함시킴.
2. **부분적 개선의 한계**: 이후 파일명에 전화번호를 넣지 않는 방식으로 변경되었으나, 이미 저장된 과거 파일들에 대해서는 이름 변경이 이루어지지 않아 여전히 개인정보 노출 위험이 존재.

---

## 7. 근본 대책

1. **보안 개발 가이드 수립**: 개인정보 취급 금지 규칙을 코드 레벨/리뷰 체크리스트에 반영.
2. **로그 기반 검증**: CloudFront, S3, ALB Request 로그를 주기적으로 분석하여 개인정보(전화번호, 주민번호 패턴 등) 포함 여부 확인.
3. **보안 스캐닝 자동화**: 저장소(S3)와 DB에 대해 정기적인 개인정보 스캐닝 배치 작업 도입.
4. **기존 데이터 정리**: 과거 이슈로 남아있는 파일명/URL을 최대한 추출해 Rename 및 DB 수정.
5. **사전 차단 로직 강화**: BE 업로드 로직에서 허용되지 않은 문자열 패턴을 사전 차단.

📌 **핵심 메시지**:

- 추가적으로 **로그 분석 및 정기 스캐닝**을 통해 과거·현재 모든 위험 요소를 점검.
- 추후에는 업로드 로직 개선과 Signed URL 도입으로 재발 방지 필요.
