# 개발자를 위한 비용 시뮬레이션 및 최적화 가이드

## 1. 개요 및 목적

### 1.1 "비용도 기능이다"

클라우드(AWS)와 SaaS(Sendbird, Bootpay 등)를 사용하는 환경에서 **비용 효율성**은 성능만큼이나 중요한 품질 지표입니다. 코드가 아무리 완벽하게 동작해도, 회사가 감당할 수 없는 비용이 청구된다면 그 설계는 **실패한 설계**입니다.

### 1.2 목적

본 가이드는 개발자가 분석/설계 및 작업 수행 전에 **반드시 비용 시뮬레이션을 수행**하여, 예기치 않은 "비용 폭탄"을 방지하고 최적의 아키텍처를 설계하는 것을 목표로 합니다. 특히 클라우드 비용 구조에 익숙하지 않은 초급 개발자를 위해 작성되었습니다.

## 2. 실제 비용 폭탄 사례 (반면교사)

다음은 실제 발생했던 사례로, 사전 시뮬레이션의 부재가 어떤 결과를 초래하는지 보여줍니다.

### Case 1. 로그 분석 비용 3,000불 ($3,000)

- **상황**: 특정 이슈 추적을 위해 AWS CloudWatch Logs Insight에서 기간을 넓게 설정하여 쿼리 실행.
- **원인**: Logs Insight는 **"스캔한 데이터 용량"** 당 과금됨을 인지하지 못함. 테라바이트(TB) 단위의 로그를 전체 스캔함.
- **교훈**: 쿼리 실행 전 **스캔 대상 용량**을 확인하고, 필요한 시간 범위만 좁혀서 실행해야 함.

### Case 2. 채팅 서비스 요금 10,000불 초과 ($10,000+)

- **상황**: Sendbird 사용 시 MUA(Monthly Active Users) 쿼터 초과.
- **원인**:
  1.  **모니터링 부재**: 서비스 성장에 따른 MUA 증가를 관찰하지 않아, 한도 임박 시점에 적절한 조치를 취하지 않음.
  2.  **설계 미스 (중복 과금)**: Dating 서비스를 분리하면서 기존 서비스와 **다른 사용자 ID를 부여**하여, 동일 인물이 MUA 2명으로 카운트됨.
  3.  **과금 체계 무지**: MUA 쿼터 초과 시 적용되는 **"징벌적 과금(Penalty Pricing)"** 체계를 인지하지 못함.
- **교훈**:
  - 서비스 분리/확장 시 **유저 식별(Identity) 체계가 비용에 미치는 영향**을 시뮬레이션해야 함.
  - SaaS 도입 시 기본 요금뿐만 아니라 **초과(Overage) 요금 정책**을 반드시 확인해야 함.

## 3. 비용 시뮬레이션 3단계 프로세스

작업을 시작하기 전(설계 단계) 또는 대규모 쿼리 실행 전, 다음 3단계를 수행하십시오.

### Step 1. 과금 모델(Pricing Model) 파악

- 사용하려는 리소스가 **무엇을 기준으로 과금**되는지 확인합니다.
- **주요 과금 요소**:
  - **컴퓨팅**: 실행 시간 (초/시간 단위), CPU/RAM 사양.
  - **스토리지**: 저장 용량 (GB/TB), 저장 기간.
  - **네트워크**: 데이터 전송량 (Data Transfer Out), 요청 횟수.
  - **SaaS**: 활성 사용자 수(MAU), API 호출 횟수, 기능별 등급.
  - **데이터 분석**: 쿼리당 스캔 용량 (Logs Insight, Athena 등).

### Step 2. 사용량 예측 (Volume Estimation & Growth)

- **최악의 시나리오(Worst Case)**를 가정하여 수치를 대입합니다.
- **기간별 성장(Growth) 예측**: 현재 시점뿐만 아니라, **6개월 후, 1년 후의 사용자/트래픽 증가**를 반영해야 합니다.
- _예: "현재 MAU 1만 명일 때 vs 6개월 후 MAU 10만 명일 때"_
- _예: "하루에 로그가 100GB 쌓이는데, 30일치를 검색하면 3TB를 스캔하게 된다."_
- _예: "이벤트가 터져서 동시 접속자가 10만 명이 되면 Sendbird 요금은 얼마가 되는가?"_

### Step 3. 비용 계산 및 공유 (Calculation & Comparison)

- 파악한 과금 모델에 예측한 사용량을 곱하여 예상 비용을 산출합니다.
- **설계안 비교 시 비용 포함**: 여러 기술적 대안(Option)을 비교할 때, **비용(Cost)**을 핵심 평가 요소로 포함해야 합니다.
- **도구 활용**:
  - [AWS Pricing Calculator](https://calculator.aws/)
  - SaaS 공식 홈페이지 Pricing 섹션
  - 간단한 엑셀 시트 작성 (아래 예시 참조)

**[설계안 비교 예시]**

| 구분                         | Option A (Self-Hosted Redis) | Option B (Managed ElastiCache)                           | Option C (DynamoDB DAX) |
| :--------------------------- | :--------------------------- | :------------------------------------------------------- | :---------------------- |
| **초기 비용**                | 낮음 (EC2 t3.medium)         | 중간 (cache.t3.medium)                                   | 높음 (Provisioned)      |
| **운영 비용**                | 높음 (패치, 백업 직접 관리)  | 낮음 (관리형)                                            | 매우 낮음               |
| **월 예상 비용 (User 1만)**  | $30                          | $50                                                      | $100                    |
| **월 예상 비용 (User 10만)** | $150 (Scale-up 필요)         | $200                                                     | $500                    |
| **선택**                     |                              | **Option B** (운영 리소스 절감 및 예측 가능한 비용 증가) |                         |

- **산출된 비용이 예상을 초과한다면 설계를 변경하거나, 상급자와 상의해야 합니다.**

## 4. 개발자를 위한 체크리스트

작업 전 아래 항목을 스스로 체크해보세요.

### ✅ AWS 리소스 사용 시

- [ ] 이 서비스가 24시간 켜져 있어야 하는가? (필요할 때만 켤 수 없는가?)
- [ ] 데이터 저장 기간(Retention)은 설정했는가? (무기한 저장은 비용의 주범)
- [ ] 데이터 전송(Traffic)이 인터넷 망을 타는가? (VPC 내부 통신으로 줄일 수 없는가?)
- [ ] CloudWatch/Athena 쿼리 시 `Limit`을 걸거나 `Time Range`를 좁혔는가?

### ✅ SaaS (Sendbird, Bootpay 등) 도입 시

- [ ] 무료(Free) 티어의 한계는 어디까지인가?
- [ ] 한계를 넘었을 때(Overage) 과금 정책은 어떻게 되는가?
- [ ] 사용자가 10배, 100배 늘었을 때 비용도 선형적으로 느는가, 기하급수적으로 느는가?

## 5. 결론: 비용 의식(Cost Awareness)을 갖자

**"내 돈이라면 이렇게 쓸까?"**

개발자는 단순히 기능을 구현하는 사람이 아니라, **비즈니스의 가치를 창출하고 지키는 사람**입니다. 설계 단계에서의 비용 시뮬레이션은 회사의 자산을 지키는 가장 확실한 방법입니다.

1.  **모르면 찾아보세요.** (Pricing Page 정독)
2.  **계산해보세요.** (단순 산수라도 좋습니다)
3.  **공유하세요.** ("이렇게 하면 월 $00 정도 나올 것 같습니다"라고 말하는 개발자가 신뢰받습니다.)
