# 기술 의사결정 및 대안 비교 가이드

## 1. 개요 및 목적

### 1.1 배경
개발 단계에서 분석/설계 시, 가장 먼저 떠오르는 **단 하나의 방법**으로 성급하게 결정하는 경향이 있습니다. 이는 현재의 요구사항은 만족할지 모르나, 미래의 확장성이나 운영 효율성을 놓치는 "최적화되지 않은 설계"로 이어질 위험이 큽니다.

### 1.2 목적
본 가이드는 개발자가 **"모든 가능한 경로(All Paths)"**를 탐색하고, 논리적인 비교 분석을 통해 **"최적의 조합"**을 선택하는 훈련을 하기 위함입니다. 이를 통해 기술적 시야를 넓히고 설득력 있는 아키텍처를 설계하는 것을 목표로 합니다.

---

## 2. 의사결정 4단계 프로세스

기술 선택이나 아키텍처 설계 시 다음 4단계를 반드시 거쳐야 합니다.

### Step 1. 확산 (Diverge): 모든 가능성 열거
*   "이거면 되겠네"라고 멈추지 말고, **"다른 방법은 없을까?"**를 3번 이상 자문합니다.
*   가능한 모든 기술, 라이브러리, 아키텍처 패턴을 나열합니다.
*   **목표**: 최소 3가지 이상의 대안(Option) 확보.

### Step 2. 수렴 (Converge): 궁합(Compatibility) 확인
*   모든 기술이 서로 섞일 수 있는 것은 아닙니다.
*   나열된 옵션들 간의 **궁합(Compatibility)**을 확인하여 실현 가능한 조합을 추려냅니다.
*   예: *특정 언어와 라이브러리의 호환성, 기존 인프라와의 통합 가능성 등.*

### Step 3. 비교 (Compare): 다각도 분석
*   추려진 대안들을 동일한 기준선상에서 비교합니다.
*   **비교 기준 예시**:
    *   **구현 비용**: 개발 공수, 난이도
    *   **운영 비용**: 유지보수 용이성, 라이선스 비용
    *   **성능**: Latency, Throughput, 리소스 사용량
    *   **확장성**: 트래픽 증가 시 대응 능력
    *   **안정성**: 장애 발생 시 복구 능력 (Failover)

### Step 4. 결정 (Decide) 및 기록
*   현재와 미래의 요구사항을 가장 잘 만족하는 하나의 안을 선택합니다.
*   **단순히 "좋아서"가 아닌, "요구사항에 부합해서" 선택해야 합니다.**

## 3. 문서화 원칙 (SRS / OnePager 필수 포함)

SRS나 OnePager 작성 시, **최종 선택된 결과만 적지 않습니다.** 의사결정의 맥락(Context)을 남겨야 합니다.

### 3.1 1 of 3 규칙
*   3가지 방법을 비교했다면, **3가지 모두 문서에 기록**해야 합니다.
*   선택하지 않은 안(Rejected Options)에 대해서도 **"왜 선택하지 않았는지"**를 명시합니다. 이는 후임자가 동일한 고민을 반복하지 않게 하는 중요한 자산이 됩니다.

### 3.2 작성 포맷 권장
| 구분 | 옵션 1 (선택됨) | 옵션 2 (기각됨) | 옵션 3 (기각됨) |
| :--- | :--- | :--- | :--- |
| **방식** | 핵심 내용 요약 | 핵심 내용 요약 | 핵심 내용 요약 |
| **장점** | ... | ... | ... |
| **단점** | ... | ... | ... |
| **선택/기각 사유** | **요구사항 A, B를 모두 충족하며 가성비가 가장 좋음** | 성능은 좋으나 구현 공수가 너무 큼 (일정 맞춤 불가) | 레거시 시스템과 호환되지 않음 |

## 4. 비교 분석 사례 (Case Studies)

다음은 실제 설계 시 반드시 비교 분석이 수행되어야 하는 대표적인 주제들입니다.

### Case 1. 세금 계산서 발행 자동화
*   **옵션 1**: 국세청 홈택스 직접 크롤링 (비용 저렴, 유지보수 고비용, 불안정)
*   **옵션 2**: Popbill 등 상용 API 서비스 이용 (비용 발생, 개발 용이, 안정적)
*   **옵션 3**: ERP 연동 모듈 도입 (대규모 인프라 필요, 초기 구축 비용 높음)
*   **결정 포인트**: 발행 건수 대비 비용, 개발 리소스, 법적 리스크 허용 범위.

### Case 2. 채팅 서버 아키텍처 (Redis 도입 여부)
*   **옵션 1 (No Redis)**: Sticky Session + DB 직접 조회. (구현 단순, 서버 확장 시 유저 분산 까다로움)
*   **옵션 2 (With Redis)**: Pub/Sub을 이용한 메시지 브로드캐스팅. (서버 무한 확장 가능, 인프라 복잡도 증가)
*   **옵션 3**: 완전 관리형 채팅 솔루션(Sendbird 등) 사용. (개발 불필요, 비용 매우 높음)
*   **결정 포인트**: 동시 접속자 수 목표, 실시간성 요구 수준, 인프라 운영 역량.

### Case 3. 메시지 큐 시스템 (Kafka vs. SQS)
*   **Kafka**: 대용량 실시간 스트리밍 처리에 적합, 운영 복잡도 높음, 메시지 영속성 우수.
*   **SQS**: 설정 간편, 완전 관리형(AWS), 대용량 처리 시 비용 증가 가능성, 단순 큐 기능.
*   **결정 포인트**: 데이터 처리량(Throughput), 운영 인력 유무, 메시지 순서 보장 및 재처리 요구사항.

### Case 4. 데이터 저장소 (RDBMS vs. NoSQL)
*   **RDBMS (PostgreSQL/MySQL)**: 데이터 정합성(ACID) 중요, 복잡한 조인 연산 필요, 정형 데이터.
*   **NoSQL (MongoDB/DynamoDB)**: 스키마 유연성 필요, 대량의 단순 읽기/쓰기, 비정형 데이터.
*   **결정 포인트**: 데이터의 구조화 정도, 트랜잭션 필요 여부, 확장성 요구사항.

### Case 5. 토큰 관리 정책 (Single vs. Multi Token)
*   **Single Token (Access only)**: 구현 단순, 보안 취약(탈취 시 만료까지 대책 없음).
*   **Multi Token (Access + Refresh)**: 보안 강화(Access 수명 단축), 구현 복잡(갱신 로직, 탈취 감지).
*   **결정 포인트**: 서비스 보안 등급, 사용자 편의성(자동 로그인) 요구사항.

### Case 6. 아키텍처 스타일 (Monolith vs. MSA)
*   **Monolith**: 초기 개발 속도 빠름, 배포/테스트 단순, 규모 커지면 유지보수 어려움.
*   **MSA (Microservices)**: 서비스별 독립 배포/확장 가능, 시스템 복잡도 급증, 통신 비용 발생.
*   **결정 포인트**: 팀 규모, 도메인 복잡도, 배포 빈도, 조직 구조.

### Case 7. tech@munto.kr 운영 및 모니터링 방안
*   **방안 1 (Owner 제도)**: 담당자를 지정하여 주기적으로 확인. (책임 명확, 휴먼 에러 가능성)
*   **방안 2 (메일 필터링)**: 중요 키워드로 라벨링/분류. (구축 저비용, 알림 누락 가능성)
*   **방안 3 (채널 통합)**: AWS SES -> SNS -> Slack 연동. (실시간 알림, 초기 구축 필요, 비용 발생)
*   **결정 포인트**: 알림의 긴급도(Emergency), 놓쳤을 때의 비즈니스 임팩트, 구축 리소스.
