# 옵션/방안 비교 및 선택 사례 정리

본 문서는 전체 문서에서 1안, 2안, 3안(옵션/방안)을 비교하여 최종 선택이 이루어진 사례를 정리한 것입니다.

## 선택 사례 목록

### 1. 채팅 서비스 SRS 리뷰 - 클라이언트 서버 선택 메커니즘

**파일**: `munto_docs/채팅/채팅SRS_리뷰_보고서_v2.md` (33-46줄)

**문제점**: 클라이언트가 채팅방에 접속할 때 어떤 서버로 연결해야 하는지 불명확

**비교 옵션**:

1. **옵션 1: 로드밸런서가 해시 기반 라우팅**

   - 클라이언트는 항상 로드밸런서로 요청
   - 로드밸런서가 `roomId`를 기반으로 해시하여 적절한 서버로 라우팅
   - 이 경우 로드밸런서 설정 및 해시 알고리즘 명시 필요

2. **옵션 2: REST API가 서버 정보 제공**
   - 클라이언트가 채팅방 ID와 함께 채널 URL 요청
   - REST API 서버가 `roomId` 해시하여 적절한 서버 결정
   - 해당 서버의 WebSocket URL 반환
   - 이 경우 REST API 서버가 모든 서버 정보를 알고 있어야 함

**최종 선택**:

- **권장: 옵션 1 또는 2 중 선택하여 명시** (193줄)
- SRS에 명시는 필요하나, 특정 옵션 강제는 하지 않음

### 2. 채팅 서비스 SRS 리뷰 - 다중 채팅방 연결 관리

**파일**: `munto_docs/채팅/채팅SRS_리뷰_보고서_v2.md` (175-193줄)

**문제점**: 사용자가 여러 채팅방에 참여할 때 각 채팅방이 다른 서버에 있을 경우 연결 관리 방식 불명확

**비교 옵션**:

1. **옵션 1: 채팅방별 독립 연결**

   - 각 채팅방마다 해당 서버에 별도 WebSocket 연결
   - 복잡도 증가하지만 서버 장애 시 영향 최소화

2. **옵션 2: 주 서버 하나만 연결**

   - 사용자가 참여한 채팅방 중 하나의 서버에만 연결
   - 다른 서버의 채팅방은 REST API로만 조회
   - 실시간 메시지는 해당 서버의 채팅방만 수신

3. **옵션 3: 게이트웨이 서버**
   - 클라이언트는 게이트웨이 서버 하나에만 연결
   - 게이트웨이가 각 채팅방 서버와 연결하여 메시지 중계
   - 이 경우 게이트웨이가 단일 장애점이 될 수 있음

**최종 선택**:

- **권장: 옵션 1 또는 2 중 선택하여 명시** (193줄)
- 옵션 3은 단일 장애점 문제로 제외
- 옵션 1 또는 2 중 선택하되 SRS에 명시 필요

### 3. AI 기반 수익 극대화 추천 시스템 - 다중 이미지 처리 전략

**파일**: `my_docs_dev/Temp/AI 기반 수익 극대화 추천 시스템 설계서.md` (320-441줄)

**문제점**: 사용자 프로필 사진이 3~6장(가변)인데, 이를 어떻게 처리할지 결정 필요

**비교 옵션**:

1. **옵션 1: 대표 이미지만 사용** (✅ 필수, Month 12-13, 모델 v2.0 초기)

   - 첫 번째 사진만 임베딩
   - 구현 가장 간단 (3일), 빠른 검증 가능
   - 단점: 나머지 사진 정보 손실

2. **옵션 2: 평균 풀링 (Mean Pooling)** (✅ 권장, Month 14, 모델 v2.0 개선, 예상 효과: +1-2%)

   - 모든 사진 임베딩 후 평균
   - 모든 사진 정보 활용, 구현 간단 (2일 추가)
   - 단점: 순서 정보 손실

3. **옵션 3: 가중 평균 (Weighted Pooling)** (✅ 권장, Month 15, 모델 v2.0 고급, 예상 효과: +2-3% 추가)

   - 순서 기반 가중 평균 (첫 사진 40% 가중치)
   - 순서 정보 반영, 모든 사진 활용
   - 단점: 가중치 하이퍼파라미터 튜닝 필요

4. **옵션 4: 연결 + 제로 패딩 (Concatenation)** (❌ 비추천)

   - Input 차원 폭발 (512 → 3072), 모델 복잡도 급증

5. **옵션 5: 어텐션 메커니즘 (Attention)** (❌ 비추천)
   - 구현 매우 복잡, AI 초보자에게 부적합

**최종 선택** (430-435줄):

**최종 권장 순서**:

1. Month 12-13: 옵션 1 (첫 사진만) ← 필수
2. Month 14: 옵션 2 (평균) ← 권장
3. Month 15: 옵션 3 (가중 평균) ← 권장
4. 옵션 4-5: 스킵 ← 비추천

**성능 예측**:

- 옵션 1: AUC 0.83-0.85
- 옵션 2: AUC 0.84-0.86 (+1-2%)
- 옵션 3: AUC 0.85-0.87 (+2-3% 추가)

**선택 이유**: 단계적 접근으로 옵션 1 → 2 → 3 순서로 도입. 옵션 4-5는 복잡도 대비 효과가 낮아 제외.

### 4. tech@munto.kr 운영 개선 방안

**파일**: `my_docs_dev/개발_자료_보고서/tech@munto.kr 운영 개선 방안 보고서.md` (14-53줄)

**문제점**: 공용 메일함(`tech@munto.kr`)의 무주공산 구조와 Email 의존성으로 인한 알림 누락

**비교 방안**:

1. **방안 1: 담당자 지정 (Owner 제도)**

   - `tech@munto.kr` 전담 관리자를 지정 (전담 관리자: 김범진)
   - 책임 소재 명확화, 운영 간소화
   - 단점: 담당자가 부재 시 리스크

2. **방안 2: 메일 필터링 및 태그 지정**

   - Gmail Workspace에서 스팸 필터 강화 + 중요 키워드 기반 라벨링
   - 단기적으로 가장 빠른 실행 가능, 비용 없음
   - 단점: 완벽하지 않아 일부 누락 가능

3. **방안 3: AWS 알림 채널 전환 (Email → SNS/Slack)**
   - AWS SES, CloudWatch 알림 등을 Email이 아닌 SNS → Slack Webhook으로 직접 연결
   - 운영팀의 실시간 대응력 향상, Email 의존도 축소
   - 단점: 세팅은 필요하지만 즉시 적용 가능

**최종 선택** (48-52줄):

**추천 전략 - 단기 (즉시 적용)**:

- **방안 1 (담당자 지정)**: 전담 책임자 지정으로 책임 소재 명확화
- **방안 2 (메일 필터/라벨링)**: Gmail 규칙으로 중요 메일 자동 분류
- **방안 3 (AWS 알림 SNS/Slack 전환)**: 주요 알림을 Slack/Jira로 즉시 전달

**선택 이유**: 3가지 방안을 모두 적용하여 단계적으로 Email 의존도를 낮추고 실시간 알림 체계 구축. 방안 1, 2, 3을 모두 선택 (병렬 적용).

## 비교만 하고 선택하지 않은 사례

다음 사례들은 옵션/방안을 비교했지만 명확한 최종 선택이 없는 경우입니다.

### 1. 결제 취소 사고 TCL 분석 - 스케줄러 테스트 방법

**파일**: `munto_docs/데이팅결제_환불사고/결제 취소 사고 TCL 분석 및 개선 방안.md` (208-213줄)

**문제점**: "30분 경과" 테스트는 실제로 매우 어려움. 스테이징 환경에서 30분을 실제로 기다리는 건 비현실적.

**비교 대안**:

- **대안 1**: 시간 조작 유틸리티 사용 (예: Jest의 `jest.useFakeTimers()`, Python의 `freezegun`)
- **대안 2**: 스케줄러 함수를 직접 호출하여 테스트
- **대안 3**: 모니터링으로 대체

**선택 여부**: ❌ 명확한 선택 없음. 3가지 대안을 제시했으나 특정 대안을 권장하지 않음.

1. **채팅 서비스 - 클라이언트 서버 선택**: 옵션 1 또는 2 중 선택 권장

3. **Portfolio 관리 방안**: 3가지 방안 나열, 선택 없음

## 패턴 분석

1. **선택 패턴**:

   - **단일 선택**: 특정 옵션 하나를 명확히 선택 (예: AI 추천 시스템 - 다중 이미지 처리)
   - **범위 선택**: 특정 옵션 중 하나를 선택하도록 권장 (예: 채팅 서비스 - 옵션 1 또는 2)
   - **병렬 선택**: 여러 옵션을 모두 선택하여 병렬 적용 (예: tech@munto.kr 운영 개선)

2. **제외 패턴**:

   - 명확한 제외 이유와 함께 특정 옵션을 제외 (예: 단일 장애점, 복잡도 대비 효과 부족)

3. **선택 기준**:
   - 구현 복잡도
   - 성능/효과 예측
   - 리스크 분석
   - 단계적 도입 가능성
