# Munto 개발 문서 리뷰

## 목적

`munto_docs` 레포의 `.md`는 **Notion·Jira를 리뷰 편의로 내려받아 모아 둔 로컬 미러**이다. 리뷰 시 본문은 이 md를 읽되, **Review·Jira 코멘트에 다른 사람이 따라갈 수 있는 링크**만 쓴다 (아래 [외부 인용 규칙](reference.md)).

`munto_docs`의 Jira 이슈 md와 Notion 본문 md를 **내용 중심**으로 리뷰한다. 매 회 동일한 산출물을 만든다.

1. **상세 리뷰 문서** (`*Review.md` 또는 `*-review.md`)
2. **Jira 이슈 마크다운** (`munto_docs/Jira/{ISSUE_KEY}.md`)에 `## comment by {리뷰어}` 추가

기본 리뷰어: **전규현**. 사용자가 다른 이름을 지정하면 그 이름을 쓴다.

## 리뷰 원칙 (요약)

- **가치 있는 의견만**: 작성자·리뷰어 시간을 쓸 만한 항목만 남긴다. “있으면 좋음”, Jira 문구 형식 충족, 상식적 체크리스트, 이미 본문으로 드러난 내용은 **의견에 넣지 않는다**.
- **포함 기준** (하나 이상 해당할 때만 항목화):
  - 잘못 따르면 **버그·과지급·오주문·배포 사고** 등 실질 피해 가능
  - 문서 **내부 모순**(숫자·정책·흐름이 앞뒤로 충돌)
  - **합의·구매·구현 전** 반드시 맞춰야 하는 수치·범위·절차
- **제외·자제**:
  - Jira와 보고서가 **의미상 정합**한데 표현만 다른 경우 (예: “기준선” ≠ “전량 즉시 약정”)
  - 엄밀한 Jira 체크리스트 미충족이나 **형식 보강**만을 위한 코멘트
  - 작성자가 이미 액션·모니터링·별도 트랙으로 처리한 운영 상식
- **예외(항상 지적)**: **보고서**에 **Executive Summary가 본문 맨 앞(메타 다음, 상세 §1 이전)에 없으면** 반드시 리뷰 항목 + **전체 평가** + **Jira 코멘트 최상단**에 명시. 팀 필수 양식이며 “형식”이 아니라 의사결정·공유 품질 이슈로 취급한다.
- **내용 Review**: 위 기준에 걸리는 것만 — 구현 가능성, conflict, 운영·마이그레이션·보안
- **작성법 Review**: 사용자가 명시하지 않으면 **제외**
- **Notion 위치 필수**: Notion 화면 기준 섹션 제목 (md `##` 문법은 작성자-facing 문장에 **쓰지 않음**)
- **문체**: Notion에 다는 **사람 댓글** 톤 — [reference.md § Review 문체](reference.md). 본문을 Notion 문서처럼 장문·다단 `###`·`1. 2. 3.`·`- **필드**: 값` 템플릿으로 쓰지 않는다
- **이모지 사용 금지**, **한국어**, **git commit / push 하지 않음**

적격 항목이 **0건**이면 “특별히 손볼 곳 없음”으로 짧게 끝낸다. 억지로 R-01을 채우지 않는다. 생략한 검토 영역은 `## 리뷰하지 않은 항목`에 한 줄씩만 적어도 되고, 없으면 생략해도 된다.

상세 체크리스트·파일 규칙: [reference.md](reference.md)  
입출력 예시: [examples.md](examples.md)

## 입력 해석

사용자가 `AWSI-83 리뷰 해줘`, `@munto_docs/Jira/AWSI-83.md 리뷰`, `WEBB-1377 2차 리뷰` 등으로 요청할 수 있다.

| 사용자 표현 | 의미 |
| --- | --- |
| `N차 리뷰`, `2차`, `v2 리뷰` | 이전 `*-review.md` 대비 **변경분·미해소 항목** 중심 |
| (없음) | **1차 리뷰** — 대상 문서 전체 |
| `Jira만`, `코멘트만` | Jira `.md` 코멘트만 작성 (Review 파일 생략 가능) |
| `Review만` | Review 파일만 (Jira 코멘트 생략) |

## 1단계: 자료 수집

다음 순서로 진행한다. MCP(Jira, Notion)가 있으면 보조로 사용하되, **레포 내 `.md`를 우선**한다.

### 1.1 Jira 이슈 문서

```
munto_docs/Jira/{ISSUE_KEY}.md
```

- 없으면 `munto_docs/**/{ISSUE_KEY}.md` 검색
- 상단 표(배경·현재 문제점·개선 방향·고려요소·기대효과)를 **리뷰 기준(SOT)** 으로 삼는다
- 기존 `## comment by …` / `# comment by …` 를 읽어 **이전 리뷰·작성자 답변** 맥락 파악

### 1.2 Notion 본문(마크다운)

같은 이슈·제목과 연결된 본문을 찾는다.

1. `munto_docs/Jira/{ISSUE_KEY}-*.md` (OnePager, 보고서 등)
2. `munto_docs/**/{ISSUE_KEY}*.md` (DB정합성, 채팅내제화 등)
3. Notion export: 파일명 끝 `{32자해시}.md` — 동일 제목·이슈 링크로 매칭
4. Jira 코멘트·본문에 있는 `notion.so` URL

**여러 후보가 있으면** 사용자에게 확인하거나, 버전 숫자가 가장 높은 문서(v3 > v2)를 기본 대상으로 한다.

### 1.3 N차 리뷰 시 이전 산출물

`Glob: **/*{ISSUE_KEY}*review*.md` 및 `**/*Review*.md` (동일 폴더)

- 가장 최근 리뷰 파일의 항목 ID(`R-01`, `C-001`, `v1-CHO-1` 등) 목록 확보
- 작성자가 “반영했다”고 한 코멘트와 대조

## 2단계: 리뷰 수행

### 2.1 1차 리뷰

1. Jira 표 5항목 대비 본문 누락·과장·모순 검사
2. 문서 유형별 체크 — [reference.md § 문서 유형별](reference.md)
3. 항목 부여:
   - OnePager/SRS 상세: `R-01` … 또는 `C-001` / `W-001` / `I-001` (Critical/Warning/Info)
   - DB 명세 N차: `v2-XXX-1` 형식 가능
4. [포함 기준](reference.md)에 해당하는 것만 항목화; 각 항목에 **Notion 위치** — [reference.md § Notion 위치 표기](reference.md)
5. **잘 된 점**: 총평 1~2문장으로 충분. 별도 섹션·bullet 다수는 **필요할 때만** (양으로 채우지 않음)

### 2.2 2차 이상 (N차)

1. **이전 리뷰 반영 표** 작성 (해소 / 미해소 / 부분 해소)
2. **신규 이슈만** 상세 기술 (수정 과정에서 생긴 conflict 우선)
3. 총평에 “N차 기준 변경 범위” 한 줄 명시

### 2.3 심각도

| 등급 | 기준 |
| --- | --- |
| Critical / HIGH | 배포·정산·결제·보안·데이터 정합에 직접 영향, 요구사항 미충족 |
| Warning / MEDIUM | 구현 전 반드시 합의 필요, 운영 리스크 |
| Info / LOW | 보완 권장, 예시·표현 명확화 |

## 3단계: 산출물 작성

### 3.1 Review 문서

파일명 규칙 — [reference.md § 파일명](reference.md)

**공통 헤더:**

```markdown
# {문서 제목} — 리뷰

- **대상 Notion 문서**: `{Notion 페이지 제목}` (버전·작성자·날짜 — md 메타에서)
- (리뷰어용 로컬 md 파일명은 Review **헤더·본문에 쓰지 않음**)
- **관련 Jira**: [{ISSUE_KEY}](https://munto.atlassian.net/browse/{ISSUE_KEY})
- **리뷰 차수**: 1차 | 2차 | …
- **리뷰 관점**: Jira 요구사항 반영, 구현 가능성, … (문서 유형에 맞게 1줄)
- **리뷰어**: {이름}
- **리뷰일**: YYYY-MM-DD
```

**본문 구조** — 문체는 [reference.md § Review 문체](reference.md):

```markdown
## 1. 전체 평가

(짧은 일반 문단. 동의·보완 요청만. “승인/반려” 문구 없음. 필요 시 bullet 2~3개)

## 2. 리뷰 의견 (Notion 댓글·수정용)

---

**[Notion 섹션 제목 — 예: 1. 권장 결정 > 권장 결정 표]**

(2~6문장. `##` 없이 Notion에 보이는 제목만. 댓글에 그대로 붙일 톤)

---
```

- `R-01` / `- **내용**:` / `- **권장 수정**:` 같은 **양식 필드 나열**은 쓰지 않는다
- 심각도는 문장 안에 자연스럽게, 또는 위치 줄 옆에만 (예: `[필수]`)
- Jira 코멘트도 동일 톤(짧은 문단 + 짧은 bullet)

**Notion 위치** — 상세는 [reference.md § Notion 위치 표기](reference.md):

- 리뷰어는 md의 `##`/`###`로 위치를 찾고, 댓글·Review에는 **Notion에 보이는 제목만** (`1. 권장 결정` > `5.2 결제 옵션` 표)
- 표·액션 번호가 있으면 함께 적기 (예: `§6 액션 아이템` > `#3`)
- Jira 이슈 표·코멘트만 관련되면 **Jira 위치**를 별도 줄로 추가
- 여러 섹션에 걸치면 **주 위치** + `(관련: …)` 로 나열

**N차 전용 상단 블록:**

```markdown
## 이전 리뷰 반영 현황
| 상태 | 건수 | ID |
| 해소 | … | … |
| 미해소 | … | … |

## 이번 차수 신규 이슈
…
```

저장 위치: **대상 Notion `.md`와 같은 디렉터리** (예외: Jira 전용 분석은 `munto_docs/Jira/`).

### 3.2 Jira 이슈 `.md` 코멘트

파일: `munto_docs/Jira/{ISSUE_KEY}.md`  
파일 **맨 아래**에 추가:

```markdown
## comment by {리뷰어}

@{작성자 멘션} {문서 유형} 리뷰했습니다. (1차|2차 요약 2~4문장)

**가장 큰 보완 (필수)** — 보고서에 Executive Summary 누락 시 **반드시 이 블록을 넣고 맨 위에 둔다** (없으면 생략)

**반영 권장** — 항목 bullet (Notion 위치는 Review.md 참고)

상세: 본 코멘트 + Review 본문 (레포 `munto_docs/...md` 경로는 **Jira·Notion 독자에게 노출하지 않음**)
```

- 기존 `## comment by {리먷 리뷰어}` 가 있으면 **같은 섹션 아래** `---` 구분 후 **새 블록** 추가 (날짜·차수 표기)
- 짧은 확인 리뷰(분석 검증 등)는 상세 Review 없이 코멘트만 가능 — [examples.md § 짧은 Jira 코멘트](examples.md)
- `# comment by` 형식이 파일에 쓰여 있으면 **새 추가분은 `##` 로 통일**해도 됨 (기존 블록은 수정하지 않음)

## 4단계: 완료 보고

사용자에게 다음을 짧게 전달한다.

1. 리뷰 차수·대상 문서 경로
2. 생성·수정한 파일 경로 (Review, Jira md)
3. 심각도별 건수
4. Jira/Notion에 **직접 붙여넣을** 코멘트가 있으면 해당 여부

**하지 않는 것**: Jira/Notion 원격 게시, commit, push.

---

## MCP·외부 도구

| 도구 | 용도 |
| --- | --- |
| user-Jira | 이슈 설명·코멘트·링크 확인 (로컬 md와 불일치 시) |
| user-Notion | 최신 본문 (로컬 md가 오래됐을 때) |

로컬 md와 Jira/Notion이 다르면 **차이를 리뷰 코멘트에 명시**하고, 어느 쪽을 기준으로 리뷰했는지 밝힌다.
