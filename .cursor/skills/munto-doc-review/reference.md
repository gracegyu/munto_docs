# Munto 문서 리뷰 — 참조

## 레포 구조

```
munto_docs/
├── Jira/                    # 이슈 요약 + comment by 스레드
│   ├── WEBB-1377.md
│   ├── WEBB-1377-… OnePager.md
│   └── WEBB-1377-… OnePager Review.md
├── DB정합성/                # WEBB-* 명세·OnePager·review
├── 채팅내제화/, 체험단/, …   # 도메인별 Notion md + Review
└── …
```

- **Jira md**: 상단 5항목 표 + 팀 코멘트 타임라인
- **Notion md**: OnePager / SRS / 보고서 / 명세서 본문 (메타: 작성자, 버전, 변경 이력) — **Notion export 미러**

정책 배경: `my_docs_dev/정책_프로세스_가이드/개발 문서 리뷰란 무엇인가.md`

## munto_docs `.md` = 리뷰용 로컬 미러 (외부 인용 규칙)

`munto_docs/**/*.md`는 Notion·Jira를 **다운로드해 리뷰어가 Cursor에서 보기 위해** 모은 것이다. 작성자·재무·팀은 보통 **Notion/Jira**만 본다.

### Review.md · Jira 코멘트에 쓸 것 / 쓰지 말 것

| 사용 | 금지 (독자가 따라갈 수 없음) |
| --- | --- |
| **Notion 문서 제목**(md `#` 제목과 동일) | `munto_docs/Jira/foo.md`, `WEBB-1328-….md` 등 **레포 파일 경로** |
| **Notion URL** — 대상·참고 문서 본문·Jira 코멘트에 `notion.so` 링크가 있으면 그대로 | `my_docs_dev/…` 등 **다른 로컬 md 경로**를 “참고하세요”로 인용 |
| **Jira 이슈** `[WEBB-1328](https://munto.atlassian.net/browse/WEBB-1328)` | Review 상세를 “`…/foo Review.md` 보라”만 하고 끝내기 |
| **Notion 위치** — Notion 화면에 보이는 **섹션 제목** (md의 `##` 기호는 **쓰지 않음**) | `## 1. 권장 결정`, `` `## …` `` 등 **export md 문법**을 댓글·Review 본문에 노출 |

### 참고 문서를 가리킬 때

1. 대상 md 상단·관련 이슈 md에 **Notion URL**이 있으면 Review/Jira에 **URL + 문서 제목**.
2. URL이 없으면 **Notion 문서 제목** + (있으면) **관련 Jira 키** + “Notion에서 동일 제목 검색” 수준. 레포 파일명은 **리뷰어 메모**로만 사용.
3. 팀 템플릿은 **템플릿 이름**(예: 사고 대응 완료 보고서 템플릿 § Executive Summary) — Confluence/Notion에 있는 실제 위치를 알 수 있을 때만 URL.

### Review 헤더

```markdown
- **대상 Notion 문서**: {제목} (v…, 작성자, 날짜)
- **Notion URL**: {있으면}
- **관련 Jira**: [KEY](https://munto.atlassian.net/browse/KEY)
```

## Review 문체 (Notion 댓글 톤)

Review.md 본문은 **리뷰어가 Notion에 comment 다는 글**이다. 대상 Notion **본문을 또 작성하는 것처럼** 길고 구조화하면 안 된다.

### 해야 할 것

- **일반 문장** 위주. “~해 주시면 좋겠습니다”, “~로 보입니다”, “동의합니다”
- **승인/반려/조건부 승인** 같은 **결재 문구는 쓰지 않음** — “괜찮아 보입니다”, “이 부분만 손보면 됩니다” 정도로 의견만
- **굵게**는 강조할 단어만 (Executive Summary, §1.2, Option A)
- 목록은 **짧을 때만** bullet 2~4개. 번호 목록·중첩 `###`·표는 N차 반영 현황 등 **꼭 필요할 때만**
- §2는 **위치 한 줄 + 본문**을 `---`로 구분해 Notion에 **복사·붙여넣기**하기 쉽게
- 메타 헤더(대상 Notion, Jira, 차수)만 YAML식 bullet — **본문에는 `- **필드**:` 템플릿 금지**

### 하지 말 것 (AI 티)

- `### 문서 구조 (본 리뷰의 핵심)` + `1. 2. 3.` 장문 체크리스트
- 항목마다 `- **Notion 위치**: …` / `- **내용**: …` / `- **권장 수정**: …` **고정 4필드**
- `판정:` / `권고 보고서 골격:` 같은 **보고서 목차형 소제목**
- `` `## 1. 권장 결정` `` — **Notion 댓글·Review 본문**에 md 제목 문법 노출 (Notion에는 `1. 권장 결정`만 보임)
- `## 잘 된 점` bullet 5개, Jira 요구사항 대비 표 (독자에게 불필요하면 생략)

### Jira `comment by`

Review §1·§2와 **같은 톤**. “**가장 큰 보완 (필수)**” 같은 라벨만 덜고, 짧은 문단 + 2~3 bullet이면 충분.

### §1과 §2 역할 분리

- **§1 전체 평가**: 구조·ES·총평 등 **문서 전체** 의견 (한곳에만)
- **§2 Notion 댓글용**: **해당 섹션에만** 달 블록별 코멘트. §1에 이미 쓴 ES·구조 이슈는 **§2에 반복하지 않음**

## 가치 있는 리뷰 의견만 (필수 원칙)

리뷰어·작성자 시간을 아끼기 위해 **반영 가치가 있는 것만** `## 2. 리뷰 의견`에 넣는다.

| 넣는다 | 넣지 않는다 |
| --- | --- |
| 오주문·잘못된 합의 수치·문서 내 모순·배포/정산/보안 차단 | Jira 문구 vs 보고서 **의미는 같고** 표현만 다른 것 |
| 수정 없이 진행 시 실제 손해·되돌리기 어려운 결정 | dev 6시나리오 표 등 **형식만** 채우는 제안 |
| 재무·PM이 문서만 보고 **잘못 결정**할 구체 지점 | “확정 문장 1줄”, “체크리스트 1줄” 등 상식·이미 액션에 있는 것 |
| **보고서 Executive Summary 누락 (팀 필수, 예외 없음)** | — |
| | **잘 된 점** bullet 나열로 분량 채우기 |

항목이 없으면: `## 1. 전체 평가`에 “특별히 손볼 곳 없음” 정도로 짧게 종료.

여러 건을 적었다가 **의미상 문제 없음**이 드러나면 항목을 **삭제**하고, 필요 시 `## 3. 리뷰하지 않은 항목`에 “검토했으나 의견 없음” 한 줄로 정리한다.

## Notion 위치 표기 (필수)

댓글은 **Notion 페이지**에 단다. Review 본문·위치 줄도 **Notion에서 보이는 말**로 쓴다.

리뷰어는 내부적으로 export `.md`의 `##`/`###`로 위치를 찾되, **작성자에게 전달하는 문장에는 `##`를 넣지 않는다** (md 저장 형식일 뿐, Notion UI에 없음).

### 형식 (Notion·댓글용)

```markdown
**[1. 권장 결정 — 권장 결정 표]**

…본문…
```

- 섹션명: md `## 1. 권장 결정` → Notion에는 **`1. 권장 결정`** (앞의 `##` 제거)
- 하위: `5. 의사결정 근거` > `5.2 결제 옵션` 표
- OnePager: `Technical Description` > `API 설계` (영문 제목은 md와 동일하게)
- 맨 위 ES: **`문서 맨 위`**, **`본문 시작부(Executive Summary 넣을 자리)`** — “메타 다음” 같은 md 용어도 피함

| 필드 | 규칙 |
| --- | --- |
| **위치 줄** | `[섹션 제목]` 또는 `[상위] > [하위] > (표·#3 행)` — **Notion에 붙일 첫 줄** |
| **Jira 위치** | Jira만 건드릴 때: `[AWSI-83] 개선 방향 1)` |
| **관련** | `(관련: 5.3 lounge 약정)` — § 기호 대신 **섹션 번호·제목** |

### 작성 절차

1. export md 목차로 **어느 블록인지** 파악 (리뷰어 전용).
2. Notion에 보일 **제목 문자열만** 뽑아 위치 줄·본문에 사용.
3. 본문에서도 “`## 1. 권장 결정` 표” ❌ → “**1. 권장 결정** 표” ⭕.

### 예시

```markdown
**[6. 액션 아이템 — #3, #4 행]**

#3 “prod RI 4종” …
```

```markdown
**[Technical Description > 통일안 > 정산 캡]**

정산 캡은 …
```

### 금지·주의

- `` `## …` ``, `` `### …` ``, “§6”만 던지기 (가능하면 **6. 액션 아이템**)
- “메타 다음”, “export”, “md” 등 **미러 파일 용어**를 작성자-facing 문장에 사용
- “전체 문서”만 적기 — 최소 한 개 **Notion에서 찾을 수 있는 제목**은 필수

## 파일명 규칙

| 유형 | 패턴 | 예 |
| --- | --- | --- |
| OnePager/SRS 1차 Review | `{본문베이스} Review.md` 또는 `{본문베이스}-review.md` | `… OnePager Review.md`, `WEBB-1328-…-review.md` |
| 이슈 키 + 차수 | `{ISSUE_KEY}-v{n}-review.md` | `WEBB-1321-v2-review.md` |
| SRS 버전 | `{문서명}_Review_v{n}.md` | `채팅 서비스 내재화 SRS_Review_v3.md` |

**우선순위**: 같은 폴더에 기존 Review가 있으면 **동일 네이밍 스타일**을 따른다.  
N차는 기존 파일을 덮어쓰지 않고 **새 파일** 또는 파일 상단에 차수·날짜를 갱신 (기존 레포 관례: `v2-review` 새 파일).

## Jira 이슈 md 코멘트 규칙

1. 경로: `munto_docs/Jira/{ISSUE_KEY}.md` (없으면 `munto_docs/**/{ISSUE_KEY}.md` 중 이슈 요약 파일)
2. 섹션: `## comment by {리뷰어}`
3. 멘션: 본문 작성자·리뷰 요청자 (`@김범진` 등) — 이전 코멘트에서 추출
4. 상세 리뷰가 길면 Jira 코멘트는 **요약 + Review 파일 상대경로**만
5. **같은 차수 중복 작성 금지** — 이미 동일 날짜·차수 코멘트가 있으면 사용자에게 확인

### 코멘트 톤

- 존댓말, 건설적, “~해 주시면 좋겠습니다” / “확인 부탁드립니다”
- 동의·잘 된 점을 먼저 짚은 뒤 이슈 나열 (가능할 때)
- 장애·인프라 리뷰: **즉시 / 병행 / 조속히** 로 액션 구분

## 문서 유형별 체크리스트

### Engineering OnePager

- [ ] Jira 5항목(배경·문제·방향·고려·효과)과 본문 정합
- [ ] As-Is → 영향 → 결정 → To-Be 순서 (마이그레이션은 통일안보다 앞 단계)
- [ ] 결정 항목·옵션 비교·권고안 근거 (정성만이면 Warning)
- [ ] Technical Description: API/DB/스키마, 에러·멱등·한도
- [ ] Resource/일정, 롤백, 모니터링, cut-over·CS·운영
- [ ] 범위 외·하위 이슈 분리 명시
- [ ] 변경 이력·버전 표기 (N차 시 diff 추적 가능한지)

### SRS / 기능 명세

- [ ] 요구사항 ID·시나리오·예외 경로
- [ ] API 요청/응답/상태코드/에러
- [ ] DB 스키마와 API 필드 매핑
- [ ] 클라이언트·서버 책임 경계
- [ ] TBD·미정 항목과 일정 리스크

### DB 정합성 / 규칙 명세 (다문서 세트)

- [ ] 문서 간 Invariant·용어·상태 전이 일치
- [ ] 환불·정산·취소 경로 cross-reference
- [ ] “1 Case = 1 Job”, SOT=PG, 원자적 check+update 등 팀 원칙
- [ ] N차: 이전 리뷰 ID 표 + 신규 conflict

### 보고서 (분류: 보고서 / Incident Report / 검증·분석 보고서)

**Executive Summary (필수 — 누락 시 HIGH + Jira 코멘트 최상단)**

- [ ] 본문 **맨 앞**: 상단 속성(이슈 링크 등) **다음**, **1. 권장 결정** 등 상세 본문 **이전**에 **Executive Summary** 섹션 존재
- [ ] ES만으로 **결론·금액/영향·리스크·즉시 액션** 파악 가능 (상세 표는 뒤로)
- [ ] 없으면: Review **전체 평가**에서 가장 크게 지적, R-01급 **[HIGH]**, Jira comment **「가장 큰 보완 (필수)」** 블록 필수
- [ ] 참고: 팀 **사고 대응 완료 보고서 템플릿** § Executive Summary / Notion에 있는 동일 제목 보고서(예: **「정산 로직 개선 검증 결과 보고서」**, Jira WEBB-1328) — **레포 md 파일명으로 인용하지 않음**

- [ ] 수치·단위·기간·리전 명시
- [ ] 권고안과 리스크(보류·예외) 분리
- [ ] Jira 배경 대비 결론 도출 여부
- [ ] Jira “**기준선**으로 ~확정”은 **전량 즉시 적용**이 아님 — 현황 인벤토리 대비 **재약정 대상 선별** 결과와 모순되는지만 본다

### 인프라·비용·분석 보고서

위 **보고서 Executive Summary** 체크를 **최우선** 적용한다.

### 짧은 분석·버그 이슈 (Review 파일 생략 가능)

- [ ] Jira 코멘트만으로 결론·동의·추가 확인 1~3점

## N차 리뷰 전용

1. **변경 범위 파악**: 본문 변경 이력, Jira “수정했습니다” 코멘트, 버전 번호
2. **반영 표** 필수 컬럼: `ID | 심각도 | 상태(해소/미해소/부분) | 비고`
3. 미해소 항목은 **재지적하지 않고** “여전히 미해소 — {이유}” 한 줄 + 신규 근거만 추가
4. 수정으로 **새 conflict** 발생 시 HIGH로 올림

## Conflict 탐지 힌트

- 같은 문서 §A “전액 환불” vs §B “환불 없음”
- Jira 표 vs OnePager 본문
- 테스트 매트릭스 vs 설계 절 (음성/양성 테스트 방향)
- Invariant vs 정합성 표 vs 예시 코드
- 비용·일정 표 vs 권고안 문단

## Notion 댓글용

Review.md 각 항목의 **Notion 위치** 줄을 댓글 첫 줄로 복사한 뒤, **내용**·**권장 수정**을 이어 붙인다.

사용자가 Notion Comment만 원하면 `Notion_Comment_리뷰_가이드.md` 스타일(P0/P1)로 요약하되, **항목마다 Notion 위치는 동일하게 유지** — 기본 산출물은 Review.md + Jira md.

## 리뷰 관점 문구 (헤더용)

- OnePager: `Jira 요구사항 반영, 결정·마이그레이션·정산/운영 영향, 구현 가능성 (작성법 제외)`
- SRS: `Jira 요구사항 반영, API/DB 스펙, 클라이언트·운영 고려 (작성법 제외)`
- DB 명세: `Invariant·문서 간 정합, v{n-1} 리뷰 반영 여부`
- 보고서: `Jira 배경 대비 결론·수치·리스크, 의사결정 지원 여부`
